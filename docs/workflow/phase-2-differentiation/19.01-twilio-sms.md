# Session 19.01 — Twilio Integration & TCPA Compliance

**Phase:** 2 — Differentiation | **Block:** 19 — SMS Notifications
**Dependencies:** 11.01
**Estimated tasks:** 12

---

## Description

Integrate Twilio for SMS notifications with full TCPA (Telephone Consumer Protection Act) compliance. This session builds the Twilio SMS service, the TCPA-compliant opt-in flow with consent recording, opt-out processing via Twilio webhooks, quiet hours enforcement, SMS templates for notification types, delivery tracking, and an admin SMS dashboard. SMS is added as a second channel to the existing notification event system from session 11.01.

## Tasks

1. Install Twilio Node.js SDK (twilio)
2. Create Twilio SMS service in packages/api
3. Configure Messaging Service per town (phone number, sender name)
4. Implement TCPA-compliant opt-in flow: consent form with required disclosures (message types, frequency, "Message and data rates may apply", STOP instructions, privacy policy link)
5. Store opt-in records: timestamp, method (web form), IP address, exact consent language shown, phone number
6. Implement opt-out handling: Twilio Messaging Service handles STOP keyword automatically -> webhook updates subscriber record -> confirmation message sent
7. Implement quiet hours enforcement: no SMS before 8:00 AM or after 9:00 PM Eastern
8. Add SMS as channel in notification event system (alongside existing email)
9. Create SMS templates for notification types: straw-poll-invite, public-hearing-notice, meeting-cancelled, meeting-schedule-change
10. Implement delivery tracking via Twilio webhooks (delivered, failed, undelivered)
11. Create admin SMS dashboard: delivery status, opt-in/opt-out counts, message history
12. Verify: TCPA consent flow works, opt-in recorded, opt-out processes, quiet hours enforced, SMS sends via Twilio, delivery tracked

```
You are building the Twilio SMS integration with full TCPA compliance for the Town Meeting Manager — a municipal meeting management platform for Maine towns. This session adds SMS as a notification channel for civic engagement features (straw polls, public hearing reminders, meeting alerts).

Read docs/advisory-resolutions/2.4-notification-providers.md for the complete Twilio architecture, TCPA compliance requirements, 10DLC registration process, notification event system architecture, and per-town configuration. This advisory defines all the compliance requirements that MUST be implemented.

PROJECT CONTEXT:

- Session 11.01 built the notification event system with Postmark email as the first channel
- SMS is the second channel added to the existing notification dispatch system
- Per advisory 2.4: one Twilio Messaging Service per town with a dedicated local phone number
- TCPA compliance is legally mandatory — opt-in, consent records, opt-out, quiet hours
- 10DLC campaign registration required for A2P SMS (handled by admin during onboarding)
- All Maine towns are in the Eastern time zone

TASK 1: Install and configure Twilio SDK
- Install in packages/api: pnpm --filter @town-meeting/api add twilio
- Create packages/api/src/services/sms/twilio-client.ts
- Initialize the Twilio client with credentials from environment variables:
  - TWILIO_ACCOUNT_SID
  - TWILIO_AUTH_TOKEN
- Create a singleton client instance with error handling for missing credentials
- Type the client using the Twilio SDK types
- Create a helper function to get the Messaging Service SID for a specific town:
  async function getTownMessagingService(townId: string): Promise<string>
  - Reads from the town_notification_config table (defined in advisory 2.4)

TASK 2: Create Twilio SMS service
- Create packages/api/src/services/sms/sms-service.ts
- Core service functions:
  async function sendSMS(params: {
    to: string;                    // E.164 format phone number
    body: string;                  // message text (max 1600 chars, 160 per segment)
    townId: string;                // determines which Messaging Service to use
    notificationEventId?: string;  // link to the notification event for tracking
  }): Promise<SMSSendResult>
- The sendSMS function:
  1. Validates the phone number format (E.164: +1XXXXXXXXXX)
  2. Checks quiet hours (Task 7) — if outside allowed hours, queue for later
  3. Checks opt-out status — if recipient has opted out, skip and log
  4. Sends via Twilio Messaging Service (not direct from a phone number):
     client.messages.create({
       messagingServiceSid: townMessagingServiceSid,
       to: params.to,
       body: params.body,
       statusCallback: `${API_BASE_URL}/api/webhooks/twilio/status`
     })
  5. Creates an SMS_MESSAGE record for tracking
  6. Returns { messageId, status, segmentCount }
- Create the SMS_MESSAGE table:
  id, town_id, notification_event_id FK, recipient_phone (hashed for privacy), twilio_message_sid, body_preview (first 50 chars), segment_count, status (queued/sent/delivered/failed/undelivered), error_code, error_message, sent_at, delivered_at, created_at

TASK 3: Configure Messaging Service per town
- Extend the TOWN_NOTIFICATION_CONFIG table (from session 11.01) with SMS fields:
  - twilio_messaging_service_sid: TEXT (the Messaging Service SID for this town)
  - twilio_phone_number: TEXT (the town's dedicated phone number, display only)
  - sms_enabled: BOOLEAN DEFAULT FALSE (admin must explicitly enable after 10DLC registration)
  - sms_quiet_hours_start: TIME DEFAULT '21:00' (9 PM ET — no messages after this)
  - sms_quiet_hours_end: TIME DEFAULT '08:00' (8 AM ET — no messages before this)
  - sms_opt_in_message: TEXT (customizable consent text, with required defaults)
  - tenDLC_campaign_registered: BOOLEAN DEFAULT FALSE (tracks whether 10DLC registration is complete)
- Create an admin settings page section for SMS configuration:
  - Messaging Service SID input (provided by admin after Twilio console setup)
  - Phone number display (populated from Twilio API after configuration)
  - Enable/disable SMS toggle (disabled until 10DLC is registered)
  - Quiet hours start/end time pickers
  - Opt-in message text editor with required elements validation
- Validation: SMS cannot be enabled unless twilio_messaging_service_sid is set and tenDLC_campaign_registered is true

TASK 4: Implement TCPA-compliant opt-in flow
- Create packages/web/src/components/notifications/SMSOptInForm.tsx
- This form appears on the public portal when residents want to receive SMS notifications
- Required TCPA disclosures (per advisory 2.4 — these are legally required):
  - Clear description of message types: "You will receive text messages about: upcoming straw polls, public hearing notices, meeting schedule changes, and meeting cancellation alerts."
  - Expected frequency: "Expect approximately 2-4 messages per month."
  - Cost notice: "Message and data rates may apply."
  - Opt-out instructions: "Reply STOP to any message to unsubscribe at any time."
  - Help instructions: "Reply HELP for help."
  - Privacy policy link: link to the town's privacy policy or a default TMM privacy policy
- Form fields:
  - Phone number input with US phone number formatting (+1 XXX-XXX-XXXX)
  - Checkboxes for notification types (straw polls, public hearings, meeting changes) — at least one required
  - Explicit consent checkbox: "I agree to receive text messages from [Town Name] at the number provided. I understand I can opt out at any time by replying STOP." — this checkbox is REQUIRED
- On submit:
  1. Validate phone number format
  2. Store the consent record (Task 5)
  3. Send a confirmation SMS: "Welcome to [Town Name] notifications. Reply STOP to unsubscribe. Reply HELP for help. Msg & data rates may apply."
  4. Show success message to the resident

TASK 5: Store opt-in consent records
- Create SMS_CONSENT table:
  id UUID PRIMARY KEY,
  town_id UUID NOT NULL REFERENCES town(id),
  resident_account_id UUID REFERENCES resident_account(id),  -- if registered resident
  phone_number TEXT NOT NULL,                                  -- E.164 format
  phone_number_hash TEXT NOT NULL,                             -- SHA-256 hash for lookups without storing plaintext in logs
  consent_granted BOOLEAN NOT NULL DEFAULT TRUE,
  consent_timestamp TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  consent_method TEXT NOT NULL,                                -- web_form, in_person, paper_form
  consent_ip_address INET,                                     -- IP address at time of web consent
  consent_user_agent TEXT,                                      -- browser user agent at time of consent
  consent_language_shown TEXT NOT NULL,                         -- exact text of the consent disclosure shown to the user
  notification_types TEXT[] NOT NULL,                           -- which types they opted into
  opted_out BOOLEAN NOT NULL DEFAULT FALSE,
  opted_out_at TIMESTAMPTZ,
  opted_out_method TEXT,                                        -- sms_stop, web_form, admin_manual
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(town_id, phone_number)
- This table is the legal record of consent — it must capture everything needed to prove compliance if challenged:
  - The exact consent language shown at opt-in time
  - The timestamp and method of consent
  - The IP address (for web form consents)
- RLS: admin and staff can read, public can insert (their own consent only)
- Create a function to check consent: async function hasActiveConsent(townId, phoneNumber): Promise<boolean>

TASK 6: Implement opt-out handling
- Twilio Messaging Service handles STOP keyword processing automatically:
  - When a recipient texts "STOP" to the town's number, Twilio intercepts it
  - Twilio sends a standard opt-out confirmation: "You have been unsubscribed..."
  - Twilio fires a webhook to our application
- Create a webhook endpoint POST /api/webhooks/twilio/opt-out:
  - Validate the webhook signature (Twilio request validation using the auth token)
  - Extract the phone number that opted out
  - Update the SMS_CONSENT record: set opted_out = true, opted_out_at = NOW(), opted_out_method = 'sms_stop'
  - Log the opt-out event
  - No further messages will be sent to this number (the sendSMS function checks opt-out status)
- Handle opt-back-in: if a recipient texts "START" or "UNSTOP":
  - Twilio fires a webhook
  - Update SMS_CONSENT: set opted_out = false, opted_out_at = null
  - Send a re-subscription confirmation SMS
- Create a web-based opt-out option: residents can also opt out via a link in the public portal
  - The opt-out link updates the SMS_CONSENT record with opted_out_method = 'web_form'

TASK 7: Implement quiet hours enforcement
- Create packages/api/src/services/sms/quiet-hours.ts
- Per advisory 2.4: no SMS before 8:00 AM or after 9:00 PM Eastern time
- Function: isWithinSendingHours(townId: string): boolean
  - Read the town's quiet hours configuration from TOWN_NOTIFICATION_CONFIG
  - Compare against current time in America/New_York timezone
  - Return true if SMS can be sent right now, false if in quiet hours
- If a message is triggered during quiet hours:
  1. Queue the message for delivery at the start of the next allowed window (8:00 AM ET next day)
  2. Create an SMS_QUEUED_MESSAGE record: id, sms_message fields, scheduled_send_at, reason ('quiet_hours')
  3. A scheduled job runs at 8:00 AM ET and processes the queue
- Edge case: if a meeting cancellation happens at 10 PM, the SMS is queued for 8 AM the next morning
  - For truly urgent notifications, the admin dashboard could show a warning: "This notification will be sent at 8:00 AM tomorrow due to quiet hours"
- Create a utility function getNextSendWindow(timezone: string): Date that returns the next valid sending time

TASK 8: Add SMS channel to notification event system
- Modify the notification dispatch system from session 11.01:
  - The existing system creates notification events and dispatches to email via Postmark
  - Add SMS as a second channel:
    - When a notification event is created, query subscribers for their channel preferences
    - For subscribers who prefer SMS (or both email+SMS):
      - Look up their SMS consent record
      - If consent is active and not opted out: send SMS
      - If no consent: skip SMS channel, send email only (if email is configured)
  - The dispatch flow per advisory 2.4:
    Event Trigger -> Notification Event Created -> Subscriber Query (filter by channel, opt-in) -> Channel Dispatch (Email via Postmark, SMS via Twilio) -> Delivery Tracking
- Update the subscriber notification preferences schema to include SMS:
  - channel: 'email' | 'sms' | 'both'
  - For SMS subscribers, consent_record_id links to the SMS_CONSENT table
- Create a shared notification dispatch function that handles both channels:
  async function dispatchNotification(event: NotificationEvent): Promise<DispatchResult>

TASK 9: Create SMS templates
- Create packages/api/src/services/sms/sms-templates.ts
- Templates for each notification type (SMS must be concise — 160 chars per segment):
  1. straw-poll-invite:
     "[Town Name]: A new poll is available — '[Poll Question]'. Vote at [short URL]. Reply STOP to opt out."
  2. public-hearing-notice:
     "[Town Name]: Public hearing on [Topic] — [Date] at [Time], [Location]. Details: [short URL]. Reply STOP to opt out."
  3. meeting-cancelled:
     "[Town Name]: [Board Name] meeting on [Date] has been cancelled. Reply STOP to opt out."
  4. meeting-schedule-change:
     "[Town Name]: [Board Name] meeting rescheduled to [New Date] at [New Time]. Details: [short URL]. Reply STOP to opt out."
- All templates must include "Reply STOP to opt out" (TCPA requirement)
- Templates are stored as functions that take variables and return the formatted message:
  function formatSMSTemplate(template: SMSTemplateType, vars: Record<string, string>): string
- Validate that the final message is within SMS limits (warn if >160 chars — will be sent as multi-segment)
- URL shortening: use a simple redirect via the app's URL (e.g., tmm.town/p/ABC123) to keep messages short

TASK 10: Implement delivery tracking via webhooks
- Create a webhook endpoint POST /api/webhooks/twilio/status:
  - Twilio sends status callbacks for each message: queued, sent, delivered, undelivered, failed
  - Validate the webhook signature using Twilio's request validation
  - Update the SMS_MESSAGE record with the delivery status and timestamp
  - Handle failure codes:
    - 30001 (Queue overflow): retry the message
    - 30002 (Account suspended): alert admin immediately
    - 30003 (Unreachable): mark as undelivered, do not retry
    - 30004 (Message blocked): recipient may have blocked the number
    - 30005 (Unknown destination): invalid phone number, update consent record
    - 30006 (Landline): cannot receive SMS, notify admin
    - 21610 (Opted out via carrier): update consent record
- Track delivery metrics per town:
  - Total sent, delivered, failed, undelivered counts
  - Delivery rate percentage
  - Average delivery time

TASK 11: Create admin SMS dashboard
- Create packages/web/src/routes/admin.sms.tsx (or add as a tab to existing admin settings)
- Dashboard sections:
  - Overview cards: Total subscribers, Active subscribers (not opted out), Messages sent (this month), Delivery rate %
  - Subscriber list: phone (masked: ***-***-1234), consent date, notification types, status (active/opted-out)
  - Message history: DataTable showing recent messages with status, type, sent_at, delivered_at
  - Delivery analytics: simple chart or stats showing sent/delivered/failed breakdown
  - Failed messages: list of failed deliveries with error codes and descriptions
  - Quiet hours indicator: "SMS sending is currently [active/paused for quiet hours]. Next window: 8:00 AM ET"
- Admin actions:
  - Manually opt-out a subscriber (with reason tracking)
  - Resend a failed message
  - Send a test SMS to verify configuration
  - View/export consent records (for compliance audits)
- Phone numbers are always displayed masked (last 4 digits only) except in the consent record detail view
```
