# Session 20.01 — Resident Accounts & Straw Polls

**Phase:** 2 — Differentiation | **Block:** 20 — Civic Engagement
**Dependencies:** 10.01, 19.01
**Estimated tasks:** 12

---

## Description

Build the resident self-registration system and the straw poll feature for civic engagement. Residents can register on the public portal to receive notifications and participate in non-binding opinion polls on upcoming agenda items. Straw polls allow town officials to gauge public sentiment before meetings. This session creates resident accounts, poll creation and management, the public voting interface, results display, and notification integration.

## Tasks

1. Create resident self-registration on public portal: name, email, phone (optional), notification preferences
2. Create PERSON + RESIDENT_ACCOUNT on registration (no USER_ACCOUNT, no app login)
3. Create straw poll system: admin creates poll linked to an agenda item, sets question, answer options, open/close dates
4. Create STRAW_POLL table: id, town_id, agenda_item_id FK, question, options JSONB, status, opens_at, closes_at, created_by FK
5. Create STRAW_POLL_RESPONSE table: id, poll_id FK, resident_account_id FK, response, submitted_at
6. Create public poll voting page: residents access via link in notification email/SMS
7. Implement one-vote-per-resident enforcement (by resident_account_id)
8. Create poll results display: aggregate results shown after poll closes (on portal and admin dashboard)
9. Trigger notifications when poll is published: email + SMS to subscribed residents
10. Create admin poll management: create, edit, close, view results
11. Implement resident subscription management: residents can manage notification preferences and unsubscribe
12. Verify: resident registers, receives poll notification, votes once, results aggregate correctly, subscription management works

## Prompt

```
You are building the Resident Accounts and Straw Poll system for the Town Meeting Manager — a municipal meeting management platform for Maine towns. Straw polls are non-binding opinion polls that let town officials gauge public sentiment on upcoming agenda items before meetings. Residents register on the public portal to receive notifications and participate in polls.

Important distinction: resident accounts are NOT user accounts. Residents do not log into the application — they interact only through the public portal and notification links. Their accounts store contact information and notification preferences, not authentication credentials.

PROJECT CONTEXT:
- Monorepo: packages/web (React 19 + React Router v7 Framework Mode), packages/api (Fastify), packages/shared (types, schemas)
- Public portal from session 10.01 (townname.townmeetingmanager.com)
- Notification event system from session 11.01 (email via Postmark)
- SMS notifications from session 19.01 (Twilio with TCPA compliance)
- PowerSync for offline-first local SQLite in the admin app
- Tailwind CSS v4 + shadcn/ui for admin components
- The public portal uses server-rendered pages (not the PowerSync-connected admin app)

TASK 1: Create resident self-registration on public portal
- Route: /portal/:townSlug/subscribe (or /portal/:townSlug/register)
- This is a public page — no authentication required
- Registration form fields:
  - First name (required, text input)
  - Last name (required, text input)
  - Email address (required, validated)
  - Phone number (optional, US format — if provided, triggers SMS opt-in flow from session 19.01)
  - Street address (optional — used for proximity notifications in Phase 3)
  - Notification preferences checkboxes:
    - Meeting agenda published (email only in Phase 2)
    - Meeting minutes approved (email only)
    - Straw polls (email + SMS if phone provided)
    - Public hearing notices (email + SMS if phone provided)
    - Meeting schedule changes (email + SMS if phone provided)
  - Preferred channel: email only, SMS only (requires phone), or both
- If phone number is provided and SMS preferences selected:
  - Show the TCPA consent form from session 19.01 inline
  - The SMS consent is recorded as part of registration
- On submit: create PERSON + RESIDENT_ACCOUNT records, send confirmation email
- Email confirmation: "Welcome to [Town Name] notifications. You're now subscribed to receive updates about [selected preferences]."
- Duplicate detection: if email already registered for this town, show message: "This email is already registered. Would you like to update your preferences?" with a link to the preference management page

TASK 2: Create PERSON + RESIDENT_ACCOUNT records
- Per the data model from advisory 1.2 (roles-permissions):
  - PERSON table already exists: id, first_name, last_name, email, phone, town_id, created_at
  - Create RESIDENT_ACCOUNT table:
    id UUID PRIMARY KEY,
    person_id UUID NOT NULL REFERENCES person(id),
    town_id UUID NOT NULL REFERENCES town(id),
    email_verified BOOLEAN DEFAULT FALSE,
    email_verification_token UUID,
    email_verification_sent_at TIMESTAMPTZ,
    notification_preferences JSONB NOT NULL DEFAULT '{}',
    preferred_channel TEXT DEFAULT 'email',        -- email, sms, both
    street_address TEXT,                            -- for future proximity matching
    status TEXT NOT NULL DEFAULT 'pending',         -- pending (awaiting email verification), active, unsubscribed
    unsubscribed_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(person_id, town_id)
  - The PERSON record has no user_account — residents are not app users
- Email verification flow:
  1. On registration, generate a verification token (UUID)
  2. Send verification email via Postmark with a link: /portal/:townSlug/verify?token=XXX
  3. When the resident clicks the link, set email_verified = true, status = 'active'
  4. Only active (verified) residents receive notifications and can vote in polls
- RLS policies: admin/staff can read all resident accounts for their town; residents can update their own preferences via token-based access

TASK 3: Create straw poll system
- Straw polls are created by admin or staff and linked to a specific agenda item
- A straw poll asks a question with predefined answer options:
  - Example: "Should the town appropriate $50,000 for playground improvements?"
  - Options: "Yes, support the appropriation" / "No, oppose the appropriation" / "Need more information"
- Polls have a lifecycle: draft -> published (open for voting) -> closed (results available)
- Polls are tied to a specific meeting's agenda item — they gather input BEFORE the meeting
- After the poll closes, results are displayed on the public portal and admin dashboard
- Results are advisory only — they do not bind the board's decision

TASK 4: Create STRAW_POLL table
- CREATE TABLE straw_poll (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    town_id UUID NOT NULL REFERENCES town(id),
    meeting_id UUID NOT NULL REFERENCES meeting(id),
    agenda_item_id UUID REFERENCES agenda_item(id),  -- optional link to specific agenda item
    question TEXT NOT NULL,                            -- the poll question
    description TEXT,                                  -- additional context about the item
    options JSONB NOT NULL,                            -- array of {id: string, label: string, sort_order: number}
    allow_comments BOOLEAN DEFAULT FALSE,              -- can voters add a comment with their vote
    status TEXT NOT NULL DEFAULT 'draft',               -- draft, published, closed
    opens_at TIMESTAMPTZ NOT NULL,                     -- when voting opens
    closes_at TIMESTAMPTZ NOT NULL,                    -- when voting closes (typically before the meeting)
    results_visible_after TEXT DEFAULT 'close',        -- 'close' (show after closes_at) or 'vote' (show immediately after voting)
    created_by UUID NOT NULL REFERENCES person(id),
    published_at TIMESTAMPTZ,
    closed_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
  );
- Validation: opens_at must be before closes_at, closes_at should be before the meeting's scheduled_date
- Add PowerSync sync rules for admin clients

TASK 5: Create STRAW_POLL_RESPONSE table
- CREATE TABLE straw_poll_response (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    poll_id UUID NOT NULL REFERENCES straw_poll(id),
    resident_account_id UUID NOT NULL REFERENCES resident_account(id),
    selected_option_id TEXT NOT NULL,                  -- matches the id from the poll's options JSONB
    comment TEXT,                                      -- optional voter comment (if allow_comments is true)
    submitted_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(poll_id, resident_account_id)               -- one vote per resident per poll
  );
- The UNIQUE constraint on (poll_id, resident_account_id) enforces one-vote-per-resident at the database level
- RLS: residents can insert their own response (via API), admin/staff can read all responses
- Responses are anonymous in the results display — the admin can see who voted (for audit) but public results show only aggregated counts

TASK 6: Create public poll voting page
- Route: /portal/:townSlug/polls/:pollId (or /portal/:townSlug/vote/:pollId)
- Accessed via link in notification email/SMS — no login required
- The voting page is token-based: the notification link includes a token that identifies the resident
  - URL format: /portal/:townSlug/polls/:pollId?token=RESIDENT_TOKEN
  - The token is a signed JWT or HMAC containing the resident_account_id (not guessable)
  - If no valid token: show the poll question and options as read-only with a "Register to vote" link
- Voting interface:
  - Poll question displayed prominently at the top
  - Description/context below the question
  - Radio buttons or card-style selectors for each option
  - Optional comment textarea (if allow_comments is true, max 500 chars)
  - "Submit Vote" button
  - Shows the related agenda item details and meeting date for context
- After voting:
  - Show confirmation: "Thank you! Your response has been recorded."
  - If results_visible_after is 'vote': show the current aggregate results immediately
  - If results_visible_after is 'close': show "Results will be available after [closes_at date]"
- If the resident has already voted: show their previous response with a message "You have already voted in this poll. Your response: [option]." Do not allow changing the vote.
- If the poll is closed: show the results, no voting allowed

TASK 7: Implement one-vote-per-resident enforcement
- Enforcement layers:
  1. Database: UNIQUE(poll_id, resident_account_id) constraint prevents duplicate rows
  2. API: POST /api/polls/:pollId/vote checks for existing response before inserting
  3. UI: the voting page checks for existing response and shows "already voted" state
- Token-based voting prevents casual double-voting but is not hardened against determined abuse (acceptable for non-binding polls)
- Create a Fastify route POST /api/polls/:pollId/vote:
  - Validate the resident token
  - Check poll status is 'published' and current time is between opens_at and closes_at
  - Check for existing response from this resident
  - If no existing response: insert STRAW_POLL_RESPONSE
  - If existing response: return 409 Conflict with the existing response
  - Rate limit: 10 requests per minute per IP to prevent brute-force attempts

TASK 8: Create poll results display
- Create packages/web/src/components/polls/PollResults.tsx (for admin dashboard)
- Create a public results page: /portal/:townSlug/polls/:pollId/results
- Results calculation:
  - Aggregate responses by selected_option_id: COUNT, percentage of total votes
  - Total votes cast
  - If allow_comments: list of comments (anonymous, without voter identity)
- Results display:
  - Horizontal bar chart showing option percentages
  - Raw counts next to each bar: "Yes: 142 votes (67%)"
  - Total participation: "213 of 850 registered residents voted (25%)"
  - If the poll is still open: show "Voting closes [date/time]" — results update in real-time (or on refresh)
  - If the poll is closed: show final results with "Voting closed [date/time]" badge
- Admin view (additional info):
  - Participation rate
  - Voter comments list (with timestamps)
  - Export results as CSV
  - "Close Poll" button (if still open)
  - Link to the related agenda item

TASK 9: Trigger notifications when poll is published
- When an admin publishes a straw poll (status changes from 'draft' to 'published'):
  1. Create a notification event with type: 'straw_poll_published'
  2. The notification dispatch system (from 11.01 + 19.01) queries subscribers:
     - Find all active resident_accounts for this town with 'straw_polls' in their notification_preferences
     - Filter by preferred_channel (email, sms, both)
  3. Dispatch notifications:
     - Email: Use Postmark template 'straw-poll-invite' with variables: poll question, voting link, open/close dates, meeting date
     - SMS: Use the SMS template from 19.01: "[Town Name]: New poll — '[Question]'. Vote at [link]. Reply STOP to opt out."
  4. Track delivery status for each notification
- The voting link in the notification includes the resident-specific token for identification
- Only send to verified (email_verified = true, status = 'active') residents

TASK 10: Create admin poll management
- Route: /meetings/:meetingId/polls (or as a tab on the meeting detail page)
- File: packages/web/src/routes/meetings.$meetingId.polls.tsx
- Poll list: DataTable showing all polls for this meeting with columns: Question, Status, Responses, Opens, Closes, Actions
- "Create Poll" button opens a creation dialog/page:
  - Select agenda item (dropdown of items in this meeting)
  - Question text (required)
  - Description/context (optional)
  - Answer options (minimum 2, maximum 6):
    - Each option has a label (text input) and sort order
    - "Add Option" button to add more options
    - Drag-and-drop reordering
  - Allow comments toggle
  - Opens at: date/time picker (default: now)
  - Closes at: date/time picker (default: 24 hours before meeting)
  - Results visibility: radio buttons (after close / immediately after voting)
- Poll lifecycle actions:
  - Publish: changes status to 'published', sends notifications, opens voting
  - Close: changes status to 'closed', stops accepting votes, makes results final
  - Delete: only in draft status, with confirmation
- Edit: modify question, options, dates (only before publishing — after publishing, only dates can be changed)

TASK 11: Implement resident subscription management
- Route: /portal/:townSlug/preferences?token=RESIDENT_TOKEN
- The link to this page is included in every notification email (in the footer)
- Preference management page:
  - Shows current notification preferences with checkboxes (matching registration form)
  - "Save Preferences" button updates the RESIDENT_ACCOUNT.notification_preferences
  - "Unsubscribe from All" button:
    - Sets status to 'unsubscribed', unsubscribed_at to now
    - If SMS was active, updates SMS_CONSENT record to opted_out
    - Shows confirmation: "You have been unsubscribed from all [Town Name] notifications."
  - Channel preference: switch between email only, SMS only, or both
- Every email includes a one-click unsubscribe link (required by CAN-SPAM):
  - URL: /portal/:townSlug/unsubscribe?token=RESIDENT_TOKEN
  - One-click: no form required — clicking the link immediately unsubscribes
  - Shows confirmation page after unsubscribing
- Email footer template (added to all notification emails):
  "You're receiving this because you subscribed to [Town Name] notifications. Manage preferences: [link] | Unsubscribe: [link]"

TASK 12: Verification
- Test the complete civic engagement workflow:
  1. Visit the public portal for a test town
  2. Register as a resident with email and phone number — PERSON + RESIDENT_ACCOUNT created
  3. Receive verification email, click link — account activated

  5. Admin creates a straw poll linked to an upcoming meeting agenda item
  6. Admin publishes the poll — notification dispatches to subscribed residents

  10. Submit a vote — response recorded, confirmation shown

  12. Admin views poll results — aggregate counts and percentages displayed
  13. Admin closes the poll — results finalized, voting disabled
  14. Public results page shows final results
  15. Resident manages preferences — updates notification types, unsubscribes
  16. Verify unsubscribed resident no longer receives notifications
```
