# Session 21.02 — Meeting Participation & Audio

**Phase:** 2 — Differentiation | **Block:** 21 — React Native Mobile App
**Dependencies:** 21.01, 17.01
**Estimated tasks:** 8

---

## Description

Build the mobile meeting participation features and audio recording capabilities. Board members can view live meeting agendas, cast votes from their devices, and record audio using the mobile microphone. The app stays synchronized with the operator's web session via PowerSync real-time sync, and integrates push notifications for meeting reminders and poll alerts.

## Tasks

1. Create mobile meeting view: agenda display, motion/vote tracking (read-only for board members, M8 vote permission)
2. Create mobile voting interface: board member can cast vote (yea/nay/abstain) from their device
3. Implement PowerSync real-time sync: mobile devices stay in sync with operator's web session
4. Create mobile audio recording using expo-audio: same segment-by-agenda-item approach as web
5. Implement background audio recording: continues recording when app is in background
6. Create audio upload: segment files upload to Supabase Storage on WiFi/cellular
7. Implement push notifications via expo-notifications for meeting reminders and poll notifications
8. Verify: mobile app shows live meeting data, voting works, audio records, push notifications receive

TASK 1: Create mobile meeting view
- File: packages/mobile/src/screens/MeetingLiveView.tsx (or app/(app)/meetings/[meetingId]/live.tsx)
- This screen shows the live meeting state as the operator runs the meeting from the web app:
  - Meeting header: board name, date, status badge (in_progress, scheduled, completed)
  - Current agenda item: prominently displayed — the item the operator is currently on
    - Title, description, presenter
    - Status: active, pending, completed
    - Section type badge (action, discussion, public_hearing, etc.)
  - Agenda overview: collapsible list of all agenda items
    - Items the meeting has already passed through are marked "completed" (dimmed)
    - The current item is highlighted
    - Upcoming items show as "pending"
  - Attendance: list of present/absent members (from the operator's attendance record)
  - Motion/vote display: when a motion is active, show:
    - Motion text
    - Mover and seconder names
    - Current vote status (if voting is in progress)
    - Vote result (after recording)
- All data comes from PowerSync queries — the mobile app reads the same tables the web operator writes to
- The screen auto-updates as the operator progresses through the meeting (PowerSync reactive queries via useQuery)
- Board members see a read-only view — they cannot advance agenda items, record motions, or modify the meeting. The exception is voting (Task 2).

TASK 2: Create mobile voting interface
- File: packages/mobile/src/components/meeting/MobileVoteCard.tsx
- Per advisory 1.2 M8 decision: board members can cast their individual vote from the mobile app
- When the operator initiates a vote on the web app:
  1. The operator creates a VOTE record with status 'in_progress'
  2. PowerSync syncs this to all connected mobile devices
  3. The mobile app detects the active vote and displays the voting card
- Mobile voting card UI:
  - Motion text displayed at the top
  - Three large buttons: "Yea" (green), "Nay" (red), "Abstain" (gray)
  - One tap to select — button highlights to show selection
  - "Submit Vote" confirmation button (prevent accidental votes)
  - After submission: shows "Vote recorded" with the member's selection
  - If the member has already voted: shows their recorded vote, no option to change
- Vote recording:
  - Create or update a VOTE_RECORD entry in PowerSync: vote_id, person_id, vote_value ('yea'|'nay'|'abstain'), submitted_via ('mobile'), submitted_at
  - PowerSync syncs this back to the server, where the operator's web app picks it up
  - The operator sees individual votes arriving in real-time on the web interface
- Permission check: only users with the 'board_member' role for the relevant board can vote
  - Use the useCurrentUser() hook to verify role and board membership
  - Non-members see the vote happening but cannot participate
- Recusal handling: if the board member has recused themselves from this item, the voting card does not appear

TASK 3: Implement PowerSync real-time sync
- Ensure PowerSync on mobile maintains real-time sync with the server during meetings:
  - The mobile app connects to the same PowerSync instance as the web app
  - Changes made by the web operator (advancing items, recording motions, starting votes) sync to mobile within 1-2 seconds
  - Changes made by mobile users (casting votes) sync back to the server within 1-2 seconds
- Configure PowerSync for optimal real-time behavior during meetings:
  - Use the streaming sync mode (not just polling) for low-latency updates
  - Handle reconnection gracefully: if the mobile device loses connectivity briefly, PowerSync reconnects and catches up
  - Show a sync status indicator on the meeting screen:
    - "Connected" (green dot): real-time sync active
    - "Syncing..." (amber dot): catching up after reconnection
    - "Offline" (red dot): no connection — data may be stale
- Create packages/mobile/src/components/SyncStatusBadge.tsx:
  - Use PowerSync's sync status API to show current state
  - Position in the header area of the meeting view
- Test scenario: operator starts a vote on web -> mobile shows voting card within 2 seconds -> board member votes on mobile -> operator sees the vote on web within 2 seconds

TASK 4: Create mobile audio recording
- File: packages/mobile/src/services/audio-recorder.ts
- Use expo-av (Audio API) for React Native audio recording:
  pnpm --filter @town-meeting/mobile add expo-av
- Create the audio recording service:
  import { Audio } from 'expo-av';

  - Request microphone permissions: Audio.requestPermissionsAsync()
  - Configure the recording:
    Audio.Recording.createAsync(
      Audio.RecordingOptionsPresets.HIGH_QUALITY,
      onRecordingStatusUpdate,
      100  // update interval in ms
    )
  - Use M4A/AAC format (native iOS/Android format, good compression, Deepgram-compatible)
- Implement the same segment-by-agenda-item approach as the web recorder (session 17.01):
  - When the current agenda item changes (detected via PowerSync sync), finalize the current segment and start a new one
  - Each segment is saved as a separate file: {meeting_id}_{segment_number}_{agenda_item_id}.m4a
  - Store segment metadata in PowerSync: AUDIO_SEGMENT table (same schema as web)
- Create packages/mobile/src/components/meeting/MobileRecordingControls.tsx:
  - Compact recording controls suitable for a mobile screen
  - Record/Pause/Stop buttons
  - Duration timer
  - Audio level indicator (simple bar based on metering data from expo-av)
  - "Recording" badge visible at all times during the meeting
- The mobile recorder is independent of the web recorder — both can record simultaneously (one operator mic, one mobile mic), or only one can record

TASK 5: Implement background audio recording
- Mobile audio recording must continue when the app is in the background:
  - Board members may switch apps during a meeting (check email, review documents)
  - Recording should not stop when the app loses focus
- Configure expo-av for background audio:
  - Set Audio.setAudioModeAsync({
      allowsRecordingIOS: true,
      staysActiveInBackground: true,
      playsInSilentModeIOS: true,
    })
  - For iOS: add UIBackgroundModes: ['audio'] to app.config.ts Info.plist configuration
  - For Android: the recording continues in the background by default with expo-av
- Show a persistent notification while recording in background:
  - iOS: the standard recording indicator (orange dot) appears in the status bar
  - Android: create a foreground service notification: "Town Meeting Manager is recording audio"
  - Tapping the notification returns to the meeting screen
- Handle app termination: if the app is killed while recording:
  - Save the current segment data before termination (use React Native's AppState listener)
  - On next app launch, detect the interrupted segment and mark it appropriately
  - Resume recording is NOT automatic after termination — the user must start a new session

TASK 6: Create audio upload service
- File: packages/mobile/src/services/audio-upload.ts
- Upload completed audio segments to Supabase Storage:
  - Same bucket and path structure as web: meeting-audio/{town_id}/meetings/{meeting_id}/segments/{segment}.m4a
  - Use Supabase Storage client for uploads:
    const { data, error } = await supabase.storage
      .from('meeting-audio')
      .upload(filePath, fileBlob, { contentType: 'audio/mp4' });
- Upload strategy:
  - Queue segments for upload as they are finalized
  - Upload over WiFi immediately if available
  - Upload over cellular with a size check: segments under 20 MB upload immediately; larger segments wait for WiFi (configurable)
  - Use NetInfo (from @react-native-community/netinfo) to detect connection type:
    pnpm --filter @town-meeting/mobile add @react-native-community/netinfo
  - Show upload progress per segment in the UI
  - Failed uploads retry with exponential backoff (3 attempts)
  - If the meeting ends and uploads are still pending, show a notification: "Uploading meeting audio... [3 of 7 segments complete]"
- Track upload status in PowerSync:
  - Update AUDIO_SEGMENT.status: recorded -> uploading -> uploaded -> failed
  - Upload progress is visible on both mobile and web (via PowerSync sync)
- Handle offline uploads:
  - Store segment files in the app's document directory (FileSystem.documentDirectory)
  - Queue upload tasks in AsyncStorage
  - When connectivity returns, process the upload queue
  - Files are deleted from local storage after successful upload (to free device space)

TASK 7: Implement push notifications
- Install expo-notifications:
  pnpm --filter @town-meeting/mobile add expo-notifications expo-device expo-constants
- Create packages/mobile/src/services/push-notifications.ts:
  - Request push notification permissions on app launch (with explanation dialog)
  - Register for push notifications via Expo Push Notification Service:
    const token = await Notifications.getExpoPushTokenAsync({
      projectId: Constants.expoConfig?.extra?.eas?.projectId
    });
  - Store the push token on the server: link to the user's person_id in a DEVICE_TOKEN table:
    CREATE TABLE device_push_token (
      id UUID PRIMARY KEY,
      person_id UUID NOT NULL REFERENCES person(id),
      town_id UUID NOT NULL REFERENCES town(id),
      expo_push_token TEXT NOT NULL,
      device_name TEXT,
      platform TEXT,         -- ios, android
      active BOOLEAN DEFAULT TRUE,
      created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
      updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
      UNIQUE(person_id, expo_push_token)
    );
- Notification types to send via push:
  - Meeting reminder: "Select Board meeting starts in 30 minutes" (scheduled notification)
  - Meeting starting: "Select Board meeting is starting now" (when operator clicks "Start Meeting")
  - Straw poll available: "New poll: [Question]. Tap to vote."
  - Vote requested: "A vote is in progress for [Motion]. Open the app to cast your vote." (when operator starts a vote during meeting)
- Create a Fastify route on the API side: POST /api/push/send
  - Uses Expo Push Notification API (expo-server-sdk on the backend) to send notifications:
    pnpm --filter @town-meeting/api add expo-server-sdk
  - Batch send to all devices registered for the relevant town/board
- Handle notification received while app is in foreground:
  - Show an in-app alert/toast for meeting-related notifications
  - For vote-requested notifications: navigate to the meeting view automatically
- Handle notification tapped:
  - Deep link to the relevant screen (meeting view, poll voting page) using Expo Router linking
