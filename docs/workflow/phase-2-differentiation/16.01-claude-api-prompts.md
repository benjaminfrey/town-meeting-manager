# Session 16.01 — Claude API Integration & Prompt Architecture

**Phase:** 2 — Differentiation | **Block:** 16 — AI Minutes Drafting
**Dependencies:** 09.01
**Estimated tasks:** 11

---

## Description

Integrate the Anthropic Claude API into the backend and build the complete prompt architecture for AI-assisted minutes drafting. This includes the system prompt, per-section prompt templates, streaming response handling, fallback logic, and configurable AI settings per town. The AI receives structured meeting JSON and produces minutes text in the town's configured style.

## Tasks

1. Install Anthropic Claude SDK (@anthropic-ai/sdk) in packages/api
2. Create AI service in packages/api with Claude API client configuration
3. Design system prompt for minutes drafting: takes structured meeting JSON (attendance, agenda items, motions, votes, timestamps) and produces minutes text in the configured style (action/summary/narrative)
4. Create per-section prompt templates: call to order, attendance, each agenda item type, executive session, adjournment
5. Implement section-type-dependent output: public comment sections identify speakers by name, board discussion identifies by role (Q16 decision)
6. Create prompt for motion/vote summary formatting (configurable per board, Q13)
7. Implement streaming response handling for real-time draft generation UI feedback
8. Create fallback handling: if AI generation fails, provide empty template for manual editing
9. Store AI-generated content in MINUTES_DOCUMENT with generated_by: 'ai'
10. Create configurable AI settings per town: minutes style preference, section verbosity, naming conventions
11. Verify: Claude API called with meeting data, minutes draft generated in correct style, streaming works, fallback handles errors

## Prompt

```
You are building the Claude API integration and prompt architecture for AI-assisted minutes drafting in the Town Meeting Manager — a municipal meeting management platform for Maine towns. This session creates the backend AI service that takes structured meeting data and produces formatted minutes drafts.

Read docs/advisory-resolutions/2.2-ai-minutes-architecture.md for the complete prompt architecture, template system, hard rules, and data pipeline. This is critical context — the advisory defines the exact data model the AI receives (Section 2), the section types and minutes behaviors (Section 1), the hard rules the AI must follow (Section 6), and the review UX expectations (Section 5).

PROJECT CONTEXT:
- Monorepo: packages/api (Fastify backend), packages/shared (types, schemas)
- Supabase PostgreSQL database with PowerSync sync
- Minutes pipeline from session 09.01: structured meeting JSON -> HTML -> PDF
- The AI inserts itself between "structured meeting JSON" and "HTML generation"
- Three minutes styles: action (decisions only), summary (decisions + brief discussion), narrative (detailed who-said-what)
- Minutes are stored in MINUTES_DOCUMENT table with section-level content

TASK 1: Install and configure the Anthropic Claude SDK
- Install @anthropic-ai/sdk in packages/api: pnpm --filter @town-meeting/api add @anthropic-ai/sdk
- Create packages/api/src/services/ai/claude-client.ts
- Initialize the Anthropic client with API key from environment variable ANTHROPIC_API_KEY
- Configure default model: claude-sonnet-4-20250514 (fast, cost-effective for structured text generation)
- Set default max_tokens: 4096 (sufficient for a single section's minutes)
- Create a type-safe wrapper that handles rate limiting, retries (3 attempts with exponential backoff), and error classification (rate_limit, invalid_request, api_error, network_error)
- Export a singleton client instance for reuse across the service

TASK 2: Create the AI minutes generation service
- Create packages/api/src/services/ai/minutes-generator.ts
- This is the main orchestrator that:
  1. Receives a complete structured meeting record (the JSON from session 09.01)
  2. Loads the town's AI configuration (minutes style, verbosity, naming conventions)
  3. Iterates through each section, applying the appropriate prompt template based on section_type and minutes_behavior
  4. Calls Claude API for each section that requires AI generation (skip sections with minutes_behavior: 'skip')
  5. Assembles the complete minutes document from section outputs
  6. Stores the result in MINUTES_DOCUMENT with generated_by: 'ai'
- Create the TypeScript interface for the generation request:
  interface MinutesGenerationRequest {
    meetingId: string;
    townId: string;
    meetingData: StructuredMeetingRecord; // from @town-meeting/shared
    style: 'action' | 'summary' | 'narrative';
    overrides?: { sectionId: string; behavior: MinutesBehavior }[];
  }
- Create the response interface:
  interface MinutesGenerationResult {
    sections: GeneratedSection[];
    metadata: { model: string; totalTokens: number; generationTimeMs: number };
  }

TASK 3: Design the system prompt
- Create packages/api/src/services/ai/prompts/system-prompt.ts
- The system prompt establishes the AI's role and constraints. It must include:
  - Role: "You are a municipal meeting minutes drafter for a Maine town."
  - Context: The town name, board name, and minutes style preference
  - Hard rules from advisory 2.2 Section 6 (these are non-negotiable):
    * Never fabricate information not present in the structured data
    * Never add opinions, interpretations, or editorial commentary
    * Use the exact names and titles as recorded in the attendance data
    * Record vote counts exactly as captured (do not infer or round)
    * If data is missing or ambiguous, flag it with [VERIFY: reason] rather than guessing
    * Executive session: record only entry/exit times and the statutory citation — never speculate on content
    * Maintain consistent past tense throughout
    * Use formal municipal language appropriate for official government records
  - Style instructions vary by minutes style:
    * Action: Record only motions, seconds, vote results, and formal decisions. No discussion summaries.
    * Summary: Record motions/votes plus 2-3 sentence summaries of significant discussion points. Attribute discussion to roles, not names (unless in public comment sections).
    * Narrative: Record motions/votes plus detailed discussion including who said what, paraphrased. Attribute by name and role.
  - Output format: Return clean text (no markdown headers — the system adds those). Use standard municipal minutes formatting conventions.
- The system prompt is assembled dynamically per generation request, incorporating town-specific configuration.
- Export a function: buildSystemPrompt(config: SystemPromptConfig): string

TASK 4: Create per-section prompt templates
- Create packages/api/src/services/ai/prompts/section-templates.ts
- Each section_type + minutes_behavior combination has a specific prompt template.
- Templates are functions that take section data and return the user message for Claude:

  For minutes_behavior: 'timestamp_only':
    - Template: "Record that [section title] occurred at [timestamp]. Output one sentence."
    - Example output: "The meeting was called to order at 7:03 PM."

  For minutes_behavior: 'action_only':
    - Template: Provide the section data (motions, votes, decisions). Instruct the AI to record only the formal actions — no discussion summary.
    - Example output: "Motion by Percy, seconded by Paz, to approve the minutes of December 22, 2025 as presented. Vote: 4-0 in favor. Motion carried."

  For minutes_behavior: 'summarize':
    - Template: Provide the full section data including operator notes, timestamps, and any transcript segments. Instruct the AI to summarize the discussion and record actions.
    - The summary length scales with the minutes style (action < summary < narrative).

  For minutes_behavior: 'full_record':
    - Template: Provide all available data. Instruct the AI to produce the most detailed account possible from the data provided.

  For minutes_behavior: 'skip':
    - No API call. Return null for this section.

- Create specific templates for special section types:
  - attendance_template: Generate the attendance paragraph from the attendance JSON (present, absent, quorum, presiding officer succession)
  - public_hearing_template: Formal hearing format with applicant statement, public comment, board discussion, vote
  - executive_session_template: Only entry/exit times and statutory citation — hard rule, never generate content about what was discussed
  - adjournment_template: "Motion by [name], seconded by [name], to adjourn at [time]. Vote: [result]. Meeting adjourned at [time]."

TASK 5: Implement section-type-dependent speaker identification
- Per advisory 2.2 Q16 decision: public comment sections identify speakers by name; board discussion sections identify by role
- In public_input and public_hearing section templates:
  - Use speaker names: "John Smith of 45 Main Street expressed concern about..."
  - Include address if provided in the meeting data
- In discussion and action section templates for summary/narrative styles:
  - Use roles: "A board member asked about..." or "The Town Manager explained..."
  - Exception: motions and seconds always use names ("Motion by Percy")
- Create a helper function: formatSpeakerReference(speaker, sectionType, minutesStyle) that returns the appropriate reference string

TASK 6: Create motion/vote formatting prompt
- Create packages/api/src/services/ai/prompts/motion-template.ts
- Per advisory 2.2 Q13: motion/vote format is configurable per board
- Support two formats:
  - Standard: "Motion by [name], seconded by [name], to [motion text]. Vote: [Y]-[N] in favor. Motion carried/failed."
  - Roll call: "Motion by [name], seconded by [name], to [motion text]. Roll call vote: [Name]: Yea, [Name]: Yea, [Name]: Nay. Vote: [Y]-[N]-[A]. Motion carried/failed."
- Handle special vote scenarios from the meeting data:
  - Unanimous voice vote: "Vote: Unanimous (4-0). Motion carried."
  - Member recusal: "[Name] recused themselves from discussion and vote on this item."
  - Member abstention: Include abstention count in tally
  - No second: "Motion by [name] to [text]. Motion failed for lack of a second."
- The motion/vote data comes from the structured meeting record — the AI formats it, it does not interpret it

TASK 7: Implement streaming response handling
- Create packages/api/src/services/ai/stream-handler.ts
- Use the Anthropic SDK's streaming API: client.messages.stream()
- Create a Fastify route POST /api/meetings/:meetingId/minutes/generate that:
  1. Validates the request (meeting exists, user has permission, meeting is completed)
  2. Loads meeting data from the database
  3. Starts streaming generation section by section
  4. Returns a Server-Sent Events (SSE) stream to the client
- SSE event format:
  - event: section_start — { sectionId, sectionTitle, sectionType }
  - event: content_delta — { sectionId, delta: string } (partial text as it streams)
  - event: section_complete — { sectionId, fullText: string, tokenCount: number }
  - event: generation_complete — { totalSections, totalTokens, generationTimeMs }
  - event: error — { sectionId?, error: string, fallbackAvailable: boolean }
- The web client can subscribe to this SSE stream to show real-time generation progress

TASK 8: Create fallback handling
- Create packages/api/src/services/ai/fallback-handler.ts
- If Claude API generation fails for any section, the system must not block the minutes workflow
- Fallback strategy per section:
  1. Retry the failed section up to 3 times with exponential backoff
  2. If all retries fail, generate a template-based fallback:
     - For timestamp_only: Use the timestamp data directly (no AI needed)
     - For action_only: Format motions/votes from structured data using a simple template (no AI)
     - For summarize/full_record: Generate a skeleton with placeholders: "[AI generation failed — please draft this section manually. Available data: {motionCount} motions, {noteCount} operator notes]"
  3. Mark the section as generated_by: 'fallback' in the stored document
- Create a simple template-based formatter for motions/votes that works without AI:
  function formatMotionFallback(motion: MotionRecord): string
  - This produces correct but unstyled motion text directly from the structured data
- Log all failures with error details for debugging

TASK 9: Store AI-generated content in MINUTES_DOCUMENT
- Integrate with the existing MINUTES_DOCUMENT table from session 09.01
- When AI generation completes (or partially completes with fallbacks):
  1. Create or update the MINUTES_DOCUMENT record for this meeting
  2. Store each section's generated text in the sections JSONB array
  3. Set status: 'ai_draft' (new status — distinct from 'draft' which means manually created)
  4. Record metadata: model used, total tokens, generation time, which sections used AI vs. fallback
  5. Track generated_by per section: 'ai' | 'fallback' | 'manual'
- The MINUTES_DOCUMENT.sections structure:
  interface MinutesSection {
    sectionId: string;
    sectionTitle: string;
    sectionType: SectionType;
    content: string;           // the generated text
    generatedBy: 'ai' | 'fallback' | 'manual';
    editedBy?: string;         // person_id if manually edited
    editedAt?: string;         // timestamp of last edit
    aiMetadata?: { model: string; tokens: number; confidence: 'high' | 'medium' | 'low' };
  }
- Add the 'ai_draft' status to the minutes status enum in @town-meeting/shared

TASK 10: Create configurable AI settings per town
- Create AI_SETTINGS table or add columns to TOWN table:
  - minutes_style: 'action' | 'summary' | 'narrative' (default from onboarding wizard)
  - ai_verbosity: 'concise' | 'standard' | 'detailed' (controls summary length)
  - speaker_naming: 'role_default' | 'name_always' (override for Q16 behavior)
  - motion_format: 'standard' | 'roll_call' (per board override possible)
  - custom_instructions: text | null (additional town-specific instructions appended to the system prompt, e.g., "Always refer to the presiding officer as 'First Selectman'")
- Create a Fastify route GET/PUT /api/towns/:townId/ai-settings for reading and updating AI configuration
- Create a settings UI component (or add to the existing TownDashboard settings):
  - Minutes style radio buttons (action/summary/narrative) with descriptions
  - Verbosity slider or dropdown
  - Custom instructions textarea with character limit (500 chars)
  - Motion format selection per board
```
