# Session 18.01 — Warrant Article Data Model & Workflow

**Phase:** 2 — Differentiation | **Block:** 18 — Warrant Article Workflow
**Dependencies:** 06.02
**Estimated tasks:** 10

---

## Description

Build the warrant article management system for annual and special town meetings. Warrant articles are the formal items that voters act on at a town meeting — each article is a specific question or proposal with structured legal text, financial data, and a voting record. This session creates the data model, management UI, article editor, PDF generation, public portal display, and integration with the town meeting agenda.

## Tasks

1. Create WARRANT_ARTICLE table: id, town_id, article_number, title, full_text, category, sponsor, estimated_cost, funding_source, status (draft/published/voted/passed/failed), town_meeting_id FK, created_at, updated_at
2. Create WARRANT_ARTICLE_VOTE table for town meeting day voting results
3. Create warrant article management page: list, create, edit, reorder articles
4. Implement article numbering (auto-incrementing per warrant)
5. Create warrant article editor: rich text for article body, structured fields for financial data
6. Create warrant document generation via Puppeteer: formatted legal text with article numbering, cover page, town seal
7. Link warrant articles to annual/special town meeting agenda
8. Create public portal view for warrant articles: residents can read articles before town meeting
9. Implement warrant article PDF generation for physical posting (legal requirement)
10. Verify: create warrant articles, number correctly, generate PDF, display on portal

## Prompt

```
You are building the Warrant Article management system for the Town Meeting Manager — a municipal meeting management platform for Maine towns. Warrant articles are the formal items that voters act on at an annual or special town meeting. Each article is a specific question or proposal (e.g., "Shall the Town appropriate $50,000 for road repairs?") that requires a vote by the assembled voters.

In Maine's town meeting form of government, the warrant is a legal document. It must be posted publicly before the meeting, and voters can only act on articles that appear in the warrant. This system manages the creation, editing, publishing, and voting workflow for warrant articles.

TASK 1: Create WARRANT_ARTICLE table
- Add to the Supabase migration in packages/api/src/db/migrations/:
  CREATE TABLE warrant_article (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    town_id UUID NOT NULL REFERENCES town(id),
    town_meeting_id UUID REFERENCES meeting(id),  -- links to the town meeting this article belongs to
    article_number INTEGER NOT NULL,               -- sequential number within the warrant (1, 2, 3...)
    title TEXT NOT NULL,                            -- short title ("Road Repair Appropriation")
    full_text TEXT NOT NULL,                        -- full legal text of the article
    plain_language_summary TEXT,                    -- optional plain-English explanation for residents
    category TEXT,                                  -- grouping: general_government, public_safety, education, public_works, debt_service, land_use, bylaw_amendment, other
    sponsor TEXT,                                   -- who submitted: selectboard, budget_committee, citizen_petition, planning_board
    estimated_cost DECIMAL(12,2),                   -- dollar amount if applicable
    funding_source TEXT,                            -- taxation, surplus, reserve_fund, bond, grant, other
    requires_ballot_vote BOOLEAN DEFAULT FALSE,     -- some articles require written/secret ballot
    recommendation TEXT,                            -- selectboard/budget committee recommendation: recommended, not_recommended, no_recommendation
    recommendation_text TEXT,                       -- explanation of recommendation
    status TEXT NOT NULL DEFAULT 'draft',           -- draft, published, voted, passed, failed, tabled, withdrawn
    sort_order INTEGER NOT NULL,                    -- display order (may differ from article_number)
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(town_id, town_meeting_id, article_number)
  );

- Add to PowerSync sync rules so warrant articles sync to connected clients

TASK 2: Create WARRANT_ARTICLE_VOTE table
- Records the voting results from town meeting day:
  CREATE TABLE warrant_article_vote (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    warrant_article_id UUID NOT NULL REFERENCES warrant_article(id),
    town_id UUID NOT NULL REFERENCES town(id),
    vote_method TEXT NOT NULL,                      -- voice, show_of_hands, standing_count, written_ballot, secret_ballot
    votes_yes INTEGER,                              -- count of yes votes (null for voice votes)
    votes_no INTEGER,                               -- count of no votes
    votes_abstain INTEGER,                          -- count of abstentions
    passed BOOLEAN NOT NULL,                        -- whether the article passed
    vote_required TEXT DEFAULT 'simple_majority',   -- simple_majority, two_thirds, three_fifths
    moderator_declared TEXT,                        -- moderator's declaration: "The motion carries" / "The motion fails"
    amendments TEXT[],                              -- any amendments adopted before the final vote
    notes TEXT,                                     -- additional notes about the vote
    recorded_by UUID REFERENCES person(id),         -- who recorded the vote
    voted_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
  );
- Add RLS policies: admin/staff can write, public can read
- Create TypeScript types in @town-meeting/shared

TASK 4: Implement article numbering
- Articles are numbered sequentially within a warrant: Article 1, Article 2, Article 3, etc.
- When a new article is created, assign the next available number
- When articles are reordered (drag-and-drop or move up/down), renumber all articles sequentially
- When an article is deleted, renumber remaining articles to close the gap
- Article numbers are always contiguous (no gaps): 1, 2, 3, 4, 5 — never 1, 2, 4, 5
- Create a utility function renumberArticles(articles: WarrantArticle[]): WarrantArticle[] in @town-meeting/shared
- The system should warn before renumbering if the warrant is already published (renumbering a published warrant could confuse residents who have already read it)

TASK 5: Create warrant article editor
- Create packages/web/src/components/warrant/WarrantArticleEditor.tsx
- Full-page or large dialog editor for a single warrant article:

  - Title: text input (required, max 200 chars)
  - Full text: rich text editor (textarea with markdown support or a simple WYSIWYG)

    - Common article patterns can be pre-filled from templates (Task 5a below)
  - Plain language summary: textarea (optional, helps residents understand legal text)
  - Category: Select dropdown (general_government, public_safety, education, public_works, etc.)
  - Sponsor: Select dropdown (selectboard, budget_committee, citizen_petition, planning_board)
  - Financial fields (shown/hidden based on whether the article involves money):
    - Estimated cost: currency input (formatted with $ and commas)
    - Funding source: Select dropdown (taxation, surplus, reserve_fund, bond, grant, other)
  - Requires ballot vote: checkbox
  - Vote required: Select dropdown (simple_majority, two_thirds, three_fifths)
  - Recommendation: radio buttons (recommended, not_recommended, no_recommendation)
  - Recommendation text: textarea (optional explanation)
- Article text templates (pre-filled patterns):
  - Appropriation: "To see if the Town will vote to raise and appropriate the sum of $_____ for the purpose of _____."
  - Bylaw amendment: "To see if the Town will vote to amend Chapter ___, Section ___ of the Town Ordinances by [adding/deleting/modifying] the following: _____."
  - Land acquisition: "To see if the Town will vote to authorize the Select Board to acquire, on such terms and conditions as it deems appropriate, the property located at _____, Map ___, Lot ___."
  - General question: "Shall the Town vote to _____?"
- Use React Hook Form + Zod validation

TASK 6: Create warrant document generation via Puppeteer
- Create an API endpoint POST /api/meetings/:meetingId/warrant/generate-pdf
- Generate a formal warrant document PDF using Puppeteer (same approach as agenda packet from session 06.03):
  - Cover page:
    - Town name and seal (from town record)
    - "WARRANT FOR [ANNUAL/SPECIAL] TOWN MEETING"
    - Date, time, location
    - Legal notice text: "To [Constable/Resident] of the Town of [Name], in the County of [County], State of Maine:"
    - "You are hereby notified to warn the inhabitants of the Town of [Name], qualified to vote in Town affairs, to assemble at [location] on [date] at [time] to act on the following articles:"
  - Article pages:
    - Each article: "ARTICLE [N]: [Title]" in bold
    - Full text indented below the title
    - If recommendation present: "Select Board Recommendation: [Recommended/Not Recommended]" with explanation
    - If financial: "Estimated Cost: $[amount] | Funding Source: [source]"
    - Category label in italics
  - Footer: page numbers, town name
- Return the PDF as a downloadable file
- Store the generated PDF in Supabase Storage: {town_id}/warrants/{meeting_id}/warrant.pdf
- Track generation: WARRANT_DOCUMENT record with generated_at, generated_by, file_path
```
