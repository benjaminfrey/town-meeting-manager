# Session 17.02 — Audio Transcription Integration

**Phase:** 2 — Differentiation | **Block:** 17 — Audio Recording & Transcription
**Dependencies:** 17.01, 16.01
**Estimated tasks:** 9

---

## Description

Integrate an audio transcription service (Deepgram or Whisper API) to convert meeting audio segments into timestamped text transcripts. This session builds the transcription pipeline, speaker diarization support, a transcript viewer with audio sync, transcript editing tools, and the integration point between transcripts and AI minutes generation.

## Tasks

1. Integrate transcription service (Deepgram or Whisper API) in packages/api
2. Create transcription pipeline: audio segment uploaded -> sent to transcription service -> transcript stored with timestamps
3. Implement speaker diarization (if service supports it): identify different speakers in the audio
4. Create transcript viewer component: timestamped text with audio playback sync (click timestamp to jump to audio position)
5. Integrate transcripts with AI minutes generation: include transcript text as additional context in Claude API prompts for richer minutes drafts
6. Create transcript editing interface: correct transcription errors, assign speaker labels
7. Store transcripts linked to agenda items and meeting records
8. Handle transcription failures: queue for retry, manual fallback
9. Verify: audio transcribes correctly, timestamps align, transcript feeds into AI minutes, editing works

## Prompt

```
You are building the audio transcription integration for the Town Meeting Manager — a municipal meeting management platform for Maine towns. This session connects audio recordings (from session 17.01) to a transcription service, builds the transcript processing pipeline, and integrates transcripts into the AI minutes generation system (from session 16.01).

Read docs/advisory-resolutions/2.2-ai-minutes-architecture.md Section 3 (AI Transcription & Summarization) for how transcripts enrich the minutes drafting process. Transcripts are an optional but valuable input — when available, they give the AI much richer context for producing detailed minutes.

PROJECT CONTEXT:
- Monorepo: packages/api (Fastify backend), packages/web (React 19), packages/shared (types, schemas)
- Audio segments stored in Supabase Storage (from session 17.01) as WebM/Opus files
- AI minutes generation via Claude API (from session 16.01) — transcripts are appended as context
- TanStack Query v5 — useQuery for reads, useMutation for writes to Supabase
- Tailwind CSS v4 + shadcn/ui for UI components
- Primary transcription provider: Deepgram (preferred for speaker diarization and real-time features) or OpenAI Whisper API (alternative)

TASK 1: Integrate the transcription service
- Create packages/api/src/services/transcription/transcription-client.ts
- Implement a provider-agnostic interface so the service can be swapped:
  interface TranscriptionProvider {
    transcribe(audioUrl: string, options: TranscriptionOptions): Promise<TranscriptionResult>;
    getStatus(jobId: string): Promise<TranscriptionStatus>;
  }
  interface TranscriptionOptions {
    language?: string;             // default: 'en-US'
    diarization?: boolean;         // default: true
    punctuate?: boolean;           // default: true
    model?: string;                // provider-specific model selection
    maxSpeakers?: number;          // hint for diarization (default: 10)
  }
  interface TranscriptionResult {
    jobId: string;
    status: 'completed' | 'failed';
    transcript: TranscriptSegment[];
    fullText: string;
    duration: number;
    confidence: number;            // 0-1, overall confidence
    speakers: number;              // detected speaker count
  }
  interface TranscriptSegment {
    text: string;
    startTime: number;             // seconds from segment start
    endTime: number;
    confidence: number;
    speakerId?: string;            // from diarization
    words?: TranscriptWord[];      // word-level timestamps if available
  }
- Create packages/api/src/services/transcription/deepgram-provider.ts:
  - Install Deepgram SDK: pnpm --filter @town-meeting/api add @deepgram/sdk
  - Configure with API key from environment variable DEEPGRAM_API_KEY
  - Use the pre-recorded transcription API (not live streaming — we transcribe after segments are uploaded)
  - Enable features: diarize=true, punctuate=true, paragraphs=true, utterances=true, model=nova-2 (or latest)
  - Map Deepgram response format to the TranscriptionResult interface
- Create packages/api/src/services/transcription/whisper-provider.ts (alternative):
  - Use OpenAI Whisper API via the OpenAI SDK
  - Whisper does not support diarization natively — note this limitation
  - Map Whisper response to TranscriptionResult (timestamps available at segment level)
- Select the active provider via environment variable TRANSCRIPTION_PROVIDER=deepgram|whisper

TASK 2: Create the transcription pipeline
- Create packages/api/src/services/transcription/transcription-pipeline.ts
- The pipeline processes audio segments after they are uploaded to Supabase Storage:
  1. Trigger: a Fastify route POST /api/meetings/:meetingId/transcribe that starts transcription for all segments of a meeting (or a single segment via query param)
  2. For each audio segment:
     a. Generate a signed URL from Supabase Storage (or download the file and send it)
     b. Send to the transcription service with appropriate options
     c. Wait for completion (poll if async, or use webhooks if supported)
     d. Store the transcript result linked to the audio segment
  3. After all segments are transcribed, merge into a meeting-level transcript
- Create a background job approach using a simple queue:
  - TRANSCRIPTION_JOB table: id, meeting_id, audio_segment_id, status (queued/processing/completed/failed), provider, started_at, completed_at, error_message, retry_count
  - Process jobs sequentially per meeting (avoid overwhelming the API)
  - Webhook endpoint POST /api/webhooks/transcription for async callbacks (if provider supports it)
- Handle audio format conversion if needed:
  - Deepgram supports WebM/Opus directly — no conversion needed
  - Whisper requires specific formats (mp3, wav, m4a) — convert WebM using ffmpeg if Whisper is selected
  - Create a utility function convertAudioFormat(inputBlob, targetFormat) using ffmpeg (via fluent-ffmpeg or similar)

TASK 3: Implement speaker diarization
- When diarization is enabled (Deepgram's diarize=true):
  - The transcription result includes speakerId for each segment (e.g., "speaker_0", "speaker_1")
  - The system does not know who each speaker is — diarization only identifies unique speakers
- Create a speaker mapping workflow:
  - After transcription, display the diarized transcript with anonymous speaker labels
  - The operator can assign speaker identities to diarization labels:
    - Show a sample utterance for each detected speaker
    - Dropdown to select from meeting attendees (members_present + staff_present from the attendance record)
    - "Speaker 0 sounds like: [Select from attendees]"
  - Store speaker mappings: TRANSCRIPT_SPEAKER_MAP table: transcript_id, diarization_label, person_id, assigned_by, assigned_at
- Create packages/web/src/components/transcription/SpeakerAssignment.tsx for the UI
- If Whisper is used (no diarization), skip speaker assignment — transcript shows as a continuous text without speaker identification
- Speaker assignment is optional — the AI can still generate minutes from unassigned transcripts, just with less speaker-specific detail

TASK 4: Create transcript viewer component
- Create packages/web/src/components/transcription/TranscriptViewer.tsx
- Displays the transcript text synchronized with audio playback:
  - Transcript segments are rendered as paragraphs with timestamps
  - Each segment shows: [HH:MM:SS] Speaker Name (if assigned): "Transcript text..."
  - During audio playback, the current segment is highlighted (scrolls into view automatically)
  - Clicking any timestamp in the transcript jumps the audio player to that position
- Layout: the transcript viewer works alongside the AudioPlayer component from 17.01
  - Option A: split view — audio player at top, transcript below (scrollable)
  - Option B: side-by-side — audio player left, transcript right (for wider screens)
- Search functionality: text search within the transcript with highlight and navigation (find next/previous)
- Segment boundaries: visual markers showing where agenda item transitions occurred
- Speaker color coding: each identified speaker gets a consistent color for easy visual scanning
- Use shadcn/ui ScrollArea with virtualization for long transcripts (meetings can produce thousands of lines)

TASK 5: Integrate transcripts with AI minutes generation
- Modify the AI minutes generation service from session 16.01 to include transcript data:
  - When generating minutes for a section, check if a transcript exists for the corresponding agenda item's audio segment
  - If a transcript is available, include it in the Claude API prompt as additional context:
    - Add a new section to the user message: "TRANSCRIPT DATA: The following is a timestamped transcript of the discussion for this agenda item..."
    - Include speaker attributions if assigned: "[Vice Chair Kostenbader]: I think we should..."
    - If speakers are not assigned, include raw diarization labels: "[Speaker 0]: I think we should..."
  - Update the system prompt to instruct the AI on how to use transcript data:
    - "Use the transcript to provide detailed discussion summaries. Paraphrase rather than quote directly."
    - "When speaker identities are known, attribute statements appropriately per the naming rules."
    - "The transcript may contain errors — cross-reference against the structured data for factual accuracy."
- Adjust per-section prompt templates:
  - For 'summarize' behavior with transcript: produce richer summaries using the discussion content
  - For 'full_record' behavior with transcript: include paraphrased key statements from each speaker
  - For 'action_only' behavior: ignore transcript content (motions/votes come from structured data)
- The transcript is optional — if no transcript exists, the AI generates from structured data alone (same as session 16.01 behavior)

TASK 6: Create transcript editing interface
- Create packages/web/src/components/transcription/TranscriptEditor.tsx
- Allows the operator to correct transcription errors before the transcript feeds into AI minutes:
  - Inline editing: click on any transcript segment to edit the text
  - Spelling corrections, name fixes, and content corrections
  - Split/merge segments: split a long segment at a specific word, or merge two adjacent segments
  - Speaker re-assignment: change the speaker for a specific segment (dropdown from attendee list)
  - Delete segment: remove a mistranscribed or irrelevant segment
- Track edits:
  - original_text: preserved for reference
  - edited_text: the corrected version
  - edited_by: person_id
  - edited_at: timestamp
- "Mark as Reviewed" button for each segment — operator can flag segments as verified
- Bulk actions:
  - "Replace All" for common transcription errors (e.g., consistently misspelled name)
  - "Accept All" to mark all segments as reviewed without individual checking
- Save edits to the TRANSCRIPT record via useMutation — changes are persisted to Supabase
- Edited transcripts are what get sent to the AI minutes generation (not the original errors)

TASK 7: Store transcripts in the database
- Create TRANSCRIPT table: id, meeting_id, audio_segment_id FK, agenda_item_id FK, town_id, full_text, segments JSONB (array of TranscriptSegment), speaker_count, duration_seconds, confidence, provider (deepgram/whisper), model_used, status (processing/completed/edited), created_at, updated_at
- Create TRANSCRIPT_EDIT table: id, transcript_id FK, segment_index, original_text, edited_text, edit_type (text_correction/speaker_change/split/merge/delete), edited_by FK (person_id), edited_at
- Create TRANSCRIPT_SPEAKER_MAP table: id, transcript_id FK, diarization_label, person_id FK, confidence, assigned_by FK (person_id), assigned_at
- Index on meeting_id and agenda_item_id for fast lookups during AI generation
- The TRANSCRIPT.segments JSONB stores the complete segment array with timestamps, allowing the transcript viewer to render without additional queries
- When a transcript is edited, store both the original and edited versions (the edited version is used for AI generation)

TASK 8: Handle transcription failures
- Create packages/api/src/services/transcription/failure-handler.ts
- Failure scenarios and handling:
  1. Provider API error (rate limit, server error): retry with exponential backoff (3 attempts, 30s/60s/120s delays)
  2. Audio quality too poor: provider returns low confidence (<0.3). Flag for manual review: "Transcript quality is low — consider reviewing or entering notes manually"
  3. Audio file corrupt or unreadable: mark segment as 'transcription_failed', notify operator
  4. Provider timeout: retry once, then mark as failed
  5. All retries exhausted: set TRANSCRIPTION_JOB.status = 'failed' with error details
- Admin notification: when transcription fails, create a notification event (email to admin) with the error details and meeting reference
- Failed transcriptions do not block the AI minutes pipeline — the AI generates from structured data alone for sections without transcripts
- Create a manual transcript entry option: if transcription fails or is unavailable, the operator can manually type notes for a segment. These manual notes are stored as a transcript with provider: 'manual'
- Retry dashboard: admin can view failed transcription jobs and retry them manually

TASK 9: Verification

  2. Trigger transcription via POST /api/meetings/:meetingId/transcribe

  4. Transcript result is stored in the TRANSCRIPT table with timestamped segments
  5. TranscriptViewer displays the transcript with timestamps and speaker labels
  6. Click a timestamp — audio player jumps to that position
  7. Speaker assignment UI allows mapping diarization labels to attendees
  8. Edit a transcript segment — correction is saved and original preserved
  9. Generate AI minutes for a meeting WITH a transcript — verify the Claude API prompt includes transcript data
  10. Generate AI minutes for a meeting WITHOUT a transcript — verify it still works (structured data only)
  11. Simulate a transcription failure — retry logic activates, manual entry fallback available
```
