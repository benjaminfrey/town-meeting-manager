# Session 25.01 — Proximity Matching for Residents

**Phase:** 3 — Scale | **Block:** 25 — Proximity Matching & Notifications
**Dependencies:** 24.02, 20.01
**Estimated tasks:** 8

---

## Description

Build the proximity matching system that connects registered residents to parcels near their address and sends targeted notifications when agenda items affect nearby properties. When a land-use board schedules a hearing about a specific parcel, the system identifies all registered residents within a configurable radius and notifies them via their preferred channel (email, SMS, or both). This fulfills the civic engagement goal of keeping residents informed about nearby development and land-use decisions.

## Tasks

1. Create resident address geocoding: when a resident registers, geocode their address to lat/long (using E911 data match or geocoding service)
2. Link residents to parcels: match resident address to nearest parcel via PostGIS spatial query
3. Create proximity notification: when an agenda item is linked to a parcel, find all residents within configurable radius and notify them
4. Implement notification preferences: residents can opt into proximity-based notifications per board/topic
5. Create notification dispatch: email + SMS (per preference) with agenda item details, meeting date/time, parcel reference
6. Create admin configuration: set notification radius per board (e.g., Planning Board: 500ft, ZBA: 250ft)
7. Create "affected properties" display on public portal agenda items
8. Verify: resident geocodes correctly, proximity search finds nearby residents, notifications dispatch to correct recipients

TASK 1: Create resident address geocoding
- When a resident registers and provides a street address (session 20.01), geocode it to a lat/long coordinate:
- Primary method — E911 data match:
  1. Parse the resident's address into components (number, street name)
  2. Search the E911_ADDRESS table for a match:
     SELECT latitude, longitude, geometry
     FROM e911_address
     WHERE town_id = :townId
       AND address ILIKE '%' || :addressQuery || '%'
     ORDER BY similarity(address, :fullAddress) DESC
     LIMIT 1;
  3. If found: use the E911 lat/long as the resident's geocoded location
  4. If not found: try a fuzzy match on street name and number separately
- Fallback method — external geocoding:
  - If no E911 match found, use a free geocoding service (Nominatim / OpenStreetMap) as fallback
  - POST to https://nominatim.openstreetmap.org/search?q=ADDRESS,TOWN,ME&format=json&limit=1
  - Respect Nominatim usage policy: max 1 request per second, include User-Agent header
  - This is a fallback only — most Maine addresses should match E911 data
- Store the geocoded result on the RESIDENT_ACCOUNT:
  ALTER TABLE resident_account ADD COLUMN geocoded_latitude DOUBLE PRECISION;
  ALTER TABLE resident_account ADD COLUMN geocoded_longitude DOUBLE PRECISION;
  ALTER TABLE resident_account ADD COLUMN geocoded_geometry GEOMETRY(Point, 4326);
  ALTER TABLE resident_account ADD COLUMN geocode_source TEXT;  -- 'e911', 'nominatim', 'manual', null
  ALTER TABLE resident_account ADD COLUMN geocode_confidence TEXT;  -- 'exact', 'approximate', 'failed'
  CREATE INDEX idx_resident_geometry ON resident_account USING GIST(geocoded_geometry);
- If geocoding fails entirely: set geocode_confidence to 'failed', resident will not receive proximity notifications (but will still receive other notification types)
- Geocoding runs at registration time and whenever the resident updates their address

TASK 2: Link residents to parcels
- After geocoding a resident's address, link them to the nearest parcel:
  SELECT p.id, p.map_bk_lot, p.prop_loc,
         ST_Distance(p.geometry::geography, ST_SetSRID(ST_MakePoint(:lng, :lat), 4326)::geography) AS distance_meters
  FROM parcel p
  WHERE p.town_id = :townId
    AND ST_DWithin(p.geometry::geography, ST_SetSRID(ST_MakePoint(:lng, :lat), 4326)::geography, 50)  -- within 50 meters
  ORDER BY distance_meters ASC
  LIMIT 1;
- Store the link:
  ALTER TABLE resident_account ADD COLUMN home_parcel_id UUID REFERENCES parcel(id);
- This link is informational — it helps identify which parcel the resident lives on
- The link does NOT restrict proximity notifications to just the home parcel — notifications use distance from the resident's geocoded point to the agenda item's linked parcel(s)
- If no parcel is found within 50 meters, the resident has no home_parcel link (acceptable — they still receive proximity notifications based on distance)

TASK 3: Create proximity notification system
- Create packages/api/src/services/notifications/proximity-notifier.ts
- When an agenda item is linked to a parcel (session 24.02 Task 4), check for proximity notifications:
  1. Get the linked parcel's geometry (centroid or boundary)
  2. Get the notification radius for this board (from admin configuration — Task 6)
  3. Query for registered residents within the radius:
     SELECT ra.id, ra.person_id, ra.notification_preferences, ra.preferred_channel,
            p.first_name, p.last_name, p.email, p.phone,
            ST_Distance(ra.geocoded_geometry::geography, parcel.geometry::geography) AS distance_meters
     FROM resident_account ra
     JOIN person p ON ra.person_id = p.id
     WHERE ra.town_id = :townId
       AND ra.status = 'active'
       AND ra.geocoded_geometry IS NOT NULL
       AND ST_DWithin(
         ra.geocoded_geometry::geography,
         (SELECT geometry::geography FROM parcel WHERE id = :parcelId),
         :radiusMeters
       )
     ORDER BY distance_meters ASC;
  4. Filter by notification preferences: only include residents who opted into proximity/land-use notifications
  5. Create notification events for each matching resident
- Timing: proximity notifications are triggered when:
  - An agenda item is linked to a parcel AND the meeting agenda is published
  - NOT immediately on parcel linking (the operator may still be drafting the agenda)
  - The agenda publication event (from session 11.01) checks for parcel-linked items and triggers proximity notifications
- De-duplication: if a resident is within range of multiple parcels on the same agenda, send only one notification listing all affected items

TASK 5: Create notification dispatch

  3. Track delivery per recipient
- Create an email template for proximity notifications:

- Create the SMS template (concise, under 160 chars if possible):

TASK 6: Create admin proximity configuration

- Configuration per board:

    - Default for Conservation Commission: 250 feet

  - Minimum distance display: "Residents within [radius] of linked parcels will be notified"
