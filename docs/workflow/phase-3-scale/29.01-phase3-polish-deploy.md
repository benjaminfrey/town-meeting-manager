TASK 3: PostGIS Performance Optimization
Optimize spatial queries for production performance:

Spatial indexes:
- Add GIST index on parcel.geometry: CREATE INDEX idx_parcel_geometry ON parcel USING GIST (geometry);
- Add GIST index on e911_address.geometry: CREATE INDEX idx_e911_geometry ON e911_address USING GIST (geometry);
- Add compound index on parcel(town_id, map_bk_lot) for text-based lookups
- Add compound index on e911_address(town_id, state_id) for linking queries

Query optimization:
- Abutters query: use ST_DWithin with geography cast for accurate distance in feet:
  SELECT p.* FROM parcel p
  WHERE ST_DWithin(p.geometry::geography, $1::geography, $2)
  AND p.town_id = $3 AND p.id != $4;
  (where $2 is radius in meters, convert from feet)
- Use ST_Simplify for map display at low zoom levels to reduce polygon complexity
- Use ST_Envelope (bounding box) for initial filtering before precise ST_DWithin

Benchmarking:
- Load 1000+ parcels for a test town (realistic for a Maine town)
- Benchmark abutters query: should complete in <100ms for 500ft radius
- Benchmark map tile rendering: should load visible parcels in <200ms
- Benchmark full-town parcel load: should complete in <500ms
- If any query exceeds targets, analyze with EXPLAIN ANALYZE and optimize

Connection pooling:
- Ensure PostGIS queries use the existing Supabase connection pool (via PostgREST or direct connection)
- Avoid creating new database connections per spatial query
