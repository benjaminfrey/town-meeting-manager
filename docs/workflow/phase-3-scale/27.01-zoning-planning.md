# Session 27.01 — Advanced Zoning & Planning Features

**Phase:** 3 — Scale | **Block:** 27 — Advanced Features
**Dependencies:** 24.02
**Estimated tasks:** 8

---

## Description

Build advanced features for zoning and planning boards, including zoning district map layers, application tracking workflows, conditions of approval management, and zoning ordinance references. These features transform the system from a general-purpose meeting management tool into a specialized platform for land-use boards, supporting the full lifecycle of planning and zoning applications from submission through hearing to compliance tracking.

## Tasks

1. Create zoning district layer: import zoning boundaries, display on map with parcel data
2. Create application tracking for planning/zoning boards: site plan review, subdivision, variance, conditional use permit
3. Create application workflow: submitted -> under review -> hearing scheduled -> approved/denied
4. Link applications to parcels and agenda items
5. Create public application portal: residents can view pending applications with maps
6. Create conditions of approval tracking: record conditions, track compliance
7. Create zoning ordinance reference: link board decisions to specific ordinance sections
8. Verify: zoning districts display, applications track through workflow, conditions recorded

TASK 1: Create zoning district layer
- Create ZONING_DISTRICT table:
  CREATE TABLE zoning_district (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    town_id UUID NOT NULL REFERENCES town(id),
    code TEXT NOT NULL,                                -- e.g., 'R1', 'R2', 'C1', 'I1', 'RP'
    name TEXT NOT NULL,                                -- e.g., 'Residential 1', 'Commercial', 'Resource Protection'
    description TEXT,                                  -- district description and purpose
    geometry GEOMETRY(MultiPolygon, 4326),             -- district boundary (MultiPolygon for non-contiguous districts)
    fill_color TEXT DEFAULT '#3b82f6',                 -- map display color
    fill_opacity DECIMAL(3,2) DEFAULT 0.15,
    stroke_color TEXT DEFAULT '#1d4ed8',
    stroke_width DECIMAL(3,1) DEFAULT 1.5,
    min_lot_size TEXT,                                 -- e.g., '40,000 sq ft'
    max_building_height TEXT,                          -- e.g., '35 feet'
    setback_requirements JSONB,                        -- { front: '30ft', side: '15ft', rear: '25ft' }
    permitted_uses TEXT[],                             -- array of permitted uses
    conditional_uses TEXT[],                            -- array of conditional/special uses
    prohibited_uses TEXT[],                             -- explicitly prohibited uses
    source TEXT DEFAULT 'manual',                      -- manual, shapefile_import
    ordinance_reference TEXT,                           -- reference to the ordinance section defining this district
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(town_id, code)
  );
  CREATE INDEX idx_zoning_district_geometry ON zoning_district USING GIST(geometry);
- Import options:
  - Manual: admin draws district boundaries on the map (using Leaflet Draw or similar)
  - Shapefile import: upload a GIS shapefile (.shp + .dbf + .shx) of the town's zoning map
    - Create an upload handler that reads shapefiles (use shapefile or shpjs library)
    - Convert shapefile geometries to PostGIS format
  - Some towns may not have digital zoning maps — the system works without them (parcels display without zoning overlay)
- Map display:
  - Add a "Zoning Districts" layer to the ParcelMap component from session 24.02
  - Each district renders as a semi-transparent colored polygon
  - Legend showing district codes with colors and names
  - Toggle the zoning layer on/off independently of the parcel layer
  - Clicking a zoning district area shows the district details in a popup:
    - District code and name
    - Permitted, conditional, and prohibited uses
    - Dimensional requirements (lot size, setbacks, height)
    - Link to the relevant ordinance section
  - Parcel-level display: when a parcel is selected, show which zoning district(s) it falls within:
    - Query: SELECT zd.* FROM zoning_district zd WHERE ST_Intersects(zd.geometry, (SELECT geometry FROM parcel WHERE id = :parcelId))

TASK 2: Create application tracking system
- Create LAND_USE_APPLICATION table:
  CREATE TABLE land_use_application (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    town_id UUID NOT NULL REFERENCES town(id),
    board_id UUID NOT NULL REFERENCES board(id),       -- which board reviews this application
    application_number TEXT NOT NULL,                   -- unique number: "SP-2026-001", "SUB-2026-003"
    application_type TEXT NOT NULL,                     -- site_plan, subdivision, variance, conditional_use, shoreland_zoning, building_permit, other
    applicant_name TEXT NOT NULL,
    applicant_address TEXT,
    applicant_phone TEXT,
    applicant_email TEXT,
    representative_name TEXT,                           -- agent/attorney if different from applicant
    representative_company TEXT,
    project_name TEXT,                                  -- e.g., "Mill Creek Subdivision"
    project_description TEXT NOT NULL,                  -- detailed project description
    parcel_id UUID REFERENCES parcel(id),              -- primary parcel for the application
    additional_parcel_ids UUID[],                       -- additional parcels if multi-parcel project
    zoning_district_id UUID REFERENCES zoning_district(id),
    status TEXT NOT NULL DEFAULT 'submitted',           -- submitted, under_review, hearing_scheduled, hearing_held, approved, approved_with_conditions, denied, withdrawn, tabled
    submitted_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    fee_amount DECIMAL(8,2),
    fee_paid BOOLEAN DEFAULT FALSE,
    fee_paid_at TIMESTAMPTZ,
    decision TEXT,                                      -- board's written decision
    decision_date DATE,
    decision_by UUID REFERENCES person(id),            -- who recorded the decision
    appeal_deadline DATE,                               -- calculated from decision date
    notes TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(town_id, application_number)
  );
  CREATE INDEX idx_application_town ON land_use_application(town_id);
  CREATE INDEX idx_application_board ON land_use_application(board_id);
  CREATE INDEX idx_application_status ON land_use_application(town_id, status);
  CREATE INDEX idx_application_parcel ON land_use_application(parcel_id);
- Application number generation:
  - Format: [TYPE_PREFIX]-[YEAR]-[SEQUENCE]
  - Type prefixes: SP (site plan), SUB (subdivision), VAR (variance), CU (conditional use), SZ (shoreland zoning)
  - Auto-increment per type per year: SP-2026-001, SP-2026-002, etc.
- Create a corresponding APPLICATION_DOCUMENT table for uploaded materials:
  - Plans, surveys, environmental assessments, application forms, etc.
  - Same Supabase Storage pattern as exhibits (session 06.02)

TASK 3: Create application workflow
- Create packages/web/src/routes/boards.$boardId.applications.tsx
- Application list page:
  - DataTable with columns: App #, Type, Applicant, Project, Parcel, Status, Submitted, Actions
  - Filter by status (tabs: All, Active, Approved, Denied, Withdrawn)
  - Sort by submitted date (most recent first)
  - Search by applicant name, application number, or address
  - "New Application" button opens the application form
- Application detail page:
  - Route: /boards/:boardId/applications/:applicationId
  - Header: application number, status badge, type badge
  - Tabs: Details, Documents, Hearing History, Conditions, Timeline
  - Status workflow buttons (context-sensitive):
    - From 'submitted': "Begin Review" -> under_review
    - From 'under_review': "Schedule Hearing" -> hearing_scheduled (links to meeting/agenda)
    - From 'hearing_scheduled': "Record Decision" -> approved/denied/tabled
    - From any active status: "Withdraw" -> withdrawn
  - Each status change is logged in a status history table:
    CREATE TABLE application_status_history (
      id UUID PRIMARY KEY,
      application_id UUID NOT NULL REFERENCES land_use_application(id),
      from_status TEXT,
      to_status TEXT NOT NULL,
      changed_by UUID NOT NULL REFERENCES person(id),
      notes TEXT,
      changed_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
    );

TASK 4: Link applications to parcels and agenda items
- When creating or editing an application:
  - "Link Parcel" button opens the ParcelSearch from session 24.02
  - Selected parcel auto-fills: property address, owner name, zoning district, lot size
  - Multiple parcels can be linked (for multi-parcel projects)
- When scheduling a hearing:
  - "Create Agenda Item" button creates an agenda item in the selected meeting:
    - Title: "[Type]: [Project Name] — [Parcel Address] (Map [X], Lot [Y])"
    - Description: project description from the application
    - Linked parcels: automatically attached to the agenda item
    - Section: appropriate section in the board's agenda template (e.g., "Public Hearings" or "Applications")
  - The agenda item links back to the application record for reference
- Create AGENDA_ITEM_APPLICATION junction:
  CREATE TABLE agenda_item_application (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    agenda_item_id UUID NOT NULL REFERENCES agenda_item(id) ON DELETE CASCADE,
    application_id UUID NOT NULL REFERENCES land_use_application(id),
    UNIQUE(agenda_item_id, application_id)
  );
- During the live meeting, the operator can open the application details from the agenda item for reference

TASK 6: Create conditions of approval tracking
- When a board approves an application with conditions:
  - Create CONDITION_OF_APPROVAL table:
    CREATE TABLE condition_of_approval (
      id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
      application_id UUID NOT NULL REFERENCES land_use_application(id),
      town_id UUID NOT NULL REFERENCES town(id),
      condition_number INTEGER NOT NULL,               -- sequential within the application
      condition_text TEXT NOT NULL,                     -- the specific condition
      category TEXT,                                   -- general, environmental, traffic, design, financial, timing
      due_date DATE,                                   -- when compliance is required
      compliance_status TEXT DEFAULT 'pending',         -- pending, in_progress, satisfied, waived, violated
      compliance_notes TEXT,
      compliance_verified_by UUID REFERENCES person(id),
      compliance_verified_at TIMESTAMPTZ,
      evidence_document_path TEXT,                      -- Supabase Storage path for compliance documentation
      created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
      updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
    );
- Conditions UI in the application detail page:
  - "Conditions" tab shows all conditions with status indicators
  - Each condition has: number, text, due date, status badge
  - Admin can update compliance status: "Mark as Satisfied" with optional notes and document upload

  - Conditions can be edited until the decision is finalized
- Compliance dashboard:
  - List of applications with outstanding conditions
  - Overdue conditions highlighted in red
  - Upcoming due dates highlighted in amber
  - Export compliance report per application or across all applications
- The conditions become part of the official record and are included in decision letters

TASK 7: Create zoning ordinance reference
- Create ZONING_ORDINANCE_SECTION table:
  CREATE TABLE zoning_ordinance_section (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    town_id UUID NOT NULL REFERENCES town(id),
    section_number TEXT NOT NULL,                      -- e.g., '4.3.2', 'Article VII, Section 3'
    title TEXT NOT NULL,                               -- section title
    summary TEXT,                                      -- brief description of what this section covers
    full_text TEXT,                                    -- optional full text of the section
    chapter TEXT,                                      -- organizational chapter: 'Dimensional Standards', 'Use Regulations'
    related_district_codes TEXT[],                     -- which zoning districts this section applies to
    ordinance_document_url TEXT,                       -- link to the full ordinance PDF (if available)
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(town_id, section_number)
  );
- Link board decisions to ordinance sections:
  - Create APPLICATION_ORDINANCE_REFERENCE junction:
    CREATE TABLE application_ordinance_reference (
      id UUID PRIMARY KEY,
      application_id UUID REFERENCES land_use_application(id),
      ordinance_section_id UUID REFERENCES zoning_ordinance_section(id),
      reference_type TEXT,                            -- basis, exception, waiver, interpretation
      notes TEXT
    );
  - When recording a decision, the admin can cite the relevant ordinance sections
  - The decision letter template includes ordinance references
- Ordinance management page:
  - Route: /admin/ordinances
  - List of ordinance sections organized by chapter
  - Add/edit/delete sections
  - Search by section number or keyword
  - Link to the full ordinance document (PDF upload or external URL)
- Ordinance lookup during meetings:
  - Quick search widget in the LiveMeetingPage sidebar (similar to the parcel lookup from 24.02)
  - Helps the board reference specific ordinance sections during discussion
