# Session 26.01 — Multi-Town Management

**Phase:** 3 — Scale | **Block:** 26 — Multi-Town Consortiums
**Dependencies:** 15.02
**Estimated tasks:** 8

---

## Description

Build the multi-town consortium system that allows a single organization (such as a regional administrative office or shared town clerk) to manage multiple towns within the Town Meeting Manager platform. This session creates the consortium data model, management UI, cross-town dashboard, town switcher, shared staff accounts, a cross-town calendar, and consortium-level reporting.

## Tasks

1. Create CONSORTIUM table: id, name, created_at and CONSORTIUM_MEMBER table: consortium_id, town_id, role (manager/member)
2. Create consortium management UI: create consortium, add/remove towns
3. Implement cross-town dashboard: consortium manager sees aggregated view across all managed towns
4. Create town switcher: users with access to multiple towns can switch between them
5. Implement shared staff accounts: a person can have user_accounts in multiple towns (e.g., regional clerk serving 3 towns)
6. Create cross-town calendar: view meetings from all consortium towns in one calendar
7. Implement consortium-level reporting: aggregate statistics across towns
8. Verify: consortium created, towns added, cross-town dashboard shows data, town switcher works

## Prompt

```
You are building the multi-town consortium management system for the Town Meeting Manager — a municipal meeting management platform for Maine towns. In rural Maine, it is common for one person (e.g., a regional town clerk or administrative assistant) to serve multiple small towns simultaneously. Consortiums allow one operator to manage meetings, agendas, and minutes for several towns from a single login.

PROJECT CONTEXT:
- Monorepo: packages/web (React 19 + React Router v7 Framework Mode), packages/api (Fastify), packages/shared (types, schemas)
- Supabase for auth and PostgreSQL database
- PowerSync for offline-first data sync
- RLS policies currently scope all data to a single town_id per user (from advisory 1.2)
- The current auth model: each USER_ACCOUNT belongs to one TOWN via the PERSON record
- This session extends the model so a PERSON can have USER_ACCOUNTs (or access) in multiple towns
- Tailwind CSS v4 + shadcn/ui for components
- Production deployment from session 15.02

TASK 1: Create consortium data model
- Create database migration for consortium tables:
  CREATE TABLE consortium (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    name TEXT NOT NULL,                               -- e.g., "Lincoln County Regional Consortium"
    description TEXT,
    primary_contact_id UUID REFERENCES person(id),    -- consortium administrator
    status TEXT NOT NULL DEFAULT 'active',             -- active, inactive
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
  );

  CREATE TABLE consortium_member (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    consortium_id UUID NOT NULL REFERENCES consortium(id) ON DELETE CASCADE,
    town_id UUID NOT NULL REFERENCES town(id),
    role TEXT NOT NULL DEFAULT 'member',               -- manager (full control), member (view only)
    joined_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    invited_by UUID REFERENCES person(id),
    status TEXT NOT NULL DEFAULT 'active',              -- active, pending, removed
    UNIQUE(consortium_id, town_id)
  );
- Create a cross-town user access table:
  CREATE TABLE cross_town_access (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    person_id UUID NOT NULL REFERENCES person(id),
    town_id UUID NOT NULL REFERENCES town(id),         -- the town they have access to
    user_account_id UUID REFERENCES user_account(id),  -- their account in that town
    access_level TEXT NOT NULL,                         -- admin, staff, board_member (may differ per town)
    granted_by UUID REFERENCES person(id),
    granted_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(person_id, town_id)
  );
- RLS policy updates:
  - Current: user sees data WHERE town_id = auth.jwt()->>'town_id'
  - Updated: user sees data WHERE town_id IN (SELECT town_id FROM cross_town_access WHERE person_id = auth.jwt()->>'person_id')
  - This expands access to all towns the user has been granted access to
  - PowerSync sync rules must also be updated to sync data for all accessible towns

TASK 2: Create consortium management UI
- Route: /admin/consortium
- File: packages/web/src/routes/admin.consortium.tsx
- Only sys_admin users can create and manage consortiums
- Consortium creation:
  - "Create Consortium" button opens a dialog
  - Fields: name (required), description (optional)
  - On creation, the creating user's town is automatically added as the first member with 'manager' role
- Consortium management page:
  - Consortium details card: name, description, member count, status
  - Member towns list: DataTable showing town name, population, role (manager/member), status, joined date
  - "Add Town" button: search for existing towns in the system
    - Search by town name
    - Select a town and send an invitation
    - The target town's admin must accept the invitation (for security — can't unilaterally add a town)
  - "Remove Town" button: removes a town from the consortium (with confirmation)
  - Role management: change a town's role between 'manager' and 'member'
- Invitation flow:
  1. Consortium manager clicks "Add Town" and selects the target town
  2. An invitation notification is sent to the target town's admin(s)
  3. Target town admin sees the invitation in their dashboard: "[Consortium Name] has invited your town to join. Accept / Decline"
  4. On accept: consortium_member record created with status 'active'
  5. On decline: invitation removed

TASK 4: Create town switcher
- Create packages/web/src/components/layout/TownSwitcher.tsx
- The town switcher appears in the app header/sidebar for users with access to multiple towns:
  - Dropdown showing all accessible towns: "[Town Name] — [Role]"
  - Current town highlighted with a check mark
  - Click a different town to switch context
- Switching towns:
  - Updates the "active town" context throughout the app
  - All routes and data queries filter to the newly selected town
  - The URL does NOT change to include a town parameter — the active town is stored in application state (context provider)
  - PowerSync data for all accessible towns is already synced locally — switching is instant (no network request)
- Create a TownContext provider:
  - Wraps the entire app
  - Provides: activeTownId, activeTown (full record), setActiveTown(townId)
  - Default: the user's "home" town (the original town from their PERSON record)
  - Persisted in localStorage so it survives page refresh
- Update existing components to read from TownContext instead of hardcoding the user's single town_id
- The town switcher is hidden for users who only have access to one town (the common case)

TASK 5: Implement shared staff accounts
- A person who serves multiple towns (e.g., a regional clerk) has:
  - One PERSON record (their identity)
  - One Supabase auth account (one login)
  - Multiple USER_ACCOUNT records (one per town) with potentially different roles
  - CROSS_TOWN_ACCESS records linking them to each town
- When a consortium manager adds a shared staff member:
  1. Search for the person by email (they may already exist in the system)
  2. If found: create a CROSS_TOWN_ACCESS record for the new town with the specified role
  3. If not found: invite them via email (same flow as session 11.02 account invitation)
  4. Create a USER_ACCOUNT record in the new town with the specified role
- The JWT token needs to be updated to support multi-town access:
  - Current JWT claims: { town_id: 'xxx', role: 'admin', person_id: 'yyy' }
  - Updated approach: the JWT still has a single town_id (the "home" town), but the RLS policies check cross_town_access for additional permissions
  - Alternative: include an array of accessible town_ids in the JWT claims (requires updating the auth hook)
- Create a shared staff management UI within the consortium management page:
  - List of staff members who have cross-town access
  - Their roles per town: "Jane Smith — Newcastle: Admin, Damariscotta: Staff, Bristol: Staff"
  - Add/remove staff access per town
```
