## Database Migration

Create `packages/web/supabase/migrations/XXX_minutes_workflow_config.sql`:

```sql
-- Town-level minutes workflow settings
ALTER TABLE town
  ADD COLUMN audio_retention_policy TEXT NOT NULL DEFAULT 'retain_30_days'
    CHECK (audio_retention_policy IN ('purge_on_approval','retain_30_days','retain_90_days','retain_indefinitely')),
  ADD COLUMN auto_publish_on_approval BOOLEAN NOT NULL DEFAULT false,
  ADD COLUMN minutes_review_window_days INTEGER NOT NULL DEFAULT 7,
  ADD COLUMN minutes_workflow_configured_at TIMESTAMPTZ;

-- Board-level overrides
ALTER TABLE board
  ADD COLUMN minutes_consent_agenda BOOLEAN NOT NULL DEFAULT false,
  ADD COLUMN minutes_requires_second BOOLEAN NOT NULL DEFAULT true,
  ADD COLUMN r4_board_member_default BOOLEAN NOT NULL DEFAULT true,
  ADD COLUMN audio_retention_policy_override TEXT
    CHECK (audio_retention_policy_override IN ('purge_on_approval','retain_30_days','retain_90_days','retain_indefinitely')),
  ADD COLUMN auto_publish_on_approval_override BOOLEAN;

-- Minutes document: amendment history
ALTER TABLE minutes_document
  ADD COLUMN amendments_history JSONB NOT NULL DEFAULT '[]';

-- Minutes addendum table
CREATE TABLE minutes_addendum (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  minutes_document_id UUID NOT NULL REFERENCES minutes_document(id) ON DELETE CASCADE,
  town_id UUID NOT NULL REFERENCES town(id),
  adopting_meeting_id UUID NOT NULL REFERENCES meeting(id),
  adopting_motion_id UUID REFERENCES motion(id),
  content_json JSONB NOT NULL DEFAULT '{}',
  html_rendered TEXT,
  description TEXT NOT NULL,
  created_by UUID REFERENCES user_account(id),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  published_at TIMESTAMPTZ
);

CREATE INDEX idx_minutes_addendum_document ON minutes_addendum(minutes_document_id);
CREATE INDEX idx_minutes_addendum_meeting ON minutes_addendum(adopting_meeting_id);
```

### Task 3 — Vote outcome automation

Update the motion result handler in the live meeting flow (`meetings.$meetingId.live.tsx` or the relevant mutation):

When a motion result is recorded (passed/failed), check if the motion is linked to a `minutes_document` approval item:

**Motion passes:**
```typescript
if (motion.type === 'approve_minutes' && motion.result === 'passed') {
  await supabase.from('minutes_document').update({
    status: 'approved',
    approved_at: new Date().toISOString(),
    approved_by_motion_id: motion.id,
  }).eq('id', motion.minutes_document_id);

  // Auto-publish if configured
  const board = await getBoard(meeting.board_id);
  const autoPublish = board.auto_publish_on_approval_override
    ?? town.auto_publish_on_approval;
  if (autoPublish) {
    await publishMinutesToPortal(motion.minutes_document_id);
  }
}
```

**Motion fails:**
```typescript
if (motion.type === 'approve_minutes' && motion.result === 'failed') {
  // Return to draft and record in history
  const currentDoc = await getMinutesDocument(motion.minutes_document_id);
  const historyEntry: AmendmentHistoryEntry = {
    event_type: 'adoption_failed',
    occurred_at: new Date().toISOString(),
    meeting_id: meeting.id,
    motion_id: motion.id,
    recorded_by: currentUser.id,
  };
  await supabase.from('minutes_document').update({
    status: 'draft',
    amendments_history: [...(currentDoc.amendments_history ?? []), historyEntry],
  }).eq('id', motion.minutes_document_id);
}
```

**Important:** Minutes content (`content_json`, `html_rendered`) is read-only in the UI when `status = 'approved'`. This is enforced at the UI layer — the edit controls are not rendered. RLS policies enforce this at the DB layer too.

### Task 5 — Addendum creation flow

On the approved minutes view, "Add Addendum" button:
1. Opens a dialog with:
   - Description field (required): brief description of what is being corrected
   - Rich text editor for the addendum content (same Tiptap setup as minutes editor)
   - "Save Draft Addendum" button
2. The draft addendum is saved to `minutes_addendum` with no `adopting_meeting_id` yet (null)
3. A separate "Attach to Agenda" action allows the admin to select an upcoming meeting and add "Motion to Amend Minutes of [date]" to its agenda, linking the draft addendum

Once attached to an agenda:
- The addendum becomes editable (R1) until the adopting meeting's motion result is recorded
- When the addendum motion **passes**: `minutes_addendum.adopting_motion_id` is set, addendum is locked, merged PDF generated and optionally published
- When the addendum motion **fails**: `minutes_addendum` record is deleted

### Task 6 — Merged PDF generation

New API endpoint: `POST /api/minutes/:minutesDocumentId/publish-with-addendum`

1. Fetch the `minutes_document` and all associated `minutes_addendum` records with `adopting_meeting_id IS NOT NULL` (adopted addenda)
2. Render each addendum as a separate PDF page: "AMENDMENT TO MINUTES OF [original meeting date]", adopted at [adopting meeting date], followed by the addendum content
3. Merge the original minutes PDF + all addendum PDFs using Puppeteer or a PDF merge library
4. Upload merged PDF to Supabase Storage at `minutes/{minutesDocumentId}/published-{timestamp}.pdf`
5. Update `minutes_addendum.published_at` for all included addenda
6. Update `minutes_document.published_at` (or a new `portal_pdf_path` column)
