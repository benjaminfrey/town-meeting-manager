# Session 09.01 — Minutes Generation Pipeline (JSON to HTML to PDF)

**Phase:** 1 — MVP | **Block:** 09 — Minutes
**Dependencies:** 07.03, 06.03
**Estimated tasks:** 12

---

## Description

Build the minutes generation pipeline that transforms structured meeting data into a complete minutes document. The pipeline follows advisory 4.3's decision: structured JSON (canonical source) to HTML (portal-ready, accessible) to PDF (download/print via Puppeteer). This session creates the generation service, JSON assembly, HTML templating, PDF rendering, and the UI trigger on the post-meeting review page.

## Tasks

1. Create minutes generation service in packages/api with a `/meetings/:meetingId/minutes/generate` endpoint
2. Build structured JSON minutes assembler from meeting data (attendance, agenda items traversed, motions, votes, timestamps, recusals, adjournment)
3. Implement minutes sections: call to order, attendance/quorum, each agenda item with discussion summary placeholder and motions/votes, executive session notation, adjournment
4. Create minutes style formatters: action (decisions only), summary (decisions + brief discussion), narrative (detailed account)
5. Include all fixed sections showing "None" if nothing recorded (Q12 decision)
6. Create section-type-dependent commenter identification (Q16: identify by name in public comment, by role in board discussion)
7. Create HTML template for minutes rendering (Handlebars, semantic HTML, WCAG 2.1 AA)
8. Implement Puppeteer PDF generation for minutes (reuse infrastructure from 06.03)
9. Create configurable certification block per board (Q17: board-level configurable)
10. Store minutes as MINUTES_DOCUMENT with content_json, html_rendered, pdf_storage_path
11. Create "Generate Minutes Draft" button on post-meeting review page
12. Verify: generate minutes JSON from completed meeting, render HTML, generate PDF, verify content accuracy

## Prompt

```
You are building the minutes generation pipeline for the Town Meeting Manager — a municipal meeting management platform for Maine towns. This pipeline transforms the structured meeting record (captured during live meetings in Block 07) into formatted minutes documents following the advisory 4.3 decision: structured JSON → HTML → PDF. Read docs/advisory-resolutions/2.2-ai-minutes-architecture.md for the full minutes architecture, sections 8 (Minutes Output), 10 (Q12, Q13, Q16, Q17 decisions), and 11 (cross-board template analysis). Also read docs/advisory-resolutions/3.3-document-generation.md for the Puppeteer PDF strategy.

PROJECT CONTEXT:
- Monorepo: packages/api (Fastify), packages/web (React 19 + React Router v7), packages/shared (types, schemas)
- Supabase PostgreSQL for persistence, PowerSync for offline-first local SQLite on the client
- Puppeteer for complex PDF generation (reuse the infrastructure created in session 06.03 for agenda packets)
- Handlebars for HTML template rendering
- Minutes format per advisory 4.3: structured JSON is the canonical source, HTML for portal display, PDF generated on demand
- Three minutes styles: action, summary, narrative (board-level configurable with town default fallback)
- Two motion display formats per advisory 2.2 Q13: block_format (Planning Board style) and inline_narrative (Select Board style)
- Certification format per Q17: prepared_by, all_member_certification, or both (board-level configurable)

TASK 1: Create the minutes generation service
- File: packages/api/src/services/minutes-generation.ts
- Create a MinutesGenerationService class with the following methods:
  - generateMinutesDraft(meetingId: string): Promise<MinutesDocument>
  - assembleMinutesJson(meetingId: string): Promise<MinutesContentJson>
  - renderMinutesHtml(contentJson: MinutesContentJson, options: MinutesRenderOptions): string
  - generateMinutesPdf(html: string, meetingId: string): Promise<string> (returns storage path)
- Create a Fastify route: POST /api/meetings/:meetingId/minutes/generate
  - Requires authentication and R2 permission (Generate AI minutes draft)
  - Validates that the meeting status is "completed" before generating
  - Calls generateMinutesDraft and returns the created MinutesDocument record
  - Returns 409 Conflict if a draft already exists (offer regenerate as separate action)

TASK 2: Build the structured JSON minutes assembler
- Create assembleMinutesJson() that queries the completed meeting data and constructs a MinutesContentJson object:
  - meeting_header: board name, town name, meeting date, time called to order, time adjourned, location, meeting type (regular/special/public hearing)
  - attendance: members_present (name, title, arrived_at), members_absent (name, title, reason), presiding_officer, presiding_officer_succession (if applicable), staff_present (name, title), quorum status (met/not met, count/required)
  - sections: ordered array matching the meeting's agenda sections, each containing:
    - section_title, section_type, minutes_behavior
    - items: array of agenda items with title, section_ref, discussion_summary (placeholder text for MVP — AI fills this in Phase 2), motions (text, moved_by, seconded_by, vote result), recusals, operator_notes
  - adjournment: method (motion or without_objection), adjourned_by, timestamp, motion details if applicable
  - certification: format type (prepared_by, all_member_certification, both), recording_secretary name/title, board members list (for all_member_certification)
- Query all relevant tables: MEETING, MEETING_ATTENDANCE, AGENDA_SECTION, AGENDA_ITEM, MOTION, VOTE, VOTE_RECORD, RECUSAL
- Use the board's effective settings (getEffectiveBoardSettings from packages/shared) to resolve minutes_style, motion_display_format, and certification_format

TASK 3: Implement minutes sections with proper content for each section type
- For each section in the meeting's agenda, generate the appropriate minutes content based on the section's minutes_behavior:
  - skip: omit entirely from the minutes output (e.g., Pledge of Allegiance)
  - timestamp_only: output only the timestamp (e.g., "Meeting called to order at 7:03 PM")
  - action_only: output only motions, votes, and decisions — no discussion summary
  - summarize: output a discussion summary placeholder + motions/votes (in MVP, the summary is a placeholder that the admin fills in during review; AI summarization comes in Phase 2)
  - full_record: output the most detailed content available
- For items with motions, format the motion according to the board's motion_display_format:
  - inline_narrative: "Paz moved to approve the minutes as presented. Percy seconded. Motion passed 4-0, 1 absent."
  - block_format: structured block with Motion / Moved by / Second / Discussion / Vote / Result fields
- Handle recusals: "[Member] recused [himself/herself] from discussion and voting on this item due to [reason]."
- Handle items without motions: for action section items with no motion, include "No motion was made." For discussion/report sections, simply summarize without mentioning missing motions.

TASK 4: Create minutes style formatters
- File: packages/api/src/services/minutes-formatters.ts
- Create three formatter functions that control the level of detail in the output:
  - formatActionMinutes(contentJson): Records only decisions, motions, and votes. Minimal prose. Example: "The Select Board approved the minutes of February 18, 2026. Selectman Jones moved; Selectman Brown seconded. Motion passed 3-0."
  - formatSummaryMinutes(contentJson): Decisions plus brief discussion summaries. Most common style. Example includes a 1-2 sentence discussion summary before the motion.
  - formatNarrativeMinutes(contentJson): Detailed account including who said what (when transcription data is available). Longest output.
- Each formatter produces a MinutesFormattedContent object with sections of formatted text ready for HTML rendering
- The formatter applies the board's member_reference_style setting (per Q5: last_name_only, title_and_last_name, full_name_first_then_last) when referencing board members in the text
- In MVP (no AI transcription yet), summary and narrative styles will include placeholder text like "[Discussion summary to be added during review]" that the admin fills in manually

TASK 5: Include all fixed sections showing "None" if nothing recorded
- Per advisory 2.2 Q12 decision: fixed sections always appear in the minutes even if nothing happened
- When a fixed section has no items and no recorded activity, output: "[Section Title]: None."
- Non-fixed sections with no items are omitted entirely
- The operator can mark a section as "None / Not Applicable" during the live meeting to explicitly confirm nothing happened — check for this flag
- Example: "Executive Session: None." / "Future Agenda Items: None."

TASK 6: Create section-type-dependent commenter identification
- Per advisory 2.2 Q16 decision: identification depends on the section type
  - public_hearing sections: comments attributed by full name (legally required — "Jane Smith, 123 Main Street, expressed concern about...")
  - public_input sections: commenter names used if recorded by the operator, otherwise "a member of the public"
  - Board discussion: members identified by the board's member_reference_style setting
- The commenter identification logic is in the formatter, not the JSON assembler — the JSON always stores the full name if available, and the formatter decides how to reference it

TASK 7: Create HTML template for minutes rendering
- File: packages/api/src/templates/minutes.hbs (Handlebars template)
- Install Handlebars: pnpm --filter @town-meeting/api add handlebars
- Create a semantic HTML template that produces WCAG 2.1 AA compliant output:
  - Use proper heading hierarchy: <h1> for town/board name, <h2> for "Meeting Minutes", <h3> for section headings
  - Use <article> for the overall minutes document
  - Use <section> elements for each agenda section
  - Use <table> with <thead>, <th>, <tbody> for attendance and vote tables
  - Use <blockquote> for motion text in block format
  - Use <time datetime="..."> for all timestamps
  - Use <footer> for the certification block
- Include a "DRAFT" watermark CSS class that is applied when the minutes status is not "approved"
- Include print-friendly CSS: page breaks between major sections, consistent fonts, proper margins
- The template receives the formatted content (from Task 4) plus metadata (town seal URL, meeting date, board name)
- Create Handlebars helpers:
  - formatDate(date, format): format dates consistently
  - formatTime(time): format times as "7:03 PM"
  - formatVoteResult(vote): "Passed 4-0, 1 absent" or "Failed 2-3"
  - memberRef(person, style): apply the member reference style

TASK 8: Implement Puppeteer PDF generation for minutes
- Reuse the Puppeteer infrastructure created in session 06.03 for agenda packets
- File: packages/api/src/services/minutes-pdf.ts
- Create generateMinutesPdf(html: string, meetingId: string, options: PdfOptions):
  - Use the existing Puppeteer browser pool / singleton from 06.03
  - Set page format: Letter (8.5" x 11"), with 1" margins
  - Add header: town name + board name (left), "Meeting Minutes" (center), date (right)
  - Add footer: "Page X of Y" (center), "DRAFT" label if not approved (right)
  - Enable CSS page break rules from the HTML template
  - Generate tagged PDF for accessibility (use Puppeteer's tagged PDF option if available)
- Upload the generated PDF to Supabase Storage bucket "minutes-pdfs" under path: {town_id}/{board_id}/{meeting_date}-minutes.pdf
- Return the storage path for the MINUTES_DOCUMENT record

TASK 9: Create configurable certification block per board
- Per advisory 2.2 Q17 decision: each board configures its certification format
- Three formats:
  - prepared_by: "Respectfully submitted, [Recording Secretary name], [title]" — Select Board style
  - all_member_certification: signature block for all board members — "We, the undersigned, certify these meeting minutes are true and accurate." with name/signature/date lines for each member — Planning Board style
  - both: prepared-by attribution plus member certification block
- The certification format is stored on the BOARD record as certification_format
- The HTML template renders the appropriate certification block based on the board's setting
- For the all_member_certification format, include signature lines for all active members at the time of the meeting (query BOARD_MEMBER records)
- The certification block is rendered in the HTML but the actual signing workflow is out of scope for MVP (physical signatures on printed copies)

TASK 10: Store minutes as MINUTES_DOCUMENT
- Create the MINUTES_DOCUMENT table (if not already in the schema):
  - id: UUID primary key
  - meeting_id: FK to MEETING (unique — one minutes document per meeting)
  - town_id: FK to TOWN (for RLS)
  - board_id: FK to BOARD
  - status: enum (draft, review, approved, published)
  - content_json: JSONB — the canonical structured minutes data
  - html_rendered: TEXT — the rendered HTML (regenerated whenever content_json changes)
  - pdf_storage_path: TEXT — path to the PDF in Supabase Storage
  - minutes_style: TEXT — which style was used (action, summary, narrative)
  - generated_at: TIMESTAMP
  - submitted_for_review_at: TIMESTAMP (null until submitted)
  - approved_at: TIMESTAMP (null until approved)
  - approved_by_motion_id: FK to MOTION (the motion that approved these minutes)
  - published_at: TIMESTAMP (null until published to portal)
  - created_by: FK to USER_ACCOUNT
  - created_at: TIMESTAMP
  - updated_at: TIMESTAMP
- Create the Supabase migration for this table
- Add RLS policies: scoped by town_id, writable by users with R1-R5 permissions
- Add the PowerSync schema definition in packages/shared for client-side sync

TASK 11: Create "Generate Minutes Draft" button on post-meeting review page
- On the post-meeting review page (built in Block 07), add a "Generate Minutes Draft" button
- The button is visible only when:
  - The meeting status is "completed"
  - No minutes document exists yet for this meeting (or the existing one was explicitly deleted)
  - The user has R2 permission
- On click:
  - Show a confirmation dialog: "Generate minutes draft for [Board Name] meeting on [date]? Minutes style: [style name]"
  - Include a dropdown to override the minutes style (defaults to the board's effective setting)
  - Call POST /api/meetings/:meetingId/minutes/generate
  - Show a loading spinner (Puppeteer PDF generation takes a few seconds)
  - On success: navigate to the MinutesReviewPage (/meetings/:meetingId/minutes) — built in session 09.02
  - On error: show toast with error message
- If minutes already exist, show "View Minutes Draft" button instead, linking to the review page
- Add a "Regenerate Minutes" option (dropdown or secondary button) that overwrites the existing draft after confirmation

TASK 12: Verification
- Create a completed meeting with full structured data: attendance, multiple agenda sections with items, motions, votes, a recusal, an executive session notation, and adjournment
- Generate minutes JSON and verify all sections are present, motions are correctly structured, recusals are noted, fixed sections with no content show "None"
- Render HTML from the JSON and verify:
  - Semantic HTML structure (proper heading hierarchy, section elements)
  - WCAG 2.1 AA compliance (test with axe-core or similar)
  - DRAFT watermark appears
  - Certification block renders correctly for each format type
- Generate PDF via Puppeteer and verify:
  - PDF is stored in Supabase Storage at the correct path
  - PDF has proper headers/footers with page numbers
  - PDF content matches the HTML
  - DRAFT label appears in footer
- Test all three minutes styles (action, summary, narrative) produce different output levels
- Test both motion display formats (block_format, inline_narrative) render correctly
- Verify the "Generate Minutes Draft" button appears on the post-meeting page and triggers generation
- Verify regeneration overwrites the existing draft after confirmation

VERIFICATION CHECKLIST:
1. assembleMinutesJson produces complete structured JSON from meeting data
2. All three minutes style formatters produce correct output
3. Fixed sections with no activity show "None" (Q12)
4. Public hearing commenters are identified by name (Q16)
5. Motion display format respects board configuration (Q13)
6. HTML template uses semantic elements and passes WCAG 2.1 AA
7. Puppeteer generates a properly formatted PDF with headers/footers
8. Certification block renders correctly for all three formats (Q17)
9. MINUTES_DOCUMENT record is created with content_json, html_rendered, pdf_storage_path
10. "Generate Minutes Draft" button works on the post-meeting review page
11. Minutes style override works in the generation dialog
12. Regeneration overwrites existing draft with confirmation
```
