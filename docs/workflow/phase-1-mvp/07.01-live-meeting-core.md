# Session 07.01 — Meeting Session Core: Start, Attendance, Navigation

**Phase:** 1 — MVP | **Block:** 07 — Live Meeting Manager
**Dependencies:** 06.02, 02.02
**Estimated tasks:** 14

---

## Description

Build the live meeting management interface — the operator's real-time tool for running a municipal board meeting. This session covers meeting start flow (presiding officer confirmation, quorum check), attendance recording, agenda navigation with current-item tracking, and Supabase Realtime multi-device sync. The live meeting page uses feature-area error boundaries so that a crash in one panel does not take down the entire meeting interface.

## Tasks

1. Create LiveMeetingPage at /meetings/:meetingId/live — the operator's real-time meeting management interface
2. Create MeetingStartFlow: button to start meeting -> prompt for presiding officer confirmation (Q8: prompt operator, default from board settings) -> record meeting start time
3. Create AttendancePanel: list all board members, mark each as present/absent/late_arrival with timestamps (Q11: simple attendance list)
4. Implement quorum check: calculate if quorum is met (>50% of seated members, or board-specific override), display quorum status prominently, block meeting start if no quorum
5. Create AgendaNavigationPanel: show all agenda sections/items from the agenda, highlight current item, allow clicking to navigate
6. Implement current-item tracking: when operator advances to an item, record the transition timestamp
7. Create AgendaItemDetailPanel: show current item's title, description, presenter, exhibits, with action buttons (motion, vote, next item, table item)
8. Implement guest speaker entry for public comment periods: name (required), address (optional) — created as meeting record only, no PERSON record (advisory 1.2 Guest Participants)
9. Create meeting timer: elapsed time display, per-item time tracking
10. Implement Supabase Realtime subscriptions: all meeting data syncs in real-time across devices via useRealtimeSubscription
11. Create feature-area error boundaries: separate boundaries for AgendaNavigationPanel, AttendancePanel, VotingPanel, so one crashing doesn't kill the others
12. Implement connection resilience: ConnectionStatusBar shows warning on disconnect, React Query cache keeps data readable
13. Create meeting status transitions: scheduled -> in_progress (on start)
14. Verify: start meeting, record attendance, navigate agenda, quorum check works, offline writes persist

TASK 2: Create MeetingStartFlow
- File: packages/web/src/components/meeting/MeetingStartFlow.tsx
- This is a stepped overlay/dialog shown before the meeting officially starts
- Step 1 — Attendance: show all board members, mark each present/absent. The operator checks off who is physically present. Late arrivals can be added later.
- Step 2 — Quorum Check: automatically calculated from attendance. Display: "X of Y members present — Quorum [Met/Not Met]." If quorum is not met, show a warning and block proceeding. The operator can wait and update attendance as members arrive.
- Step 3 — Presiding Officer: show a prompt: "Who is presiding over this meeting?"
  - If the board Chair is marked present: pre-select the Chair, show as default with "Confirm" button
  - If the board Chair is marked absent: show "Chair [name] is absent. Who is presiding?" with Vice Chair pre-selected in a Select dropdown listing all present members, requiring explicit confirmation
  - The presiding officer selection is stored on the MEETING record (presiding_officer_id)
  - If the presiding officer differs from the Chair, record presiding_officer_succession data: original_chair, presiding_officer, reason
- Step 4 — Recording Secretary: prompt: "Who is serving as recording secretary?" with the board's default recording secretary pre-selected (from board_member.is_default_rec_sec). Select dropdown lists all present members and staff.
- "Start Meeting" button: sets meeting status to "in_progress", records started_at timestamp, writes presiding_officer_id and recording_secretary to the MEETING record, creates MEETING_ATTENDANCE records for all members

TASK 4: Implement quorum check
- File: packages/web/src/hooks/useQuorumCheck.ts (uses shared utility from @town-meeting/shared)
- Quorum calculation: majority of seated members = floor(totalSeatedMembers / 2) + 1
- totalSeatedMembers = count of active board_members for this board (not including vacant seats)
- presentCount = count of MEETING_ATTENDANCE records with status "present" or "late_arrival"
- hasQuorum = presentCount >= quorumNeeded
- Display quorum status prominently in the AttendancePanel header and in the LiveMeetingPage header
- If quorum is lost during the meeting (a member departs), show an alert: "Quorum has been lost. [X] of [Y] members present. [Z] needed for quorum."
- Quorum loss does not auto-end the meeting — the chair decides what to do — but it prevents new votes from being recorded until quorum is restored
- The quorum check is reactive: recalculates whenever attendance changes

TASK 6: Implement current-item tracking
- The MEETING record maintains a current_agenda_item_id field
- When the operator clicks an item in the AgendaNavigationPanel or clicks "Next Item" in the detail panel, the current item changes
- Record a timestamp for each item transition in an AGENDA_ITEM_TRANSITION table:

  - When transitioning to a new item, set ended_at on the previous transition and create a new record with started_at = now
- This data feeds the per-item time tracking (Task 9) and the structured meeting record for AI minutes
- "Next Item" button advances to the next item in sort_order within the current section, then to the first item of the next section
- "Previous Item" button goes back (for corrections)
- Keyboard shortcuts: Right arrow = next item, Left arrow = previous item (only when no text input is focused)

  2. Current item time: counts up from the current item's transition started_at timestamp (format: "0:45")

TASK 10: Implement Supabase Realtime subscriptions for live sync
- All meeting data is loaded via TanStack Query (useQuery with Supabase fetch functions)
- Live multi-device sync uses the useRealtimeSubscription hook from session 02.02
- When the operator writes data (attendance, transitions, notes), it writes to Supabase via useMutation
- Other devices receive the update in real-time via Supabase Realtime postgres_changes events
- Key Realtime subscriptions (each calls queryClient.invalidateQueries() in its onEvent callback):
  - useRealtimeSubscription({ table: 'meeting_attendance', filter: `meeting_id=eq.${meetingId}`, ... })
  - useRealtimeSubscription({ table: 'agenda_item', filter: `meeting_id=eq.${meetingId}`, ... })
  - useRealtimeSubscription({ table: 'agenda_item_transition', filter: `meeting_id=eq.${meetingId}`, ... })
  - useRealtimeSubscription({ table: 'motion', filter: `meeting_id=eq.${meetingId}`, ... }) (built in 07.02)
- Use typed query hooks from lib/api/ for the initial data load:
  const { data: meeting } = useMeeting(meetingId);  // includes attendance, agenda items, motions via JOIN

TASK 11: Create feature-area error boundaries
- Per advisory 1.3a, wrap each panel in its own ErrorBoundary from react-error-boundary:
  - AgendaNavigationPanel error boundary: if the nav crashes, attendance and detail panels still work
  - AttendancePanel error boundary: if attendance crashes, agenda navigation and detail still work
  - AgendaItemDetailPanel error boundary: if the detail panel crashes, navigation and attendance still work
  - MeetingTimer error boundary: timer crash must never take down meeting operations
  - ConnectionStatusBar error boundary: connection status crash must never affect the meeting interface
- Each error boundary shows a compact error fallback:
  - "[Panel name] encountered an error" with a "Retry" button
  - The compact fallback does NOT take the full panel width — it fits in the panel's space
- Error boundaries do NOT catch async errors — use try/catch with local error state for useMutation errors

TASK 12: Implement connection resilience
- All meeting operations (attendance, item transitions, notes, guest speakers) write to Supabase via useMutation
- The ConnectionStatusBar (from session 02.02) shows Supabase Realtime connection state:
  - Connected (no display — silence is good news)
  - Connecting... (amber, pulsing) — reconnecting after a brief drop
  - Disconnected (red) — "Meeting data may be outdated. Reconnecting..."
- If the network disconnects during a meeting:
  - React Query cache keeps meeting data readable — users can still view the agenda and prior records
  - Mutation calls (writes) fail with an error toast: "Unable to save — no connection. Please wait for reconnection."
  - ConnectionStatusBar updates immediately when the Realtime connection drops
  - When connectivity returns, Supabase Realtime reconnects automatically and React Query refetches stale data
- Write resilience: useMutation calls should include retry: 1 for transient failures
- Test the disconnect flow: disconnect network → ConnectionStatusBar shows Disconnected → attempt a write → error toast appears → reconnect → Realtime reconnects → data refetches automatically

TASK 13: Create meeting status transitions
- MEETING status enum: draft | scheduled | in_progress | completed | cancelled
- This session handles: scheduled -> in_progress
- Transition happens when the operator clicks "Start Meeting" in the MeetingStartFlow
- The transition writes to the MEETING record: status = "in_progress", started_at = now()
- Reverse transition (in_progress -> scheduled) is NOT supported — once started, a meeting can only move forward
- The LiveMeetingPage checks the meeting status on load:
  - "scheduled": show MeetingStartFlow overlay
  - "in_progress": show the live meeting interface
  - "completed": show read-only review mode
  - "draft" or "cancelled": redirect to the AgendaBuilder or meeting list
