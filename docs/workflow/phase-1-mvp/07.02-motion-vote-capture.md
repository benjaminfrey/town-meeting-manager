# Session 07.02 — Motion Capture & Vote Recording

**Phase:** 1 — MVP | **Block:** 07 — Live Meeting Manager
**Dependencies:** 07.01
**Estimated tasks:** 13

---

## Description

Build the motion capture and vote recording system for the live meeting manager. This includes the motion dialog with pre-populated suggested motions, the vote panel for recording each member's vote, recusal handling with quorum impact checks, motion amendments, tabling, and configurable motion display formats. All data writes to PowerSync local SQLite for offline resilience.

## Tasks

1. Create MotionCaptureDialog: motion_text (free text), motion_type (main, amendment, tabling, postponement, reconsideration, procedural), moved_by (select from present members), seconded_by (select from present members)
2. Create MotionPanel showing all motions for the current agenda item with status badges
3. Implement motion workflow: pending -> discussion -> vote -> passed/failed/tabled/withdrawn
4. Create VotePanel: for each present board member, record vote (yea/nay/abstain)
5. Implement recusal recording: member declares recusal -> capture reason (free text) -> member excluded from vote count -> record in meeting record per Maine law 2605(4)
6. Implement quorum impact check on recusal: if recusal drops participating members below quorum, alert chair immediately, block vote
7. Calculate and display vote results: yeas, nays, abstentions, recusals, pass/fail based on majority of those voting
8. Create motion amendment flow: amend motion text, record amendment vote separately
9. Implement motion to table: remove item from current discussion, mark as tabled
10. Create motion display format: configurable per board (Q13: board-level configurable) — compact or detailed
11. Record all motion/vote data in PowerSync local SQLite for offline resilience
12. Create vote summary display for completed items: motion text, who moved/seconded, vote breakdown, result
13. Verify: create motion, record votes for all members, handle recusal with quorum check, amendment flow, tabling, vote tallies correct

TASK 1: Create MotionCaptureDialog

TASK 2: Create MotionPanel
- File: packages/web/src/components/meeting/MotionPanel.tsx
- Displayed in the AgendaItemDetailPanel below the item details
- Lists all motions for the current agenda item, ordered by created_at
- Each motion card shows:
  - Motion text (truncated to 2 lines, expandable)
  - Motion type Badge (color-coded: main=blue, amendment=purple, table=amber, procedural=gray)
  - Status Badge: pending (gray), discussion (blue), voting (amber), passed (green), failed (red), tabled (amber), withdrawn (gray strikethrough)
  - Moved by / Seconded by names
  - Vote summary (after voting): "Passed 4-1" or "Failed 2-3"
- Clicking a motion expands it to show full details and vote breakdown
- Amendments to a motion are displayed as nested cards under the parent motion
- If no motions exist: show a subtle message "No motions recorded for this item"
- Use PowerSync useQuery on the MOTION table filtered by agenda_item_id for reactive updates

TASK 3: Implement motion workflow
- Motion status transitions:
  - pending -> discussion (automatic when motion is recorded — motion is "on the floor")
  - discussion -> voting (operator clicks "Call the Vote" button on the motion card)
  - voting -> passed | failed (calculated from vote results)
  - discussion -> tabled (via motion to table — separate motion that passes)
  - discussion -> withdrawn (mover requests to withdraw, operator confirms)
  - pending -> withdrawn (motion withdrawn before discussion)
- Each status transition writes to the MOTION record in PowerSync
- "Call the Vote" button appears on motions in "discussion" status
- Clicking "Call the Vote" opens the VotePanel (Task 4) for this specific motion
- "Withdraw Motion" button appears on motions in "pending" or "discussion" status
- Withdraw requires confirmation Dialog: "The mover wishes to withdraw this motion. Confirm?"
- Status transitions are logged with timestamps for the structured meeting record

TASK 4: Create VotePanel
- File: packages/web/src/components/meeting/VotePanel.tsx
- Displayed when "Call the Vote" is clicked on a motion in discussion status
- Shows a card for each board member with their name and three vote buttons: Yea, Nay, Abstain
- Members are listed in a consistent order (alphabetical by last name, or by seat_title)
- Vote button behavior:
  - Clicking a vote button highlights it and records the vote
  - Clicking a different button on the same member changes their vote
  - All members must have a vote recorded before the vote can be finalized
  - Absent members are auto-populated with vote: "absent" (grayed out, not clickable)
  - Members who have recused from this item (Task 5) show vote: "recusal" with their reason
- "Record Vote" button at the bottom: finalizes the vote, calculates the result, updates the motion status
- Before finalizing: display the vote tally prominently: "Yea: 4, Nay: 1, Abstain: 0, Recused: 1, Absent: 0"
- After finalizing: the VotePanel collapses and the motion card shows the result
- Each vote creates a VOTE_RECORD in PowerSync: one record per board member per motion

TASK 5: Implement recusal recording
- File: packages/web/src/components/meeting/RecusalDialog.tsx
- A board member can declare a recusal at any point during an agenda item
- "Record Recusal" button in the AttendancePanel (next to each member) or in the VotePanel
- RecusalDialog fields:
  - Member: pre-filled based on which member's recusal button was clicked
  - Reason: Textarea (required — per Maine law 2605(4), the disclosure must be recorded)
  - Scope: radio buttons — "This item only" (recused from the current agenda item) or "Remaining items" (rare, but possible)
- On submit: create a RECUSAL record linked to the meeting_id, agenda_item_id, and board_member_id
  - Fields: id, meeting_id, agenda_item_id, board_member_id, person_id, reason, scope, town_id, created_at
- When a member is recused from an item:
  - They appear in the VotePanel with "Recused" status (not clickable)
  - Their vote is recorded as "recusal" with the reason text
  - They are excluded from the vote count denominator for pass/fail calculation
  - The AttendancePanel shows a small recusal indicator next to their name for the duration of that item
- Per Maine law 2605(4): the software satisfies the requirement that "disclosure and abstention must be recorded with the clerk or secretary" by capturing the recusal reason in the meeting record

TASK 6: Implement quorum impact check on recusal
- When a recusal is recorded, immediately check if the remaining participating members still form a quorum
- Use the shared quorumAfterRecusal utility from @town-meeting/shared:
  quorumAfterRecusal(presentCount, recusalCount, totalSeatedMembers)
- If the recusal drops participating members below quorum:
  - Show a prominent alert (red Banner at the top of the center panel): "QUORUM ALERT: Recusal of [member name] drops participating members to [X]. Quorum requires [Y]. This vote cannot proceed."
  - Disable the "Call the Vote" and "Record Vote" buttons
  - The chair must decide: adjourn the item, table it, or seek the recused member's return to the vote (if the conflict is resolved)
- The alert persists until either:
  - Another member arrives (late_arrival) restoring quorum
  - The item is tabled or deferred
  - The recusal is rescinded (rare — requires Dialog confirmation)
- This check is critical: a vote taken without quorum is legally voidable

TASK 7: Calculate and display vote results
- File: packages/web/src/hooks/useVoteCalculation.ts
- Vote calculation rules:
  - Eligible voters = present members - recused members (absent members are NOT eligible)
  - Yea count = votes where vote === "yea"
  - Nay count = votes where vote === "nay"
  - Abstain count = votes where vote === "abstain" (abstentions do NOT count toward either side)
  - Majority = floor(eligibleVoters / 2) + 1 (simple majority by default)
  - Passed = yeaCount >= majority of those actually voting (yea + nay, excluding abstentions)
  - Some votes may require 2/3 majority (e.g., suspend rules) — support a requiredMajority parameter
- Display vote results in two formats:
  - Compact: "Passed 4-1" or "Failed 2-3" or "Passed unanimously"
  - Detailed: "Yea: Smith, Jones, Brown, Davis (4). Nay: Wilson (1). Abstained: None. Recused: Taylor (conflict of interest). Absent: Johnson."
- The result format is controlled by the board's motion_display_format setting (Q13):
  - block_format: structured block with each element on its own line
  - inline_narrative: single paragraph narrative style
- Update the MOTION record with the result: status = "passed" or "failed", vote_summary JSONB

TASK 8: Create motion amendment flow
- When a motion is in "discussion" status, the operator can record an amendment
- "Amend Motion" button appears on the motion card during discussion
- Amendment creates a NEW motion with:
  - motion_type: "amendment"
  - parent_motion_id: the original motion's id
  - motion_text: the amended text (can be the full revised text or just the amendment language)
- The amendment goes through its own vote: discussion -> voting -> passed/failed
- If the amendment passes:
  - The original motion's text is updated to reflect the amendment
  - The original amendment text is preserved in the amendment motion record (audit trail)
  - Voting on the original motion (as amended) continues
- If the amendment fails:
  - The original motion text remains unchanged
  - Discussion on the original motion continues
- Multiple amendments can be proposed sequentially (one at a time — you cannot amend an amendment per Robert's Rules)
- Amendments display as nested cards under the parent motion in the MotionPanel

TASK 9: Implement motion to table
- "Table Item" button appears in the AgendaItemDetailPanel
- Clicking "Table Item" opens the MotionCaptureDialog pre-filled with:
  - motion_type: "table"
  - motion_text: "to table [item title]"
- The tabling motion goes through the standard motion/vote workflow
- If the tabling motion passes:
  - The agenda item's status changes to "tabled"
  - The item appears dimmed in the AgendaNavigationPanel with a "Tabled" badge
  - The operator is advanced to the next item automatically
  - The tabled item is added to the "items for next meeting" queue (built in session 07.03)
- If the tabling motion fails:
  - The agenda item remains active
  - Discussion continues on the original item
- A tabled item can be untabled later via a "Motion to Untable" (motion_type: "untable")

TASK 10: Create configurable motion display format
- Per advisory Q13: each board template specifies a motion_display_format
- Two formats:
  - block_format (Planning Board style):
    Motion: [motion text]
    Moved by: [member name]
    Seconded by: [member name]
    Discussion: [summary or "None"]
    Vote: [Yea: X, Nay: Y, Abstain: Z]
    Result: [Passed/Failed]
  - inline_narrative (Select Board style):
    "[Member] moved to [motion text]. [Member] seconded. Motion [passed/failed] [vote count]."
- The format is used in:
  - The MotionPanel display in the live meeting (real-time view)
  - The vote summary display after a vote is completed
  - The structured meeting record passed to the AI for minutes generation
- The format setting is stored on the BOARD record (motion_display_format field)
- Read the board's setting via PowerSync useQuery on the BOARD table

TASK 11: Record all motion/vote data in PowerSync
- Every write operation in this session writes to PowerSync local SQLite via powersync.execute()
- Operations that must work offline:
  - Creating a motion (MOTION table insert)
  - Updating motion status (MOTION table update)
  - Recording a vote (VOTE_RECORD table insert)
  - Recording a recusal (RECUSAL table insert or dedicated field on VOTE_RECORD)
  - Calculating vote results (computed locally, stored on MOTION record)
- Use Kysely for type-safe writes:
  await db.insertInto('motion').values({ id: generateId(), agenda_item_id, meeting_id, ... }).execute()
- PowerSync automatically queues all writes for sync when connected
- Use writeTransaction for multi-record operations (e.g., creating vote records for all members + updating motion status should be atomic):
  await powersync.writeTransaction(async (tx) => { ... })

TASK 12: Create vote summary display

- Summary includes:

- For completed agenda items: clicking the item in the AgendaNavigationPanel shows the full vote summary in the detail panel (read-only)
- The summary data is the source for the structured meeting record sent to the AI minutes pipeline
