# Session 03.01 — Wizard Framework & Stage 1 (Your Town)

**Phase:** 1 — MVP | **Block:** 03 — Onboarding Wizard
**Dependencies:** 02.03
**Estimated tasks:** 11

---

## Description

Build the onboarding wizard framework with multi-stage navigation, client-side state management, and browser exit protection. Implement Stage 1 (Your Town) with full field validation, sensible defaults, and the first working wizard step that captures municipality identity.

## Tasks

1. Create WizardProvider context to hold all wizard state across 5 stages (React state, no database writes)
2. Create WizardLayout component with progress indicator (step 1 of 5, 2 of 5, etc.), back/next navigation, stage title
3. Implement browser back button interception with confirmation dialog ("If you leave, your setup progress will be lost")
4. Create WizardStage1 (Your Town) form with fields: town name, state, municipality type, population range, primary contact name, primary contact role
5. Create Zod schema for Stage 1 validation (import from shared package)
6. Implement field-level validation with error messages
7. Implement Next button disabled until validation passes
8. Configure sensible defaults: state=Maine, type=Town, population="Under 1,000"
9. Style wizard with shadcn/ui Card, Input, Select, Button components
10. Create wizard route at /setup, accessible only to authenticated users with no town_id
11. Verify: wizard renders, Stage 1 form validates, defaults are set, navigation works, browser back shows confirmation

You are building the onboarding wizard for the Town Meeting Manager — the first-run experience where a town administrator configures their municipality. The wizard is a 5-stage, all-or-nothing flow. No data is saved to the database until the final stage is completed.

Read docs/advisory-resolutions/2.1-onboarding-wizard-ux-spec.md thoroughly — it is the authoritative specification for every field, validation rule, default value, and behavior in the wizard. Key rules from that document:

WIZARD BEHAVIOR RULES:
- Single-session, all-or-nothing flow — no partial saves
- If the user closes the browser, navigates away, or logs out, all progress is lost
- State lives in client-side memory only (React state). No database writes until completion.
- Back button: users can go back to any previous stage. Data in later stages is preserved.
- Forward navigation: only by completing current stage's required fields. "Next" is disabled until validation passes.
- Browser back button: intercepted with confirmation dialog.
- Every field has a sensible default so a confused user can click "Next" without getting stuck.

TASK 1: Create WizardProvider context
- Create src/providers/WizardProvider.tsx
- Define the complete wizard state shape covering all 5 stages:

  interface WizardState {
    stage1: {
      townName: string;
      state: string;
      municipalityType: string;
      populationRange: string;
      contactName: string;
      contactRole: string;
    };
    stage2: {
      boardName: string;
      memberCount: number;
      electionMethod: 'at_large' | 'role_titled';
      seatTitles: string[];
      officerElectionMethod: string;
      districtBased: boolean;
      staggeredTerms: boolean;
    };
    stage3: {
      presidingOfficer: string;
      minutesRecorder: string;
      staffRolesPresent: string[];
    };
    stage4: {
      boards: Array<{
        id: string;
        name: string;
        memberCount: number;
        electedOrAppointed: 'elected' | 'appointed';
        isCustom: boolean;
        checked: boolean;
      }>;
    };
    stage5: {
      meetingFormality: 'informal' | 'semi_formal' | 'formal';
      minutesStyle: 'action' | 'summary' | 'narrative';
    };
    currentStage: number; // 1-5
    completedStages: Set<number>; // tracks which stages have been completed
  }

- Initialize with sensible defaults from the advisory:
  - stage1: state="ME", municipalityType="town", populationRange="under_1000"
  - stage2: memberCount=3, electionMethod="at_large", officerElectionMethod="vote_of_board", districtBased=false, staggeredTerms=false
  - stage3: presidingOfficer="chair_of_board", minutesRecorder="town_clerk", staffRolesPresent=[]
  - stage4: boards=[] (no boards pre-checked per advisory)
  - stage5: meetingFormality="informal", minutesStyle="summary"

- Provide context functions:
  - updateStage(stageNumber, data) — merge partial data into a stage
  - goToStage(stageNumber) — navigate to a specific stage
  - goNext() — advance to next stage (only if current stage is valid)
  - goBack() — go to previous stage
  - markStageComplete(stageNumber) — mark a stage as validated and complete
  - getWizardData() — return the complete state for the final submission
  - resetWizard() — clear all state (used if user wants to start over)

- Use React useState (not useReducer) for simplicity — the state shape is defined but updates are straightforward
- Do NOT use React Hook Form at the provider level — each stage has its own form instance. The provider stores the RESULTS of each stage's form submission.

TASK 5: Create Zod schema for Stage 1
- Create the schema in @town-meeting/shared if possible (so it can be reused server-side), or in src/lib/schemas/wizard.ts:

  const stage1Schema = z.object({
    townName: z.string()
      .min(2, "Town name must be at least 2 characters")
      .max(100, "Town name must be less than 100 characters")
      .regex(/^[a-zA-Z0-9\s\-'.]+$/, "Town name can only contain letters, numbers, spaces, hyphens, apostrophes, and periods"),
    state: z.enum(["ME", "NH", "VT", "MA", "CT", "RI"]),
    municipalityType: z.enum(["town", "city", "plantation"]),
    populationRange: z.enum(["under_1000", "1000_2500", "2500_5000", "5000_10000", "over_10000"]),

      .max(100, "Contact name must be less than 100 characters"),
    contactRole: z.string()

      .max(100, "Contact role must be less than 100 characters"),
  });
