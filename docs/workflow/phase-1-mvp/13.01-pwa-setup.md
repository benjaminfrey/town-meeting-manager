# Session 13.01 — Progressive Web App Setup

**Phase:** 1 — MVP | **Block:** 13 — PWA & Offline
**Dependencies:** 02.02
**Estimated tasks:** 9

---

## Description

Configure the Town Meeting Manager web application as a Progressive Web App (PWA) with a web app manifest, Service Worker for offline caching, app install prompt, and proper icons. The PWA setup complements the PowerSync offline-first data layer (session 02.02) by caching the application shell and static assets, enabling the app to load and function even without a network connection — critical for rural Maine town halls with unreliable connectivity.

## Tasks

1. Create web app manifest (manifest.json): name, short_name, description, icons, theme color, display: standalone, start_url
2. Create Service Worker for offline caching using Workbox (or vite-plugin-pwa)
3. Configure cache strategies: Cache-First for static assets (JS, CSS, images), Network-First for API responses
4. Implement offline fallback page: displayed when navigating to uncached routes while offline
5. Create app install prompt: detect beforeinstallprompt event, show custom install banner
6. Generate PWA icons in required sizes (192x192, 512x512, maskable)
7. Configure Vite build for PWA output (service worker registration, manifest injection)
8. Test PWA installability: Chrome DevTools → Application → Manifest, Lighthouse PWA audit
9. Verify: app installable on mobile and desktop, offline caching works, service worker registered, Lighthouse PWA score passes

TASK 2: Create Service Worker with Workbox
- Use vite-plugin-pwa (recommended approach for Vite-based projects):
  - Install: pnpm --filter @town-meeting/web add -D vite-plugin-pwa
  - vite-plugin-pwa uses Workbox internally but provides a simpler configuration interface
- Alternatively, configure Workbox directly with workbox-build if more control is needed
- Configure in vite.config.ts:
  ```ts
  import { VitePWA } from 'vite-plugin-pwa';

  export default defineConfig({
    plugins: [
      // ... other plugins
      VitePWA({
        registerType: 'autoUpdate',
        includeAssets: ['icons/*.png', 'fonts/*.woff2'],
        manifest: false, // we manage manifest.json manually
        workbox: {
          // See Task 3 for cache strategy configuration
          globPatterns: ['**/*.{js,css,html,ico,png,svg,woff2}'],
          runtimeCaching: [
            // See Task 3
          ],
          navigateFallback: '/offline.html',
          navigateFallbackDenylist: [/^\/api\//],
        },
      }),
    ],
  });
  ```
- The Service Worker should:
  - Precache the application shell (all JS, CSS, and HTML from the build output)
  - Use runtime caching for dynamic content
  - Support the autoUpdate strategy: when a new version is deployed, the SW updates in the background and takes effect on next page load
  - Show a toast notification when a new version is available: "A new version is available. Refresh to update."

TASK 3: Configure cache strategies
- Static assets (JS, CSS, fonts, images) — CacheFirst strategy:
  - These assets have content hashes in their filenames (Vite's default behavior)
  - Cache them indefinitely — a new hash means a new file, and the old cached version is never served for the new URL
  - Configuration:
    ```ts
    {
      urlPattern: /\.(?:js|css|woff2|png|svg|ico)$/,
      handler: 'CacheFirst',
      options: {
        cacheName: 'static-assets',
        expiration: { maxEntries: 100, maxAgeSeconds: 365 * 24 * 60 * 60 },
      },
    }
    ```
- HTML pages — NetworkFirst strategy:
  - Try the network first to get the latest HTML
  - Fall back to cached version if offline
  - This ensures fresh navigation when online but still works offline
- API responses — NetworkFirst with timeout:
  - For non-PowerSync API calls (e.g., portal search, PDF generation requests)
  - Try network first with a 3-second timeout, fall back to cache
  - Note: most data is served via PowerSync (local SQLite), not API calls, so this is a secondary concern
  - Configuration:
    ```ts
    {
      urlPattern: /^https:\/\/api\.townmeetingmanager\.com\/.*/,
      handler: 'NetworkFirst',
      options: {
        cacheName: 'api-responses',
        networkTimeoutSeconds: 3,
        expiration: { maxEntries: 50, maxAgeSeconds: 24 * 60 * 60 },
      },
    }
    ```
- PowerSync / Supabase API calls — NetworkOnly:
  - PowerSync manages its own caching through local SQLite
  - Do NOT cache PowerSync sync requests in the Service Worker (it would interfere with PowerSync's sync protocol)
  - Exclude: /powersync/*, /rest/v1/*, /auth/v1/*
- Images (town seals, exhibits) — StaleWhileRevalidate:
  - Serve from cache immediately, fetch updated version in background
  - Good for images that change rarely (town seals) but should update eventually
  - Configuration:
    ```ts
    {
      urlPattern: /\.(?:jpg|jpeg|gif|webp)$/,
      handler: 'StaleWhileRevalidate',
      options: {
        cacheName: 'images',
        expiration: { maxEntries: 50, maxAgeSeconds: 30 * 24 * 60 * 60 },
      },
    }
    ```

TASK 4: Implement offline fallback page
- Create: packages/web/public/offline.html
- This page is displayed when the user navigates to a route that is not cached while offline
- Content:
  - Town Meeting Manager branding
  - Message: "You're currently offline"
  - Explanation: "This page isn't available offline, but your meeting data is still accessible."
  - "Go to Dashboard" button (the dashboard route and its data are available offline via PowerSync)
  - "Your changes will sync when you're back online."
- The offline page is precached by the Service Worker (always available)
- Style it consistently with the app using inline CSS (since external stylesheets might not be cached for this page)
- The navigateFallback in Workbox config points to this file
- Note: the main app shell (dashboard, meeting pages) should load offline because they are precached. This fallback is for edge cases where a specific route's chunk was never cached.

TASK 5: Create app install prompt
- File: packages/web/src/components/pwa/InstallBanner.tsx
- Listen for the beforeinstallprompt event:
  - Store the event reference for later use
  - Show a custom install banner when the event fires
- Custom install banner design:
  - Fixed banner at the bottom of the screen (above the footer)
  - Message: "Install Town Meeting Manager for quick access and offline use"
  - "Install" button — triggers the native install prompt via deferredPrompt.prompt()
  - "Not now" dismiss button — hides the banner for the session
  - "Don't show again" option — sets a localStorage flag to suppress the banner permanently
- Show the banner only:
  - On desktop and mobile devices (not in already-installed standalone mode)
  - After the user has been on the app for at least 30 seconds (avoid immediately pestering first-time visitors)
  - If the user hasn't dismissed it permanently
- After successful installation:
  - Hide the banner
  - Detect standalone mode: window.matchMedia('(display-mode: standalone)').matches
  - Optionally show a welcome toast: "Town Meeting Manager installed successfully"
- For iOS (Safari doesn't support beforeinstallprompt):
  - Detect iOS: check userAgent for iPhone/iPad
  - Show a custom instruction banner: "To install, tap the Share button and then 'Add to Home Screen'"
  - Include a visual guide (share icon illustration)
