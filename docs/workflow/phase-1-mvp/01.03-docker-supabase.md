# Session 01.03 — Docker Compose & Self-Hosted Supabase

**Phase:** 1 — MVP | **Block:** 01 — Infrastructure & Monorepo Foundation
**Dependencies:** 01.01
**Estimated tasks:** 11

---

## Description

Set up the self-hosted Supabase stack via Docker Compose, including PostgreSQL with PostGIS, GoTrue authentication, PostgREST, Realtime, Storage, Kong API gateway, and Studio. This provides the complete backend infrastructure for local development and mirrors the production deployment architecture.

## Tasks

1. Create docker/ directory at repo root
2. Create docker-compose.yml for Supabase stack (PostgreSQL + PostGIS, GoTrue, PostgREST, Realtime, Storage, Kong, Meta)
3. Create .env.example with all required Supabase environment variables (JWT secret, anon/service keys, database credentials, API URLs)
4. Configure PostgreSQL with PostGIS extension enabled
5. Configure Kong API gateway with route definitions for auth, rest, realtime, storage
6. Configure GoTrue (auth) service with email provider settings
7. Configure Storage service with local file storage
8. Create volumes configuration for persistent PostgreSQL data and storage
9. Create docker/scripts/init.sql for initial database setup (enable extensions: uuid-ossp, postgis, pgjwt)
10. Create a Makefile or package.json scripts for common Docker operations (up, down, reset, logs)
11. Test: docker compose up, verify all services healthy, verify PostGIS extension active

## Prompt

```
You are setting up the self-hosted Supabase infrastructure for the Town Meeting Manager project using Docker Compose. Read docs/advisory-resolutions/3.2-supabase-hosting.md for the infrastructure layout and hosting decisions.

KEY CONTEXT (from advisory 3.2):
- Self-hosted Supabase via Docker Compose from day one
- Components: PostgreSQL + PostGIS, GoTrue (auth), PostgREST, Realtime, Storage, Kong (API gateway), Studio (admin UI), Meta
- Multi-town isolation via RLS (not separate databases) — all towns share one PostgreSQL instance
- PostGIS is required for Phase 3 parcel data features
- This is for local development; production deployment will be on a dedicated server

IMPORTANT: Use the official Supabase self-hosting Docker configuration as the reference. The Supabase team maintains a docker-compose.yml in the supabase/supabase repository (https://github.com/supabase/supabase/tree/master/docker). Base your configuration on their latest stable version but customize for this project.

TASK 1: Create docker/ directory
- Create docker/ at the repo root
- Create docker/volumes/ directory (will hold persistent data, gitignored)

TASK 2: Create docker-compose.yml
Create docker/docker-compose.yml with these services:

db (PostgreSQL + PostGIS):
- Image: supabase/postgres:15.8.1.78 (or latest stable with PostGIS)
- Port: 5432 (mapped to host 54322 to avoid conflicts with local PostgreSQL)
- Health check: pg_isready
- Volume: ./volumes/db/data:/var/lib/postgresql/data
- Environment: POSTGRES_PASSWORD, POSTGRES_DB (postgres), JWT_SECRET, JWT_EXP
- Init script: ./scripts/init.sql mounted to /docker-entrypoint-initdb.d/

kong (API Gateway):
- Image: kong:2.8.1
- Port: 8000 (mapped to host 54321 — this is the main Supabase API URL)
- Depends on: auth, rest, realtime, storage
- Environment: KONG_DATABASE=off, KONG_DECLARATIVE_CONFIG
- Volume: mount kong.yml config file

auth (GoTrue):
- Image: supabase/gotrue:v2.164.0 (or latest stable)
- Depends on: db
- Environment: GOTRUE_DB_DATABASE_URL, GOTRUE_SITE_URL, GOTRUE_JWT_SECRET, GOTRUE_JWT_EXP, GOTRUE_EXTERNAL_EMAIL_ENABLED, GOTRUE_MAILER_AUTOCONFIRM (true for dev), API_EXTERNAL_URL
- No direct port mapping (accessed through Kong)

rest (PostgREST):
- Image: postgrest/postgrest:v12.2.3 (or latest stable)
- Depends on: db
- Environment: PGRST_DB_URI, PGRST_DB_SCHEMAS=public,storage, PGRST_DB_ANON_ROLE=anon, PGRST_JWT_SECRET

realtime (Supabase Realtime):
- Image: supabase/realtime:v2.33.70 (or latest stable)
- Depends on: db
- Environment: DB_HOST, DB_PORT, DB_USER, DB_PASSWORD, DB_NAME, PORT, JWT_SECRET, REPLICATION_MODE=RLS, SECURE_CHANNELS=true

storage (Supabase Storage):
- Image: supabase/storage-api:v1.14.5 (or latest stable)
- Depends on: db, rest
- Environment: POSTGREST_URL, DATABASE_URL, PGRST_JWT_SECRET, STORAGE_BACKEND=file, FILE_STORAGE_BACKEND_PATH=/var/lib/storage
- Volume: ./volumes/storage:/var/lib/storage

meta (Supabase Meta):
- Image: supabase/postgres-meta:v0.84.2 (or latest stable)
- Depends on: db
- Environment: PG_META_DB_HOST, PG_META_DB_PORT, PG_META_DB_NAME, PG_META_DB_USER, PG_META_DB_PASSWORD

studio (Supabase Studio):
- Image: supabase/studio:20241202-5765c67 (or latest stable)
- Port: 3000 (mapped to host 54323)
- Depends on: meta
- Environment: STUDIO_PG_META_URL, SUPABASE_URL (Kong URL), SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY

TASK 3: Create .env.example
Create docker/.env.example with all required variables. Generate placeholder values where appropriate:
```env
# PostgreSQL
POSTGRES_PASSWORD=your-super-secret-db-password
POSTGRES_DB=postgres
POSTGRES_PORT=5432

# Supabase JWT — IMPORTANT: Generate real secrets for production
# Generate with: openssl rand -base64 32
JWT_SECRET=your-super-secret-jwt-token-with-at-least-32-characters-long
JWT_EXPIRY=3600

# Supabase API Keys — generate from JWT_SECRET
# These are JWTs signed with JWT_SECRET containing role claims
ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
SERVICE_ROLE_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

# URLs
SITE_URL=http://localhost:3000
API_EXTERNAL_URL=http://localhost:54321
SUPABASE_PUBLIC_URL=http://localhost:54321

# Studio
STUDIO_PORT=54323

# GoTrue
GOTRUE_SMTP_HOST=inbucket
GOTRUE_SMTP_PORT=2500
GOTRUE_SMTP_USER=
GOTRUE_SMTP_PASS=
GOTRUE_SMTP_ADMIN_EMAIL=admin@townmeetingmanager.local
GOTRUE_MAILER_AUTOCONFIRM=true

# (PowerSync removed — TanStack Query is the data layer, no sync service needed)
# POWERSYNC_URL=http://localhost:8080
```

Also create a docker/.env file that is a copy of .env.example with working development values. Generate actual JWT tokens for ANON_KEY and SERVICE_ROLE_KEY using the JWT_SECRET. The anon key JWT payload should be: { "role": "anon", "iss": "supabase", "iat": <timestamp>, "exp": <far-future-timestamp> }. The service role key JWT payload should be: { "role": "service_role", "iss": "supabase", "iat": <timestamp>, "exp": <far-future-timestamp> }.

TASK 4: Configure PostgreSQL with PostGIS
The init.sql script (Task 9) handles enabling PostGIS. Ensure the PostgreSQL image used includes PostGIS. The supabase/postgres image includes it by default. Verify the image tag includes PostGIS support.

TASK 5: Configure Kong API gateway
Create docker/volumes/api/kong.yml with declarative route definitions:
- /auth/v1/* routes to auth service (GoTrue)
- /rest/v1/* routes to rest service (PostgREST)
- /realtime/v1/* routes to realtime service
- /storage/v1/* routes to storage service
- /pg/* routes to meta service (Studio)
Include CORS headers in the Kong configuration for local development (allow localhost origins).

TASK 6: Configure GoTrue auth service
- Enable email provider (GOTRUE_EXTERNAL_EMAIL_ENABLED=true)
- Set GOTRUE_MAILER_AUTOCONFIRM=true for development (no email verification required)
- Set GOTRUE_DISABLE_SIGNUP=false (allow signups during development)
- Configure JWT settings: secret, expiry, algorithm (HS256 for dev; RS256 for production per advisory 1.3a)

TASK 7: Configure Storage service
- Backend: local file storage for development (FILE_STORAGE_BACKEND_PATH=/var/lib/storage)
- Create storage volume mapping for persistent file storage
- Storage will hold: agenda packets (PDF), minutes documents (PDF), exhibit attachments, audio recordings (Phase 2)

TASK 8: Create volumes configuration
- docker/volumes/db/data/ — PostgreSQL data (gitignored)
- docker/volumes/storage/ — Supabase Storage files (gitignored)
- docker/volumes/api/ — Kong configuration
- Add docker/volumes/db/ and docker/volumes/storage/ to .gitignore

TASK 9: Create docker/scripts/init.sql

CREATE EXTENSION IF NOT EXISTS "pgjwt";
CREATE EXTENSION IF NOT EXISTS "pgcrypto";
CREATE EXTENSION IF NOT EXISTS "pg_stat_statements";

-- Create custom types for the application
-- (These will be expanded in session 01.05 when we create the full schema)

```

TASK 10: Create convenience scripts
Add to the root package.json scripts:
```json
{
  "supabase:up": "cd docker && docker compose up -d",
  "supabase:down": "cd docker && docker compose down",
  "supabase:reset": "cd docker && docker compose down -v && docker compose up -d",
  "supabase:logs": "cd docker && docker compose logs -f",
  "supabase:status": "cd docker && docker compose ps",
  "supabase:db": "docker exec -it town-meeting-db psql -U postgres"
}
```
Set container names in docker-compose.yml for easy reference:
- db: town-meeting-db
- kong: town-meeting-kong
- auth: town-meeting-auth
- rest: town-meeting-rest
- realtime: town-meeting-realtime
- storage: town-meeting-storage
- meta: town-meeting-meta
- studio: town-meeting-studio

TASK 11: Test the stack
- Run `pnpm supabase:up` (or `cd docker && docker compose up -d`)
- Wait for all services to become healthy
- Verify PostgreSQL is accessible: `docker exec town-meeting-db psql -U postgres -c "SELECT version();"`
- Verify PostGIS is active: `docker exec town-meeting-db psql -U postgres -c "SELECT PostGIS_version();"`
- Verify Kong is routing: `curl http://localhost:54321/rest/v1/` (should return PostgREST response)
- Verify Studio is accessible: open http://localhost:54323 in a browser
- Verify GoTrue is running: `curl http://localhost:54321/auth/v1/health`
- Run `pnpm supabase:down` to stop all services

VERIFICATION:

4. `docker exec town-meeting-db psql -U postgres -c "SELECT extname FROM pg_extension;"` shows uuid-ossp, postgis, pgjwt, pgcrypto
5. `curl http://localhost:54321/auth/v1/health` returns a healthy status
6. `curl http://localhost:54321/rest/v1/` returns a PostgREST response (empty or schema info)

8. docker/.env.example exists and is documented
9. docker/volumes/db/ and docker/volumes/storage/ are in .gitignore
