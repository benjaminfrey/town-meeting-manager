# Session 01.05 — Database Schema: Core Tables

**Phase:** 1 — MVP | **Block:** 01 — Infrastructure & Monorepo Foundation
**Dependencies:** 01.03
**Estimated tasks:** 15

---

## Description

Create the PostgreSQL database schema for all core entities in the Town Meeting Manager: towns, persons, user accounts, boards, board members, meetings, agenda items, motions, votes, and attendance. Each table gets proper foreign keys, indexes, constraints, and an updated_at trigger. A seed data script provides sample data for development.

## Prompt

```
You are creating the core database schema for the Town Meeting Manager. Read docs/advisory-resolutions/1.2-roles-permissions.md for the PERSON entity architecture, role definitions, and field requirements. The schema must be created as SQL migration files that run against the self-hosted Supabase PostgreSQL instance (Docker Compose from session 01.03).

KEY ARCHITECTURAL DECISIONS (from advisory 1.2):
- PERSON is the identity anchor — one person can have user_account (0-1), board_members (0-many), resident_account (0-1)
- 4 roles: sys_admin, admin, staff, board_member
- staff and board_member are mutually exclusive on the same PERSON
- Permissions stored as JSONB on user_account (not computed from templates at query time)
- Per-board permissions with global defaults: { global: {...}, board_overrides: [...] }
- gov_title is a display label with no effect on permissions
- Multi-tenant: all tables have town_id FK, isolation enforced via RLS (session 01.07)
- recording_secretary is tracked on MEETING_ATTENDANCE, not on user_account
- Minutes approval is a board vote action, not a single-person permission

MIGRATION NAMING CONVENTION:
Use timestamp-based naming: YYYYMMDDHHMMSS_description.sql
Example: 20260308000001_create_town_table.sql

All migrations go in supabase/migrations/ directory.

TASK 5: Create USER_ACCOUNT table (20260308000004_create_user_account.sql)
```sql
CREATE TABLE user_account (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  person_id UUID NOT NULL UNIQUE REFERENCES person(id) ON DELETE CASCADE,
  town_id UUID NOT NULL REFERENCES town(id) ON DELETE CASCADE,
  role user_role NOT NULL,
  gov_title TEXT,
  permissions JSONB NOT NULL DEFAULT '{"global": {}, "board_overrides": []}',
  auth_user_id UUID UNIQUE REFERENCES auth.users(id) ON DELETE SET NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  archived_at TIMESTAMPTZ,

  -- Ensure town_id matches the person's town_id (denormalized for RLS performance)
  CONSTRAINT user_account_town_match CHECK (town_id IS NOT NULL)
);

COMMENT ON TABLE user_account IS 'Application login account linked to a PERSON. One person has at most one user_account.';
COMMENT ON COLUMN user_account.role IS 'App role: sys_admin, admin, staff, board_member. Staff and board_member are mutually exclusive.';
COMMENT ON COLUMN user_account.gov_title IS 'Display label only (Town Clerk, Treasurer, etc.) — has no effect on permissions.';
COMMENT ON COLUMN user_account.permissions IS 'JSONB permissions matrix: { global: { action: bool }, board_overrides: [{ board_id, permissions: { action: bool } }] }';
COMMENT ON COLUMN user_account.auth_user_id IS 'FK to Supabase Auth (auth.users). Null if account is archived or pending invitation.';
```

TASK 14: Create updated_at trigger (20260308000013_create_updated_at_trigger.sql)
```sql
-- Trigger function to auto-update updated_at on row modification
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ language 'plpgsql';

-- Apply to all tables with updated_at columns
CREATE TRIGGER update_town_updated_at BEFORE UPDATE ON town
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

CREATE TRIGGER update_meeting_updated_at BEFORE UPDATE ON meeting
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

CREATE TRIGGER update_agenda_item_updated_at BEFORE UPDATE ON agenda_item
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
```
