# Session 04.03 — Member Management

**Phase:** 1 — MVP | **Block:** 04 — Town Profile & Board Management
**Dependencies:** 04.02
**Estimated tasks:** 13

---

## Description

Build the member management system covering board member rosters, staff account creation with configurable permissions, the visual PermissionMatrixEditor, and role transition workflows. This session enforces the critical mutual exclusivity rule between staff and board_member roles, implements the five default permission templates, and handles account lifecycle transitions -- all grounded in the PERSON entity architecture from advisory 1.2.

## Tasks

1. Create MemberRoster component on BoardDetailPage showing current members (name, seat title, term dates, status, role indicator)
2. Create AddMemberDialog: search existing PERSONs by name/email, or create new PERSON + USER_ACCOUNT
3. Implement role selection with mutual exclusivity enforcement: if PERSON is staff, cannot be added as board_member (and vice versa) -- show clear error message referencing conflict of interest
4. Create staff account creation flow: select permission template (Town Clerk, Deputy Clerk, Board-Specific Staff, General Staff, Recording Secretary Only), then customize permissions via matrix editor
5. Create PermissionMatrixEditor component: visual grid of 30 governable actions organized by functional area (Town & System, Agenda & Meeting Prep, Live Meeting, Minutes & Records, Civic Engagement, View & Download), with Y/N/Board-specific toggles
6. Implement per-board permission overrides: staff can have different permissions on different boards
7. Create board_member account creation with term dates, seat title, and default permissions
8. Implement recording secretary default assignment per board member (is_default_rec_sec flag)
9. Create MemberTransitionDialog: handle board member to archived, staff to board member transitions with proper mutual exclusivity checks
10. Create government title field (display label only -- no permission effect)
11. Implement account invitation: generate invitation that can be sent via email (placeholder -- full email integration in session 11.01)
12. Create member archival flow: archive board membership (set term_end, status=archived), optionally archive user_account
13. Verify: add members, assign permissions, enforce mutual exclusivity, template application works, archival preserves records

## Prompt

```
You are building the Member Management system for the Town Meeting Manager — a municipal meeting management platform for Maine towns. This covers board member rosters, staff accounts with configurable permissions, and the PERSON entity lifecycle. Read docs/advisory-resolutions/1.2-roles-permissions.md thoroughly — it defines the complete permissions matrix, five templates, mutual exclusivity rules, PERSON entity architecture, and account lifecycle. Implement the following.

PROJECT CONTEXT:
- Monorepo: packages/web (React 19 + React Router v7 Framework Mode), packages/shared (types, schemas)
- PowerSync for offline-first local SQLite — use @powersync/react hooks (useQuery)
- Kysely ORM via @powersync/kysely-driver for type-safe SQL
- Tailwind CSS v4 + shadcn/ui (Radix UI) for components
- React Hook Form + Zod for form validation
- 4 roles: sys_admin, admin, staff, board_member
- PERSON entity with linked user_account (0 or 1), board_members (0 to many), resident_account (0 or 1)
- Permissions stored as JSONB on user_account — 30 governable actions with per-board overrides
- Government title is a display label only — has no effect on permissions

PERSON & ACCOUNT DATA MODEL (from advisory 1.2):
- PERSON: id, name, email (unique), town_id, created_at, archived_at
- USER_ACCOUNT: id, person_id, town_id, role (sys_admin|admin|staff|board_member), gov_title, permissions (JSONB), status, created_at, archived_at
- BOARD_MEMBERS: id, person_id, board_id, town_id, seat_title, term_start, term_end, status (active|archived), is_default_rec_sec, created_at

MUTUAL EXCLUSIVITY RULE (critical — from advisory 1.2):
- A PERSON's user_account role of 'staff' and 'board_member' are MUTUALLY EXCLUSIVE
- The system MUST prevent assigning both simultaneously
- Grounded in Maine conflict-of-interest law (30-A M.R.S.A. section 2605)
- A person must leave one role before assuming the other
- Allowed combinations: admin + board memberships, any role + resident_account

TASK 1: Create MemberRoster component

  - Term (term_start — term_end formatted as dates, or "No term set")
  - Role indicator: Badge showing "Staff" (blue) or "Board Member" (green)

TASK 2: Create AddMemberDialog

  Step 1 — Find or Create Person:

  - Role selection: radio buttons for "Board Member" or "Staff"
  - The admin role is not selectable here — admins are created separately

TASK 3: Implement mutual exclusivity enforcement
- Before allowing role assignment, check the PERSON's existing user_account
- If PERSON has user_account with role='staff' and user tries to add as board_member:
  - Show AlertDialog with clear explanation:
    - Title: "Role Conflict — Staff and Board Member"
    - Body: "[Person Name] currently has a staff account. Under Maine conflict-of-interest law (30-A M.R.S.A. section 2605), a person cannot simultaneously serve as staff and a board member. To add [Person Name] as a board member, their staff account must be archived first."
    - Actions: "Archive Staff Account & Continue" or "Cancel"
  - If user confirms: archive staff user_account, then proceed with board_member creation
- Same logic in reverse: if PERSON is board_member and user tries to create staff account
- Also check during MemberTransitionDialog (Task 9)
- This enforcement must be implemented in packages/shared as a utility function checkRoleMutualExclusivity(person, targetRole) so it can be reused

TASK 4: Create staff account creation flow
- File: packages/web/src/components/members/StaffAccountFlow.tsx
- Step 1: Select permission template from 5 options (from advisory 1.2):
  1. Town Clerk — Full operational access (A1-A6, M1-M7, R1-R6, C1-C5 on all boards)
  2. Deputy Clerk — Minutes/records focused (A2,A3,A6, M1-M5, R1-R4,R6 on all boards)
  3. Board-Specific Staff — Full operational on designated boards only (A1-A6, M1-M7, R1-R6 on specific boards)
  4. General Staff — View-oriented (A3, R4,R6 on all boards)
  5. Recording Secretary Only — Meeting recording on designated boards (M2-M5, R1-R4,R6 on specific boards)
- Show template description and summary of included permissions
- Step 2: Customize with PermissionMatrixEditor (Task 5)
- Step 3: If template is Board-Specific or Recording Secretary Only, select which boards the permissions apply to
- On complete: create USER_ACCOUNT with role='staff', populated permissions JSONB, link to PERSON

TASK 5: Create PermissionMatrixEditor component
- File: packages/web/src/components/members/PermissionMatrixEditor.tsx
- Visual grid organized by functional area (from advisory 1.2 permissions matrix):
  - Town & System (T1-T4): manage town profile, user accounts, board config, permissions
    - NOTE: T1-T4 are admin-only and CANNOT be delegated. Show them grayed out with a lock icon and tooltip "Admin only"
  - Agenda & Meeting Prep (A1-A7): create meeting, build agenda, upload attachments, publish agenda, generate packet
  - Live Meeting Operations (M1-M8): start meeting, attendance, motions, recusal, rec sec, exec session, speaker queue, vote
  - Minutes & Records (R1-R6): edit draft, generate AI draft, submit for review, view draft, publish, export
  - Civic Engagement (C1-C5): straw polls, notifications, resident accounts, moderate comments, portal config
  - View & Download (V1-V5): view agenda, download PDF, view minutes, calendar, directory — these are always Y for everyone
- Each action row has three toggle states:
  - Y (allowed on all boards) — green check
  - N (denied) — red X
  - Board-specific (allowed on designated boards only) — blue icon with board selector
- When "Board-specific" is selected, show a multi-select of active boards for that action
- Pre-fill from selected template; admin can adjust any permission
- Store as JSONB structure on user_account.permissions:

    "global": { "A1": true, "A2": true, "M1": false, ... },
    "board_overrides": {
      "[board_id]": { "A1": true, "M1": true, ... }
    }
  }

TASK 6: Implement per-board permission overrides
- The JSONB permissions structure supports global defaults + per-board overrides
- Create a utility function hasPermission(permissions, action, boardId) in packages/shared:
  - If board_overrides[boardId] has the action defined, use that value
  - Otherwise, fall back to global[action]
  - Admin role always returns true (full access)
  - Board member role: check the fixed board_member permissions (V1-V5: Y, A4: Y, A7: Y, M8: Y, R4: Y default, R6: Y approved only)
- Create a PermissionOverrideView component that shows a staff member's effective permissions per board
- Display on the member detail view: a tab or accordion per board showing which permissions differ from global

TASK 7: Create board_member account configuration

TASK 8: Implement recording secretary default

TASK 9: Create MemberTransitionDialog
- File: packages/web/src/components/members/MemberTransitionDialog.tsx
- Handle these transitions (from advisory 1.2 lifecycle):

  - Staff -> Board member: mutual exclusivity check, archive staff account, create board_member
```
