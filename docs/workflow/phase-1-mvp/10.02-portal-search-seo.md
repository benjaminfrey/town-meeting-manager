# Session 10.02 — Public Portal: Search & SEO

**Phase:** 1 — MVP | **Block:** 10 — Public Portal
**Dependencies:** 10.01
**Estimated tasks:** 8

---

## Description

Add full-text search across published agendas and approved minutes on the public portal, plus comprehensive SEO optimization including meta tags, sitemaps, structured data, and robots.txt. Search uses PostgreSQL's native full-text search (tsvector/tsquery) to enable residents to find specific meeting content by keyword. SEO ensures meeting records are discoverable via search engines, increasing government transparency.

## Tasks

1. Implement full-text search across published agendas and approved minutes using PostgreSQL full-text search (tsvector/tsquery)
2. Create search UI on public portal: search bar with results showing meeting title, date, matched content snippet
3. Create search results page with filters (date range, board, content type: agenda/minutes)
4. Implement SEO meta tags per page: title, description, Open Graph tags for social sharing
5. Create sitemap.xml generation per town (dynamic, includes all published agendas and approved minutes URLs)
6. Create robots.txt per town subdomain
7. Implement structured data (JSON-LD) for government organization and meeting events (Schema.org)
8. Verify: search returns relevant results, meta tags present, sitemap generates, structured data validates

```

- Portal content: only published agendas and approved/published minutes are searchable

TASK 1: Implement PostgreSQL full-text search
- Create a full-text search infrastructure in PostgreSQL:
  - Add a tsvector column (search_vector) to the MINUTES_DOCUMENT table
  - Add a tsvector column (search_vector) to the AGENDA_ITEM table
  - Create a GIN index on each search_vector column for fast full-text queries
  - Create a trigger function that automatically updates search_vector when content changes:
    - For MINUTES_DOCUMENT: build the tsvector from content_json text content (extract all text from the JSON structure — section titles, discussion summaries, motion texts, member names)
    - For AGENDA_ITEM: build from title, background, recommendation, suggested_motion
  - Use the 'english' text search configuration for stemming and stop words
  - Create a Supabase migration with all schema changes, indexes, and triggers
- Create a search function in PostgreSQL (or Fastify):
  - portal_search(town_id UUID, query TEXT, content_type TEXT, board_id UUID, date_from DATE, date_to DATE)
  - Returns: matching records with headline() snippets showing the matched text in context
  - Uses ts_rank() to order results by relevance
  - Filters by town_id (mandatory), content_type (agenda/minutes/all), board_id (optional), date range (optional)
  - Only searches published agendas and approved/published minutes (enforced in the query WHERE clause)
- Create a Fastify API endpoint: GET /api/portal/:townId/search
  - Query params: q (search query, required), type (agenda|minutes|all, default: all), board (board_id, optional), from (date, optional), to (date, optional), page (pagination, default: 1), limit (default: 20)
  - Returns: { results: [{ type, meeting_id, meeting_date, board_name, title, snippet, url }], total, page, pages }
  - The snippet is generated by PostgreSQL's ts_headline() function, showing matched terms with <mark> tags for highlighting
  - Validate and sanitize the search query to prevent injection (use parameterized queries)

TASK 4: Implement SEO meta tags
- Create a meta tag system for the portal that sets page-specific metadata:
  - File: packages/web/src/lib/portal/seo.ts

  - Agenda view: title="[Board Name] Agenda - [Meeting Date] - Town of [Name]", description="Published agenda for [Board Name] meeting on [date]"

- Open Graph tags for social sharing:

- Twitter Card tags: twitter:card="summary", twitter:title, twitter:description
- Canonical URL: <link rel="canonical" href="..."> for each page
- Use React Router v7's meta export to set these tags per route (or React Helmet / document head API)

TASK 5: Create dynamic sitemap.xml
- Create a Fastify API endpoint: GET /api/portal/:townId/sitemap.xml
  - Content-Type: application/xml
  - Generates a sitemap following the Sitemap Protocol (sitemaps.org)
  - Include all publicly accessible portal URLs:
    - Portal home: / (priority: 1.0, changefreq: weekly)
    - Meetings list: /meetings (priority: 0.8, changefreq: daily)
    - Each meeting with a published agenda: /meetings/:meetingId/agenda (priority: 0.6, changefreq: monthly)
    - Each meeting with approved minutes: /meetings/:meetingId/minutes (priority: 0.7, changefreq: never — approved minutes don't change)
    - Board directory: /boards (priority: 0.7, changefreq: monthly)
    - Each active board: /boards/:boardId (priority: 0.5, changefreq: monthly)
    - Calendar: /calendar (priority: 0.6, changefreq: daily)
  - Each URL entry includes: <loc>, <lastmod>, <changefreq>, <priority>
  - <lastmod> uses the actual last-modified date of the content (published_at for agendas, approved_at for minutes)
  - Only include URLs for published/approved content
- The Nginx configuration should serve the sitemap at /sitemap.xml by proxying to this API endpoint
- Cache the sitemap XML with a TTL of 1 hour (regenerate on demand but cache the result)

TASK 6: Create robots.txt
- Create a Fastify API endpoint: GET /api/portal/:townId/robots.txt
  - Content-Type: text/plain
  - Content:
    ```
    User-agent: *
    Allow: /
    Disallow: /api/
    Sitemap: https://[subdomain].townmeetingmanager.com/sitemap.xml
    ```
  - The sitemap URL includes the correct subdomain for the resolved town
- Nginx serves this at /robots.txt by proxying to the API endpoint
- For unknown subdomains (no matching town), return:
  ```
  User-agent: *
  Disallow: /
  ```

TASK 7: Implement Schema.org structured data (JSON-LD)
