# Session 12.01 — RBAC UI Enforcement & Route Guards

**Phase:** 1 — MVP | **Block:** 12 — RBAC & Permissions
**Dependencies:** 04.03, 07.03, 09.02
**Estimated tasks:** 10

---

## Description

Implement comprehensive role-based access control enforcement across the entire web application UI. This creates the usePermission hook, PermissionGate component, and route-level guards that hide or disable UI elements based on the current user's JSONB permissions matrix. Every page, button, and menu item is audited against the 30-action permissions matrix defined in advisory 1.2, with board-scoped permission checks for staff members who have different access levels per board.

## Tasks

1. Create usePermission(actionCode) hook: checks current user's JSONB permissions for a specific action, with optional board_id scope
2. Create PermissionGate component: conditionally renders children based on permission check (wraps action buttons, menu items, etc.)
3. Implement route-level guards: protect routes based on role and permissions (admin-only routes, staff routes based on matrix, board-member routes)
4. Hide UI elements the user cannot access: menu items, action buttons, create/edit controls
5. Implement board-scoped permission checks: staff member sees different UI per board based on their per-board permissions
6. Create "Access Denied" page for direct URL access without permission
7. Audit all existing pages and components for proper permission gates: town settings (T1-T4), agenda building (A1-A6), meeting operations (M1-M7), minutes (R1-R6), civic (C1-C5)
8. Ensure admin always has full access (bypass permission checks for admin role)
9. Ensure view permissions (V1-V5) are always granted for all authenticated users
10. Verify: staff with limited permissions sees restricted UI, board-specific permissions work, direct URL access blocked, admin sees everything

## Prompt

```
You are implementing RBAC (role-based access control) UI enforcement across the Town Meeting Manager web application — a municipal meeting management platform for Maine towns. Read docs/advisory-resolutions/1.2-roles-permissions.md for the complete permissions model: 4 roles (sys_admin, admin, staff, board_member), 30 governable actions across 5 functional areas (T1-T4, A1-A7, M1-M8, R1-R6, C1-C5, V1-V5), JSONB permissions storage, and board-scoped permission overrides for staff.

PROJECT CONTEXT:
- Monorepo: packages/web (React 19 + React Router v7 Framework Mode), packages/shared (types, schemas)
- TanStack Query v5 — user permissions loaded via useCurrentUser hook and cached by React Query
- Tailwind CSS v4 + shadcn/ui for components
- User permissions stored as JSONB on USER_ACCOUNT: { "global": { "A1": true, "A2": true, ... }, "boards": { "board-uuid-1": { "A1": true, "A2": false, ... } } }
- Effective permission for a board action: board-specific override if exists, else global default, else deny
- Admin role always has full access — bypass all permission checks
- Board member role has fixed permissions per advisory 1.2 (A4, A7, M8, R4 default Y, R6 approved only, V1-V5 always)
- View permissions (V1-V5) are always granted for all authenticated users
- The permissions matrix was configured by the admin during user account setup (session 04.03)

TASK 1: Create usePermission hook
- File: packages/web/src/hooks/usePermission.ts
- Create a hook: usePermission(actionCode: string, boardId?: string): { allowed: boolean, loading: boolean }
  - Reads the current user's role and permissions from the auth context (useCurrentUser hook, cached by React Query)
  - Permission resolution logic:
    1. If user role is 'admin': return { allowed: true } — admin bypasses all checks
    2. If user role is 'sys_admin': return { allowed: true } — sys_admin also bypasses
    3. If actionCode is V1-V5: return { allowed: true } — view permissions always granted
    4. If user role is 'board_member':
       - Check the fixed board_member permissions from advisory 1.2:
         - A4 (upload files for admin review): always allowed
         - A7 (submit questions/comments): always allowed
         - M8 (vote during meeting): always allowed if member of the board
         - R4 (view draft minutes): configurable, default Y
         - R6 (export minutes): allowed for approved minutes only
         - All other actions: check the user's JSONB permissions (admin may have granted additional permissions like A2 or M5)
       - If boardId provided: check that the user is an active member of that board
    5. If user role is 'staff':
       - If boardId provided: check board-specific override first (permissions.boards[boardId][actionCode])
       - If no board override: check global default (permissions.global[actionCode])
       - If neither exists: denied
    6. For any unrecognized role or missing permissions: denied
  - The hook should be fast — it reads from the React Query cache (no network round-trip after initial load)
  - Return loading: true while the user data is being loaded

- Create a companion hook: usePermissions(actionCodes: string[], boardId?: string): Record<string, boolean>
  - Checks multiple permissions at once (useful for pages that need to show/hide multiple elements)
  - Returns an object mapping each action code to its allowed status

- Create a utility function (non-hook, for use in clientLoader):
  - checkPermission(userAccount: UserAccount, actionCode: string, boardId?: string): boolean
  - Same logic as the hook but takes the user account as a parameter instead of reading from context
  - Used in clientLoader functions where hooks are not available

TASK 2: Create PermissionGate component
- File: packages/web/src/components/auth/PermissionGate.tsx
- A component that conditionally renders children based on permission:
  ```tsx
  <PermissionGate action="A2" boardId={boardId}>
    <Button onClick={handleEditAgenda}>Edit Agenda</Button>
  </PermissionGate>
  ```
- Props:
  - action: string (required) — the action code to check (e.g., "A2", "M1", "R5")
  - boardId?: string — optional board scope for the permission check
  - fallback?: ReactNode — optional content to render if permission is denied (default: render nothing)
  - showDisabled?: boolean — if true, render children but disabled/grayed out instead of hiding (useful for menu items where the user should know the action exists but they can't use it)
- Implementation:
  - Uses usePermission internally
  - If allowed: render children
  - If not allowed and showDisabled: render children with disabled styles (pointer-events: none, opacity: 0.5) and a tooltip: "You don't have permission to [action description]"
  - If not allowed and no showDisabled: render fallback or nothing
- Create a lookup for human-readable action descriptions:
  - File: packages/shared/src/constants/permission-labels.ts
  - Map each action code to a description: { A1: "Create a new meeting", A2: "Build or edit agenda items", ... }
  - Used in tooltips and the Access Denied page

TASK 3: Implement route-level guards
- Create a route guard component:
  - File: packages/web/src/components/auth/RouteGuard.tsx
  - Wraps route components to check permissions before rendering
  - Props: requiredRole?: string, requiredAction?: string, requiredBoardId?: string
- Create a clientLoader guard utility:
  - File: packages/web/src/lib/auth/route-guard.ts
  - Function: requirePermission(userAccount, action, boardId?): throws redirect to /access-denied if not allowed
  - Used in clientLoader functions:
    ```ts
    export async function clientLoader() {
      const user = await getCurrentUser();
      requirePermission(user, 'T1'); // admin-only route
      return { ... };
    }
    ```
- Route protection mapping:
  - /admin/* routes: require admin role
  - /admin/notifications: require admin role
  - /settings/town: require T1 permission
  - /boards/create, /boards/:boardId/edit: require T3 permission
  - /meetings/:meetingId/agenda/edit: require A2 permission + board scope
  - /meetings/:meetingId/run: require M1 permission + board scope
  - /meetings/:meetingId/minutes: require R1 permission for edit mode, R4 for view mode
  - /meetings/:meetingId/minutes (publish action): require R5 permission
- If the user navigates directly to a protected URL without permission, redirect to /access-denied with context about what permission is needed

TASK 4: Hide UI elements based on permissions
- Audit the main navigation / sidebar:
  - "Town Settings" menu item: visible only to admin (T1)
  - "User Management" menu item: visible only to admin (T2)
  - "Notifications" admin dashboard: visible only to admin

- Audit action buttons throughout the app:

TASK 5: Implement board-scoped permission checks
- Staff members can have different permissions per board (per advisory 1.2):
  - Example: Town Planner has full access to Planning Board but view-only on Select Board
  - Example: Deputy Clerk has minutes access on all boards but cannot publish agendas
- When a staff member navigates to a board-specific page:
  - The permission check uses the board's ID to resolve board-specific overrides
  - If the staff member has a board override for the action, use it
  - If not, fall back to their global default for that action
- UI behavior per board:
  - On the boards list page: show all boards, but action buttons (create meeting, edit agenda) are gated per board
  - On a board detail page: the action bar shows only the buttons the user can use for this specific board
  - On the meeting list: filter actions per meeting's board
- Create a helper: getUserBoardPermissions(userAccount, boardId): returns the effective permissions for a specific board
  - Merges global defaults with board-specific overrides
  - Used by the UI to determine which buttons/actions to show
