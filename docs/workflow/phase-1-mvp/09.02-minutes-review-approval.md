# Session 09.02 — Minutes Review & Approval Workflow

**Phase:** 1 — MVP | **Block:** 09 — Minutes
**Dependencies:** 09.01
**Estimated tasks:** 11

---

## Description

Build the minutes review and approval workflow that takes a generated minutes draft through editing, board review, formal approval via motion at the next meeting, and publication to the public portal. This implements the Maine legal requirement that minutes are approved by the board as a body (not by a single person), with the approval recorded as a formal vote on the next meeting's agenda.

## Tasks

1. Create MinutesReviewPage at /meetings/:meetingId/minutes — shows generated minutes with editing tools
2. Create inline editor for minutes sections: edit text, add/remove content, correct names/times
3. Implement tracked changes: show modifications from the generated draft (visual diff)
4. Create minutes status workflow: draft → review → approved → published
5. Implement "Submit for Board Review" action: makes minutes visible to board members (based on R4 permission)
6. Create board-member minutes review view: read-only view of draft minutes
7. Implement minutes approval as agenda item: create "Approval of Minutes from [date]" item on next meeting's agenda
8. When the approval motion passes, automatically update minutes status to "approved" and lock the document
9. Create "Publish to Portal" action for approved minutes (requires R5 permission)
10. Implement amendment workflow: if board requests changes during approval, minutes go back to draft, edits applied, re-submitted
11. Verify: generate draft → review/edit → submit for review → approve via motion → publish, amendment cycle works

## Prompt

```
You are building the minutes review and approval workflow for the Town Meeting Manager — a municipal meeting management platform for Maine towns. This follows session 09.01 which created the minutes generation pipeline. Read docs/advisory-resolutions/2.2-ai-minutes-architecture.md section 5 (Review & Approval) and docs/advisory-resolutions/1.2-roles-permissions.md for the minutes approval process (it is a board vote, not a single-person action). Also reference the permissions matrix: R1 (edit draft minutes), R3 (submit for board review), R4 (view draft minutes), R5 (publish approved minutes), R6 (export minutes PDF/DOCX).

PROJECT CONTEXT:
- Monorepo: packages/web (React 19 + React Router v7 Framework Mode), packages/api (Fastify), packages/shared (types, schemas)
- PowerSync for offline-first local SQLite — use @powersync/react hooks (useQuery)
- Tailwind CSS v4 + shadcn/ui for components
- MINUTES_DOCUMENT table created in 09.01 with status: draft → review → approved → published
- Minutes stored as structured JSON (content_json) with rendered HTML (html_rendered) and PDF path
- Per advisory 1.2: "Minutes are approved by the board, not by a single person" — approval requires a formal vote at a meeting
- The review UX per advisory 2.2: split-panel editor with rich text on one side and source data on the other

TASK 1: Create MinutesReviewPage
- Route: /meetings/:meetingId/minutes
- File: packages/web/src/routes/meetings.$meetingId.minutes.tsx
- Use clientLoader to load the MINUTES_DOCUMENT record and the associated MEETING data from PowerSync
- Page layout:
  - Header: board name, meeting date, minutes status badge (Draft/Under Review/Approved/Published)
  - Action bar: status-dependent action buttons (Edit, Submit for Review, Publish, Export PDF, Regenerate)
  - Main content area: rendered minutes (from html_rendered) displayed in a readable format
  - Breadcrumb: Dashboard > Boards > [Board Name] > Meetings > [Meeting Date] > Minutes
- If no minutes document exists, show a prompt to generate one (link back to post-meeting review page)
- If minutes are approved or published, show them in read-only mode with a locked banner

TASK 2: Create inline editor for minutes sections
- File: packages/web/src/components/minutes/MinutesEditor.tsx
- When the user clicks "Edit" (requires R1 permission), switch from read-only to edit mode
- The editor is a split-panel layout (per advisory 2.2 section 5):
  - LEFT panel: rich text editor for the minutes content. Use a lightweight rich text editor — either TipTap or a contentEditable implementation with basic formatting (bold, italic, lists)
  - RIGHT panel: source data panel showing the original structured meeting data for the current section (motions, votes, attendance, timestamps). This panel is read-only reference material.
- The editor operates on the content_json — edits modify the structured JSON, and the HTML is regenerated on save
- Section-by-section editing: the user can click into any section to edit it. The right panel shows the corresponding source data for that section.
- Editing capabilities:
  - Edit discussion summary text for each agenda item
  - Correct names, times, and titles
  - Add or remove content within a section
  - Reorder content within a section (move paragraphs up/down)
  - Cannot modify motions or vote counts directly in the editor — these are sourced from the structured meeting data and must be corrected there if wrong
- Save button writes the updated content_json to the MINUTES_DOCUMENT record and regenerates html_rendered
- Auto-save draft every 30 seconds while editing (write to PowerSync local, queued for sync)

TASK 3: Implement tracked changes
- File: packages/web/src/components/minutes/TrackedChanges.tsx
- Track modifications between the originally generated draft and the current edited version
- Store the original content_json as original_content_json on the MINUTES_DOCUMENT (set at generation time, never modified)
- Compare current content_json with original_content_json to produce a visual diff:
  - Added text: highlighted in green with a "+" marker
  - Removed text: shown in red with strikethrough
  - Modified text: show both old and new versions
- Toggle button: "Show Changes" / "Hide Changes" — default is to show changes during editing, hide during final review
- The diff is computed client-side using a text diffing library (e.g., diff-match-patch or similar)
- Changes are informational — they help the reviewer see what was modified from the generated draft but do not need to be individually accepted/rejected (this is not a track-changes workflow like Word)

TASK 4: Create minutes status workflow
- Status transitions with guards:
  - draft → review: requires R3 permission (Submit for Board Review). Sets submitted_for_review_at timestamp.
  - review → draft: if board requests amendments, admin returns to draft. Clears submitted_for_review_at.
  - review → approved: ONLY via a formal motion passing at a meeting (see Task 8). Sets approved_at and approved_by_motion_id.
  - approved → published: requires R5 permission (Publish to Portal). Sets published_at timestamp.
  - approved → draft: NOT ALLOWED. Once approved, minutes are locked.
  - published → unpublished: requires admin role. Removes from portal but keeps approved status.
- Display the current status as a prominent Badge on the MinutesReviewPage
- Show a status timeline: "Generated [date] → Edited [date] → Submitted for review [date] → Approved [date] → Published [date]"
- When status is "approved" or "published," the content_json is locked — no edits allowed

TASK 5: Implement "Submit for Board Review" action
- On the MinutesReviewPage, when status is "draft" and user has R3 permission, show a "Submit for Board Review" button
- On click: confirmation dialog explaining what happens:
  - "Submit these minutes to board members for review before the next meeting?"
  - "Board members with R4 permission will be able to view the draft."
- On confirm:
  - Update MINUTES_DOCUMENT status to "review"
  - Set submitted_for_review_at timestamp
  - Trigger a notification event: minutes_review (this dispatches email to board members via the notification system built in Block 11)
  - Regenerate the HTML and PDF (in case edits were made since last generation)
  - Show success toast: "Minutes submitted for board review"

TASK 6: Create board-member minutes review view
- File: packages/web/src/routes/meetings.$meetingId.minutes-review.tsx
- Route: /meetings/:meetingId/minutes-review (or use the same route with a different view mode)
- This page is visible to users with R4 permission (default: board members can view draft minutes)
- Read-only view of the minutes HTML — no editing capabilities
- Show "DRAFT — Pending Board Approval" banner at the top
- Include buttons:
  - "Download PDF" (if user has R6 permission)
  - "Back to Meeting" link
- Board members access this from:
  - A link in the minutes_review notification email
  - The meeting detail page (if the meeting has minutes in "review" status)
- Do NOT show this page if minutes are still in "draft" status (not yet submitted for review)

TASK 7: Implement minutes approval as agenda item
- When minutes are in "review" status, the system should facilitate placing them on the next meeting's agenda
- Create a helper function: createMinutesApprovalAgendaItem(meetingId, nextMeetingId)
  - Creates an agenda item titled "Approval of Minutes from [Meeting Date]"
  - Places it in the "Minutes of Previous Meeting(s)" section (minutes_approval section type)
  - Sets the section_ref (e.g., "3A")
  - Attaches the minutes PDF as an exhibit
  - Pre-populates the suggested_motion: "to approve the minutes of the [Board Name] meeting of [date] as presented"
- The auto-generated minutes_approval section (per advisory 2.2 section 1) should automatically query for minutes in "review" status for this board and list them as sub-items
- When building a new agenda, the system checks for any pending minutes and adds them automatically
- The admin can remove or modify these auto-generated items if needed

TASK 8: Auto-approve minutes when the approval motion passes
- During a live meeting, when the "Approval of Minutes from [date]" agenda item is processed and a motion to approve passes:
  - The system identifies the linked MINUTES_DOCUMENT record
  - Updates the status from "review" to "approved"
  - Sets approved_at to the current timestamp
  - Sets approved_by_motion_id to the motion that passed
  - Locks the content_json — no further edits allowed
  - Regenerates the final HTML and PDF (without DRAFT watermark)
  - Triggers a notification event: minutes_approved (dispatches email via Block 11)
- If the motion includes "as amended" (motion text contains "as amended" or "with corrections"):
  - Set a flag: approved_as_amended = true
  - The amendments should have been applied before the vote (see Task 10)
  - The final PDF includes "Approved as amended on [date]" in the footer
- If the motion fails or is tabled:
  - The minutes remain in "review" status
  - The operator can initiate the amendment workflow (Task 10)

TASK 9: Create "Publish to Portal" action
- On the MinutesReviewPage, when status is "approved" and user has R5 permission, show a "Publish to Portal" button
- On click: confirmation dialog:
  - "Publish these approved minutes to the public portal?"
  - "They will be visible at [town subdomain].townmeetingmanager.com"
- On confirm:
  - Update MINUTES_DOCUMENT status to "published"
  - Set published_at timestamp
  - The public portal (Block 10) reads from this status to determine what to display
  - Trigger a notification event: minutes_published (optional — for subscribers who want to know when new minutes are posted)
  - Show success toast: "Minutes published to public portal"
- The published minutes are served as HTML on the portal and as PDF for download
- Add an "Unpublish" action (admin only) that removes from the portal but preserves the approved status

TASK 10: Implement amendment workflow
- If the board requests changes during the approval discussion (before or after a failed vote):
  - The admin clicks "Request Amendments" on the minutes
  - Confirmation dialog: "Return these minutes to draft for amendments? They will be re-submitted for review at a future meeting."
  - On confirm:
    - Status reverts from "review" to "draft"
    - A notes field records: "Returned for amendments on [date] during [Meeting Name]. Board requested: [admin enters description of requested changes]"
    - The admin edits the minutes (Task 2) to apply the requested changes
    - The admin re-submits for review (Task 5)
    - The minutes appear on the next meeting's agenda for re-approval
- The amendment history is tracked: each round of amendments adds an entry to an amendments_history JSONB array on the MINUTES_DOCUMENT:
  - { round: 1, returned_at: timestamp, reason: "Board requested correction to vote count on Item 6A", resubmitted_at: timestamp }
- The final approved minutes include an amendment note if applicable: "Minutes approved as amended on [date]"

TASK 11: Verification
- Generate a minutes draft from a completed meeting (via session 09.01)
- Open the MinutesReviewPage — minutes display correctly in read-only mode
- Click Edit — split-panel editor opens with minutes content on left, source data on right
- Edit a discussion summary and a name correction — save, verify changes persist
- Toggle "Show Changes" — visual diff shows modifications from the original draft
- Click "Submit for Board Review" — status changes to "review," notification event fires
- Log in as a board member with R4 permission — verify the review view is accessible and read-only
- Board member without R4 permission — verify access is denied
- Create a new meeting with the same board — verify "Approval of Minutes from [date]" appears automatically on the agenda in the minutes_approval section
- During the live meeting, process the minutes approval item — record a motion to approve, pass the vote
- Verify: MINUTES_DOCUMENT status changes to "approved," content is locked, DRAFT watermark removed from HTML/PDF
- Click "Publish to Portal" — status changes to "published," published_at is set
- Test the amendment cycle: return minutes to draft, edit, re-submit, re-approve
- Verify amendments_history tracks each round
- Verify "Unpublish" action (admin only) removes from portal but keeps approved status

VERIFICATION CHECKLIST:
1. MinutesReviewPage renders minutes with correct status badge
2. Split-panel editor allows section-by-section editing with source data reference
3. Tracked changes show visual diff between original and edited content
4. Status workflow enforces correct transitions (draft → review → approved → published)
5. "Submit for Board Review" requires R3 permission and triggers notification
6. Board member review view is read-only and only visible when status is "review"
7. Minutes approval agenda item auto-generates on next meeting's agenda
8. Passing the approval motion auto-updates status to "approved" and locks content
9. "Publish to Portal" requires R5 permission and sets published_at
10. Amendment workflow returns to draft, tracks history, and supports re-approval
11. Approved/published minutes cannot be edited (locked state)
```
