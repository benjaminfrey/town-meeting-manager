# Session 02.02 — PowerSync Web Integration

**Phase:** 1 — MVP | **Block:** 02 — Web Application Foundation
**Dependencies:** 02.01, 01.04, 01.08
**Estimated tasks:** 10

---

## Description

Integrate PowerSync into the web application as the offline-first sync engine. Configure the PowerSync database with SQLite WASM, create the SupabaseConnector for authentication and upload handling, set up Kysely for type-safe SQL queries, and build the sync status UI. This establishes the data layer that replaces traditional API fetching and state management.

## Tasks

1. Install PowerSync web packages: @powersync/web, @powersync/react, @powersync/kysely-driver
2. Install Supabase client: @supabase/supabase-js
3. Create SupabaseConnector class implementing PowerSyncBackendConnector (fetchCredentials using Supabase auth session, uploadData for write queue)
4. Create PowerSync schema definition (AppSchema) importing table definitions from @town-meeting/shared
5. Create PowerSyncProvider wrapper component that initializes PowerSync with the web database (WASQLiteOpenFactory or WASQLitePowerSyncDatabaseOpenFactory)
6. Configure Kysely instance with PowerSync driver for type-safe queries
7. Create useSupabase() and usePowerSync() context hooks
8. Create a SyncStatusBar component showing online/offline status, pending uploads count (from ps_crud table), sync state
9. Add PowerSyncProvider to the root layout, wrapping all routes
10. Verify: PowerSync initializes, connects to local SQLite, sync status displays correctly, Kysely queries work

TASK 3: Create SupabaseConnector
- Create src/lib/powersync/SupabaseConnector.ts
- The connector implements PowerSyncBackendConnector interface from @powersync/web:

  fetchCredentials():
  - Get the current Supabase auth session via supabase.auth.getSession()
  - If no session exists, throw an error (user must be logged in before PowerSync initializes)
  - Return: { endpoint: VITE_POWERSYNC_URL, token: session.access_token, expiresAt: decoded JWT expiry }
  - PowerSync calls this automatically when the token expires — Supabase auto-refreshes the token via its onAuthStateChange listener

  uploadData(database):
  - This processes the local write queue (ps_crud table) — called by PowerSync when local writes need to sync to Supabase
  - For each transaction in the upload queue:
    - Read the operation (PUT, PATCH, DELETE) and the table/row data
    - Map to Supabase PostgREST operations:
      - PUT → supabase.from(table).upsert(record)
      - PATCH → supabase.from(table).update(record).eq('id', id)
      - DELETE → supabase.from(table).delete().eq('id', id)
    - On success, mark the transaction as complete via database.setCrudProcessed()
    - On failure, throw the error — PowerSync will retry automatically
  - Handle conflict resolution: if the server rejects a write (409 Conflict), log the conflict and decide whether to retry or discard. For MVP, log and retry with exponential backoff.
