# Session 11.02 — Email Preferences & Account Invitation Emails

**Phase:** 1 — MVP | **Block:** 11 — Notifications
**Dependencies:** 11.01, 04.03
**Estimated tasks:** 8

---

## Description

Build the email preference management interface for authenticated users and implement the account invitation email flow. Users can toggle email notifications per event type. Admins invite new users via email with a magic link that leads to account setup. All broadcast emails include a CAN-SPAM compliant unsubscribe link. Meeting lifecycle events (created, cancelled, agenda published, minutes approved) are wired to trigger the appropriate notification events.

## Tasks

1. Create email preference management page for authenticated users: toggle email notifications per event type (meeting scheduled, agenda published, minutes approved, etc.)
2. Store preferences in SUBSCRIBER_NOTIFICATION_PREFERENCE table
3. Create account invitation email flow: admin clicks "Invite" on member → system sends invite-user email via Postmark → email contains magic link → recipient clicks → creates password → account activated
4. Implement invitation tracking: pending/sent/accepted/expired status
5. Create invitation resend capability
6. Create email unsubscribe link in all broadcast emails (CAN-SPAM compliance) → link updates subscriber preferences
7. Wire notification triggers to meeting lifecycle events: meeting created/cancelled → meeting-notice email, agenda published → agenda-published email, minutes approved → minutes-approved email
8. Verify: user can manage email preferences, invitation email sends and links work, unsubscribe works, lifecycle events trigger emails

```

- CAN-SPAM compliance requires an unsubscribe mechanism in all broadcast emails
- Email event types (Phase 1 broadcast): meeting_scheduled, meeting_cancelled, agenda_published, minutes_approved, minutes_published

TASK 1: Create email preference management page
- Route: /settings/notifications
- File: packages/web/src/routes/settings.notifications.tsx
- Accessible to all authenticated users (admin, staff, board members)
- Display a list of notification event types with toggle switches:
  - "Meeting Scheduled/Cancelled" — receive emails when meetings are scheduled or cancelled for boards you're associated with
  - "Agenda Published" — receive emails when an agenda is published for your boards
  - "Minutes Ready for Review" — receive emails when draft minutes are submitted for board review (board members only)
  - "Minutes Approved" — receive emails when minutes are approved for your boards
  - "Minutes Published" — receive emails when minutes are published to the public portal
  - "Admin Alerts" — receive system alerts (admin role only — this preference is shown only to admins)
- Each toggle maps to a SUBSCRIBER_NOTIFICATION_PREFERENCE record
- When a user changes a toggle:
  - Update the preference in PowerSync local SQLite (instant, queued for sync)
  - Show a toast: "Notification preference updated"
- Default state for new users: all applicable notifications enabled
- Include a section explaining what each notification does and who receives it
- Add a "Disable all broadcast notifications" master toggle at the bottom

TASK 2: Store preferences in SUBSCRIBER_NOTIFICATION_PREFERENCE table
- Create the table (Supabase migration):
  - id: UUID primary key
  - user_account_id: FK to USER_ACCOUNT
  - town_id: FK to TOWN (for RLS)
  - event_type: TEXT (matches the notification event type enum)
  - channel: TEXT (email for Phase 1; sms added in Phase 2)
  - enabled: BOOLEAN (default true)
  - updated_at: TIMESTAMP
  - UNIQUE constraint on (user_account_id, event_type, channel)
- Add RLS policies:
  - Users can read and update their own preferences only
  - Admin can read all preferences for their town (for debugging)
- Add the PowerSync schema definition in packages/shared for client-side sync
- When a new user account is created, initialize default preferences:
  - Create a function initializeDefaultPreferences(userAccountId, role):
    - Admin: all event types enabled
    - Staff: meeting_scheduled, meeting_cancelled, agenda_published, minutes_approved enabled
    - Board member: meeting_scheduled, meeting_cancelled, agenda_published, minutes_review, minutes_approved enabled
  - Call this function during user account creation (session 04.03)
- The notification pipeline (from 11.01) checks these preferences before dispatching

TASK 3: Create account invitation email flow
- The invitation flow per advisory 1.2:
  1. Admin navigates to the member management page and clicks "Invite" on a PERSON record that has a USER_ACCOUNT but no active login
  2. System creates an INVITATION record with a unique token and expiration (48 hours)
  3. System sends the invite-user email via Postmark with a magic link: https://app.townmeetingmanager.com/invite/accept?token=xxx
  4. Recipient clicks the link → lands on the account setup page
  5. Recipient creates a password (meets complexity requirements)
  6. Account is activated — user can log in
- Create a Fastify route: POST /api/invitations
  - Body: { personId, email, role, boardIds (optional) }
  - Requires T2 permission (Manage user accounts — admin only)
  - Creates the invitation record
  - Sends the invite-user email via the NotificationService
  - Returns the invitation record
- Create the INVITATION table (Supabase migration):
  - id: UUID primary key
  - town_id: FK to TOWN
  - person_id: FK to PERSON
  - email: TEXT
  - token: TEXT (unique, random, URL-safe — use crypto.randomBytes or UUID v4)
  - status: enum (pending, sent, accepted, expired, revoked)
  - role: TEXT (admin, staff, board_member)
  - invited_by: FK to USER_ACCOUNT (the admin who sent the invitation)
  - sent_at: TIMESTAMP
  - accepted_at: TIMESTAMP
  - expires_at: TIMESTAMP (48 hours from creation)
  - created_at: TIMESTAMP
- Create the invitation acceptance page:
  - Route: /invite/accept?token=xxx
  - File: packages/web/src/routes/invite.accept.tsx
  - Validate the token: call GET /api/invitations/validate?token=xxx
    - Check: token exists, status is "sent" or "pending," not expired
    - Return: invitation details (town name, role, inviter name) or error
  - If valid: show a welcome message and password creation form
    - "Welcome to [Town Name]'s Meeting Manager!"
    - "You've been invited as a [role] by [inviter name]."
    - Password field + confirm password field
    - Password requirements: minimum 8 characters, at least one uppercase, one lowercase, one number
  - On submit: POST /api/invitations/accept
    - Body: { token, password }
    - Server-side: validate token again, create/update the Supabase Auth account with the email and password, update USER_ACCOUNT to active, mark invitation as accepted
    - Redirect to the login page with a success message: "Your account is ready. Please log in."
  - If expired: show message "This invitation has expired. Please contact your administrator for a new one."
  - If already accepted: show message "This invitation has already been used."

TASK 4: Implement invitation tracking
- On the member management page (built in session 04.03), show invitation status for each user:
  - No invitation sent: show "Invite" button
  - Invitation pending/sent: show "Invited [date]" with status badge and "Resend" option
  - Invitation accepted: show "Active" status
  - Invitation expired: show "Expired" with "Resend" option
- Create a Fastify route: GET /api/invitations?townId=xxx
  - Returns all invitations for the town with status
  - Requires T2 permission
- On the admin dashboard, add an "Invitations" section or card:
  - "X pending invitations, Y accepted this week"
  - Link to the full invitation list

TASK 5: Create invitation resend capability
- On the member management page, when an invitation is pending, sent, or expired, show a "Resend" button
- On click: POST /api/invitations/:invitationId/resend
  - Revokes the old invitation (sets status to "revoked")
  - Creates a new invitation with a fresh token and expiration
  - Sends a new invite-user email
  - Returns the new invitation record
- Rate limit: maximum 3 resends per user per 24 hours (prevent abuse)
- Each resend generates a new token — old tokens become invalid

TASK 6: Create email unsubscribe link
- Per CAN-SPAM requirements, all broadcast emails must include an unsubscribe mechanism
- In the email templates (from 11.01), the footer of broadcast emails includes:
  - "Manage your notification preferences" link → https://app.townmeetingmanager.com/settings/notifications
  - "Unsubscribe from this type of email" link → one-click unsubscribe URL
- Create a one-click unsubscribe endpoint:
  - Route: GET /api/notifications/unsubscribe?token=xxx&event_type=xxx
  - The token is a signed JWT (or HMAC-signed URL parameter) containing the user_account_id
  - On access: disable the specified event_type for this subscriber
  - Redirect to a confirmation page: "You've been unsubscribed from [event type description] emails."
  - Include a link to manage all preferences: "Want to adjust other notifications? Log in to manage preferences."
- Generate the unsubscribe URL per subscriber per email:
  - In the EmailSenderService, generate a signed unsubscribe URL for each email sent
  - Include both the subscriber ID and event type in the signed token
  - The signature prevents unauthorized unsubscription (someone can't unsubscribe other people)
- Also support the List-Unsubscribe email header:
  - Add List-Unsubscribe and List-Unsubscribe-Post headers to broadcast emails
  - This enables one-click unsubscribe in email clients that support it (Gmail, Apple Mail)
  - Postmark supports custom headers via the Headers array in the API call

TASK 7: Wire notification triggers to meeting lifecycle events
- Create event trigger functions that fire at key moments in the meeting lifecycle:
  - File: packages/api/src/services/notification-triggers.ts
  - triggerMeetingNotice(meetingId, action: 'scheduled' | 'cancelled'):
    - Creates a notification event: meeting_scheduled or meeting_cancelled
    - Payload: { meetingId, boardName, meetingDate, meetingTime, location, action }
    - Subscribers: users associated with the meeting's board
  - triggerAgendaPublished(meetingId):
    - Creates a notification event: agenda_published
    - Payload: { meetingId, boardName, meetingDate, agendaUrl, portalUrl }
    - Subscribers: users associated with the meeting's board
  - triggerMinutesReview(minutesDocumentId):
    - Creates a notification event: minutes_review
    - Payload: { minutesId, boardName, meetingDate, reviewUrl }
    - Subscribers: board members with R4 permission
  - triggerMinutesApproved(minutesDocumentId):
    - Creates a notification event: minutes_approved
    - Payload: { minutesId, boardName, meetingDate, minutesUrl, portalUrl }
    - Subscribers: users associated with the meeting's board
- Integrate these triggers into the existing application flows:
  - When a meeting is created (session 05.01 or equivalent): call triggerMeetingNotice(meetingId, 'scheduled')
  - When a meeting is cancelled: call triggerMeetingNotice(meetingId, 'cancelled')
  - When an agenda is published (session 06.02): call triggerAgendaPublished(meetingId)
  - When minutes are submitted for review (session 09.02, Task 5): call triggerMinutesReview(minutesDocId)
  - When minutes are approved (session 09.02, Task 8): call triggerMinutesApproved(minutesDocId)
- Each trigger function includes the portal URL for the town so email recipients can click through to the public portal
```
