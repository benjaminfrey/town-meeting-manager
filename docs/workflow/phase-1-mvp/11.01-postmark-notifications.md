# Session 11.01 — Postmark Integration & Notification Event System

**Phase:** 1 — MVP | **Block:** 11 — Notifications
**Dependencies:** 04.01
**Estimated tasks:** 12

---

## Description

Integrate Postmark as the email delivery provider and build the notification event system that powers all email communication in the platform. This includes the event processing pipeline (event trigger to subscriber query to channel dispatch to delivery tracking), Postmark sender configuration with per-town branding, HTML email templates for all Phase 1 notification types, webhook handling for delivery/bounce/complaint callbacks, and an admin notification dashboard.

## Tasks

1. Install Postmark Node.js SDK (postmark package)
2. Create notification event system in packages/api: event triggers → subscriber query → channel dispatch → delivery tracking
3. Create NOTIFICATION_EVENT processing: when an event is created, find all subscribers for that event type + town, filter by channel preference and opt-in status
4. Create Postmark email sender service: template rendering, variable substitution, send via Postmark API
5. Create email templates (Handlebars HTML): meeting-notice, agenda-published, minutes-review, minutes-approved, admin-alert, invite-user, password-reset
6. Configure Postmark message streams: transactional (invites, password resets, review requests) and broadcast (meeting notices, agenda/minutes publication)
7. Implement per-town sender configuration: notifications@townname.townmeetingmanager.com
8. Create Postmark webhook handler: process bounce, delivery, and complaint callbacks → update NOTIFICATION_DELIVERY status
9. Create delivery tracking: store delivery status per notification per subscriber
10. Implement retry logic: failed deliveries retried with exponential backoff (3 attempts), permanent failures (hard bounce) update subscriber record
11. Create admin notification dashboard: show recent notifications, delivery status, bounce/complaint summary
12. Verify: send test email via Postmark, webhook processes delivery callback, retry logic works, admin dashboard shows delivery status

## Prompt

```
You are building the notification system for the Town Meeting Manager — a municipal meeting management platform for Maine towns. This integrates Postmark for email delivery and creates the event-driven notification pipeline. Read docs/advisory-resolutions/2.4-notification-providers.md for the complete Postmark integration architecture, message streams, email templates, and notification event system design.

PROJECT CONTEXT:
- Monorepo: packages/api (Fastify), packages/web (React 19 + React Router v7), packages/shared (types, schemas)
- Supabase PostgreSQL for persistence
- Postmark for email delivery (Phase 1 — email only; Twilio SMS deferred to Phase 2)
- Per-town sender: notifications@townname.townmeetingmanager.com (per advisory 4.4)
- Two message streams: transactional (invites, password resets, review requests) and broadcast (meeting notices, agenda/minutes publication)
- Seven Phase 1 email templates: invite-user, password-reset, meeting-notice, agenda-published, minutes-review, minutes-approved, admin-alert
- Postmark webhooks for delivery tracking: bounce, delivery, complaint callbacks
- Phase 1 subscribers are authenticated users only (board members, staff, admin) — no public subscriber management yet

TASK 1: Install Postmark SDK
- Install in packages/api: pnpm --filter @town-meeting/api add postmark
- The Postmark npm package provides: ServerClient for sending emails, AccountClient for account-level operations
- Create a Postmark client factory:
  - File: packages/api/src/lib/postmark.ts
  - Initialize with the server API token from environment: POSTMARK_SERVER_TOKEN
  - For multi-town deployments, each town can have its own Postmark server token (stored encrypted in TOWN_NOTIFICATION_CONFIG table) or share a single server with per-town sender signatures
  - Export a function getPostmarkClient(townId: string) that retrieves the appropriate API token and returns a configured ServerClient instance

TASK 2: Create the notification event system
- File: packages/api/src/services/notification-service.ts
- Create a NotificationService class implementing the event pipeline from advisory 2.4:
  - createNotificationEvent(eventType, townId, payload): Creates a NOTIFICATION_EVENT record and triggers processing
  - processNotificationEvent(eventId): Main processing function:
    1. Load the event record
    2. Find all subscribers for this event_type + town_id
    3. Filter by channel preference (email enabled) and opt-in status
    4. For each subscriber: create a NOTIFICATION_DELIVERY record and dispatch via the appropriate channel
    5. Mark the event as processed
  - dispatchEmail(delivery, template, variables): Renders the email template and sends via Postmark
- Database tables (create Supabase migration):
  - NOTIFICATION_EVENT: id, town_id, event_type (enum), payload (JSONB), status (pending/processing/completed/failed), created_at, processed_at
  - NOTIFICATION_DELIVERY: id, event_id, subscriber_id (FK to USER_ACCOUNT), channel (email/sms), status (pending/sent/delivered/bounced/failed), postmark_message_id, sent_at, delivered_at, error_message, retry_count, next_retry_at
  - TOWN_NOTIFICATION_CONFIG: id, town_id, postmark_server_token (encrypted), postmark_sender_email, postmark_sender_name, created_at, updated_at
- Event types enum (Phase 1):
  - meeting_scheduled, meeting_cancelled, agenda_published, minutes_review, minutes_approved, minutes_published, admin_alert, user_invited, password_reset

TASK 3: Create NOTIFICATION_EVENT processing pipeline
- When a notification event is created, process it asynchronously:
  - Use a Fastify background task or a simple queue mechanism (for MVP, process inline with the request but return early — use setImmediate or a simple in-process queue)
  - For production, consider a proper job queue (Bull/BullMQ with Redis) — but for MVP, inline processing is acceptable
- Subscriber query:
  - For meeting_scheduled, meeting_cancelled: find all users associated with the meeting's board (board members + staff with permissions on that board)
  - For agenda_published: all users associated with the board
  - For minutes_review: board members with R4 permission (view draft minutes)
  - For minutes_approved, minutes_published: all users associated with the board
  - For admin_alert: all users with admin role for the town
  - For user_invited: single recipient (the invited user's email)
  - For password_reset: single recipient (the user requesting reset)
- Filter subscribers:
  - Check SUBSCRIBER_NOTIFICATION_PREFERENCE for each subscriber + event_type + channel
  - If no preference exists, use the default for the event type (most event types default to enabled for associated users)
  - Skip subscribers who have explicitly disabled this notification type
  - Skip subscribers with hard-bounced email addresses (email_bounced flag on USER_ACCOUNT)

TASK 4: Create Postmark email sender service
- File: packages/api/src/services/email-sender.ts
- Create an EmailSenderService class:
  - sendEmail(options: SendEmailOptions): Promise<PostmarkResponse>
    - options: { to, from, subject, htmlBody, textBody, tag, messageStream, metadata }
    - Uses the Postmark ServerClient.sendEmail() method
    - Sets the From address from the town's notification config (notifications@townname.townmeetingmanager.com)
    - Sets the appropriate message stream (transactional or broadcast)
    - Tags emails with the notification event type for tracking in Postmark
    - Includes metadata: { town_id, event_id, delivery_id } for webhook correlation
  - sendBatchEmail(messages: SendEmailOptions[]): Promise<PostmarkResponse[]>
    - Uses Postmark's batch API for sending up to 500 emails at once
    - For subscriber lists larger than 500, batch in groups of 500
  - renderTemplate(templateName: string, variables: Record<string, unknown>): { html: string, text: string }
    - Renders the Handlebars HTML template with variables
    - Also generates a plain-text version (strip HTML tags, format for text)
- Create a helper to determine the message stream for each event type:
  - Transactional: user_invited, password_reset, minutes_review, admin_alert
  - Broadcast: meeting_scheduled, meeting_cancelled, agenda_published, minutes_approved, minutes_published

TASK 5: Create email templates
- Directory: packages/api/src/templates/email/
- Create Handlebars HTML templates for each Phase 1 email:
  - All templates share a common layout: header with town name/seal, content area, footer with unsubscribe link (for broadcast) and town contact info
  - Common layout file: packages/api/src/templates/email/layout.hbs
  - Template files:
    1. invite-user.hbs: "You've been invited to [Town Name]'s meeting management system. Click to set up your account." Variables: recipientName, townName, role, inviterName, setupUrl, expiresAt
    2. password-reset.hbs: "Reset your password for [Town Name] Meeting Manager." Variables: recipientName, townName, resetUrl, expiresAt
    3. meeting-notice.hbs: "[Board Name] meeting [scheduled/cancelled] for [date]." Variables: recipientName, townName, boardName, meetingDate, meetingTime, location, meetingType, action (scheduled/cancelled), meetingUrl
    4. agenda-published.hbs: "The agenda for [Board Name]'s [date] meeting has been published." Variables: recipientName, townName, boardName, meetingDate, agendaUrl, itemCount, portalUrl
    5. minutes-review.hbs: "Draft minutes from [Board Name]'s [date] meeting are ready for your review." Variables: recipientName, townName, boardName, meetingDate, reviewUrl
    6. minutes-approved.hbs: "Minutes from [Board Name]'s [date] meeting have been approved." Variables: recipientName, townName, boardName, meetingDate, minutesUrl, portalUrl
    7. admin-alert.hbs: Generic admin notification. Variables: recipientName, townName, alertTitle, alertMessage, actionUrl, actionLabel
- Template design guidelines:
  - Clean, professional government aesthetic — no flashy marketing email design
  - Use inline CSS (email client compatibility) with a consistent color scheme
  - Include the town seal in the header
  - Mobile-responsive (use max-width: 600px container, single column layout)
  - All links use absolute URLs
  - Broadcast emails include an unsubscribe link in the footer: "Manage your notification preferences" linking to the app
  - Transactional emails do NOT include unsubscribe links (they are necessary system communications)

TASK 6: Configure Postmark message streams
- In the Postmark sender service, route emails to the correct message stream:
  - Transactional stream: system-critical emails that users cannot unsubscribe from
    - invite-user: account invitation
    - password-reset: password reset
    - minutes-review: review request (internal workflow)
    - admin-alert: system alerts to admins
  - Broadcast stream: notification emails that users can opt out of
    - meeting-notice: meeting scheduled/cancelled
    - agenda-published: agenda published
    - minutes-approved: minutes approved
    - minutes-published: minutes published to portal
- The message stream is passed to Postmark via the MessageStream header in the API call
- Document the stream configuration: the Postmark account needs two message streams configured (default transactional stream + a broadcast stream named "broadcast" or "notifications")

TASK 7: Implement per-town sender configuration
- Per advisory 4.4: emails send from notifications@townname.townmeetingmanager.com
- Store sender configuration in TOWN_NOTIFICATION_CONFIG:
  - postmark_sender_email: "notifications@newcastle.townmeetingmanager.com"
  - postmark_sender_name: "Town of Newcastle"
- Postmark requires sender signatures (verified domains or email addresses):
  - The wildcard subdomain *.townmeetingmanager.com should be verified once in Postmark
  - SPF, DKIM, and DMARC records configured for the wildcard domain
  - Each town's sender email is automatically valid under the wildcard
- When sending emails, use the town-specific sender:
  - From: "Town of Newcastle <notifications@newcastle.townmeetingmanager.com>"
  - Reply-To: the town's primary contact email (from the town profile)
- Create an admin UI section (in town settings) for configuring notification sender:
  - Display the sender email (auto-generated from town subdomain)
  - Allow overriding the sender display name
  - Show Postmark connection status (API token valid/invalid)

TASK 8: Create Postmark webhook handler
- Create a Fastify route: POST /api/webhooks/postmark
  - This receives webhook callbacks from Postmark for delivery events
  - Verify the webhook authenticity (Postmark includes a server token or you can restrict by IP)
- Handle webhook event types:
  - Delivery: email successfully delivered to recipient's mail server
    - Update NOTIFICATION_DELIVERY status to "delivered"
    - Set delivered_at timestamp
  - Bounce: email bounced
    - Update NOTIFICATION_DELIVERY status to "bounced"
    - Set error_message from the bounce description
    - For hard bounces (Type: "HardBounce"): set email_bounced flag on the subscriber's USER_ACCOUNT to prevent future sends
    - For soft bounces (Type: "SoftBounce", "Transient"): trigger retry logic (Task 10)
  - SpamComplaint: recipient marked email as spam
    - Update NOTIFICATION_DELIVERY status to "complained"
    - Set email_complained flag on the subscriber's USER_ACCOUNT
    - Add to suppression list — do not send further broadcast emails to this address
  - Open: email opened (optional tracking — useful for admin dashboard)
    - Update NOTIFICATION_DELIVERY with opened_at timestamp
  - Click: link clicked (optional tracking)
    - Log the clicked URL and timestamp
- Parse the webhook payload, extract the metadata (town_id, event_id, delivery_id) to correlate with the NOTIFICATION_DELIVERY record
- Return 200 OK to Postmark quickly — process the webhook asynchronously if needed

TASK 9: Create delivery tracking
- The NOTIFICATION_DELIVERY table (created in Task 2) tracks each email send:
  - One record per notification event per subscriber
  - Status lifecycle: pending → sent → delivered (success path) or pending → sent → bounced/failed (failure path)
  - Store the Postmark message ID (returned from the send API call) for correlation with webhooks
- Create a delivery summary function:
  - getDeliverySummary(eventId): returns { total, sent, delivered, bounced, failed, pending }
  - Used by the admin dashboard (Task 11) to show delivery stats per notification event
- Create a subscriber delivery history function:
  - getSubscriberDeliveryHistory(userId, limit): returns recent deliveries for a specific user
  - Useful for debugging delivery issues

TASK 10: Implement retry logic
- For failed email deliveries (soft bounces, temporary errors):
  - Retry up to 3 times with exponential backoff:
    - Attempt 1: immediate
    - Attempt 2: 5 minutes after first failure
    - Attempt 3: 30 minutes after second failure
  - After 3 failed attempts: mark as permanently failed, log the error
- Implement retry scheduling:
  - On failure, increment retry_count on NOTIFICATION_DELIVERY
  - Set next_retry_at based on the backoff schedule
  - Create a periodic retry processor (Fastify cron job or setInterval) that queries for deliveries with next_retry_at <= now and retries them
  - On permanent failure (hard bounce, 3 retries exhausted): set status to "failed" and stop retrying
- For hard bounces: do NOT retry — immediately mark as failed and flag the subscriber
- Log all retry attempts for debugging
```
