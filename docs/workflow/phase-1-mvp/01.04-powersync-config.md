## Description

Add PowerSync Open Edition to the Docker Compose stack and configure sync rules for multi-tenant data isolation. PowerSync provides bidirectional offline-first sync between Supabase PostgreSQL and local SQLite on client devices, which is the critical foundation for the live meeting manager's offline capability.

## Tasks

1. Add PowerSync Open Edition service to docker-compose.yml
2. Configure PowerSync connection to Supabase PostgreSQL (replication)
3. Create powersync/ directory with configuration files
4. Create sync-rules.yaml with initial Sync Rules for multi-tenant data isolation (scope by town_id)
5. Define sync rules for core tables: town, board, person, user_account, board_member
6. Define sync rules for meeting tables: meeting, agenda_item, motion, vote_record, attendance
7. Define sync rules for minutes tables: minutes_document, minutes_section
8. Configure JWT authentication between PowerSync and Supabase Auth (RS256 JWKS)
9. Test: verify PowerSync connects to PostgreSQL and sync rules load without errors

## Prompt

```
You are adding PowerSync Open Edition to the Town Meeting Manager's Docker Compose stack. PowerSync provides bidirectional offline-first sync between Supabase PostgreSQL and local SQLite on client devices. Read docs/advisory-resolutions/1.3-tech-stack-evaluation.md for the PowerSync architecture decision.

KEY CONTEXT (from advisory 1.3):
- PowerSync is the offline-first sync layer — it replaces Supabase Realtime as the primary data sync mechanism
- Local SQLite on every client device means the live meeting manager works with zero connectivity
- Writes go into a local upload queue and push to Supabase when connectivity returns
- Sync Rules control which data syncs to which users — maps to multi-tenant + per-board permissions model
- PowerSync Open Edition is self-hostable alongside Supabase
- JWT authentication: client authenticates with Supabase Auth, PowerSync validates the Supabase-issued JWT

MULTI-TENANT DATA ISOLATION:
- All data is scoped by town_id
- Each user's JWT contains their town_id claim (configured in session 01.08)
- Sync Rules use the JWT town_id to filter which rows sync to which users
- A user should only ever receive data for their own town

TASK 1: Add PowerSync to docker-compose.yml
Add the PowerSync service to docker/docker-compose.yml:

```yaml
powersync:
  container_name: town-meeting-powersync
  image: journeyapps/powersync-service:latest
  depends_on:
    db:
      condition: service_healthy
  ports:
    - "8080:8080"
  environment:
    - POWERSYNC_CONFIG_PATH=/config/powersync.yaml
  volumes:
    - ../powersync/powersync.yaml:/config/powersync.yaml:ro
    - ../powersync/sync-rules.yaml:/config/sync-rules.yaml:ro
  restart: unless-stopped
```

TASK 2: Configure PowerSync connection to PostgreSQL
Create powersync/powersync.yaml at the repo root (not inside docker/):

```yaml
# PowerSync configuration for Town Meeting Manager
# Docs: https://docs.powersync.com/self-hosting/configuration

replication:
  # Connection to Supabase PostgreSQL for logical replication
  connections:
    - type: postgresql
      uri: "postgresql://postgres:${POSTGRES_PASSWORD}@db:5432/postgres"
      # Slot name for logical replication
      slot_name: "powersync_replication"
      # Publication name — PowerSync will create this if it doesn't exist
      publication_name: "powersync_publication"

# Client API configuration
api:
  port: 8080

# Sync Rules file location
sync_rules:
  path: /config/sync-rules.yaml

# JWT authentication — validates Supabase Auth JWTs
client_auth:
  # Use JWKS endpoint for RS256 JWT verification
  # This auto-discovers Supabase's public keys
  jwks:
    - uri: "http://kong:8000/auth/v1/.well-known/jwks.json"
      # For development, you can alternatively use HS256 with the JWT secret:
  # supabase:
  #   jwt_secret: "${JWT_SECRET}"

# Storage configuration
storage:
  type: mongodb
  uri: "mongodb://mongo:27017/powersync"

# Telemetry
telemetry:
  disable_telemetry: true
```

Note: PowerSync Open Edition requires MongoDB for internal state storage. Add a MongoDB service to docker-compose.yml:

```yaml
mongo:
  container_name: town-meeting-mongo
  image: mongo:7
  ports:
    - "27017:27017"
  volumes:
    - ./volumes/mongo/data:/data/db
  restart: unless-stopped
```

Add mongo/data to .gitignore.

TASK 4: Create sync-rules.yaml — Core structure and tenant isolation
Create powersync/sync-rules.yaml. PowerSync Sync Rules define which data syncs to which client. The fundamental isolation principle: every query is scoped by town_id from the user's JWT token_parameters.

```yaml
# PowerSync Sync Rules for Town Meeting Manager
# All data is scoped by town_id extracted from the user's JWT claims.
# A user only receives data for their own town.

bucket_definitions:

  # ─── TOWN DATA ───────────────────────────────────────────────
  # The user's own town record. Every authenticated user gets this.
  town_data:
    parameters:
      - SELECT token_parameters.town_id as town_id
    data:
      - SELECT id, name, state, municipality_type, population_range,
               contact_name, contact_role, meeting_formality, minutes_style,
               presiding_officer_default, minutes_recorder_default, subdomain,
               created_at, updated_at
        FROM town
        WHERE id = bucket.town_id
```

TASK 5: Define sync rules for core tables (person, user_account, board, board_member)
Continue in sync-rules.yaml:

```yaml
  # ─── PEOPLE & ACCOUNTS ──────────────────────────────────────
  # All persons, user accounts, and board members for the user's town.
  # Staff and admin see all; board members need to see fellow members for meeting context.
  people_data:
    parameters:
      - SELECT token_parameters.town_id as town_id
    data:
      - SELECT id, town_id, name, email, created_at, archived_at
        FROM person
        WHERE town_id = bucket.town_id

      - SELECT id, person_id, town_id, role, gov_title, permissions,
               auth_user_id, created_at, archived_at
        FROM user_account
        WHERE town_id = bucket.town_id

      - SELECT id, town_id, name, board_type, member_count,
               election_method, officer_election_method, district_based,
               staggered_terms, is_governing_board, meeting_formality_override,
               minutes_style_override, created_at, archived_at
        FROM board
        WHERE town_id = bucket.town_id

      - SELECT id, person_id, board_id, town_id, seat_title,
               term_start, term_end, status, is_default_rec_sec, created_at
        FROM board_member
        WHERE town_id = bucket.town_id
```

TASK 8: Configure JWT authentication
PowerSync needs to validate JWTs issued by Supabase Auth. There are two approaches:

For development (HS256 — simpler):
- PowerSync validates using the shared JWT_SECRET from Supabase
- Configure in powersync.yaml under client_auth.supabase.jwt_secret

For production (RS256 — more secure, recommended by advisory 1.3a):
- PowerSync auto-discovers Supabase's public JWKS endpoint
- Configure in powersync.yaml under client_auth.jwks.uri
- The JWKS endpoint is exposed through Kong at /auth/v1/.well-known/jwks.json

For the initial development setup, use HS256 (shared secret) since it is simpler to configure with Docker Compose. Add a comment noting the production switch to RS256 via JWKS.

The JWT must contain custom claims that PowerSync uses in sync rules:
- token_parameters.town_id — the user's town_id (configured in session 01.08)
- token_parameters.user_id — the user's person_id
- token_parameters.role — the user's role (admin, staff, board_member)

These claims will be added to the JWT by a custom access token hook in session 01.08. For now, ensure the powersync.yaml references token_parameters correctly.

Update the docker/.env.example with PowerSync-related variables:

# PowerSync
POWERSYNC_PORT=8080

TASK 9: Test PowerSync integration
- Ensure PostgreSQL has logical replication enabled. Add to PostgreSQL config:
  ```
  wal_level = logical
  max_replication_slots = 4
  max_wal_senders = 4
  ```
  This can be done via the db service's command in docker-compose.yml:
  ```yaml
  command: >
    postgres
    -c wal_level=logical
    -c max_replication_slots=4
    -c max_wal_senders=4
  ```
- Start the full stack: `pnpm supabase:up`
- Check PowerSync logs: `docker logs town-meeting-powersync`
- Verify PowerSync connects to PostgreSQL (look for "Connected to PostgreSQL" in logs)
- Verify sync rules load (look for "Sync rules loaded" or similar)
- PowerSync will report "no tables found" initially since the schema hasn't been created yet (session 01.05) — this is expected
- Verify the PowerSync diagnostics endpoint: `curl http://localhost:8080/api/diagnostics`
