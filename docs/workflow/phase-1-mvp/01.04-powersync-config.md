# Session 01.04 — React Query & Push Notification Foundation

**Phase:** 1 — MVP | **Block:** 01 — Infrastructure & Monorepo Foundation
**Dependencies:** 01.03
**Estimated tasks:** 9

---

## Description

Install TanStack Query v5 as the client data layer, create the QueryClient singleton and QueryProvider, define typed query key factories for all entities, set up the Supabase typed client, generate VAPID keys for Web Push Notifications, create the `push_subscriptions` database migration, and install the `web-push` package in the API. This session establishes the client data management foundation and the Push Notification infrastructure.

## Tasks

1. Install TanStack Query v5 packages: `@tanstack/react-query`, `@tanstack/react-query-devtools`
2. Create `packages/web/src/lib/queryClient.ts` — QueryClient singleton with default options
3. Create `packages/web/src/providers/QueryProvider.tsx` — wraps app in QueryClientProvider + ReactQueryDevtools
4. Create `packages/web/src/lib/queryKeys.ts` — typed query key factory for all entities
5. Create/expand `packages/web/src/lib/supabase.ts` — typed Supabase client with Database generic
6. Generate VAPID key pair: `npx web-push generate-vapid-keys` — store in `docker/.env`
7. Create `push_subscriptions` table migration: `supabase/migrations/20260312000003_push_subscriptions.sql`
8. Install `web-push` in packages/api for server-side push dispatch
9. Verify: QueryClient initializes, VAPID keys generated, migration runs cleanly

TASK 2: Create QueryClient singleton
Create packages/web/src/lib/queryClient.ts:

  import { QueryClient } from '@tanstack/react-query';

  export const queryClient = new QueryClient({
    defaultOptions: {
      queries: {
        staleTime: 60 * 1000,        // 1 minute — data is fresh for 1 minute
        gcTime: 5 * 60 * 1000,       // 5 minutes — unused cache entries GC'd after 5 min
        retry: 2,                    // retry failed queries twice
        refetchOnWindowFocus: true,  // refetch when user returns to the tab
      },
      mutations: {
        retry: 0,                    // don't auto-retry mutations
      },
    },
  });

This singleton is imported by QueryProvider and can also be imported directly in clientLoaders
for queryClient.ensureQueryData() calls.

TASK 3: Create QueryProvider
Create packages/web/src/providers/QueryProvider.tsx:

  import { QueryClientProvider } from '@tanstack/react-query';
  import { ReactQueryDevtools } from '@tanstack/react-query-devtools';
  import { queryClient } from '@/lib/queryClient';

  export function QueryProvider({ children }: { children: React.ReactNode }) {
    return (
      <QueryClientProvider client={queryClient}>
        {children}
        {import.meta.env.DEV && (
          <ReactQueryDevtools initialIsOpen={false} position="bottom-right" />
        )}
      </QueryClientProvider>
    );
  }

The ReactQueryDevtools panel is only included in development builds (import.meta.env.DEV).
In production, the panel is completely excluded from the bundle.

TASK 4: Create typed query key factory
Create packages/web/src/lib/queryKeys.ts. Typed query keys ensure consistent cache management
across the app — every useQuery and invalidateQueries call uses the same key structure.

  export const queryKeys = {
    // Towns
    towns: () => ['towns'] as const,
    town: (townId: string) => ['towns', townId] as const,

    // Boards
    boards: (townId: string) => ['towns', townId, 'boards'] as const,
    board: (boardId: string) => ['boards', boardId] as const,

    // Board members
    boardMembers: (boardId: string) => ['boards', boardId, 'members'] as const,

    // Persons
    persons: (townId: string) => ['towns', townId, 'persons'] as const,
    person: (personId: string) => ['persons', personId] as const,

    // User accounts
    currentUser: () => ['currentUser'] as const,
    userAccount: (userId: string) => ['userAccounts', userId] as const,

    // Meetings
    meetings: (boardId: string) => ['boards', boardId, 'meetings'] as const,
    meeting: (meetingId: string) => ['meetings', meetingId] as const,

    // Agenda items
    agendaItems: (meetingId: string) => ['meetings', meetingId, 'agendaItems'] as const,
    agendaItem: (itemId: string) => ['agendaItems', itemId] as const,

    // Agenda templates
    agendaTemplates: (boardId: string) => ['boards', boardId, 'agendaTemplates'] as const,

    // Attendance
    meetingAttendance: (meetingId: string) => ['meetings', meetingId, 'attendance'] as const,

    // Motions
    motions: (meetingId: string) => ['meetings', meetingId, 'motions'] as const,

    // Votes
    votes: (motionId: string) => ['motions', motionId, 'votes'] as const,
    meetingVotes: (meetingId: string) => ['meetings', meetingId, 'votes'] as const,

    // Minutes
    minutesDocument: (meetingId: string) => ['meetings', meetingId, 'minutes'] as const,

    // Public portal
    publicMeetings: (townSubdomain: string) => ['public', townSubdomain, 'meetings'] as const,
    publicMeeting: (townSubdomain: string, meetingId: string) =>
      ['public', townSubdomain, 'meetings', meetingId] as const,

    // Notifications
    notificationPreferences: (userId: string) => ['notificationPreferences', userId] as const,
    pushSubscription: (userId: string) => ['pushSubscription', userId] as const,
  } as const;

The hierarchical key structure allows broad cache invalidation:
- queryClient.invalidateQueries({ queryKey: queryKeys.boards(townId) }) invalidates all boards for a town
- queryClient.invalidateQueries({ queryKey: ['meetings', meetingId] }) invalidates all meeting-related queries

TASK 5: Create/expand the Supabase typed client
Create (or expand) packages/web/src/lib/supabase.ts:

  import { createClient } from '@supabase/supabase-js';
  import type { Database } from '@town-meeting/shared/database.types';

  const supabaseUrl = import.meta.env.VITE_SUPABASE_URL;
  const supabaseAnonKey = import.meta.env.VITE_SUPABASE_ANON_KEY;

  if (!supabaseUrl || !supabaseAnonKey) {
    throw new Error('Missing Supabase environment variables');
  }

  export const supabase = createClient<Database>(supabaseUrl, supabaseAnonKey, {
    auth: {
      autoRefreshToken: true,
      persistSession: true,
      detectSessionInUrl: true,
    },
    realtime: {
      params: {
        eventsPerSecond: 10, // appropriate for live meeting updates
      },
    },
  });

The Database type is generated from the Supabase schema by running:
  npx supabase gen types typescript --local > packages/shared/src/database.types.ts

TASK 6: Generate VAPID key pair
VAPID (Voluntary Application Server Identification) keys authenticate the push notification server to
browser push services. They must be generated once and stored securely.

Run:
  npx web-push generate-vapid-keys

This outputs:
  Public Key: <base64url string>
  Private Key: <base64url string>

Add to docker/.env and docker/.env.example:
  # Web Push Notifications (VAPID)
  VAPID_PUBLIC_KEY=<your-public-key>
  VAPID_PRIVATE_KEY=<your-private-key>
  VAPID_CONTACT_EMAIL=admin@townmeetingmanager.com

Add to packages/web/.env.example:
  # Web Push — public key only (safe to expose to browser)
  VITE_VAPID_PUBLIC_KEY=<your-public-key>

IMPORTANT: The VAPID_PRIVATE_KEY must never be exposed to the browser. It stays in the API server environment only.
The VITE_VAPID_PUBLIC_KEY (public key) is used by the browser's PushManager.subscribe() call.

If npx web-push is not available, install it temporarily:
  pnpm add -g web-push
  web-push generate-vapid-keys

Then uninstall: pnpm remove -g web-push

TASK 7: Create push_subscriptions migration
Create supabase/migrations/20260312000003_push_subscriptions.sql:

  -- Push notification subscriptions for Web Push API
  -- One row per device per user — a user may have multiple devices/browsers subscribed
  CREATE TABLE IF NOT EXISTS push_subscriptions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_account_id UUID NOT NULL REFERENCES user_account(id) ON DELETE CASCADE,
    town_id UUID NOT NULL REFERENCES town(id) ON DELETE CASCADE,
    endpoint TEXT NOT NULL,           -- The push endpoint URL from PushSubscription.endpoint
    p256dh TEXT NOT NULL,             -- ECDH public key from PushSubscription.getKey('p256dh')
    auth TEXT NOT NULL,               -- Auth secret from PushSubscription.getKey('auth')
    user_agent TEXT,                  -- Browser/device identifier (for display in preferences)
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT push_subscriptions_user_endpoint_unique UNIQUE (user_account_id, endpoint)
  );

  -- Users can only see/manage their own subscriptions
  ALTER TABLE push_subscriptions ENABLE ROW LEVEL SECURITY;

  CREATE POLICY "push_subscriptions_own_access" ON push_subscriptions
    FOR ALL
    TO authenticated
    USING (user_account_id IN (
      SELECT id FROM user_account WHERE auth_user_id = auth.uid()
    ));

  -- Index for efficient lookup by user when dispatching pushes
  CREATE INDEX push_subscriptions_user_account_idx ON push_subscriptions (user_account_id);
  CREATE INDEX push_subscriptions_town_idx ON push_subscriptions (town_id);

Run the migration:
  npx supabase db push --local

Or if using direct psql (see MEMORY.md for the correct approach):
  PGPASSWORD=$(grep POSTGRES_PASSWORD docker/.env | cut -d= -f2) psql -U postgres -h localhost -p 5432 -f supabase/migrations/20260312000003_push_subscriptions.sql
