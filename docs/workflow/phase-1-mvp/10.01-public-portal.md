# Session 10.01 — Public Portal: Per-Town Subdomain & Content

**Phase:** 1 — MVP | **Block:** 10 — Public Portal
**Dependencies:** 09.02, 01.03
**Estimated tasks:** 13

---

## Description

Build the public-facing portal accessible at townname.townmeetingmanager.com, where residents and the public can view upcoming meetings, published agendas, approved minutes, and board directories. The portal uses wildcard subdomain routing via Nginx, resolves the town from the subdomain, and serves only published/approved content in a clean, accessible layout. This implements advisory 4.2 (subdomain per town) and advisory 4.3 (minutes as HTML for portal display).

## Tasks

1. Create public portal route structure: accessible at townname.townmeetingmanager.com
2. Configure Nginx wildcard subdomain routing (*.townmeetingmanager.com)
3. Create subdomain resolution middleware: extract town name from subdomain → look up town record → scope all queries
4. Create public portal layout: clean, accessible design with town name/seal, navigation (meetings, minutes, boards, contact)
5. Create public meetings page: list of upcoming and past meetings with agenda links
6. Create public agenda view: formatted agenda for a specific meeting (items, exhibits marked public)
7. Create public minutes view: approved minutes rendered as HTML (from minutes_document html_rendered)
8. Create public minutes PDF download: serve stored PDF from Supabase Storage
9. Create public board directory: list of boards with current member roster (names, titles, term dates — public record)
10. Create meeting calendar view: calendar showing scheduled meetings for all boards
11. Handle draft/unapproved content: only published agendas and approved minutes visible on portal
12. Create 404 page for unknown subdomains
13. Verify: portal accessible via subdomain, meetings/agendas/minutes display, only approved/published content shown, unknown subdomain shows 404

## Prompt

```
You are building the public-facing portal for the Town Meeting Manager — a municipal meeting management platform for Maine towns. Each town gets its own subdomain (e.g., newcastle.townmeetingmanager.com) where residents can view meetings, agendas, minutes, and board information. Read docs/advisory-resolutions/4-simple-decisions.md for decision 4.2 (subdomain per town) and 4.3 (minutes as HTML for portal). Read docs/advisory-resolutions/3.2-supabase-hosting.md for the infrastructure layout showing Nginx routing.

PROJECT CONTEXT:
- Monorepo: packages/web (React 19 + React Router v7 Framework Mode), packages/api (Fastify), packages/shared (types, schemas)
- Self-hosted on dedicated server with Nginx reverse proxy
- Nginx routes: app.townmeetingmanager.com → web app, api.townmeetingmanager.com → Fastify, *.townmeetingmanager.com → public portal
- Supabase PostgreSQL for data, Supabase Storage for PDFs and documents
- The portal is PUBLIC — no authentication required. It serves only published/approved content.
- Minutes are stored as HTML in MINUTES_DOCUMENT.html_rendered (per advisory 4.3)
- Agendas are published via a publish action that sets a published_at timestamp
- Multi-tenant: all towns share the same database, isolated by town_id + RLS

TASK 1: Create public portal route structure
- The public portal is a separate section of the React Router v7 application, served at wildcard subdomains
- Create a route layout for the portal: packages/web/src/routes/portal.tsx (or a separate route tree under portal/)
- Route structure:
  - / → portal home (upcoming meetings, recent minutes)
  - /meetings → list of meetings
  - /meetings/:meetingId → meeting detail with agenda
  - /meetings/:meetingId/agenda → published agenda view
  - /meetings/:meetingId/minutes → approved minutes view
  - /boards → board directory
  - /boards/:boardId → board detail with member roster
  - /calendar → meeting calendar
- The portal routes do NOT use PowerSync (no offline-first requirement for the public portal)
- Portal data is fetched server-side or via API calls to the Fastify backend using standard fetch (not PowerSync queries)

TASK 2: Configure Nginx wildcard subdomain routing
- Create or update the Nginx configuration to handle wildcard subdomains:
  - File: infrastructure/nginx/portal.conf (or similar location)
  - Server block:
    - listen 443 ssl
    - server_name *.townmeetingmanager.com
    - Exclude: app.townmeetingmanager.com, api.townmeetingmanager.com, supabase.townmeetingmanager.com (these have their own server blocks)
    - Proxy pass to the same web app, but with a header indicating the subdomain
    - Add header: X-Town-Subdomain extracted from $host
  - SSL: uses the wildcard certificate from Let's Encrypt (configured in session 15.02)
- The web application receives the X-Town-Subdomain header and uses it to determine the town context
- Alternatively, the React app can read the subdomain directly from window.location.hostname on the client side

TASK 3: Create subdomain resolution middleware
- File: packages/web/src/lib/portal/subdomain-resolver.ts
- Create a function resolveTownFromSubdomain(hostname: string):
  - Extract the subdomain from the hostname (e.g., "newcastle" from "newcastle.townmeetingmanager.com")
  - Handle edge cases: www prefix, localhost for development, missing subdomain
  - Query the TOWN table for a record where portal_subdomain matches the extracted subdomain
  - Return the town record (id, name, seal_url, portal_subdomain) or null if not found
- Create a Fastify API endpoint: GET /api/portal/resolve?subdomain=newcastle
  - Returns the town record if found, 404 if not
  - This endpoint is public (no authentication required)
  - Response includes: town_id, town_name, municipality_type, seal_url, contact info
- Create a React context: PortalTownContext that holds the resolved town data
- The portal layout component calls the resolver on mount and wraps children in the context
- All portal data queries include the town_id from this context to scope results

TASK 4: Create public portal layout
- File: packages/web/src/routes/portal/layout.tsx
- Clean, accessible layout for public consumption:
  - Header:
    - Town seal image (if uploaded) or placeholder icon
    - Town name prominently displayed (e.g., "Town of Newcastle")
    - Municipality type subtitle (e.g., "Municipal Government")
    - Navigation bar: Meetings, Minutes, Boards, Calendar, Contact
  - Main content area with consistent padding and max-width for readability
  - Footer:
    - "Powered by Town Meeting Manager" link
    - Town contact information (from town profile)
    - Accessibility statement link
  - Mobile-responsive: hamburger menu for navigation on small screens
- Use Tailwind CSS v4 for styling — clean, government-appropriate design
- WCAG 2.1 AA compliance: proper contrast ratios, focus indicators, skip-to-content link, semantic landmarks
- No dark mode on the portal (keep it simple and official-looking)
- If the town is not found (subdomain doesn't match any town), render the 404 page (Task 12)

TASK 5: Create public meetings page
- Route: /meetings (within the portal layout)
- File: packages/web/src/routes/portal/meetings.tsx
- Fetch meetings for the resolved town via the Fastify API: GET /api/portal/:townId/meetings
  - Only return meetings with status "scheduled" (upcoming) or "completed" (past, with published agenda or approved minutes)
  - Do NOT return draft or cancelled meetings
  - Sort: upcoming meetings first (ascending by date), then past meetings (descending by date)
  - Paginate: 20 per page with load-more or pagination controls
- Display as a list with two sections:
  - "Upcoming Meetings" — meetings with scheduled_at in the future
    - Show: board name, date, time, location, "View Agenda" link (if agenda is published)
  - "Past Meetings" — completed meetings
    - Show: board name, date, "View Agenda" link, "View Minutes" link (if approved), PDF download icon
- Filter by board (dropdown showing all active boards)
- Empty state: "No scheduled meetings at this time."

TASK 6: Create public agenda view
- Route: /meetings/:meetingId/agenda
- File: packages/web/src/routes/portal/meetings.$meetingId.agenda.tsx
- Fetch the published agenda via API: GET /api/portal/:townId/meetings/:meetingId/agenda
  - Returns the agenda only if it has been published (published_at is not null)
  - Returns 404 if the agenda is not published
- Display the agenda in a formatted, readable layout:
  - Meeting header: board name, date, time, location
  - Sections with items, each showing:
    - Section title and number
    - Item title with section reference (e.g., "6A")
    - Background text (if provided and marked public)
    - Exhibits list with download links (only exhibits marked as public — per advisory 2.2 Q6: all public by default, admin can mark internal-only)
    - Suggested motion text (if provided)
  - Staff resource listed per item
- Include a "Download PDF" button that serves the agenda packet PDF from Supabase Storage
- Print-friendly styling: a print media query that formats the page nicely for printing directly from the browser

TASK 7: Create public minutes view
- Route: /meetings/:meetingId/minutes
- File: packages/web/src/routes/portal/meetings.$meetingId.minutes.tsx
- Fetch the approved/published minutes via API: GET /api/portal/:townId/meetings/:meetingId/minutes
  - Returns the minutes only if MINUTES_DOCUMENT status is "published" (or "approved" — decide based on portal visibility policy)
  - Returns 404 if minutes are not approved/published
- Display the minutes using the stored html_rendered content from the MINUTES_DOCUMENT record
  - Render the pre-generated HTML directly inside a container element (using dangerouslySetInnerHTML or a sanitized HTML renderer)
  - The HTML was generated by the Handlebars template in session 09.01 and is already semantic and accessible
  - Do NOT re-render from content_json on the portal — use the pre-rendered HTML
- Include:
  - Meeting metadata header (board, date, time)
  - "Approved on [date]" notation
  - "Download PDF" button linking to the stored PDF
  - Print button
- No DRAFT watermark (only approved/published minutes are shown)

TASK 8: Create public minutes PDF download
- Create a Fastify API endpoint: GET /api/portal/:townId/meetings/:meetingId/minutes/pdf
  - Validates that the minutes are approved/published
  - Fetches the PDF from Supabase Storage using the pdf_storage_path on the MINUTES_DOCUMENT
  - Returns the PDF with appropriate headers:
    - Content-Type: application/pdf
    - Content-Disposition: inline; filename="[board-name]-minutes-[date].pdf" (inline for browser preview, attachment for download)
  - Returns 404 if minutes are not published or PDF doesn't exist
- Add a similar endpoint for agenda packet PDFs: GET /api/portal/:townId/meetings/:meetingId/agenda/pdf
- Both endpoints are public (no authentication required)
- Consider caching: set Cache-Control headers for PDFs (they don't change once published)

TASK 9: Create public board directory
- Route: /boards
- File: packages/web/src/routes/portal/boards.tsx
- Fetch all active boards via API: GET /api/portal/:townId/boards
  - Returns only active boards (not archived)
  - For each board: name, type (elected/appointed), member count
- Display as a grid of cards, each linking to the board detail page
- Route: /boards/:boardId
- File: packages/web/src/routes/portal/boards.$boardId.tsx
- Fetch board detail with member roster: GET /api/portal/:townId/boards/:boardId
  - Board info: name, type, meeting schedule, contact
  - Active members: name, seat_title (if role-titled), term start/end dates
  - These are public records — names, titles, and term dates of board members are public information
- Display the member roster in a table: Name, Title/Seat, Term Start, Term End
- Link to the board's upcoming meetings
- Do NOT display email addresses or phone numbers of board members on the public portal

TASK 10: Create meeting calendar view
- Route: /calendar
- File: packages/web/src/routes/portal/calendar.tsx
- Fetch scheduled meetings via API: GET /api/portal/:townId/calendar?start=YYYY-MM-DD&end=YYYY-MM-DD
  - Returns meetings within the date range
  - Includes: board name, date, time, location, meeting type
- Display as a monthly calendar grid:
  - Days with meetings show colored dots or indicators
  - Clicking a day shows the meetings for that day in a panel or popover
  - Each meeting links to its detail page
  - Color-code by board (consistent colors per board)
- Navigation: previous/next month, today button
- Use a lightweight calendar component — either build with CSS Grid and Tailwind or use a minimal calendar library
- Mobile view: switch to a list view on small screens (show meetings as a chronological list for the selected month)

TASK 11: Handle draft/unapproved content visibility
- Create a strict content visibility policy for the portal:
  - Agendas: visible only when published_at is not null
  - Minutes: visible only when MINUTES_DOCUMENT.status is "published" (use "published" rather than just "approved" to give admin explicit control over portal visibility)
  - Meetings: visible when they have a published agenda OR are in "scheduled" status with a future date
  - Boards: visible when status is "active"
  - Board members: visible when status is "active" on an active board
- All portal API endpoints enforce these visibility rules server-side (not just client-side filtering)
- Create RLS policies or API middleware that enforces portal visibility:
  - Portal API endpoints use a special portal_read role (or just filter by visibility columns)
  - Never expose draft agendas, draft minutes, internal exhibits, or archived boards on the portal
- Display appropriate messages when content is not yet available:
  - "Agenda not yet published" on meeting pages without a published agenda
  - "Minutes pending approval" on past meeting pages without approved minutes

TASK 12: Create 404 page for unknown subdomains
- File: packages/web/src/routes/portal/not-found.tsx
- When the subdomain doesn't match any registered town:
  - Display a friendly 404 page: "Town Not Found"
  - Message: "We couldn't find a portal for '[subdomain].townmeetingmanager.com'. Please check the URL and try again."
  - Include a link: "Are you a town administrator? Set up your portal at app.townmeetingmanager.com"
  - Clean design consistent with the portal branding
- Also handle 404 for valid towns with invalid routes (e.g., /nonexistent-page)
  - "Page Not Found" with navigation links back to the portal home
```
