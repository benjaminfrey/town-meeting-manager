**Phase:** 1 — MVP | **Block:** 01 — Infrastructure & Monorepo Foundation
**Dependencies:** 01.07

## Description

Configure Supabase Auth (GoTrue) with custom JWT claims, auth hooks for user lifecycle management, and email templates. The auth system links Supabase Auth users to the application's PERSON/user_account records and injects town_id, role, and person_id into JWTs so that RLS policies can enforce data isolation without additional database lookups.

## Tasks

1. Configure GoTrue email provider settings in docker-compose environment
2. Create auth hook/trigger: on user signup, link to PERSON record or create new one
3. Create auth hook/trigger: on JWT generation, include custom claims (town_id, role, person_id) from user_account
4. Configure JWT token settings (expiry, refresh token rotation)
5. Create database function: handle_new_user() -- triggered on auth.users insert, creates person + user_account records
6. Create database function: custom_access_token_hook() -- adds town_id, role, permissions to JWT claims
7. Set up email templates for Supabase Auth (invite, confirm, reset) -- placeholder templates
8. Test auth flow: sign up, verify JWT contains custom claims, verify RLS works with claims
9. Document the auth configuration and JWT claims structure

TASK 2: Create handle_new_user() function (20260308000037)
This function is triggered when a new auth.users record is created. In the Town Meeting Manager, users are NOT created via self-signup — they are invited by an admin. The admin first creates the PERSON and user_account records, then invites the user. When the invite is accepted, auth.users is created and we link it to the existing user_account.

However, we also need to handle the initial admin account created during onboarding (before any user_account exists).

```sql
-- Handle new user creation in auth.users
-- Links the auth user to an existing user_account (invited users)
-- or creates initial records for the first admin (onboarding)
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER AS $$
DECLARE
  existing_account user_account%ROWTYPE;
  new_person_id UUID;
  new_account_id UUID;
BEGIN
  -- Check if a user_account already exists for this email
  -- (admin created the account before sending the invite)
  SELECT ua.* INTO existing_account
  FROM user_account ua
  JOIN person p ON ua.person_id = p.id
  WHERE p.email = NEW.email
  AND ua.auth_user_id IS NULL
  LIMIT 1;

  IF existing_account.id IS NOT NULL THEN
    -- Link the auth user to the existing user_account
    UPDATE user_account
    SET auth_user_id = NEW.id
    WHERE id = existing_account.id;

    -- Set app_metadata on the auth user for JWT claims
    UPDATE auth.users
    SET raw_app_meta_data = raw_app_meta_data ||
      jsonb_build_object(
        'town_id', existing_account.town_id::TEXT,
        'role', existing_account.role::TEXT,
        'person_id', existing_account.person_id::TEXT,
        'user_account_id', existing_account.id::TEXT
      )
    WHERE id = NEW.id;

  ELSE
    -- No pre-existing account — this is the initial admin setup (onboarding)
    -- Create PERSON and user_account records
    -- The town_id must be provided in user_metadata during signup
    IF NEW.raw_user_meta_data ->> 'town_id' IS NOT NULL THEN
      new_person_id := gen_random_uuid();
      new_account_id := gen_random_uuid();

      INSERT INTO person (id, town_id, name, email, created_at)
      VALUES (
        new_person_id,
        (NEW.raw_user_meta_data ->> 'town_id')::UUID,
        COALESCE(NEW.raw_user_meta_data ->> 'name', split_part(NEW.email, '@', 1)),
        NEW.email,
        now()
      );

      INSERT INTO user_account (id, person_id, town_id, role, auth_user_id, permissions, created_at)
      VALUES (
        new_account_id,
        new_person_id,
        (NEW.raw_user_meta_data ->> 'town_id')::UUID,
        'admin',
        NEW.id,
        '{"global": {}, "board_overrides": []}',
        now()
      );

      -- Set app_metadata
      UPDATE auth.users
      SET raw_app_meta_data = raw_app_meta_data ||
        jsonb_build_object(
          'town_id', NEW.raw_user_meta_data ->> 'town_id',
          'role', 'admin',
          'person_id', new_person_id::TEXT,
          'user_account_id', new_account_id::TEXT
        )
      WHERE id = NEW.id;
    END IF;
  END IF;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- Trigger on auth.users insert
CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
```

TASK 3: Create custom_access_token_hook() function (20260308000038)
This hook runs every time Supabase Auth generates a JWT (login, token refresh). It injects custom claims from the user_account record into the JWT's app_metadata. This is what makes the RLS helper functions and RLS policies work.

```sql
-- Custom access token hook
-- Injects town_id, role, person_id, user_account_id into JWT app_metadata
-- Called by GoTrue on every token generation (login, refresh)
CREATE OR REPLACE FUNCTION public.custom_access_token_hook(event JSONB)
RETURNS JSONB AS $$
DECLARE
  claims JSONB;
  user_account_record RECORD;
BEGIN
  -- Extract current claims
  claims := event -> 'claims';

  -- Look up the user's account details
  SELECT
    ua.id as user_account_id,
    ua.town_id,
    ua.role,
    ua.person_id
  INTO user_account_record
  FROM user_account ua
  WHERE ua.auth_user_id = (event ->> 'user_id')::UUID
  AND ua.archived_at IS NULL
  LIMIT 1;

  IF user_account_record IS NOT NULL THEN
    -- Inject custom claims into app_metadata
    claims := jsonb_set(
      claims,
      '{app_metadata}',
      COALESCE(claims -> 'app_metadata', '{}'::JSONB) ||
      jsonb_build_object(
        'town_id', user_account_record.town_id::TEXT,
        'role', user_account_record.role::TEXT,
        'person_id', user_account_record.person_id::TEXT,
        'user_account_id', user_account_record.user_account_id::TEXT
      )
    );
  ELSE
    -- No active user_account found — user may have been archived
    -- Leave claims unchanged; RLS will deny access since town_id will be null
    NULL;
  END IF;

  -- Return the modified event with updated claims
  RETURN jsonb_set(event, '{claims}', claims);
END;
$$ LANGUAGE plpgsql STABLE SECURITY DEFINER;

-- Grant execute permission to the supabase_auth_admin role
GRANT EXECUTE ON FUNCTION public.custom_access_token_hook TO supabase_auth_admin;

-- Revoke from public for security
REVOKE EXECUTE ON FUNCTION public.custom_access_token_hook FROM PUBLIC;
REVOKE EXECUTE ON FUNCTION public.custom_access_token_hook FROM anon;
REVOKE EXECUTE ON FUNCTION public.custom_access_token_hook FROM authenticated;
```

TASK 5: Create invite_user function (20260308000040)
Create a server-side function that the admin calls to invite a new user. This is the primary account creation flow:

```sql
-- Invite a new user to the platform
-- Called by admin after creating PERSON and user_account records
-- Triggers Supabase Auth to send an invite email
CREATE OR REPLACE FUNCTION public.invite_user(
  p_email TEXT,
  p_town_id UUID,
  p_person_id UUID,
  p_user_account_id UUID,
  p_role TEXT,
  p_redirect_url TEXT DEFAULT NULL
)
RETURNS JSONB AS $$
DECLARE
  result JSONB;
BEGIN
  -- Verify the calling user is an admin
  IF NOT is_admin() THEN
    RAISE EXCEPTION 'Only admins can invite users';
  END IF;

  -- Verify the person and user_account exist and belong to this town
  IF NOT EXISTS (
    SELECT 1 FROM person WHERE id = p_person_id AND town_id = p_town_id
  ) THEN
    RAISE EXCEPTION 'Person not found in this town';
  END IF;

  IF NOT EXISTS (
    SELECT 1 FROM user_account
    WHERE id = p_user_account_id
    AND person_id = p_person_id
    AND town_id = p_town_id
  ) THEN
    RAISE EXCEPTION 'User account not found for this person in this town';
  END IF;

  -- The actual invite email is sent via the Supabase Auth admin API
  -- This function validates preconditions; the API call is made from the application layer
  RETURN jsonb_build_object(
    'success', true,
    'email', p_email,
    'person_id', p_person_id,
    'user_account_id', p_user_account_id
  );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

Note: The actual Supabase Auth invite is sent via the Admin API (using the service_role key) from the application layer, not from a database function. This function validates preconditions.
