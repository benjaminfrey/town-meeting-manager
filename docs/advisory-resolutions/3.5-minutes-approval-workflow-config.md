# Advisory 3.5 тАФ Minutes Approval Workflow Configuration

**Status:** Approved
**Applies to:** Session 14.x
**Extends:** Session 09.02 (Minutes Review & Approval Workflow), Advisory 2.2 (AI Minutes Architecture)

---

## 1. Overview

Session 09.02 implemented the core minutes review/approval workflow at the meeting level (draft тЖТ review тЖТ approved тЖТ published). This advisory defines the **configurable settings** that govern how that workflow behaves, surfaced through:

1. A **town-level settings page** at `/settings/minutes-workflow` (global defaults)
2. **Board-level overrides** in the board detail page's Settings tab
3. The **amendment and addendum data model** for already-adopted minutes

---

## 2. Settings Architecture

### 2.1 Town-Level Defaults (`/settings/minutes-workflow`)

These settings apply to all boards unless overridden at the board level. They are stored as columns on the `town` table.

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| `audio_retention_policy` | enum | `retain_30_days` | `purge_on_approval` / `retain_30_days` / `retain_90_days` / `retain_indefinitely` |
| `auto_publish_on_approval` | boolean | `false` | Automatically publish to portal when motion passes |
| `minutes_review_window_days` | integer | `7` | Days before next meeting that draft minutes must be distributed to board members |

New columns:

```sql
ALTER TABLE town
  ADD COLUMN audio_retention_policy TEXT NOT NULL DEFAULT 'retain_30_days'
    CHECK (audio_retention_policy IN ('purge_on_approval','retain_30_days','retain_90_days','retain_indefinitely')),
  ADD COLUMN auto_publish_on_approval BOOLEAN NOT NULL DEFAULT false,
  ADD COLUMN minutes_review_window_days INTEGER NOT NULL DEFAULT 7,
  ADD COLUMN minutes_workflow_configured_at TIMESTAMPTZ;
  -- Set when admin saves the settings page; used for ProgressChecklist completion signal
```

### 2.2 Board-Level Overrides (Board Settings Tab)

These settings inherit from town defaults and can be overridden per board. Stored on the `board` table.

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| `minutes_style` | enum | (from town) | `action` / `summary` / `narrative` тАФ already exists |
| `certification_format` | enum | `prepared_by` | `prepared_by` / `all_member_certification` / `both` тАФ already exists |
| `member_reference_style` | text | `first_last` | How members are referenced in minutes тАФ already exists |
| `minutes_consent_agenda` | boolean | `false` | Allow approval of minutes by consent agenda (no separate motion) |
| `minutes_requires_second` | boolean | `true` | Whether motion to approve minutes requires a second |
| `r4_board_member_default` | boolean | `true` | Can board members view draft minutes before approval (R4 permission default) |
| `audio_retention_policy` | enum / null | `null` (inherit) | Board-specific override; null = use town default |
| `auto_publish_on_approval` | boolean / null | `null` (inherit) | Board-specific override; null = use town default |

New columns:

```sql
ALTER TABLE board
  ADD COLUMN minutes_consent_agenda BOOLEAN NOT NULL DEFAULT false,
  ADD COLUMN minutes_requires_second BOOLEAN NOT NULL DEFAULT true,
  ADD COLUMN r4_board_member_default BOOLEAN NOT NULL DEFAULT true,
  ADD COLUMN audio_retention_policy_override TEXT
    CHECK (audio_retention_policy_override IN ('purge_on_approval','retain_30_days','retain_90_days','retain_indefinitely')),
  ADD COLUMN auto_publish_on_approval_override BOOLEAN;
  -- null = inherit from town
```

### 2.3 ProgressChecklist Completion Signal

"Configure minutes approval workflow" is complete when `town.minutes_workflow_configured_at IS NOT NULL`. This timestamp is set the first time the admin saves the `/settings/minutes-workflow` page, even if they accept all defaults.

## 3. Approval Process Rules

### 3.1 Edit Window

Any staff user with **R1 permission** may edit draft minutes for any meeting, regardless of which meeting they originally recorded. Editing is permitted from the time minutes are created until the board chair calls for the vote (i.e., up to the moment the motion result is recorded in the live meeting).

### 3.2 Lock Behavior

When a motion to approve minutes **passes**:
- `minutes_document.status` тЖТ `approved`
- `minutes_document.approved_at` тЖТ timestamp
- `minutes_document.approved_by_motion_id` тЖТ motion record ID
- Content (`content_json`, `html_rendered`) becomes **fully read-only** тАФ no edits by any role, including admin
- Metadata operations remain available: publish to portal, export PDF, attach addendum

When a motion to approve minutes **fails**:
- `minutes_document.status` тЖТ `draft` (automatically, triggered by the motion result)
- Content is unlocked and editable again (R1 permission)
- Minutes remain in draft until staff attaches them to a future meeting's agenda
- The failed adoption is recorded in `amendments_history` (see ┬з4)

### 3.3 Consent Agenda Approval

If `board.minutes_consent_agenda = true`, the agenda builder may group minutes approval under the consent agenda item. When the consent agenda motion passes, minutes are approved using the same lock logic. If the consent agenda is pulled to be voted separately, it reverts to standard approval flow.

## 4. Amendment & Addendum Data Model

### 4.1 Two Distinct Cases

| Case | Scenario | Resolution |
|------|----------|------------|
| **Pre-adoption amendment** | Minutes are in `draft` or `review` and need changes | Staff edits in place (R1 permission). Normal workflow continues. |
| **Post-adoption addendum** | Minutes have been `approved` and a formal motion to amend them passes at a subsequent meeting | Separate addendum record created. Original document unchanged. |

### 4.2 Addendum Record

New table:

```sql
CREATE TABLE minutes_addendum (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  minutes_document_id UUID NOT NULL REFERENCES minutes_document(id) ON DELETE CASCADE,
  town_id UUID NOT NULL REFERENCES town(id),
  adopting_meeting_id UUID NOT NULL REFERENCES meeting(id),  -- meeting where amendment was adopted
  adopting_motion_id UUID REFERENCES motion(id),
  content_json JSONB NOT NULL,             -- addendum content
  html_rendered TEXT,
  description TEXT NOT NULL,               -- brief description, e.g. "Corrected vote tally on item 4"
  created_by UUID REFERENCES user_account(id),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  published_at TIMESTAMPTZ                 -- when merged PDF was published to portal
);
```

### 4.3 Addendum Workflow

1. At a future meeting, an agenda item "Motion to Amend Minutes of [date]" is added (same type as regular minutes approval item)
2. The recording secretary drafts the addendum content in the `minutes_addendum` editor (accessible from `/meetings/:meetingId/minutes`)
3. The addendum is editable until the chair calls for the vote
4. When the motion passes: addendum is locked; a merged PDF (original + addendum) is generated and optionally published to portal
5. When the motion fails: addendum record is deleted; no changes to the original

### 4.4 `amendments_history` on `minutes_document`

The `minutes_document` table gains an `amendments_history` JSONB column to track all amendment events (failed adoption attempts + successful addenda):

```sql
ALTER TABLE minutes_document
  ADD COLUMN amendments_history JSONB NOT NULL DEFAULT '[]';
```

Each history entry:

```typescript
interface AmendmentHistoryEntry {
  event_type: 'adoption_failed' | 'addendum_adopted';
  occurred_at: string;           // ISO timestamp
  meeting_id: string;            // meeting where event occurred
  motion_id?: string;
  addendum_id?: string;          // for addendum_adopted events
  description?: string;
  recorded_by: string;           // user_account.id
}
```

## 5. Public Portal Behavior

### 5.1 Standard Approved Minutes

When `auto_publish_on_approval` is true (town or board level), publishing happens automatically when the motion passes. Otherwise, a staff user with R5 permission manually triggers publishing.

### 5.2 Amended Minutes (with Addendum)

When a `minutes_addendum` is adopted:
1. A merged PDF is generated: original minutes PDF + addendum PDF, concatenated
2. The merged PDF replaces the published document on the portal
3. The original un-amended PDF remains accessible in Supabase Storage for audit purposes
4. The meeting's portal page shows a badge: "Amended [date of amendment meeting]"

Portal display for a meeting with amendments:
- Primary document: merged PDF (original + addendum, labelled clearly)
- Historical documents section: original adopted minutes (for transparency)

## 6. Settings UI Design

### 6.1 `/settings/minutes-workflow` (Town Defaults)

Sections:
1. **Recording & Audio Retention** тАФ audio_retention_policy radio group with descriptions of each option
2. **Portal Publishing** тАФ auto_publish_on_approval toggle, explanation of R5 permission
3. **Review Window** тАФ minutes_review_window_days number input, description of how it affects notifications
4. Save button тЖТ sets `minutes_workflow_configured_at` on first save

### 6.2 Board Settings Tab тАФ Minutes Section

Shows inherited town defaults with "Override" toggles:
- Each setting shows the town default with a "Use town default / Override" toggle
- When override is active, the per-board value input appears
- Additional board-only settings: `minutes_consent_agenda`, `minutes_requires_second`, `r4_board_member_default`

---

## 7. Interaction with Existing Permissions

No changes to the permission matrix. The configurable settings (consent agenda, second required) inform the **validation logic** during live meeting agenda item processing тАФ they do not grant or remove permissions.

The `r4_board_member_default` setting controls the initial value of the R4 permission for new board members added to that board, not existing members' permissions.
