# Advisory 2.1 — Onboarding Wizard UX Specification

**Advisory Item:** The Onboarding Wizard Needs a Full UX Specification
**Date:** March 2026
**Status:** Resolved — all states, transitions, validation rules, and behaviors defined

---

## Decisions Summary

| Question | Decision |
|----------|----------|
| Mid-wizard exit | Nothing is saved. User must start over. Town is not created in the database until the final stage is completed. |
| Role-titled vs. at-large branching | No branching needed. This is a configuration setting changeable at any time by the admin. |
| Review screen before commit | No. Completing Stage 5 immediately creates the town. Everything is editable from the dashboard afterward. |
| Post-wizard landing | Town Profile dashboard with a brief 3–4 step tour, then the progress checklist |
| Empty states | Contextual guidance per screen throughout the app |
| Quorum rules | Simple majority default (>50% of seated members), overridable per board in settings |

---

## Wizard Behavior Rules

### Session Management
- The wizard is a **single-session, all-or-nothing flow.** No partial saves.
- If the user closes the browser, navigates away, or logs out during the wizard, all progress is lost and they must restart from Stage 1.
- This is intentional: the wizard takes under 10 minutes, and partial town configurations in the database create orphaned data and broken references. It is cleaner to require completion.
- The wizard state lives in client-side memory only (React state / React Hook Form). No database writes occur until the final completion action.

### Navigation Within the Wizard
- **Back button:** Users can navigate back to any previous stage. Data entered in later stages is preserved in client-side state — going back does not discard forward progress.
- **Forward navigation:** Users can only advance forward by completing the current stage's required fields. The "Next" button is disabled until validation passes.
- **Stage indicator:** A progress bar or step indicator shows which stage the user is on (1 of 5, 2 of 5, etc.) and which stages are complete.
- **Browser back button:** Should be intercepted with a confirmation dialog: "If you leave, your setup progress will be lost. Are you sure?"

### Sensible Defaults
- Every field has a sensible default so a confused user can hit "Next" and fix it later.
- Stage 1: State defaults to Maine. Municipality type defaults to Town. Population range defaults to "Under 1,000."
- Stage 2: Member count defaults to 3. Election method defaults to at-large. Officer election defaults to "Vote of the board."
- Stage 3: Presiding officer defaults to "Chair of the board." Minutes recorder defaults to "Town Clerk."
- Stage 4: No boards pre-checked (user must actively select their boards).
- Stage 5: Formality defaults to "Open (informal)." Minutes style defaults to "Summary."

## Stage-by-Stage Specification

### Stage 1 — Your Town

**Purpose:** Identify the municipality and establish billing/labeling context.

| Field | Type | Required | Validation | Default |
|-------|------|----------|------------|---------|
| Town name | Text input | Yes | Min 2 characters, max 100 characters. No special characters except hyphens, apostrophes, and periods (e.g., "Isle au Haut", "St. George"). | Empty |
| State | Dropdown | Yes | Options: Maine, New Hampshire, Vermont, Massachusetts, Connecticut, Rhode Island | Maine |
| Municipality type | Dropdown | Yes | Options: Town, City, Plantation | Town |
| Population range | Dropdown | Yes | Options: Under 1,000 · 1,000–2,500 · 2,500–5,000 · 5,000–10,000 · Over 10,000 | Under 1,000 |
| Primary contact name | Text input | Yes | Min 2 characters, max 100 characters | Empty |
| Primary contact role | Text input | Yes | Free text, max 100 characters. Examples shown as placeholder: "Town Manager", "Town Clerk", "Select Board Chair" | Empty |

**Downstream effects:**
- Population range sets the billing tier
- Town name + municipality type used in all generated labels ("Town of Newcastle" vs. "City of Belfast" vs. "Plantation of Monhegan")
- State will eventually determine which compliance rules apply (Maine Open Meetings Law for Phase 1)

---

### Stage 2 — Your Governing Board

**Purpose:** Configure the primary governing board (Select Board / Town Council). This is the board that runs the town — every town has one.

| Field | Type | Required | Validation | Default |
|-------|------|----------|------------|---------|
| Board name | Text input with suggestions | Yes | Free text, min 2 characters, max 100 characters. Dropdown suggestions: "Select Board", "Board of Selectmen", "Town Council", "Board of Aldermen" — user can type a custom name | Empty (suggestions shown) |
| Member count | Number input | Yes | Integer, minimum 0, maximum 15. **A board can be created with 0 members** but cannot hold a meeting unless it has at least 3 members and a quorum of those members is present. | 3 |
| Election method | Radio buttons | Yes | Options: "At-large" (all members hold equal seats) · "Role-titled" (each seat has a title, e.g., 1st Selectman, 2nd Selectman) | At-large |
| Seat titles | Dynamic list | Conditional — shown only if role-titled is selected | One text field per member count. Each min 2 characters, max 50 characters. | Pre-filled with "1st [Board name singular]", "2nd [Board name singular]", etc. |
| Officer election method | Dropdown | Yes | Options: "Vote of the board" · "Highest vote-getter in election" · "Appointed by authority" · "Rotation" | Vote of the board |
| District-based seats | Toggle (yes/no) | Yes | If yes, shows a district assignment interface (per-seat district label) | No |
| Staggered terms | Toggle (yes/no) | Yes | Informational — affects how term dates are tracked for members added later | No |

**Validation notes:**
- If member count is 0, a notice appears: "You can add members later. Note: a board must have at least 3 members to hold a meeting."
- If role-titled is selected and member count changes, seat title fields are added or removed to match. Existing titles are preserved.
- If the user switches from role-titled to at-large, any entered seat titles are discarded silently (they are cosmetic labels, not structural data — and the entire wizard is discardable anyway).
- Election method and officer election method are **display/configuration settings**, not permission determinants (per 1.2 — government titles ≠ app roles).

### Stage 3 — Who Runs Your Meetings

**Purpose:** Identify the default roles for meeting operations. This does not create user accounts — it tells the system what the town's structure looks like so it can set intelligent defaults.

| Field | Type | Required | Validation | Default |
|-------|------|----------|------------|---------|
| Presiding officer | Dropdown | Yes | Options: "Chair of the board" · "Town Manager" · "Town Administrator" · "Mayor" · "Moderator" | Chair of the board |
| Minutes recorder | Dropdown | Yes | Options: "Town Clerk" · "Deputy Clerk" · "Recording Secretary (board member)" · "Recording Secretary (staff)" · "Other staff" | Town Clerk |
| Staff roles present | Multi-select checkboxes | No | Options: "Town Manager" · "Town Administrator" · "Town Clerk" · "Deputy Clerk" · "None — volunteer board with no staff" | None checked |

**Validation notes:**
- All fields have defaults, so this stage can be completed by clicking "Next" without changing anything.
- "None — volunteer board with no staff" is a valid configuration for very small towns and plantations. Selecting it unchecks all other options.
- The staff roles selected here inform what staff permission templates are shown during user account creation later — but they do not create accounts.

**Downstream effects:**
- Presiding officer sets the default label in generated agenda headers and minutes
- Minutes recorder sets the default recording secretary assignment for new meetings
- Staff roles present informs the onboarding checklist (if no Town Clerk role exists, the "Set up Town Clerk account" checklist item is not shown)

### Stage 4 — Your Boards & Committees

**Purpose:** Register all boards and committees beyond the governing board. Each checked board gets a minimal configuration.

**Standard Maine boards checklist:**

| Board | Type | Notes |
|-------|------|-------|
| Planning Board | Elected or Appointed | Required by statute in most towns |
| Zoning Board of Appeals (ZBA) | Appointed | Required if town has zoning ordinance |
| Budget Committee | Elected or Appointed | Common but not universal |
| Conservation Commission | Appointed | |
| Parks & Recreation Committee | Appointed | |
| Harbor Committee | Appointed | Coastal towns |
| Shellfish Conservation Commission | Appointed | Coastal towns |
| Cemetery Committee | Appointed or Elected | |
| Road Committee | Appointed | |
| Comprehensive Plan Committee | Appointed | Formed as needed |
| Broadband Committee | Appointed | Increasingly common |

**Custom board option:** A free-text "Add another board or committee" field that can be used unlimited times.

**Per-board sub-configuration (shown inline after checking a board):**

| Field | Type | Required | Validation | Default |
|-------|------|----------|------------|---------|
| Board name | Pre-filled, editable | Yes | Min 2, max 100 characters | Standard name from checklist, or custom name |
| Member count | Number input | Yes | Integer, minimum 0, maximum 25 | 5 for standard boards, 0 for custom |
| Elected or Appointed | Radio buttons | Yes | Options: "Elected" · "Appointed" | Varies by board type (standard boards have known defaults) |

**Validation notes:**
- A town is not required to check any boards in this stage. A town with only a Select Board is valid.
- Appointing authority and formality default are **not collected in the wizard** — they are deferred to the Town Profile settings. This keeps the wizard fast.
- The 0-member count rule from Stage 2 applies here too: boards can exist with 0 members but cannot hold meetings.

### Stage 5 — Meeting Style & Minutes

**Purpose:** Set the town-level defaults for how meetings run and how minutes are formatted. These apply to all boards as defaults and can be overridden per board later.

| Field | Type | Required | Validation | Default |
|-------|------|----------|------------|---------|
| Meeting formality | Radio buttons with descriptions | Yes | Options below | Open (informal) |
| Minutes style | Radio buttons with descriptions | Yes | Options below | Summary |

**Meeting formality options:**

| Option | Label | Description shown to user |
|--------|-------|--------------------------|
| `informal` | Open (informal) | Discussion flows freely. The chair manages the conversation. Motions are made when the board is ready. Most small Maine towns operate this way. |
| `semi_formal` | Structured (semi-formal) | The chair follows the agenda in order. Members are recognized before speaking. Motions follow a clear pattern but without strict parliamentary procedure. |
| `formal` | Formal (Robert's Rules) | Strict parliamentary procedure. Members must be recognized. All discussion occurs within the context of a motion. Formal points of order are observed. |

**Minutes style options:**

| Option | Label | Description shown to user |
|--------|-------|--------------------------|
| `action` | Action minutes | Records only decisions, motions, votes, and assignments. Does not include discussion content. Shortest format. |
| `summary` | Summary minutes | Records decisions and a brief summary of key discussion points. The most common style for Maine municipal boards. |
| `narrative` | Narrative minutes | Records decisions and a detailed account of discussion, including who said what. Most thorough but most time-consuming to produce. |

**Downstream effects:**
- Meeting formality sets the live meeting manager's behavior mode (how the push-to-speak / motion capture / agenda flow UX works)
- Minutes style sets the AI minutes drafting template and the recording secretary's capture expectations
- Both are stored at the town level as defaults. Per-board overrides are configured in board settings after the wizard.

## Wizard Completion — What Happens

### The Moment of Creation

When the user clicks "Complete Setup" (or equivalent) on Stage 5:

1. **All wizard data is submitted to the backend in a single transaction.** This creates:
   - The TOWN record
   - The BOARD records (governing board + any boards checked in Stage 4)
   - The admin user account linked to the PERSON record (the person who ran the wizard)
   - Default town-level settings (formality, minutes style, presiding officer defaults, etc.)

2. **If the transaction fails** (network error, validation error), the wizard shows an error with a "Retry" button. The user does not have to re-enter data — it's still in client-side state.

3. **On success**, the wizard closes and the user transitions to the Town Profile dashboard.

### The Quick Tour (3–4 steps)

Immediately after the wizard completes, a brief overlay tour highlights the key areas of the app:

| Step | Highlight | Message |
|------|-----------|---------|
| 1 | Sidebar navigation | "Here's your navigation. You'll find your boards, meetings, agendas, and settings here." |
| 2 | Progress checklist | "This checklist shows what's left to set up. Start with adding your board members." |
| 3 | Board section | "Each board has its own area for meetings, agendas, and minutes." |
| 4 | Public portal link | "When you're ready, your town's public portal will be live at [subdomain].townmeetingmanager.com" |

- The tour is **skippable** ("Skip tour" link always visible).
- The tour is **one-time only** — it does not show again on subsequent logins.
- After the tour completes (or is skipped), the user is on the Town Profile dashboard.

### The Town Profile Dashboard

The dashboard is the home screen of the app. After the wizard, it shows:

**Progress Checklist (top of dashboard):**
A card or banner showing setup completion. Items are checked off as completed. Each item links to the relevant section of the app.

| Checklist Item | Required? | Links To |
|---------------|-----------|----------|
| Add board members (0 of N added) | Yes (before first meeting) | Board member management |
| Set meeting notice template | No | Town Profile settings |
| Configure minutes approval workflow | No | Town Profile settings |
| Upload town seal | No | Town Profile settings |
| Set public portal subdomain | No | Town Profile settings |
| Acknowledge retention policy | Yes (before first meeting) | Modal with policy text + acknowledge button |

**Collapsible sections (one per wizard stage):**
Each section shows the configured values with an "Edit" button. Changes take effect immediately — no re-running the wizard.

**Key principle:** The checklist items are **gentle prompts, not blockers.** The town can begin exploring the app, building agenda templates, and familiarizing themselves with the interface immediately. However, two items are required before the town can hold its first meeting:
1. At least 3 members must be added to a board
2. The retention policy must be acknowledged

### Empty States Throughout the App

Every screen in the app has a contextual empty state message that guides the user toward the prerequisite action:

| Screen | Empty State Message | Action Button |
|--------|-------------------|---------------|
| Meetings list | "No meetings scheduled yet. Add board members to your [Board Name], then create your first meeting." | "Add Board Members" (if none) or "Create Meeting" (if members exist) |
| Agenda builder | "No agendas yet. Create a meeting first, then build its agenda." | "Go to Meetings" |
| Minutes archive | "No minutes yet. Minutes are created automatically when you complete a meeting." | "Go to Meetings" |
| Public portal preview | "Your public portal is empty. Once you publish your first agenda or approved minutes, they'll appear here." | "Learn about the public portal" |
| Board member roster | "No members added yet. Add your [Board Name] members to get started." | "Add Member" |
| Straw polls | "No polls yet. Create a poll to gather resident input on an upcoming agenda item." | "Create Poll" |

Each empty state is specific to its context and tells the user what prerequisite must be met before this section has content.

## Validation Rules — Complete Reference

### Cross-Stage Rules

| Rule | Enforcement |
|------|------------|
| Town name must be unique within the platform | Checked on completion (not during typing — avoid premature validation calls) |
| A board cannot hold a meeting without at least 3 members | Enforced at meeting creation time, not during wizard |
| A meeting cannot start without a quorum present | Enforced at meeting start time |
| Quorum = simple majority (>50% of seated members) by default | Calculated automatically; overridable per board in settings |
| Retention policy must be acknowledged before first meeting | Enforced at meeting creation time |

## Resolution Status

**Item 2.1 is resolved.** The onboarding wizard now has a complete UX specification covering:

- **Session behavior:** All-or-nothing, no partial saves, client-side state only until completion
- **Navigation:** Back/forward with state preservation, browser back interception
- **Every field:** Type, required/optional, validation rules, defaults
- **Completion flow:** Single-transaction database creation → quick tour → dashboard with checklist
- **Empty states:** Contextual guidance per screen throughout the app
- **Post-wizard editing:** All settings editable from Town Profile without re-running wizard
- **Quorum:** Simple majority default, overridable per board
- **Meeting prerequisites:** Minimum 3 members + retention policy acknowledgment
