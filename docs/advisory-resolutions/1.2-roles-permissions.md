# Advisory 1.2 — Roles & Permissions Model

**Advisory Item:** Resolve the User Roles & Permissions Model
**Date:** March 2026
**Status:** Resolved — all decisions made, permissions matrix complete

---

## The Core Problem with the Current Spec

The current specification conflates **app roles** (what you can do in the software) with **government titles** (what you're called in town government). The spec defines five USER roles: `admin`, `town_clerk`, `staff`, `board_member`, `recording_secretary`. This is wrong in several ways:

1. **`town_clerk` is a government title, not a permission level.** A Town Clerk's permission needs within the app vary by town. In some towns the Clerk runs everything; in others, the Town Manager is the primary operator and the Clerk handles records. Making `town_clerk` a role bakes in an assumption about how every town operates.

2. **`recording_secretary` is not a role — it's a function.** A board member, a staff member, or even the admin can serve as recording secretary for a given meeting. It's a capability applied to a person for a meeting, not a permanent identity in the system.

3. **There is no system-level administrator concept.** Self-hosted deployments need someone who manages infrastructure — backups, deployment, configuration — separate from the town-level admin who manages boards and agendas.

4. **No permissions granularity within `staff`.** The spec treats all staff the same, but a Town Clerk and a Code Enforcement Officer have wildly different needs in the application.

---

## Revised Role Architecture

Four app-level roles, cleanly separated from government titles:

### 1. `sys_admin` — System Administrator
- Controls setup, deployment, backups, system configuration
- Relevant only for self-hosted deployments; in the SaaS model, this maps to the platform operator — individual towns never see this role
- Not a user of the meeting management features
- May not even be a town employee (could be an IT contractor)
- The app should detect which deployment model it's running in and hide `sys_admin` from the role picker in the SaaS version

### 2. `admin` — Town Administrator
- Full control over all app features
- Manages user accounts, initial wizard setup, town-level settings
- Also a regular user of the system — builds agendas, manages meetings, runs the tool
- Likely the person who initially sets up the town in the system
- Can set permissions for staff and board member accounts via the permissions matrix
- **Multiple admin accounts per town are allowed.** Restricting to a single admin creates a single point of failure. An audit log tracks which admin performed which action.

### 3. `staff` — Staff Account
- Permissions are **configurable per account** via a matrix controlled by the admin
- Permissions are **per-board with global defaults** — a Town Planner can have full agenda-building access for the Planning Board but view-only on the Select Board; a Finance Officer can have access to the Budget Committee but not the Conservation Commission
- Implementation: global default permissions + per-board overrides
- Different staff members have different needs:
  - Town Clerk → may need nearly admin-level access
  - Deputy Clerk → reduced set, focused on minutes and records
  - Treasurer / Finance Officer → may only need access to specific boards
  - Code Enforcement Officer → may need agenda building for Planning Board only
  - Town Planner → similar to CEO, board-specific access
- Default permission templates exist for quick account setup (see Permission Templates below)
- The government title (Town Clerk, Treasurer, etc.) is a **display label**, not a permission determinant
- **A staff member cannot simultaneously hold a board_member role.** See Conflict of Interest section below.

### 4. `board_member` — Board Member Account
- Default permissions: view agenda, download agenda as PDF, upload files to admin for review for inclusion on agenda items, submit questions/comments on agenda items before the meeting
- **All board member comments/questions flow through the admin.** There is no direct board member to board member communication within the app. The admin reviews submissions and decides whether to share them with the full board. This is absolute — it prevents any appearance of deliberation outside a public meeting.
- Cannot edit agendas unless admin grants permission through the matrix
- May serve on multiple boards simultaneously
- **A board member cannot simultaneously hold a staff role.** See Conflict of Interest section below.

### Cross-cutting: Recording Secretary
- **Not a role.** It is a permission/capability that can be applied to any `admin`, `staff`, or `board_member` account
- Can be assigned per meeting (ad hoc) or as a standing default
- A board member who is also recording secretary needs access to all recording secretary features during the meeting **in addition to** their board member capabilities
- A dedicated recording secretary (someone who only takes minutes and has no other role) gets an account with recording secretary permissions only — no board member voting/commenting features
- Tracked on the MEETING_ATTENDANCE record, not on the user account

### Cross-cutting: Government Title
- A **display label** stored on the user account or member record
- Examples: Town Clerk, Deputy Clerk, Treasurer, Finance Officer, Code Enforcement Officer, Town Planner, 1st Selectman, Chair, Vice Chair
- Has **no effect on permissions** — permissions come from the role + matrix
- Appears in meeting records, public rosters, and generated documents

## The PERSON Entity — Unified Identity Architecture

**Decision: Adopt the PERSON entity approach.**

Rather than separate USER and RESIDENT_ACCOUNT tables with no connection, a **PERSON** entity serves as the identity anchor:

```
┌─────────────────────────────────────────────────────────────┐
│                      PERSON (identity)                       │
│                                                              │
│  name, email (unique), created_at, archived_at               │
│                                                              │
│  ┌──────────────┐  ┌───────────────┐  ┌──────────────────┐  │
│  │ user_account │  │ board_members │  │ resident_account │  │
│  │ (0 or 1)     │  │ (0 to many)   │  │ (0 or 1)         │  │
│  │              │  │               │  │                  │  │
│  │ role:        │  │ board_id      │  │ subscriptions    │  │
│  │  sys_admin   │  │ seat_title    │  │ notification_    │  │
│  │  admin       │  │ term_start    │  │   preferences    │  │
│  │  staff       │  │ term_end      │  │ parcel_match     │  │
│  │  board_member│  │ status:       │  │ topic_tags       │  │
│  │              │  │  active       │  │ channel_prefs    │  │
│  │ gov_title    │  │  archived     │  │                  │  │
│  │ permissions  │  │               │  │                  │  │
│  │  (matrix)    │  │ rec_sec_      │  │                  │  │
│  │              │  │  default: T/F │  │                  │  │
│  └──────┬───────┘  └───────┬───────┘  └────────┬─────────┘  │
│         │                  │                    │            │
│         └──────────────────┼────────────────────┘            │
│                            │                                 │
│               All linked to one PERSON                       │
│               Historical records tied to PERSON              │
│               (never orphaned by account changes)            │
└─────────────────────────────────────────────────────────────┘
```

### What This Enables

- **Promotion:** A resident gets elected → admin adds a board_member role and board membership(s) to their existing PERSON record. Their resident notification preferences are preserved.
- **Demotion:** A board member's term ends → admin archives their board membership. If they have a resident_account, they continue as a resident. If not, they can be prompted to create one. Their historical votes, motions, and attendance remain tied to their PERSON record.
- **Multi-board:** One PERSON with multiple active entries in board_members, each with its own board_id, seat_title, and term dates.
- **Role transition:** A board member who later becomes staff (after leaving the board — see conflict-of-interest rules) gets their role changed on the same PERSON record. All history is preserved.
- **Archival:** When someone leaves town government entirely, their user_account is archived (login disabled), but their PERSON record and all historical data remain. Name and government title are retained indefinitely (public record). Personal contact info is scrubbed after a configurable retention period. If they return to government, their account is reactivated on the same PERSON record.

### Mutual Exclusivity Rule

A PERSON's user_account role of `staff` and a role of `board_member` are **mutually exclusive.** The system must enforce this — a person cannot hold both simultaneously. This is grounded in Maine conflict-of-interest law (see below). The system should prevent the admin from assigning both and display a clear explanation of why.

A PERSON *can* simultaneously have:
- `admin` + board memberships (the admin may also be an elected official, though this is uncommon)
- `staff` + resident_account (a staff member who also wants civic engagement notifications)
- `board_member` + resident_account (a board member who wants notifications about other boards they don't sit on)
- `admin` + resident_account

## Maine Conflict-of-Interest Law — Application to the Software

### Core Statute: 30-A M.R.S.A. §2605

This is the central conflict-of-interest statute for all Maine municipal officials. Key provisions the software must account for:

**Pecuniary Interest (§2605(1), (4)):**
- A vote is **voidable** when any official votes on a question in which they have a direct or indirect pecuniary interest
- Pecuniary interest is defined as: the official is an officer, director, partner, associate, employee, or stockholder of a private entity related to the question **AND** owns at least 10% of the stock or interest in that entity
- Both conditions must be met for a statutory conflict

**Disclosure & Abstention Safe Harbor (§2605(4)):**
- A vote is NOT voidable if the official: (1) makes full disclosure of the interest before action is taken, (2) abstains from voting, (3) abstains from negotiation/award of any contract, and (4) abstains from otherwise attempting to influence the decision
- **The disclosure and abstention must be recorded with the clerk or secretary** — this is a direct requirement the software can and should fulfill

**Appearance of Conflict (§2605(6)):**
- Every municipal official "shall attempt to avoid the appearance of a conflict of interest by disclosure or by abstention"
- This is broader than the pecuniary interest threshold — it covers situations that don't meet the 10% test but still look bad

**Municipal Ethics Policy (§2605(8)):**
- Municipalities may adopt their own ethics policies that are stricter than state law
- The software should allow configuration of additional local rules per town

**Former Officials — Revolving Door (§2605(7)):**
- No former official may, for one year after leaving office, act as agent or attorney or participate in proceedings before a municipal body in connection with matters that were pending before them and directly within their responsibilities

### Related Statutes

**Family Disqualification — 1 M.R.S.A. §71(6):**
- A board member related by blood or marriage within the 6th degree (2nd cousins) to an interested party is automatically disqualified unless written consent is obtained from all parties

**Board of Appeals Conflict Procedure — 30-A M.R.S.A. §2691:**
- For Boards of Appeal / Zoning Boards of Appeal only: whether a conflict exists is decided by majority vote of the remaining members (excluding the challenged member)
- This is the only board type with a statutory procedure for adjudicating conflicts

**Town Manager Incompatibility — 30-A M.R.S.A. §2632:**
- A Town Manager may NOT serve as moderator, Select Board member, assessor, or school committee member

**Treasurer/Tax Collector Incompatibility — 30-A M.R.S.A. §2526:**
- Treasurers and tax collectors cannot simultaneously serve as municipal officers or assessors

**Select Board Authority — 30-A M.R.S.A. §2635:**
- In towns with a Town Manager, the Select Board must deal with administrative services solely through the manager — they cannot give direct orders to subordinates of the manager

### Staff/Board Member Mutual Exclusivity — Legal Basis

While Maine statute does not explicitly prohibit a town employee from serving on a board, the legal landscape strongly discourages it:

- **1980 AG Opinion (Macri):** Selectman and town employee are incompatible because the selectman has power over hiring, firing, and compensation
- **2008 Superior Court (Harpswell v. Wallace):** Permitted dual service but only with recusal on employment-related matters
- **2011 AG Opinion (Gardiner):** Said the positions are not incompatible
- The area remains legally ambiguous with conflicting guidance

**For purposes of this software:** Staff and board member roles are treated as **mutually exclusive.** This is the conservative position that protects towns from conflict-of-interest challenges. A person must leave one role before assuming the other. The system enforces this and provides a clear explanation referencing the conflict-of-interest concern.

### Quasi-Judicial Board Requirements (Planning Board, ZBA)

Boards acting in a quasi-judicial capacity have additional obligations beyond §2605:
- **Bias/prejudgment** (not just financial interest) can require recusal
- **Ex parte communications** — members should not receive information about pending matters outside the public hearing
- Members must sit together, hear evidence together, and decide based on the same record

### What the Software Should Do

Based on this legal framework, the software should implement:

1. **Recusal Recording:** When a member recuses from a vote, the system captures: the member's name, the agenda item, the stated reason for recusal, and timestamps. This is recorded in the meeting minutes automatically. (Required by §2605(4) — disclosure and abstention must be recorded with the clerk.)

2. **Quorum Impact Alert:** When a recusal drops the participating members below quorum, the chair is immediately alerted. The vote cannot proceed.

3. **Role Incompatibility Enforcement:** The system prevents assigning both `staff` and `board_member` roles to the same PERSON. Clear error message explains the conflict-of-interest basis.

4. **Town Manager Incompatibility:** If the town has a Town Manager (captured during onboarding), the system prevents the Town Manager's account from being assigned as a member of the Select Board, school committee, or assessor role.

5. **Former Official Tracking:** When a user account is archived, the system records the archive date. For one year after, the PERSON record carries a flag indicating revolving-door restrictions. (Informational — the software flags it but doesn't need to enforce it beyond display.)

6. **Configurable Ethics Policy:** A town-level settings area where the admin can document additional local conflict-of-interest rules. These appear as reminders during meeting management but are not programmatically enforced (too varied across towns to codify).

7. **Board-Specific Conflict Procedures:** For Boards of Appeal, the system should support the §2691 workflow — when a conflict challenge is raised, the system facilitates a vote by the remaining members on whether the challenged member must recuse.

## Minutes Approval — How It Works in Maine

**Decision: Minutes are approved by the board, not by a single person.**

In Maine, minutes must be approved by the elected/appointed officials of the board, typically at the next regularly scheduled meeting of that same board. This means:

- "Approve minutes" is **not a permission assigned to one person** — it is a board action taken by vote during a meeting
- The workflow is: draft minutes are prepared → placed on the next meeting's agenda → the board reviews and votes to approve (with or without corrections) → the vote is recorded → minutes status changes to "approved"
- The software should support this as an agenda item type: "Approval of Minutes from [date]" with the draft attached
- Once approved by board vote, the minutes are locked (no further edits) and eligible for publication to the public portal
- The admin or designated staff member handles the mechanical steps (placing it on the agenda, publishing after approval), but the approval itself is a board action

## Board Member Comment/Question Flow

**Decision: All board member communications flow through the admin. No direct board-to-board communication within the app.**

This is a firm architectural constraint driven by Maine Open Meetings Law (1 M.R.S.A. §401–412). Board members deliberating outside of a public meeting — even through a software tool — could constitute an illegal meeting if a quorum of members participates.

### How It Works

1. A board member submits a question or comment on an agenda item through the app
2. The submission goes to the **admin only** — no other board member sees it
3. The admin reviews the submission and decides:
   - **Share with the board:** The admin can choose to distribute the comment/question to all board members on that board. The admin may add context or edit for clarity before sharing. The shared version is attributed to the original submitter.
   - **Respond privately:** The admin responds to the submitting member only
   - **Hold for meeting:** The admin notes it for discussion during the meeting but does not distribute it beforehand
4. Board members **never** see each other's submissions unless the admin explicitly shares them
5. Shared comments are **not published to the public portal** — they are internal working communications

### Why This Matters

- Prevents any claim that the board is deliberating outside a public meeting
- Gives the admin full visibility into what board members are thinking before the meeting
- Allows the admin to identify consensus or concerns before the meeting starts
- Protects the town legally

## Account Lifecycle

### Creation Paths

| Starting point | How the account is created |
|----------------|---------------------------|
| **New admin** | Created during initial town setup (onboarding wizard) or by an existing admin |
| **New staff member** | Created by an admin, template selected, permissions adjusted |
| **New board member** | Admin creates the PERSON (or finds existing), adds board membership with term dates, creates user_account with board_member role if login access is needed |
| **New resident** | Self-service through the public portal — name, contact method, interest profile. Creates a PERSON + resident_account. No user_account (no app login). |
| **Resident elected to board** | Admin finds the existing PERSON record (by email match), adds a user_account with board_member role and board membership. Resident_account and its preferences are preserved. |

### Transition Paths

| Transition | What happens |
|------------|-------------|
| **Board member → resident** | Board membership(s) archived (term_end set, status → archived). If PERSON has a resident_account, it continues. If not, admin or system prompts: "Would you like to continue receiving town notifications as a resident?" If yes, resident_account is created on the same PERSON. User_account login can be disabled or retained depending on whether they have other active roles. |
| **Board member → different board** | New board_members entry added. Old one may remain active (multi-board) or be archived (term ended). Same PERSON, same user_account. |
| **Board member → staff** | Board membership(s) must be archived first (mutual exclusivity). Then role on user_account changes from board_member to staff. Admin sets permissions via matrix. |
| **Staff → board member** | Staff role must be removed first (mutual exclusivity). Role changes to board_member. Board membership(s) added. |
| **Anyone → archived** | user_account login disabled. PERSON record retained. Name and government title retained indefinitely (public record). Personal contact info (email, phone) scrubbed after configurable retention period (default: 1 year after departure). Historical records (votes, motions, attendance, minutes actions) remain tied to the PERSON and are never deleted. |
| **Archived → reactivated** | If the person returns to government, admin reactivates the same PERSON record. Full history is preserved. A new user_account is created (or the old one re-enabled) on the same PERSON. |

### Data Retention Rules

| Data type | Retention |
|-----------|-----------|
| Name, government title | Indefinite (public record) |
| Vote records, motions, attendance | Indefinite (public record, legally required) |
| Meeting minutes references | Indefinite (public record) |
| Email address | Scrubbed after configurable period post-departure (default 1 year) |
| Phone number | Scrubbed after configurable period post-departure (default 1 year) |
| Login credentials | Deleted immediately upon archival |
| Resident account preferences | Retained as long as resident_account is active; deleted if resident_account is deleted |

## Guest Participants — Public Comment

During public hearings and public comment periods, non-account-holders may speak. The system handles this as:

- **Guest speaker entry:** The meeting operator creates a guest speaker record during the meeting — name required, address optional (speakers often state their name and address for the record)
- **No account creation:** Guest speaker entries are part of the meeting record only, not linked to any PERSON or account
- **No persistence:** If the same person speaks at a future meeting, the operator enters their name again. No attempt is made to link repeat guest appearances (these are infrequent and the overhead isn't worth it)
- **Audio capture consideration (future):** When the distributed audio capture feature is built (see 1.1), guest speakers don't have devices running the app. Their audio is captured by the room mic (operator's device) and attributed manually by the operator during or after the meeting

## Supabase RLS Policy Implications

The revised role model changes how Row Level Security policies must be structured:

### Current Spec (flat roles)
```
-- Simple role check
WHERE auth.jwt() ->> 'role' IN ('admin', 'town_clerk', 'staff')
```

### Revised Model (PERSON + matrix)
```
-- Must check: PERSON → user_account → role + permissions matrix + board scope
-- Pseudocode:
1. Identify the PERSON from the authenticated user
2. Check the user_account role (admin/staff/board_member)
3. If admin → allow (admin has full access)
4. If staff → look up permissions matrix for this staff account
   → Check if the action is permitted
   → Check if the action is permitted for this specific board
5. If board_member → check board membership is active for this board
   → Check the limited set of board_member permissions
   → Check recording_secretary assignment if relevant
```

### Key RLS Considerations

- **Multi-tenant isolation remains unchanged:** All queries are still scoped by `town_id`
- **Board-scoped permissions add a layer:** Staff with board-specific access need RLS policies that check both the permission matrix AND the board_id context
- **Recording secretary is contextual:** RLS for meeting-related writes must check MEETING_ATTENDANCE for recording_secretary assignment, not just the user_account role
- **PERSON → user_account join:** RLS policies will need to traverse the PERSON → user_account relationship, which means the permissions matrix should be stored in a way that's efficient for RLS evaluation (denormalized or cached, not computed on every query)

**Recommendation:** Store the effective permissions as a JSONB column on the user_account (or a related permissions table) rather than computing them from templates at query time. When the admin changes permissions, the JSONB is updated. RLS policies check the JSONB directly. This avoids complex joins in hot paths.
