# Advisory 1.3 — Tech Stack Evaluation

**Advisory Item:** Supabase Realtime Stress Test for the Live Meeting Manager
**Expanded Scope:** Full tech stack evaluation before committing to any framework or service
**Date:** March 2026
**Status:** Resolved — recommended stack identified, with alternatives documented

---

## Why This Evaluation Matters

The original advisory item (1.3) recommended a Supabase Realtime proof-of-concept to validate real-time sync during meetings. Before building that proof-of-concept, a more fundamental question must be answered: **is the assumed tech stack (Supabase + React + React Native + Fastify) the right foundation at all?**

The project has specific technical constraints that not every stack handles well:

1. **Resilient live meeting manager.** The live meeting manager must survive brief network interruptions gracefully. Rural Maine town halls sometimes have poor WiFi, but this is an increasingly rare edge case in 2026. The requirement is reconnection resilience and graceful degradation — not full offline capability.
2. **Real-time multi-device sync.** During a meeting, the operator's device and board members' devices must stay in sync — agenda item transitions, votes, motions.
3. **Complex relational data model.** 13+ entities with deep relationships — the PERSON model (1.2), board memberships, meetings, agenda items, motions, votes, minutes.
4. **Complex permissions.** Per-board, per-role permissions matrix with row-level security across a multi-tenant system.
5. **PostGIS required.** Parcel proximity matching in Phase 3 requires geographic queries.
6. **Self-hostable.** Part of the business model — towns may want to own their data.
7. **Audio recording.** Future phases require device-level audio capture with sync.
8. **Web + Mobile.** Both browser and native app (or PWA) needed.
9. **Small team.** Likely solo developer or very small team — stack complexity is a real cost.

---

## The Network Resilience Requirement

The live meeting manager must be resilient to brief network interruptions, but full offline-first capability is not required.

**What "resilient" means for this app:**
- Brief network drops (30 seconds or less) should not disrupt the meeting in progress
- Supabase Realtime reconnects automatically when connectivity returns
- React Query cache keeps meeting data readable during a brief disconnect
- Writes during a disconnect show a warning and block for up to 30 seconds, then surface an error
- A complete network outage during a meeting is an edge case for 2026 — graceful degradation (read-only mode with clear status) is the appropriate response

**What this means for the technology choice:** Full offline-first sync (local SQLite on every client, bidirectional write queuing, conflict resolution) adds significant engineering complexity for a failure mode that affects a small minority of meetings. Supabase Realtime with TanStack Query's built-in caching provides adequate resilience for the actual use case.

**Technologies evaluated for offline-first that are NOT needed:**

| Technology | Why it's not required |
|-----------|----------------------|
| PowerSync | Full offline sync — more than needed; adds Docker service, MongoDB, sync rules YAML |
| ElectricSQL | Read-only sync — not bidirectional |
| WatermelonDB | Mobile-first, weak web support |
| TinyBase | CRDT complexity not warranted for this use case |
| Triplit | Replaces Postgres entirely — loses PostGIS and Supabase Auth |

## Backend / Database Evaluation

### Option A: Supabase + TanStack Query (Recommended)

**What each piece does:**
- **Supabase** provides: PostgreSQL database, authentication (Supabase Auth), file storage (Supabase Storage for agenda packets, minutes PDFs, audio files), Edge Functions, and the server-side API
- **TanStack Query** provides: client-side server state management — caching, background refetch, optimistic updates, loading/error states, and cache invalidation. It queries Supabase directly via the Supabase client.
- **Supabase Realtime** provides: live multi-device sync during meetings via `postgres_changes` subscriptions — the primary mechanism for real-time sync in the live meeting manager

**Why this combination works:**
- Direct Supabase client queries with full JOIN support — no need for multiple queries and JS merging
- Proper TypeScript types throughout — no 0/1 boolean coercion, no JSON.stringify/parse for JSONB
- TanStack Query's stale-while-revalidate caching means data stays readable during brief network interruptions
- Supabase Realtime is purpose-built for exactly this use case: multi-device live sync with RLS-aware postgres_changes subscriptions
- React Query + Supabase Realtime together: initial load via React Query, live updates via Realtime subscriptions
- react-hook-form works without workarounds — no dual React instance issues
- PostgreSQL gives you PostGIS for Phase 3 parcel matching
- Supabase is self-hostable via Docker Compose — matches the business model
- Simpler infrastructure: Supabase + Fastify API, no additional sync service

**Supabase Realtime for Live Meeting Manager:**

Supabase Realtime with `postgres_changes` subscriptions is the primary real-time mechanism for multi-device meeting sync. It handles the core use case well:

- When the operator advances an agenda item, all board member devices receive the update within ~100ms under normal connectivity
- RLS policies apply to Realtime subscriptions — users only receive changes for their own town's data
- If the connection drops, the Supabase Realtime client reconnects automatically
- React Query's cache keeps the UI functional and readable during the brief reconnection window
- The `ConnectionStatusBar` component shows the Realtime connection state (Connected / Connecting / Disconnected)

**What changes from the original spec:**
- Supabase Realtime is promoted back to the primary real-time mechanism (it was demoted when PowerSync was the chosen approach)
- react-hook-form works directly — no useWizardForm workaround needed
- JOINs are now available — complex relational queries can be done in a single Supabase client call
- Proper boolean and JSONB types throughout — no SQLite type coercion

**Concerns:**
- A complete network outage during a meeting means writes block (no local queue). This is an acceptable tradeoff for 2026 where persistent offline meetings are an edge case for any SaaS product.
- Supabase Realtime + RLS has known quirks that need to be tested with the actual multi-tenant data model — this is why the proof-of-concept (see below) is still recommended.

**Verdict: Strongly recommended.** This combination provides real-time multi-device sync, full relational query support, proper TypeScript types, and straightforward development patterns — with significantly less infrastructure complexity than the PowerSync alternative.

### Option B: Supabase + TanStack Query + Offline Sync Layer

This is the approach with a dedicated offline sync library (ElectricSQL, TinyBase, etc.) added on top of the recommended stack.

**Why it's weaker than Option A:**
- Full offline sync requires local SQLite, sync protocol implementation, conflict resolution, and migration handling for offline devices
- These engineering costs are not justified by the actual failure mode — rural Maine WiFi outages during town meetings are rare and getting rarer
- The complexity penalty is paid by every developer working on the project, not just when offline scenarios occur
- Graceful degradation (read-only mode + clear status) is the appropriate response to a complete network outage in 2026

**When to consider it:** If pilot towns report persistent connectivity failures during real meetings. The architecture can be extended with an offline layer later — it is much easier to add than to remove.

**Verdict: Not recommended for Phase 1.** Build with graceful degradation first. Revisit if real-world usage data shows offline capability is needed.

### Option C: Triplit (Full Replacement)

Triplit is a CRDT-based full-stack database with built-in offline sync and real-time collaboration. It would replace both Supabase and TanStack Query.

**What it offers:**
- True offline-first with CRDT-based conflict resolution (per-attribute, not per-row)
- Real-time sync via WebSockets
- TypeScript-native, runs everywhere (browser, Node, React Native, Cloudflare Workers)
- Simpler architecture — one system instead of Supabase + TanStack Query

**Why it's risky:**
- Replaces PostgreSQL entirely — you lose PostGIS (Phase 3 parcel matching), mature query planning, decades of Postgres tooling
- No Supabase Auth equivalent — you must build or integrate authentication separately
- No file storage equivalent — you need a separate solution for PDFs, audio files, attachments
- Smaller ecosystem and community — fewer resources, harder to hire developers who know it
- Lock-in to Triplit's proprietary data layer
- No equivalent to Supabase's RLS — you must implement authorization in application code
- 1.0 released April 2025 — young for a municipal SaaS product where reliability is paramount

**When to consider it:** If you were building a greenfield collaborative app with no geographic data requirements, no complex file storage needs, and were comfortable with a newer database platform.

**Verdict: Not recommended.** The loss of PostgreSQL, PostGIS, Supabase Auth, and Supabase Storage creates more problems than it solves.

### Option D: Custom Backend (Fastify/Express + PostgreSQL) Without Supabase

Build everything from scratch: custom API server, custom auth, custom file storage, custom real-time.

**What it offers:**
- Complete control over every layer
- No vendor dependencies
- Exact permissions model without working around BaaS abstractions

**Why it's risky:**
- Enormous engineering surface area for a small team — auth, RLS-equivalent middleware, file storage, real-time pub/sub, all from scratch
- The time spent building auth, storage, and real-time is time not spent on the meeting management features that differentiate the product
- Supabase's value is precisely that it provides these foundations so you can focus on business logic

**When to consider it:** If Supabase's abstractions become a bottleneck — particularly if RLS policies can't express the complex permissions matrix from 1.2, or if Edge Functions are too limited for the AI/document generation pipeline.

**Verdict: Not recommended for Phase 1.** Keep this as a migration path if Supabase becomes constraining. The API layer (Fastify or similar) should exist for complex operations (AI minutes, document generation, notification dispatch), but use Supabase for auth, storage, and the database.

---

### Option E: Django + PostgreSQL

Python-based full-stack framework with a mature ORM, built-in admin panel, and GeoDjango for PostGIS.

**What it offers:**
- GeoDjango is the most mature PostGIS integration in any web framework
- Built-in admin panel could accelerate the admin/staff interface
- Django REST Framework for API development
- Mature auth system with granular permissions
- Large developer community

**Why it doesn't fit:**
- Python, not TypeScript — you lose the shared type system across web, mobile, and API that was identified as the primary reason for a monorepo
- Real-time support (Django Channels) exists but is less mature than Supabase Realtime
- React Native integration requires a separate API layer anyway
- Two language ecosystems (Python backend + TypeScript frontend) increases complexity for a small team

**When to consider it:** If Phase 3 parcel matching becomes the primary feature and PostGIS query complexity demands GeoDjango's abstractions. But PostGIS works directly through Supabase's Postgres, so this advantage is marginal.

**Verdict: Not recommended.** The TypeScript full-stack advantage outweighs Django's maturity for a project that shares types across web, mobile, and API.

## Frontend Framework Evaluation

### Option A: React + React Router v7 (fka Remix) (Recommended)

**What it offers:**
- Progressive enhancement — forms work without JavaScript
- Loader/action pattern maps well to React Query's `queryClient.ensureQueryData()` pattern in `clientLoader`
- React ecosystem enables code sharing with React Native
- Less Vercel lock-in than Next.js
- Remix merged into React Router v7 in December 2024 — the same tool, now unified

**Concerns:**
- Smaller community than Next.js
- Less corporate backing than Next.js (Vercel)

**Verdict: Recommended for web frontend.** The loader/action pattern integrates cleanly with TanStack Query's prefetching patterns.

---

### Option B: Next.js

**What it offers:**
- Largest React ecosystem, most developer availability
- Server Components, API routes, built-in optimizations
- Massive community, abundant tutorials and libraries

**Concerns:**
- Server Components model adds complexity for a client-heavy SaaS app
- Increasingly coupled to Vercel's infrastructure
- App Router complexity adds overhead for a small team

**Verdict: Acceptable but not ideal.** If hiring is the top concern, Next.js is safer. Use Pages Router if you go this route — App Router adds unnecessary complexity for this use case.

---

### Option C: SvelteKit

**What it offers:**
- Smallest bundle sizes (compiler approach)
- Svelte 5 runes system is clean and productive
- Less boilerplate than React

**Concerns:**
- No React Native equivalent — you'd need a different technology for mobile (Flutter, Capacitor)
- Smaller developer community
- Breaks the shared-code advantage between web and mobile

**Verdict: Not recommended.** The loss of code sharing with React Native is a dealbreaker.

---

### Option D: Progressive Web App (PWA) Instead of React Native

**What it offers:**
- One codebase for web and mobile
- Service Workers provide offline caching
- No app store distribution friction
- Simpler development and deployment
- Web Push Notifications now supported on iOS 16.4+ (PWA installed to home screen)

**Concerns:**
- Audio recording API access varies by platform/browser
- "Feels like a website, not an app" — may matter for board members during meetings
- Board members may expect a native app experience

**Verdict: Worth considering as a Phase 1 alternative to React Native.** If the mobile experience is primarily "view agenda, submit comments, vote during meeting," a PWA is sufficient. React Native can be added in Phase 2 if native capabilities are needed (background audio recording). Phase 1 now includes Web Push Notifications for the PWA (meeting reminders, agenda published, minutes approved).

## Mobile Strategy

| Option | Network Resilience | Audio Recording | Push Notifications | Code Sharing with Web | Development Effort |
|--------|-------------------|----------------|-------------------|--------------------|-------------------|
| **React Native + Expo** | React Query cache | Native API access | Yes (expo-notifications) | High (React + shared types) | Moderate |
| **PWA** | React Query cache + SW | Web Audio API (limited) | Yes (Web Push, iOS 16.4+) | Maximum (same codebase) | Low |

**Recommendation:** Start with a PWA for Phase 1. The board member experience (view agenda, submit comments, vote) doesn't require native capabilities. Web Push Notifications are a Phase 1 feature for the PWA. Add React Native via Expo in Phase 2 when audio recording is a requirement.

## Supabase Realtime Proof-of-Concept

The original advisory recommended building a Supabase Realtime proof-of-concept before the live meeting manager. This remains the right approach — and is now more important than ever, since Realtime is the primary real-time mechanism.

**Proof-of-concept scope:**
1. Set up Supabase project with meeting-related tables (meeting, agenda_item, vote_record)
2. Enable `postgres_changes` Realtime on these tables with RLS policies active
3. Build a minimal React web client with TanStack Query initial load + Realtime subscription for live updates
4. Simulate a meeting: one "operator" device advancing agenda items, two "board member" devices receiving updates
5. Test RLS on Realtime: verify board member devices only receive changes for their own town's data
6. Test reconnection: disconnect the operator device, reconnect, verify the Realtime subscription resumes and React Query refetches
7. Measure sync latency under normal connectivity (target: <200ms)
8. Test the ConnectionStatusBar: verify it shows accurate connection state during disconnect/reconnect

This is a 2–3 day exercise that validates the core architecture before building the live meeting manager.

## Resolution Status

**Item 1.3 is resolved.** The tech stack evaluation identified TanStack Query + Supabase Realtime as the recommended client data architecture. Supabase Realtime is the primary real-time sync mechanism for the live meeting manager. Graceful degradation replaces the offline-first survival requirement.

### Key Decisions Made

1. **Supabase remains the backend foundation** — database, auth, storage, edge functions
2. **TanStack Query for all client data management** — caching, background refetch, optimistic updates, loading/error states
3. **Supabase Realtime for live meeting sync** — primary mechanism, promoted from secondary role
4. **React Router v7 (Remix) for web frontend** — progressive enhancement, integrates with React Query prefetching
5. **PWA for Phase 1 mobile + Web Push Notifications** — reduces initial complexity while including push
6. **React Native for Phase 2** — audio recording, native performance
7. **Graceful degradation for network outages** — read-only mode + clear status, not offline write queue
8. **Proof-of-concept scope updated** — validate Supabase Realtime + TanStack Query + RLS together

### Remaining Work
1. Build the Supabase Realtime proof-of-concept (2–3 days)
2. Validate RLS on Realtime channels with the actual multi-tenant data model
3. Design ConnectionStatusBar behavior for disconnect/reconnect scenarios
4. Define the 30-second write-blocking window for brief disconnections
5. Prototype PWA offline experience and Web Push Notification flow for board members
