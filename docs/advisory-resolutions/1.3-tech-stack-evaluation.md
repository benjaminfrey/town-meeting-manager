# Advisory 1.3 — Tech Stack Evaluation

**Advisory Item:** Supabase Realtime Stress Test for the Live Meeting Manager
**Expanded Scope:** Full tech stack evaluation before committing to any framework or service
**Date:** March 2026
**Status:** Resolved — recommended stack identified, with alternatives documented

---

## Why This Evaluation Matters

The original advisory item (1.3) recommended a Supabase Realtime proof-of-concept to validate real-time sync during meetings. But before building that proof-of-concept, a more fundamental question must be answered: **is the assumed tech stack (Supabase + React + React Native + Fastify) the right foundation at all?**

The project has specific technical constraints that not every stack handles well:

1. **Offline-first is a survival requirement.** The live meeting manager must work completely without network connectivity. Rural Maine town halls often have poor WiFi. If the app freezes during a meeting because it lost connectivity, that town will never use it again.
2. **Real-time multi-device sync.** During a meeting, the operator's device and board members' devices must stay in sync — agenda item transitions, votes, motions.
3. **Complex relational data model.** 13+ entities with deep relationships — the PERSON model (1.2), board memberships, meetings, agenda items, motions, votes, minutes.
4. **Complex permissions.** Per-board, per-role permissions matrix with row-level security across a multi-tenant system.
5. **PostGIS required.** Parcel proximity matching in Phase 3 requires geographic queries.
6. **Self-hostable.** Part of the business model — towns may want to own their data.
7. **Audio recording.** Future phases require device-level audio capture with sync.
8. **Web + Mobile.** Both browser and native app (or PWA) needed.
9. **Small team.** Likely solo developer or very small team — stack complexity is a real cost.

---

## The Decisive Constraint: Offline-First

The offline-first requirement eliminates a surprising number of options and reshapes how the remaining options fit together.

**What "offline-first" means for this app:**
- The live meeting manager must be fully functional with no network connectivity
- Data entered during a meeting (agenda transitions, motions, votes, timestamps) must be stored locally on the device
- When connectivity is restored, local changes must sync to the server without data loss
- Conflict resolution must be handled gracefully if multiple devices were recording simultaneously

**Technologies eliminated by the offline-first requirement:**

| Technology | Why it's eliminated |
|-----------|-------------------|
| Phoenix LiveView | Requires a constant WebSocket connection — no server = no UI |
| Convex | Cloud-first, server-authoritative — offline writes are alpha/experimental only |
| Zero (Rocicorp) | Explicitly does not support offline writes — writes are rejected when disconnected |
| PocketBase | No offline sync capabilities, pre-1.0 stability, solo maintainer |
| Supabase Realtime alone | Realtime is a server push mechanism — it goes silent when the network drops |

**Technologies that survive the offline-first filter:**

| Technology | Offline writes | Production ready | Postgres compatible | Web + Mobile |
|-----------|---------------|-----------------|-------------------|-------------|
| **PowerSync** | Yes | Yes | Native | Both |
| **ElectricSQL** | No (read-only sync) | Yes (1.0+) | Native | Both |
| **WatermelonDB** | Yes | Yes | Via custom endpoints | Mobile-first, weak web |
| **TinyBase** | Yes (CRDT) | Yes | Via persistence | Both |
| **Triplit** | Yes (CRDT) | Yes (1.0) | No — replaces Postgres | Both |
| **Replicache** | Yes | Yes (but sunsetting) | Via custom backend | Web + community RN |

## Backend / Database Evaluation

### Option A: Supabase + PowerSync (Recommended)

**What each piece does:**
- **Supabase** provides: PostgreSQL database, authentication (Supabase Auth), file storage (Supabase Storage for agenda packets, minutes PDFs, audio files), Edge Functions, and the server-side API
- **PowerSync** provides: Bidirectional offline-first sync between Supabase Postgres and local SQLite on client devices, with Sync Rules for per-tenant data isolation and an upload queue for writes made while offline

**Why this combination works:**
- PowerSync is an official Supabase integration partner — well-documented, tested together
- Local SQLite on every client device means the live meeting manager works with zero connectivity
- Writes go into a local upload queue and push to Supabase when connectivity returns
- Sync Rules control which data syncs to which users — maps directly to the multi-tenant + per-board permissions model
- Custom conflict resolution runs in the Supabase backend (you write the logic)
- PostgreSQL gives you PostGIS for Phase 3 parcel matching
- Supabase is self-hostable via Docker Compose — matches the business model
- PowerSync has a self-hosted Open Edition — can be co-deployed with Supabase

**What changes from the original spec:**
- Supabase Realtime becomes less critical — PowerSync handles the sync that Realtime was supposed to do
- Supabase Realtime can still be used for lightweight presence/notification features (who's in the meeting room) while PowerSync handles data sync
- The Fastify API layer may be partially replaced by Supabase Edge Functions for simpler operations, though a standalone API is still needed for complex business logic (AI minutes generation, document generation, notification dispatch)

**Concerns:**
- Adding PowerSync introduces another moving part (though it replaces a harder problem — building offline sync from scratch)
- PowerSync Cloud pricing scales with daily active users (~$51/mo for 5K DAU, ~$399/mo for 100K DAU) — acceptable for the municipal market
- Conflict resolution logic must be carefully designed for the meeting context (what happens if two devices record different vote counts?)
- Self-hosting PowerSync requires running their Open Edition alongside Supabase — more infrastructure to manage

**Verdict: Strongly recommended.** This is the only production-ready combination that provides offline-first bidirectional sync, native Postgres integration, web + React Native support, multi-tenant capability, and self-hosting — all of which are non-negotiable requirements.

### Option B: Supabase Only (No Sync Layer)

This is the original spec's approach — use Supabase Realtime for multi-device sync and IndexedDB for offline storage.

**Why it's weaker:**
- You must build offline sync from scratch: local storage, change tracking, upload queue, conflict resolution, retry logic — this is months of engineering
- Supabase Realtime + RLS has known quirks (acknowledged in the original advisory)
- IndexedDB is a key-value store — querying it like a relational database requires an abstraction layer (Dexie.js or similar)
- No sync protocol means you're inventing one — every edge case (partial sync, interrupted uploads, schema migrations on offline devices) must be handled manually

**When to consider it:** Only if PowerSync introduces unacceptable complexity or cost. For Phase 1 MVP, you could build with Supabase only and add PowerSync later — but retrofitting offline sync is significantly harder than building with it from the start.

**Verdict: Not recommended as the primary approach.** The offline-first requirement is too critical to build from scratch.

---

### Option C: Triplit (Full Replacement)

Triplit is a CRDT-based full-stack database with built-in offline sync and real-time collaboration. It would replace both Supabase and PowerSync.

**What it offers:**
- True offline-first with CRDT-based conflict resolution (per-attribute, not per-row)
- Real-time sync via WebSockets
- TypeScript-native, runs everywhere (browser, Node, React Native, Cloudflare Workers)
- Simpler architecture — one system instead of Supabase + PowerSync

**Why it's risky:**
- Replaces PostgreSQL entirely — you lose PostGIS (Phase 3 parcel matching), mature query planning, decades of Postgres tooling
- No Supabase Auth equivalent — you must build or integrate authentication separately
- No file storage equivalent — you need a separate solution for PDFs, audio files, attachments
- Smaller ecosystem and community — fewer resources, harder to hire developers who know it
- Lock-in to Triplit's proprietary data layer
- No equivalent to Supabase's RLS — you must implement authorization in application code
- 1.0 released April 2025 — young for a municipal SaaS product where reliability is paramount

**When to consider it:** If you were building a greenfield collaborative app with no geographic data requirements, no complex file storage needs, and were comfortable with a newer database platform.

**Verdict: Not recommended.** The loss of PostgreSQL, PostGIS, Supabase Auth, and Supabase Storage creates more problems than it solves.

### Option D: Custom Backend (Fastify/Express + PostgreSQL) Without Supabase

Build everything from scratch: custom API server, custom auth, custom file storage, custom real-time.

**What it offers:**
- Complete control over every layer
- No vendor dependencies
- Exact permissions model without working around BaaS abstractions

**Why it's risky:**
- Enormous engineering surface area for a small team — auth, RLS-equivalent middleware, file storage, real-time pub/sub, all from scratch
- Still need an offline sync layer (PowerSync or equivalent) — PowerSync works with any Postgres, so this part doesn't change
- The time spent building auth, storage, and real-time is time not spent on the meeting management features that differentiate the product
- Supabase's value is precisely that it provides these foundations so you can focus on business logic

**When to consider it:** If Supabase's abstractions become a bottleneck — particularly if RLS policies can't express the complex permissions matrix from 1.2, or if Edge Functions are too limited for the AI/document generation pipeline.

**Verdict: Not recommended for Phase 1.** Keep this as a migration path if Supabase becomes constraining. The API layer (Fastify or similar) should exist for complex operations (AI minutes, document generation, notification dispatch), but use Supabase for auth, storage, and the database.

---

### Option E: Django + PostgreSQL

Python-based full-stack framework with a mature ORM, built-in admin panel, and GeoDjango for PostGIS.

**What it offers:**
- GeoDjango is the most mature PostGIS integration in any web framework
- Built-in admin panel could accelerate the admin/staff interface
- Django REST Framework for API development
- Mature auth system with granular permissions
- Large developer community

**Why it doesn't fit:**
- Python, not TypeScript — you lose the shared type system across web, mobile, and API that was identified as the primary reason for a monorepo
- Still need an offline sync layer — Django doesn't solve this
- Real-time support (Django Channels) exists but is less mature than Supabase Realtime
- React Native integration requires a separate API layer anyway
- Two language ecosystems (Python backend + TypeScript frontend) increases complexity for a small team

**When to consider it:** If Phase 3 parcel matching becomes the primary feature and PostGIS query complexity demands GeoDjango's abstractions. But PostGIS works directly through Supabase's Postgres, so this advantage is marginal.

**Verdict: Not recommended.** The TypeScript full-stack advantage outweighs Django's maturity for a project that shares types across web, mobile, and API.

## Frontend Framework Evaluation

### Option A: React + React Router v7 (fka Remix) (Recommended)

**What it offers:**
- Progressive enhancement — forms work without JavaScript, which aids offline resilience
- Loader/action pattern maps well to PowerSync's upload queue model
- React ecosystem enables code sharing with React Native
- Less Vercel lock-in than Next.js
- Remix merged into React Router v7 in December 2024 — the same tool, now unified

**Concerns:**
- Smaller community than Next.js
- Less corporate backing than Next.js (Vercel)

**Verdict: Recommended for web frontend.** The progressive enhancement model and loader/action pattern are the best architectural fit for an offline-first app.

---

### Option B: Next.js

**What it offers:**
- Largest React ecosystem, most developer availability
- Server Components, API routes, built-in optimizations
- Massive community, abundant tutorials and libraries

**Concerns:**
- Server Components model conflicts with offline-first — RSC requires a server
- Increasingly coupled to Vercel's infrastructure
- App Router complexity adds overhead for a small team
- Fighting the framework's server-centric architecture when implementing offline features

**Verdict: Acceptable but not ideal.** If hiring is the top concern, Next.js is safer. But its architecture fights the offline-first requirement. Use Pages Router if you go this route — App Router adds unnecessary complexity for this use case.

---

### Option C: SvelteKit

**What it offers:**
- Smallest bundle sizes (compiler approach)
- Svelte 5 runes system is clean and productive
- Less boilerplate than React

**Concerns:**
- No React Native equivalent — you'd need a different technology for mobile (Flutter, Capacitor)
- Smaller developer community
- PowerSync has React and React Native SDKs — Svelte support would need the generic JS client
- Breaks the shared-code advantage between web and mobile

**Verdict: Not recommended.** The loss of code sharing with React Native is a dealbreaker.

---

### Option D: Progressive Web App (PWA) Instead of React Native

**What it offers:**
- One codebase for web and mobile
- Service Workers provide offline caching
- No app store distribution friction
- Simpler development and deployment

**Concerns:**
- PWA on iOS still has significant limitations (limited background processing, no push notifications without workarounds, storage quotas)
- Audio recording API access varies by platform/browser
- "Feels like a website, not an app" — may matter for board members during meetings
- Board members may expect a native app experience

**Verdict: Worth considering as a Phase 1 alternative to React Native.** If the mobile experience is primarily "view agenda, submit comments, vote during meeting," a PWA may be sufficient. React Native can be added in Phase 2 if native capabilities are needed (push notifications, background audio recording). PowerSync supports both PWA (web SDK) and React Native.

## Mobile Strategy

| Option | Offline Support | Audio Recording | Push Notifications | Code Sharing with Web | Development Effort |
|--------|----------------|----------------|-------------------|--------------------|-------------------|
| **React Native + Expo** | Via PowerSync RN SDK | Native API access | Yes | High (React + shared types) | Moderate |
| **PWA** | Via Service Workers | Web Audio API (limited) | Limited on iOS | Maximum (same codebase) | Low |
| **Capacitor (Ionic)** | Via PowerSync web SDK | Native plugin available | Yes | High (web stack) | Low–Moderate |

**Recommendation:** Start with a PWA for Phase 1. The board member experience (view agenda, submit comments, vote) doesn't require native capabilities. Add React Native via Expo in Phase 2 when audio recording and push notifications become requirements.

## Recommended Tech Stack

### Core Stack

| Layer | Technology | Rationale |
|-------|-----------|-----------|
| **Database** | Supabase (PostgreSQL) | Relational model, PostGIS, RLS, Auth, Storage, self-hostable |

## What Changed from the Original Spec

| Original Spec | Revised | Why |
|--------------|---------|-----|
| Supabase Realtime for device sync | PowerSync for offline-first data sync | Realtime requires connectivity; PowerSync works offline |

## Impact on the Supabase Realtime Proof-of-Concept

The original advisory recommended building a Supabase Realtime proof-of-concept before the live meeting manager. With PowerSync as the sync layer, this proof-of-concept changes:

**New proof-of-concept scope:**
1. Set up Supabase project with meeting-related tables (meeting, agenda_item, vote_record)
2. Set up PowerSync with Sync Rules scoping data to a single meeting
3. Build a minimal React web client with PowerSync local SQLite
4. Simulate a meeting: one "operator" device advancing agenda items, two "board member" devices receiving updates
5. Test offline scenario: disconnect the operator device, make changes, reconnect, verify sync
6. Test conflict scenario: two devices submit changes while disconnected, reconnect, verify resolution
7. Measure sync latency under normal connectivity

This is still a 2–3 day exercise, but it validates PowerSync + Supabase rather than Supabase Realtime alone.

## Resolution Status

**Item 1.3 is resolved.** The tech stack evaluation identified PowerSync as a critical addition to the Supabase foundation, replacing Supabase Realtime as the primary sync mechanism for the live meeting manager. The offline-first requirement was the decisive constraint.

### Key Decisions Made

1. **Supabase remains the backend foundation** — database, auth, storage, edge functions
2. **PowerSync added for offline-first sync** — bidirectional, local SQLite, production-ready
3. **React Router v7 (Remix) for web frontend** — progressive enhancement fits offline-first
4. **PWA for Phase 1 mobile, React Native for Phase 2** — reduces initial complexity
5. **Supabase Realtime demoted to presence/notifications** — not relied upon for data sync
6. **Proof-of-concept scope updated** — validate PowerSync + Supabase, not Realtime alone

### Remaining Work
1. Build the PowerSync + Supabase proof-of-concept (2–3 days)
2. Design PowerSync Sync Rules for multi-tenant data isolation
3. Define conflict resolution strategy for meeting data (what happens when two devices disagree)
4. Evaluate PowerSync Cloud pricing vs. self-hosted Open Edition for the business model
5. Prototype PWA offline experience for board member view
