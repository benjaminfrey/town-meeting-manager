## The TanStack Query Question

**Short answer: No, TanStack Query is not needed. PowerSync replaces it.**

TanStack Query (React Query) solves a specific problem: managing server state in a client application — caching API responses, handling loading/error/stale states, deduplicating requests, background refetching, and cache invalidation. Every component that needs data makes a network request, and TanStack Query manages the lifecycle of those requests.

**With PowerSync, that problem doesn't exist.** The data lives in local SQLite on the device. Components query the local database, not a remote API. There is no network request to cache, no response to invalidate, no loading state for a round-trip that never happens. PowerSync handles sync in the background — your components don't know or care whether the data came from the network 5 seconds ago or 5 hours ago.

### What PowerSync provides instead

PowerSync's `@powersync/react` package includes reactive hooks that directly replace TanStack Query:

| TanStack Query | PowerSync Equivalent | Difference |
|---------------|---------------------|------------|
| `useQuery()` | `useQuery(sql, params)` | Queries local SQLite, auto-rerenders when underlying tables change |
| `useSuspenseQuery()` | `useSuspenseQuery(sql, params)` | Same, but delegates loading/error to React Suspense + ErrorBoundary |
| `useMutation()` | `powersync.execute()` / `powersync.writeTransaction()` | Writes directly to local SQLite; PowerSync queues for sync automatically |
| `useQueryClient().invalidateQueries()` | Not needed | PowerSync watches SQLite tables; when data changes (locally or via sync), queries rerender automatically |
| Query caching | SQLite IS the cache | Persistent across page reloads, no cache invalidation logic |
| Background refetching | PowerSync background sync | Continuous, handled by the SDK |
| Optimistic updates | Writes ARE local-first | Every write is instantly reflected because it goes to local SQLite first |

### When TanStack Query IS still useful

There is a narrow category of operations that don't go through PowerSync — API calls that aren't database read/write operations:

- **AI minutes generation** — calling the Claude API to draft minutes from structured meeting data
- **Document generation** — requesting a PDF from the Fastify API
- **File uploads** — uploading audio recordings or attachments to Supabase Storage
- **One-off server queries** — e.g., checking whether a subdomain is available during onboarding

For these, you have two options:
1. **Use TanStack Query** for these specific non-PowerSync API calls
2. **Use simple `fetch` + local `useState`** for loading/error states

**Recommendation:** Don't add TanStack Query. These non-database API calls are infrequent and simple enough to handle with `fetch` + `useState` or a lightweight custom `useFetch` hook. Adding TanStack Query for 5% of your data operations while PowerSync handles 95% introduces an unnecessary dependency and two competing data-fetching paradigms.

PowerSync does offer an official `@powersync/tanstack-react-query` integration package, but it's in **alpha** status and is designed for teams migrating from TanStack Query to PowerSync, not for new projects.

## State Management — What Goes Where

PowerSync eliminates the need for global state management libraries. Here's the complete state taxonomy:

| State Type | Examples in the Meeting Manager | Where It Lives |
|-----------|-------------------------------|---------------|
| **Persisted application data** | Meetings, agenda items, motions, votes, persons, board memberships, minutes | PowerSync local SQLite (single source of truth) |
| **Component-local UI state** | Is the motion dialog open? Which tab is active? Sidebar collapsed? | `useState` / `useReducer` |
| **Form state** | Current field values, validation errors, dirty tracking | React Hook Form |
| **Navigation state** | Current route, URL parameters, breadcrumbs | React Router v7 |
| **Cross-component session state** | Active meeting context, toast notification queue, theme preference | React Context (2–3 contexts max) |
| **Derived/computed state** | Filtered agenda items, vote tallies, quorum calculations | `useMemo` over PowerSync query results |
| **Sync status indicators** | Online/offline, pending uploads count, "syncing..." | PowerSync's `useStatus()` hook |

**No Redux. No Zustand. No Jotai. No Recoil.** The complexity of these libraries is not justified when PowerSync handles all server-state concerns and SQLite handles persistence. If cross-component UI state ever becomes unwieldy (unlikely for this app's scope), Zustand (3.5 KB) is the fallback — but start without it.

## React Router v7 — How It Works with PowerSync

### Framework Mode with clientLoader/clientAction

React Router v7 Framework Mode provides automatic code splitting and structured data loading. Since data comes from local SQLite (not a server), use `clientLoader` and `clientAction` instead of `loader` and `action`:

**`clientLoader`** runs in the browser before the route renders — it queries PowerSync's local SQLite and returns data to the component. This prevents layout shift (the route doesn't render until data is ready).

**`clientAction`** handles form submissions by writing to local SQLite. PowerSync automatically queues the write for sync.

**When both clientLoader and useQuery are needed:** `clientLoader` provides initial data on navigation (one-time load). Within the component, `useQuery` hooks subscribe to the same tables for live updates as other devices sync changes during a meeting. The typical flow:

1. User navigates to `/meeting/123`
2. `clientLoader` queries SQLite → returns meeting + agenda items + motions → route renders
3. Inside the component, `useQuery` watches the same tables for real-time sync changes
4. When another device syncs a new motion, `useQuery` detects the change and rerenders

This dual pattern (clientLoader for initial data, useQuery for reactivity) is the correct architecture for a PowerSync + React Router v7 app.

## Forms — React Hook Form + Zod

**React Hook Form** remains the dominant form library (8.6 KB, zero dependencies, 8.1M weekly npm downloads). It uses uncontrolled inputs with `ref` instead of controlled state, meaning forms don't cause rerenders on every keystroke — critical for an operator typing during a live meeting where input lag would be unacceptable.

**Zod** provides schema validation that can be shared between:
- Client-side form validation (React Hook Form + `@hookform/resolvers/zod`)
- Server-side validation in Supabase Edge Functions
- React Native forms (same Zod schemas, different form rendering via React Hook Form's `Controller`)

**How forms interact with PowerSync:**
1. User fills out form → React Hook Form manages field state
2. On submit → Zod validates → if valid, write to local SQLite via `powersync.execute()`
3. The write is instant (local database) — no loading spinner needed
4. PowerSync queues the write for sync in the background
5. If the server rejects the write (validation failure, constraint violation), the error surfaces through the sync cycle, not through the form submission. PowerSync recommends communicating server-side errors via a synced `write_errors` table.

## Component Library & Styling

### shadcn/ui + Radix UI + Tailwind CSS v4

**shadcn/ui** is the recommended component library:
- Built on **Radix UI** primitives — WAI-ARIA compliant headless components that handle keyboard navigation, focus management, screen reader support, and ARIA attributes
- **WCAG 2.1 AA accessibility** comes from Radix and persists regardless of visual customization
- **50+ components** — Dialog, DataTable, Form (React Hook Form integration), Tabs, Select, Toast, Command (search palette), Accordion, Sheet, Badge, and more
- **Copy-paste model** — components are copied into your `components/ui/` directory, not installed as a black-box dependency. You own the code.
- Components are tree-shaken — unused components add zero to the bundle

**Tailwind CSS v4** (released January 2025):
- CSS-first configuration — no `tailwind.config.js` needed
- 5x faster full builds, 100x faster incremental builds (Rust engine)
- `@import "tailwindcss"` replaces `@tailwind base/components/utilities`
- Automatic template file detection — no content paths to configure

**Radix UI** has consolidated into a single `radix-ui` package (February 2026) — no more individual `@radix-ui/react-*` packages.

## Testing Strategy

### Unit & Integration Testing

| Platform | Runner | Component Testing | Why |
|----------|--------|-------------------|-----|
| Web | **Vitest** | React Testing Library | 10x faster than Jest, native ESM, same API |
| React Native | **Jest** | React Native Testing Library | Vitest doesn't support the RN runtime |

**Testing PowerSync components:** PowerSync does not provide official test utilities for React. Two strategies:

1. **Mock the PowerSync context** — for unit tests verifying component rendering and behavior. Create a mock database object with `vi.fn()` stubs for `execute`, `getAll`, `watch`, etc.

2. **In-memory SQLite** — for integration tests verifying actual SQL queries. Create a PowerSync database with `dbFilename: ':memory:'`, seed test data, and test against real queries.

### E2E Testing

**Playwright** (not Cypress) for E2E:
- Cross-browser: Chrome, Firefox, Safari (WebKit) — Cypress can't test Safari
- Free parallel execution — Cypress requires paid plan
- Offline simulation via `page.route()` network blocking — critical for testing the offline-first experience
- Service Worker interception for PWA testing

**Key offline test scenarios to automate:**
- App loads and displays data after initial sync
- Network disconnects → app continues to function → changes saved locally
- Network reconnects → queued changes sync → UI updates
- Conflict resolution when two devices sync contradictory changes

### Testing budget for the meeting manager

| Test Type | Coverage Target | Tools |
|-----------|----------------|-------|
| Unit tests | Business logic hooks, validation schemas, utility functions | Vitest |
| Component tests | Key UI interactions (motion form, vote panel, agenda navigation) | Vitest + React Testing Library |
| Integration tests | PowerSync queries, data flow through the upload queue | Vitest + in-memory SQLite |
| E2E tests | Critical paths: start meeting → record motion → vote → end meeting | Playwright |
| E2E offline tests | Offline meeting operation, sync recovery | Playwright |

## Error Handling & Resilience

### Error Boundaries — Strategic Placement

Use the `react-error-boundary` library. Place boundaries around feature areas, not the entire app — if the agenda panel crashes, voting and minutes should still work:

- **AgendaPanel** — own error boundary
- **VotingPanel** — own error boundary
- **MinutesPanel** — own error boundary
- **SyncStatusBar** — own error boundary (must never crash and take the meeting view with it)

Error boundaries do NOT catch async errors (PowerSync writes, fetch calls). Use `try/catch` with local error state for those.

### Sync Status UI

PowerSync's `useStatus()` hook provides: `connected` (boolean), `hasSynced` (boolean), and `dataFlowStatus` (uploading/downloading booleans). Combine with a query against PowerSync's internal `ps_crud` table to show pending upload count:

```sql
SELECT count(*) as pending_count FROM ps_crud
```

Display states:
- **Offline — changes saved locally** (amber) — `!connected`
- **Initial sync in progress...** (blue) — `connected && !hasSynced`
- **N changes pending sync** (blue) — `pendingWrites > 0`
- **Synced** (green) — `connected && hasSynced && pendingWrites === 0`

### Error Monitoring — Sentry

Sentry provides an official **offline transport** that queues error events in IndexedDB when the browser is offline and flushes them when connectivity returns. This is essential for an app that operates in low-connectivity environments — errors that occur during an offline meeting are captured and reported when the device comes back online.

## Authentication Flow

The client authenticates with **Supabase Auth directly**, not through PowerSync. PowerSync validates the Supabase-issued JWT to authorize sync operations.

**Flow:**
1. User logs in via Supabase Auth (email/password, magic link, or SSO)
2. Supabase issues a JWT (access token) + refresh token
3. Client initializes PowerSync with a `SupabaseConnector` that implements `fetchCredentials()` → returns the JWT
4. PowerSync validates the JWT against Supabase's JWKS endpoint
5. PowerSync authorizes the sync connection and begins downloading data scoped to the user's permissions via Sync Rules
6. When the JWT expires, PowerSync calls `fetchCredentials()` again → Supabase client auto-refreshes the token

**Key point:** The Supabase client library handles token refresh automatically. No manual refresh logic needed.

**JWT verification:** Use RS256 asymmetric keys (JWKS auto-detect) — PowerSync auto-discovers the JWKS endpoint from the Supabase project. More secure than legacy HS256 shared secrets.

## Performance Decisions

### React Server Components — Not Used

RSC requires a server to render components. For an offline-first app where the local device is the source of truth and must function with zero connectivity, RSC is architecturally incompatible. All components are client-side React.

### Code Splitting

React Router v7 Framework Mode provides automatic route-level code splitting. Each route becomes a separate bundle — visiting `/meeting/123` loads only the meeting route's code. Admin routes, citizen portal routes, and board management routes are loaded on demand.

### Bundle Size Estimate

| Package | Size (min+gzip) |
|---------|----------------|
| React 19 + ReactDOM | ~42 KB |
| PowerSync Web (includes SQLite WASM) | ~100–150 KB |
| PowerSync React hooks | ~5 KB |
| React Hook Form | ~9 KB |
| Zod | ~14 KB |
| react-error-boundary | ~2 KB |
| Sentry React (tree-shaken) | ~25–30 KB |
| shadcn/ui components | 0 KB base (tree-shaken) |
| Tailwind CSS v4 (purged CSS) | ~5–10 KB |
| **Estimated core bundle** | **~250–300 KB gzipped** |

The SQLite WASM binary is the largest single item and is cached by the service worker after first load.

## Type-Safe SQL — Kysely ORM

PowerSync supports **Kysely** via `@powersync/kysely-driver` for type-safe queries against local SQLite. Instead of raw SQL strings:

```typescript
// Raw SQL (works but no type checking)
const items = await powersync.getAll(
  'SELECT * FROM agenda_items WHERE meeting_id = ?', [meetingId]
);

// Kysely (type-safe, autocomplete, compile-time errors)
const items = await db
  .selectFrom('agenda_items')
  .selectAll()
  .where('meeting_id', '=', meetingId)
  .execute();
```

Kysely queries can also be passed to PowerSync's `useQuery` hook. This gives type-safe reactive queries in components.

**Recommendation:** Use Kysely from day one. The type safety catches SQL errors at compile time rather than runtime — critical for a meeting manager where a query error during a live meeting would be catastrophic.
