## The TanStack Query Question

**Short answer: Yes, TanStack Query is used. It is the client data layer.**

TanStack Query (React Query) solves a specific problem: managing server state in a client application — caching API responses, handling loading/error/stale states, deduplicating requests, background refetching, and cache invalidation. Every component that needs data makes a network request, and TanStack Query manages the lifecycle of those requests.

**What TanStack Query provides for this app:**
- **Caching**: query results are cached in memory. Navigate away from a meeting page and back — the data is instant from cache, with a background refetch in progress.
- **Background refetch**: stale data is refreshed automatically when the user refocuses the window or when the network reconnects. No manual polling needed.
- **Optimistic updates**: mutations can immediately update the UI before the server responds, then reconcile with the actual server response. Forms feel instant.
- **Loading and error states**: every `useQuery` call returns `isLoading`, `isError`, and `error` — no `useState` boilerplate for async operations.
- **Cache invalidation**: after a mutation, `queryClient.invalidateQueries()` causes all stale queries to refetch. The UI stays consistent.
- **Parallel and dependent queries**: fetch multiple resources in parallel, or chain queries that depend on each other.

### How TanStack Query + Supabase work together

Components don't call `fetch` directly. They call typed query hooks that wrap Supabase client calls:

```typescript
// A query hook
function useMeeting(meetingId: string) {
  return useQuery({
    queryKey: queryKeys.meeting(meetingId),
    queryFn: () => supabase
      .from('meeting')
      .select('*, board(*), agenda_items(*)')
      .eq('id', meetingId)
      .single()
      .then(({ data, error }) => { if (error) throw error; return data; }),
  });
}

// A mutation hook
function useUpdateAgendaItem() {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: ({ id, ...data }) => supabase
      .from('agenda_item')
      .update(data)
      .eq('id', id),
    onSuccess: (_, { meetingId }) => {
      queryClient.invalidateQueries({ queryKey: queryKeys.meeting(meetingId) });
    },
  });
}
```

JOINs are fully supported in Supabase PostgREST — a single query can fetch a meeting with its agenda items, motions, votes, and attendance records. No more "separate queries + merge in JS."

### When to use Supabase Realtime alongside TanStack Query

For most pages, React Query's background refetch is sufficient. For the live meeting manager — where multiple devices must see changes within ~100ms of each other — use Supabase Realtime `postgres_changes` subscriptions alongside React Query:

1. `useQuery` fetches the initial meeting state on page load
2. A `useRealtimeSubscription` hook subscribes to `postgres_changes` on the relevant tables
3. When another device makes a change, the Realtime event fires, and `queryClient.invalidateQueries()` triggers a refetch of the affected data
4. All connected devices update within ~100ms under normal connectivity

## State Management — What Goes Where

TanStack Query handles all server state. Here is the complete state taxonomy:

| State Type | Examples in the Meeting Manager | Where It Lives |
|-----------|-------------------------------|---------------|
| **Server state (persisted data)** | Meetings, agenda items, motions, votes, persons, board memberships, minutes | TanStack Query (cache backed by Supabase) |
| **Component-local UI state** | Is the motion dialog open? Which tab is active? Sidebar collapsed? | `useState` / `useReducer` |
| **Form state** | Current field values, validation errors, dirty tracking | React Hook Form |
| **Navigation state** | Current route, URL parameters, breadcrumbs | React Router v7 |
| **Cross-component session state** | Active meeting context, toast notification queue, theme preference | React Context (2–3 contexts max) |
| **Derived/computed state** | Filtered agenda items, vote tallies, quorum calculations | `useMemo` over React Query results |
| **Realtime connection status** | Connected / Connecting / Disconnected | Supabase Realtime client state + `ConnectionStatusBar` |

**No Redux. No Zustand. No Jotai. No Recoil.** TanStack Query handles server state concerns entirely. If cross-component UI state ever becomes unwieldy (unlikely for this app's scope), Zustand (3.5 KB) is the fallback — but start without it.

## React Router v7 — How It Works with TanStack Query

### Framework Mode with clientLoader

React Router v7 Framework Mode provides automatic code splitting and structured data loading. The standard pattern for TanStack Query integration uses `clientLoader` with `queryClient.ensureQueryData()`:

**`clientLoader`** runs in the browser before the route renders — it ensures data is in the React Query cache before the component renders. This prevents layout shift.

```typescript
// route: meetings.$meetingId.tsx
export async function clientLoader({ params }: ClientLoaderFunctionArgs) {
  const meetingId = params.meetingId!;
  // Ensure the data is in cache before rendering
  await queryClient.ensureQueryData({
    queryKey: queryKeys.meeting(meetingId),
    queryFn: () => fetchMeeting(meetingId),
  });
  return { meetingId };
}

export default function MeetingPage() {
  const { meetingId } = useLoaderData<typeof clientLoader>();
  // useQuery reads from cache — no loading state on initial render
  const { data: meeting } = useQuery({
    queryKey: queryKeys.meeting(meetingId),
    queryFn: () => fetchMeeting(meetingId),
  });
  // Also subscribe to Realtime for live updates
  useRealtimeSubscription('agenda_item', `meeting_id=eq.${meetingId}`, () => {
    queryClient.invalidateQueries({ queryKey: queryKeys.meeting(meetingId) });
  });
  return <MeetingView meeting={meeting} />;
}
```

The dual pattern (clientLoader for initial load, useQuery for reactivity, Realtime for live updates) is the correct architecture for this stack.

## Forms — React Hook Form + Zod

**React Hook Form** is the form library of choice (8.6 KB, zero dependencies, 8.1M weekly npm downloads). It uses uncontrolled inputs with `ref` instead of controlled state, meaning forms don't cause rerenders on every keystroke — critical for an operator typing during a live meeting where input lag would be unacceptable.

**Zod** provides schema validation that can be shared between:
- Client-side form validation (React Hook Form + `@hookform/resolvers/zod`)
- Server-side validation in Supabase Edge Functions
- React Native forms (same Zod schemas, different form rendering via React Hook Form's `Controller`)

**How forms interact with TanStack Query:**
1. User fills out form → React Hook Form manages field state
2. On submit → Zod validates → if valid, `useMutation` sends to Supabase
3. On success → `queryClient.invalidateQueries()` refreshes affected queries
4. Optimistic updates: `useMutation`'s `onMutate` can update the cache immediately for instant UI feedback
5. If the server rejects the write, the optimistic update is rolled back and the error surfaces via the mutation's `onError`

No workarounds needed — react-hook-form works directly with this stack without any dual React instance issues.

## Supabase Realtime in Components

Supabase Realtime subscriptions are set up with `useEffect` and cleaned up on unmount. The `useRealtimeSubscription` hook in `packages/web/src/hooks/useRealtimeSubscription.ts` provides a clean abstraction:

```typescript
// Generic hook — sets up postgres_changes subscription
function useRealtimeSubscription(
  table: string,
  filter: string,
  onEvent: (payload: RealtimePostgresChangesPayload) => void
) {
  const supabase = useSupabase();
  useEffect(() => {
    const channel = supabase
      .channel(`${table}-${filter}`)
      .on('postgres_changes', { event: '*', schema: 'public', table, filter }, onEvent)
      .subscribe();
    return () => { supabase.removeChannel(channel); };
  }, [table, filter, onEvent]);
}

// Usage in live meeting component
function LiveMeetingPage({ meetingId }: { meetingId: string }) {
  const queryClient = useQueryClient();

  // Invalidate meeting data when any agenda item changes
  useRealtimeSubscription('agenda_item', `meeting_id=eq.${meetingId}`, () => {
    queryClient.invalidateQueries({ queryKey: queryKeys.meeting(meetingId) });
  });

  // Invalidate votes when any vote is recorded
  useRealtimeSubscription('vote_record', `meeting_id=eq.${meetingId}`, () => {
    queryClient.invalidateQueries({ queryKey: queryKeys.meetingVotes(meetingId) });
  });
}
```

## Testing Strategy

### Unit & Integration Testing

| Platform | Runner | Component Testing | Why |
|----------|--------|-------------------|-----|
| Web | **Vitest** | React Testing Library | 10x faster than Jest, native ESM, same API |
| React Native | **Jest** | React Native Testing Library | Vitest doesn't support the RN runtime |

**Testing TanStack Query components:** Wrap components in a `QueryClientProvider` with a fresh `QueryClient` for each test. Use `msw` (Mock Service Worker) to intercept Supabase API calls and return test data.

```typescript
// Test utility
function createWrapper() {
  const queryClient = new QueryClient({
    defaultOptions: { queries: { retry: false } },
  });
  return ({ children }: { children: React.ReactNode }) => (
    <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>
  );
}
```

### E2E Testing

**Playwright** (not Cypress) for E2E:
- Cross-browser: Chrome, Firefox, Safari (WebKit) — Cypress can't test Safari
- Free parallel execution — Cypress requires paid plan
- Network simulation via `page.route()` — test connection loss and recovery scenarios
- Service Worker interception for PWA testing

**Key test scenarios to automate:**
- App loads and displays data correctly after login
- Network disconnects → ConnectionStatusBar shows warning → data remains readable from cache
- Network reconnects → ConnectionStatusBar updates → React Query refetches
- Realtime: one device updates agenda item → second device receives update within 500ms

### Testing budget for the meeting manager

| Test Type | Coverage Target | Tools |
|-----------|----------------|-------|
| Unit tests | Business logic hooks, validation schemas, utility functions | Vitest |
| Component tests | Key UI interactions (motion form, vote panel, agenda navigation) | Vitest + React Testing Library |
| Integration tests | React Query + Supabase fetch functions, mutation + invalidation flow | Vitest + msw |
| E2E tests | Critical paths: start meeting → record motion → vote → end meeting | Playwright |
| Realtime tests | Multi-device sync via Supabase Realtime | Playwright (two browser contexts) |

## Error Handling & Resilience

### Error Boundaries — Strategic Placement

Use the `react-error-boundary` library. Place boundaries around feature areas, not the entire app — if the agenda panel crashes, voting and minutes should still work:

- **AgendaPanel** — own error boundary
- **VotingPanel** — own error boundary
- **MinutesPanel** — own error boundary
- **ConnectionStatusBar** — own error boundary (must never crash and take the meeting view with it)

Error boundaries do NOT catch async errors (React Query mutations, fetch calls). Use `try/catch` with local error state, or React Query's `onError` callbacks, for those.

### Connection Status UI

The `ConnectionStatusBar` component shows Supabase Realtime connection state:

- **Connected** (green dot) — Realtime subscription is active
- **Connecting...** (amber dot, pulsing) — reconnecting after a disconnect
- **Disconnected** (red dot, with warning text) — "Meeting data may be outdated. Reconnecting..."

The component subscribes to the Supabase Realtime client's `onClose` and `onOpen` events to track connection state. During a disconnected state, writes show an error toast: "Unable to save — no connection. Please wait for reconnection or check your network."

React Query's cache keeps data readable during a disconnect — users can still view the meeting, agenda, and prior votes even when the Realtime connection is down.

### Error Monitoring — Sentry

Sentry provides error capture that works during brief connectivity interruptions. Errors are queued in memory and flushed when connectivity returns. Essential for capturing errors that occur during live meetings.

## Authentication Flow

The client authenticates with **Supabase Auth directly**.

**Flow:**
1. User logs in via Supabase Auth (email/password, magic link, or SSO)
2. Supabase issues a JWT (access token) + refresh token
3. Client stores the session; Supabase client handles auto-refresh via `onAuthStateChange`
4. After login, React Query prefetches the user's town and board data: `queryClient.prefetchQuery()`
5. Supabase Realtime uses the same JWT for subscription authorization (RLS policies apply to subscriptions)
6. When the JWT expires, Supabase client auto-refreshes the token; Realtime subscriptions reconnect with the new token

**Key point:** The Supabase client library handles token refresh automatically. No manual refresh logic needed.

**JWT verification:** Use RS256 asymmetric keys (JWKS auto-detect) — Supabase auto-discovers the JWKS endpoint. More secure than legacy HS256 shared secrets.

## Performance Decisions

### React Server Components — Not Used

RSC requires a server to render components. The app is a client-side SPA where all data flows through TanStack Query and Supabase Realtime. All components are client-side React.

### Code Splitting

React Router v7 Framework Mode provides automatic route-level code splitting. Each route becomes a separate bundle — visiting `/meeting/123` loads only the meeting route's code. Admin routes, citizen portal routes, and board management routes are loaded on demand.
