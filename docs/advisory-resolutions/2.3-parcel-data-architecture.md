# Advisory 2.3 — Parcel & Address Data Architecture

**Advisory Item:** The System Needs a Parcel Data Integration Strategy Using Maine GeoLibrary
**Date:** March 2026
**Status:** Resolved — all data sources identified, architecture decisions made

---

## Context

Land-use boards (Planning Board, Zoning Board of Appeals, etc.) reference parcels by Map/Lot in their agenda items, applications, and minutes. The system needs parcel boundary and address data to:

1. Display parcel locations on maps when building agendas or reviewing applications
2. Auto-populate parcel details (address, acreage, owner) when an operator enters a Map/Lot reference
3. Provide geographic context in the public portal (e.g., "this agenda item concerns the parcel at 123 Main St")

Per the user's direction:
- **Parcel data is refreshed once per year** — no need to handle mid-year updates before new data is available
- **Data should be loaded per town**, not statewide
- **Maine GeoLibrary is the sole data source** — no third-party APIs
- **Parcel data and E911 data should be combined** to get complete property information
- **The coordinate system needs to be tested** at integration time to confirm behavior

## Coordinate Systems

| Service | WKID | Projection | Units |
|---------|------|-----------|-------|
| Parcels (Organized Towns) | 26919 | NAD 1983 UTM Zone 19N | Meters |
| Parcels (Merged) | 102388 / 6348 | NAD 1983 State Plane Maine | Meters |
| E911 Addresses/Roads | 26919 | NAD 1983 UTM Zone 19N | Meters |
| Town Boundaries | 26919 | NAD 1983 UTM Zone 19N | Meters |
| **GeoJSON output** | **4326** | **WGS 84** | **Decimal degrees** |

**Key insight:** When requesting data with `f=geojson`, coordinates are automatically reprojected to WGS 84 (EPSG:4326) — standard lat/long used by web mapping libraries (Leaflet, Mapbox GL, etc.). This eliminates the need to handle projection transformations in application code.

The coordinate system should still be tested at integration time to confirm this behavior, per the user's note.

## Data Linking Strategy

The parcel and E911 datasets share two linking fields:

- **`STATE_ID`** — State-level parcel identifier (primary join key)
- **`MAP_BK_LOT`** — Map/Book/Lot reference (secondary join key, what boards use verbally)

### Combined data model per parcel:

```
Parcel (from Parcels layer)
├── Geometry (polygon boundary)
├── TOWN, COUNTY, GEOCODE
├── STATE_ID, MAP_BK_LOT
├── PROP_LOC (street address from parcel data)
├── Shape__Area (acreage calculation)
│
├── Ownership (from ADB table, joined on STATE_ID)
│   ├── OWNER1, OWNER2
│   ├── Owner mailing address
│   ├── Last sale date/price/deed reference
│   ├── LAND_VAL, BLDG_VAL
│   └── LAND_USE, LAND_USE_D
│
└── E911 Addresses (from Addresses layer, joined on STATE_ID or MAP_BK_LOT)
    ├── Full structured address (number, street, suffix, unit)
    ├── LATITUDE, LONGITUDE (point location)
    ├── ZIPCODE
    └── BUILDING, LANDMARK, PLACE_TYPE
```

This combination gives the system:
- **Parcel boundaries** (polygon geometry for map display)
- **Property address** (from E911, more reliable than parcel PROP_LOC)
- **Owner information** (from ADB table)
- **Assessed values** (from ADB table)
- **Land use classification** (from ADB table)
- **Precise point location** (from E911 lat/long)

## Update Frequency & Data Refresh Strategy

### Current Update Cadence

| Dataset | Frequency | Notes |
|---------|-----------|-------|
| Parcels | Irregular / voluntary | Towns submit updates on a non-regular basis. Currency varies widely (2001–2023+). ~1/3 of records have unknown update dates. Each record has an `FMUPDAT` field. |
| ADB Ownership | Irregular | Submitted alongside parcel geometry. Ownership data may be up to 5 years old. |
| E911 Addresses | Ongoing | Last edited January 2025. Maintained by Maine Addressing Officers. More frequently updated than parcels. |

### Proposed Refresh Architecture

1. **Monthly automated check:** A scheduled job runs monthly, queries the GeoLibrary API for each configured town, and compares record counts and update timestamps against the last import. If changes are detected, the system imports the updated data and notifies the admin with a summary.

2. **Per-town loading:** Data is fetched per town using `where=TOWN='TOWNNAME'` queries against the Merged parcels service. Only towns configured in the system are loaded.

3. **Pagination:** The API returns max 2,000 records per request. The import job must paginate using `resultOffset` and `resultRecordCount` parameters.

4. **Storage:** Imported parcel and E911 data is stored in PostGIS-enabled PostgreSQL, not re-queried from the API at runtime. This ensures fast lookups and offline resilience.

5. **Admin dashboard:** Shows last refresh date per town, import summary (record counts, changes detected), and a "Refresh Now" button for on-demand imports.

### Example API Queries for a Town Import

```
# Parcel geometries + ownership for Newcastle (Merged service)
GET .../Maine_Parcels_Merged/FeatureServer/1/query
  ?where=TOWN='NEWCASTLE'
  &outFields=*
  &f=geojson
  &resultOffset=0
  &resultRecordCount=2000

# E911 addresses for Newcastle
GET .../Addresses/FeatureServer/0/query
  ?where=TOWN='NEWCASTLE'
  &outFields=*
  &f=geojson
  &resultOffset=0
  &resultRecordCount=2000
```

## Town Identification

The `GEOCODE` field (5-character string) is the authoritative municipal identifier across all GeoLibrary datasets. The town boundaries service provides the master list:

```
GET .../Maine_Town_and_Townships_Boundary_Polygons/FeatureServer/0/query
  ?where=1=1
  &outFields=TOWN,COUNTY,GEOCODE
  &returnDistinctValues=true
  &f=json
```

The system should store each town's `GEOCODE` during onboarding and use it for all GeoLibrary queries.

## Important Caveats

1. **Voluntary submission:** Parcel data is submitted voluntarily by towns. Coverage and currency vary significantly. Some towns have data from 2001; others from 2023+. Approximately one-third of records have unknown update dates.

2. **Not survey-grade:** Digital parcel data does not represent or replace survey-grade legal descriptions. Boundaries are approximate and should only be used for visual reference.

3. **ADB ownership lag:** Ownership data in the ADB table may be up to 5 years outdated depending on when the town last submitted.

4. **Town name matching:** Town names in the GeoLibrary data use uppercase (e.g., `NEWCASTLE`, `PORTLAND`). The system must normalize casing when matching against the application's town records. The `GEOCODE` field is more reliable for matching.

## Open Questions

### Q1: Should the parcel data refresh be automated or admin-triggered?

**Decision: Automated monthly check.** A scheduled job runs monthly, queries the GeoLibrary API, and compares against the last import to detect new or updated records. If changes are found, the system automatically imports the updated data and notifies the admin with a summary. The admin dashboard also shows the last refresh date and a "Refresh Now" button for on-demand imports.

### Q2: Which parcel service should be the primary source?

**Decision: Merged (FeatureServer/1).** Use the `Maine_Parcels_Merged` service as the primary source. Single-layer query with 51 fields including ownership data inline. Simpler integration — one query per town instead of separate parcel + ADB table joins. Filter by town name to exclude unorganized territory data the app doesn't need.

### Q3: How should the system handle towns with no parcel data in the GeoLibrary?

**Decision: Option B (E911 fallback + manual parcel entry).** If no parcel data exists in the GeoLibrary, the system still imports E911 address points for address lookups and point-based map display. The admin sees a persistent notice: "No parcel boundary data available for [town]. Address lookups use E911 data only." Additionally, the admin can manually create parcel records (MAP_BK_LOT, address, owner) without geometry — covering cases where the operator knows parcel details from town records but GeoLibrary hasn't received the digital data yet. Manual records are flagged and automatically replaced when GeoLibrary data becomes available. The monthly check continues to look for newly submitted parcel data.

### Q4: Should the system store parcel geometry or just metadata?

**Decision: Full PostGIS geometry.** Store parcel polygons in PostGIS. This enables map display of parcel boundaries, spatial queries (e.g., abutters within a radius), and click-to-identify on maps. No runtime dependency on the GeoLibrary API for map features.

### Q5: Should E911 address data be refreshed more frequently than parcels?

**Decision: Same monthly check as parcels.** E911 data follows the same automated monthly check-and-update schedule as parcel data (per Q1). All GeoLibrary data refreshes on the same cycle. Since E911 is updated more frequently upstream, the monthly check will naturally pick up new addresses sooner than parcel changes. The admin can also trigger a manual refresh at any time via the dashboard.
