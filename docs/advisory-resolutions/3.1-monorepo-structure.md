# Advisory 3.1 — Monorepo vs. Separate Repositories

**Advisory Item:** Monorepo vs. Separate Repositories
**Date:** March 2026
**Status:** Resolved — monorepo with pnpm workspaces + Turborepo

---

## Decision

**Option A — Monorepo.** Single GitHub repository with packages for `web`, `mobile`, `api`, and `shared`. Tooling: pnpm workspaces + Turborepo (already confirmed in advisory 1.3).

### Rationale

- **Shared TypeScript types** across all packages — critical for type-safe API contracts between frontend, mobile, and backend
- **Single CI/CD pipeline** — coordinated builds, tests, and deployments
- **Atomic changes** — a data model change that affects the API response and the frontend component can be a single commit with a single PR
- **Dependency consistency** — shared dependencies stay in sync across packages

### Package Structure

```
town-meeting-manager/
├── packages/
│   ├── web/          # React + Tailwind CSS (web application)
│   ├── mobile/       # React Native (mobile app)
│   ├── api/          # Node.js / Fastify (backend API)
│   └── shared/       # Shared TypeScript types, validators, constants
├── pnpm-workspace.yaml
├── turbo.json
├── package.json
└── tsconfig.base.json
```

### What Goes in `shared/`

- TypeScript type definitions for API request/response shapes
- Zod schemas for runtime validation (shared between API and frontend)
- Enum/constant definitions (board types, role names, meeting statuses, motion types, etc.)
- Utility functions with no platform-specific dependencies

### What Stays Package-Specific

- UI components (web vs. mobile have different component libraries)
- Platform-specific APIs (Supabase client config, React Native modules)
- Route definitions and middleware (API-only)
- Styling (Tailwind for web, StyleSheet for mobile)
