## Context

The notification system serves two distinct communication channels:

1. **Email** (Phase 1) — Meeting notices, agenda publication alerts, minutes approval notifications, admin alerts (e.g., parcel data refresh summaries), password resets, account invitations
2. **SMS** (Phase 2) — Straw poll notifications, public hearing reminders, meeting schedule changes, opt-in civic engagement alerts

The provider choice affects the notification event system architecture, opt-in/opt-out flows, and compliance implementation. Choosing providers before building the notification infrastructure avoids costly migrations later.

---

## Provider: Postmark (Email)

### Why Postmark

- **Deliverability focus.** Postmark is purpose-built for transactional email with industry-leading delivery rates. They prohibit bulk marketing email, which keeps their IP reputation high — critical for government notifications that must actually reach recipients.
- **Simple API.** Clean REST API and official Node.js SDK. Template system supports server-side templates with variable substitution.
- **Message Streams.** Separate transactional and broadcast streams with independent tracking — useful for separating system emails (password resets, invitations) from notification emails (meeting alerts, agenda publication).
- **Inbound processing.** Webhook-based inbound email processing if needed later (e.g., reply-to-approve minutes).
- **Compliance features.** Built-in suppression list management, bounce handling, and unsubscribe processing that help with CAN-SPAM compliance.

### Postmark Configuration

- **Sender domain:** Configured per town (e.g., `notifications@newcastle.townmeetingmanager.com` or custom domain)
- **DKIM/SPF/DMARC:** Required for deliverability — Postmark provides DNS records to configure
- **Webhooks:** Bounce, delivery, open, and click tracking events posted back to the application for delivery status tracking
- **Suppression management:** Postmark automatically suppresses hard bounces and spam complaints

## Provider: Twilio (SMS)

### Why Twilio

- **TCPA compliance tooling.** Twilio provides built-in opt-in/opt-out management, consent tracking, and required STOP/HELP keyword handling — critical for government SMS notifications.
- **10DLC registration.** Twilio handles A2P 10DLC campaign registration (required for application-to-person SMS in the US). This is a regulatory requirement, not optional.
- **Messaging Services.** Logical grouping of phone numbers with built-in opt-out management, sticky sender for consistent caller ID, and rate limiting.
- **Mature API.** Most comprehensive SMS API available with extensive Node.js SDK, detailed delivery status callbacks, and robust error handling.
- **Regulatory compliance.** Twilio's Compliance Center guides through TCPA requirements, opt-in record keeping, and campaign registration.

### TCPA Compliance Architecture

The Telephone Consumer Protection Act requires explicit prior consent before sending SMS to consumers. The system must implement:

1. **Express written consent.** Residents must explicitly opt in to receive SMS notifications. A checkbox during straw poll enrollment is not sufficient — the consent must include:
   - Clear disclosure of what messages will be sent
   - Expected message frequency
   - "Message and data rates may apply" notice
   - Instructions to text STOP to opt out
   - Link to privacy policy

2. **Opt-in record keeping.** The system must store:
   - Timestamp of consent
   - Method of consent (web form, in-person, etc.)
   - IP address (for web consent)
   - Exact language shown at time of consent
   - Phone number consented

3. **Opt-out processing.** When a recipient texts STOP:
   - Twilio's Messaging Service handles this automatically
   - The application receives a webhook and updates the subscriber record
   - A confirmation message is sent: "You have been unsubscribed. Reply START to re-subscribe."
   - No further messages are sent to that number

4. **Quiet hours.** No SMS before 8:00 AM or after 9:00 PM in the recipient's local time zone (Eastern for Maine).

### Twilio Integration Points

| Use Case | Phase | Message Type |
|----------|-------|-------------|
| Straw poll notification | Phase 2 | One-way (notification with link) |
| Public hearing reminder | Phase 2 | One-way |
| Meeting cancellation alert | Phase 2 | One-way |
| Meeting schedule change | Phase 2 | One-way |

### Twilio Configuration

- **Messaging Service:** One per town, with a dedicated phone number
- **10DLC Campaign:** Registered as "Government" use case (higher throughput, better deliverability)
- **Webhooks:** Delivery status callbacks (delivered, failed, undelivered) posted to the application
- **Phone number type:** Local US number (not short code — too expensive for small-town volumes)

### 10DLC Registration Requirements

A2P 10DLC is mandatory for application-to-person SMS. The registration process:

1. **Brand registration** — Register the business/organization with The Campaign Registry (TCR) via Twilio
2. **Campaign registration** — Register the specific use case (government notifications) with message samples
3. **Number assignment** — Assign the town's phone number to the registered campaign

Government use cases typically receive favorable trust scores and higher message throughput. Twilio handles the TCR submission through their API/console.

## Notification Event System Architecture

Both providers integrate through a unified notification event system:

```
Event Trigger (e.g., agenda published)
  │
  ├── Notification Event Created
  │   ├── event_type: "agenda_published"
  │   ├── town_id
  │   ├── payload: { meeting_id, agenda_url, ... }
  │   └── created_at
  │
  ├── Subscriber Query
  │   ├── Find all subscribers for this event type + town
  │   ├── Filter by channel preference (email, sms, both)
  │   └── Filter by opt-in status (TCPA for SMS)
  │
  ├── Channel Dispatch
  │   ├── Email → Postmark API (template + variables)
  │   └── SMS → Twilio Messaging Service
  │
  └── Delivery Tracking
      ├── Webhook callbacks update delivery status
      └── Admin dashboard shows delivery summary
```

### Subscriber Preferences

Each subscriber (board member, staff, or public resident) has per-channel, per-event-type preferences:

```
subscriber_notification_preferences
├── subscriber_id
├── channel: "email" | "sms"
├── event_type: "agenda_published" | "meeting_cancelled" | "straw_poll" | ...
├── enabled: boolean
├── consent_timestamp (SMS only)
├── consent_method (SMS only)
└── consent_record (SMS only — exact text shown)
```

### Rate Limiting & Batching

- **Email:** Postmark allows 500 emails/batch via the batch API. For towns with large subscriber lists, emails are batched in groups of 500.
- **SMS:** Twilio Messaging Service handles rate limiting automatically based on 10DLC throughput. Typical government campaigns allow 75 messages/second.
- **Retry logic:** Failed deliveries are retried with exponential backoff (3 attempts for email, 2 for SMS). Permanent failures (hard bounce, invalid number) update the subscriber record.

## Cost Estimates

### Postmark Pricing

| Tier | Monthly Emails | Price |
|------|---------------|-------|
| Free | 100 | $0 |
| Starter | 10,000 | $15/mo |
| Standard | 50,000 | $50/mo |

For a small town (500–2,000 residents), the Starter tier is likely sufficient. Most towns will send fewer than 1,000 emails/month (meeting notices, agenda alerts, minutes notifications).

### Twilio Pricing

| Component | Cost |
|-----------|------|
| Local phone number | $1.15/mo |
| Outbound SMS | $0.0079/segment |
| 10DLC campaign registration | $15 one-time |
| 10DLC brand registration | $4 one-time |

For a small town sending 200 SMS/month (straw polls + meeting alerts), the monthly cost is approximately $2.73 ($1.15 number + $1.58 messages).

## Phase 1 vs. Phase 2 Implementation

### Phase 1 (MVP) — Email Only

- Integrate Postmark for transactional and broadcast email
- Build the notification event system with email as the only channel
- Implement email preference management for board members and staff
- No public subscriber management yet (email goes to authenticated users only)

### Phase 2 — Add SMS + Public Subscribers

- Integrate Twilio for SMS notifications
- Add SMS as a second channel to the existing notification event system
- Build TCPA-compliant opt-in flows for public residents
- Add straw poll creation and notification dispatch
- Build public subscriber self-service (manage preferences, opt out)
- Register 10DLC campaigns for each town
