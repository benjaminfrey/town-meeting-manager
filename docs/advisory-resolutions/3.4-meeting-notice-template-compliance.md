# Advisory 3.4 — Meeting Notice Template System & Notice Compliance Engine

**Status:** Approved
**Applies to:** Sessions Pre-13.01-B, 13.x
**Supersedes:** Session 06.03 Task 8 (simple pdf-lib notice — upgraded by this advisory)

---

## 1. Overview

Meeting notices in Maine and other New England states are governed by specific statutory requirements that vary by meeting type, action type, and municipal charter. This advisory defines:

1. A **per-board block-based notice template system** for generating formatted PDF notices
2. A **notice compliance engine** — a shared library with state-specific rules, deadline calculators, and earliest-date forecasting
3. The **notice generation workflow** and how it integrates with the existing meeting lifecycle

The compliance engine is **advisory only** — it surfaces warnings and guidance but never hard-blocks actions.

---

## 2. Notice Template System

### 2.1 Template Scope

Each board has its own notice template. Templates are not shared at the town level, though a **copy-from-board** action is provided for rapid setup.

Templates live at two access points:
- `/settings/meeting-notices` — overview page showing all boards with template status (configured / not configured) and copy-from-board action; entry point from the ProgressChecklist
- `/boards/:boardId` Settings tab — per-board template editor (primary editing surface)

### 2.2 Template Structure (Block-Based)

A notice template is composed of ordered blocks. Available block types:

| Block Type | Description | Configurable Fields |
|------------|-------------|---------------------|
| `letterhead` | Town name, board name, town seal/logo | Logo position (left/center/right), show seal (yes/no), font size |
| `meeting_details` | Date, time, location, virtual link | Label format (e.g., "DATE:" vs "Meeting Date:"), show virtual link |
| `agenda_summary` | Pulled from the meeting's agenda items | Include sub-items (yes/no), max items to show (number or "all") |
| `rich_text` | Free-form body text with merge fields | Full rich-text editing (bold, italic, lists, headings) |
| `statutory_footer` | Compliance citation and posting reference | Statute citation (auto-populated from compliance engine or manual override) |
| `signature_block` | Prepared-by line | Name field (manual or auto from `presiding_officer_default`) |
| `spacer` | Visual spacing | Height (sm/md/lg) |

Blocks are **ordered and reorderable**. Admins can add multiple `rich_text` blocks. Only one `letterhead`, `meeting_details`, `agenda_summary`, `statutory_footer`, and `signature_block` are permitted per template.

### 2.3 Merge Fields

Rich text and other blocks support merge fields using `{{field_name}}` syntax:

| Field | Value |
|-------|-------|
| `{{town_name}}` | Town name |
| `{{board_name}}` | Board name |
| `{{meeting_date}}` | Formatted scheduled date |
| `{{meeting_time}}` | Formatted scheduled time |
| `{{meeting_location}}` | Location |
| `{{meeting_type_label}}` | Human-readable meeting type |
| `{{notice_deadline}}` | Deadline date for posting (from compliance engine) |
| `{{statute_citation}}` | Applicable statute citation (from compliance engine) |
| `{{presiding_officer}}` | Board's default presiding officer |

### 2.4 Template Storage

Templates are stored as JSONB on the `board` table:

```sql
ALTER TABLE board ADD COLUMN notice_template_blocks JSONB;
-- null = not configured; array of block objects = configured
```

Each block object: `{ type, order, config: { ...block-specific fields } }`

### 2.5 ProgressChecklist Completion Signal

The "Set meeting notice template" checklist item is complete when **every board in the town** has `notice_template_blocks IS NOT NULL`. The checklist shows a sub-count: "X of Y boards configured."

### 2.6 Template Preview

The Settings tab and `/settings/meeting-notices` editor show a live HTML preview of the rendered notice using sample/placeholder data. A "Generate Sample PDF" button triggers the API to render a real PDF for download.

## 3. Notice Generation Workflow

### 3.1 Flow

The "Generate Meeting Notice" action on the AgendaBuilder page:

1. Renders the board's template with the meeting's real data
2. Generates a PDF via the Fastify API (`POST /api/meetings/:meetingId/meeting-notice`)
3. Stores the PDF in Supabase Storage (`notices/{meetingId}/notice-{timestamp}.pdf`)
4. Records `notice_generated_at` and `notice_pdf_storage_path` on the `meeting` table
5. Presents two actions: **Download PDF** and **Publish to Portal**

### 3.2 Portal Publishing

Publishing is a separate explicit action (requires user confirmation). When confirmed:
- The notice is made publicly accessible on the meeting's portal page
- `notice_published_at` is recorded on the `meeting` table
- The meeting status transitions to `noticed`

**The `noticed` status can only be reached through the notice generation + publish flow.** There is no manual "mark as noticed" shortcut.

### 3.3 Notice Tracking Columns (new on `meeting` table)

```sql
ALTER TABLE meeting
  ADD COLUMN notice_generated_at TIMESTAMPTZ,
  ADD COLUMN notice_pdf_storage_path TEXT,
  ADD COLUMN notice_published_at TIMESTAMPTZ;
```

### 3.4 Cancellation Notices

When a meeting is cancelled, the system generates a cancellation notice using the same template with a `rich_text` block populated with "NOTICE OF CANCELLATION" language. The admin can edit the cancellation body before generating. The same publish flow applies.

## 4. Notice Compliance Engine

### 4.1 Architecture

The compliance engine lives in `packages/shared/src/notice-rules/` as a pure TypeScript library with no external dependencies. It is importable by both frontend components and the Fastify API.

```
packages/shared/src/notice-rules/
├── index.ts                  # Public API
├── types.ts                  # NoticeRule, ComplianceResult, ForecastResult
├── rules/
│   ├── maine.ts              # Maine-specific rules (primary)
│   ├── new-hampshire.ts      # NH stub (extensible)
│   ├── vermont.ts            # VT stub (extensible)
│   └── index.ts              # State registry
├── calculator.ts             # Deadline + earliest-date logic
└── formatter.ts              # Human-readable output strings
```

### 4.2 Rule Data Model

```typescript
interface NoticeRule {
  id: string;                    // e.g., "ME_OPEN_MEETINGS_BOARD"
  state: string;                 // "ME", "NH", "VT", etc.
  meetingTypes: MeetingType[];   // which meeting types this applies to
  actionTypes?: string[];        // optional: specific action types (e.g., "ordinance_change")
  minimumNoticeHours: number;    // minimum hours before meeting
  minimumNoticeDays?: number;    // for calendar-day rules (e.g., "14 days")
  calendarDays: boolean;         // true = calendar days, false = hours
  excludeWeekends?: boolean;     // some rules exclude weekends from counting
  statutes: string[];            // e.g., ["1 M.R.S.A. §403"]
  description: string;           // human-readable rule description
  notes?: string;                // additional legal context
  requiresNewspaper?: boolean;   // additional publication requirement
  requiresPosting?: boolean;     // physical posting required
}
```

### 4.3 Maine Rules (initial dataset — requires legal review before shipping)

> ⚠️ **Legal Review Required:** The statute citations and notice periods below are based on current understanding of Maine law and must be verified by legal counsel before shipping to production. The rule data is structured for easy correction.

| Rule ID | Meeting/Action Type | Notice Period | Statute | Notes |
|---------|--------------------|--------------|---------|----|
| `ME_OPEN_MEETINGS_BOARD` | All board meetings (regular, special) | 24 hours | 1 M.R.S.A. §403 | Minimum; charters may require more |
| `ME_TOWN_MEETING_WARRANT` | Annual town meeting | 14 calendar days | 30-A M.R.S.A. §2521 | Posted in at least 2 public places |
| `ME_SPECIAL_TOWN_MEETING` | Special town meeting | 14 calendar days | 30-A M.R.S.A. §2521 | Warrant must specify the article |
| `ME_ZONING_ORDINANCE_HEARING` | Public hearing — zoning/land use ordinance | 14 calendar days | 30-A M.R.S.A. §4352 | Notice in newspaper of general circulation required |
| `ME_SUBDIVISION_HEARING` | Planning Board subdivision hearing | 14 calendar days | 30-A M.R.S.A. §4403 | Abutters must be notified by mail |
| `ME_BUDGET_COMMITTEE` | Budget committee public hearing | 10 calendar days | 30-A M.R.S.A. §2902 | Verify against local charter |
| `ME_REFERENDUM` | Referendum/Special election | 42 days | 21-A M.R.S.A. §§621–627 | Secretary of State notification required |

### 4.4 Compliance Engine Public API

```typescript
// Get applicable rules for a meeting
function getRulesForMeeting(params: {
  state: string;
  meetingType: MeetingType;
  actionTypes?: string[];
}): NoticeRule[]

// Calculate the posting deadline for a meeting
function getNoticeDeadline(params: {
  meetingDate: Date;
  meetingTime?: string;   // "HH:MM"
  state: string;
  meetingType: MeetingType;
  actionTypes?: string[];
}): ComplianceResult

interface ComplianceResult {
  deadline: Date;              // Latest date/time notice must be posted
  daysUntilDeadline: number;  // Negative if overdue
  isOverdue: boolean;
  rules: NoticeRule[];         // Rules that apply
  citations: string[];         // Formatted statute citations
  warningLevel: 'ok' | 'warning' | 'danger' | 'overdue';
  // 'ok' = > 3 days remaining
  // 'warning' = 1-3 days remaining
  // 'danger' = < 24 hours remaining
  // 'overdue' = past deadline
  advisoryMessage: string;     // Human-readable guidance
}

// Forecast earliest possible meeting date for a given action type
function forecastEarliestMeetingDate(params: {
  fromDate: Date;              // Today or desired notice posting date
  state: string;
  meetingType: MeetingType;
  actionTypes?: string[];
}): ForecastResult

interface ForecastResult {
  earliestMeetingDate: Date;
  requiredNoticePostByDate: Date;  // fromDate in most cases
  rules: NoticeRule[];
  explanation: string;             // e.g., "A special town meeting regarding a land use ordinance requires 14 days notice per 30-A M.R.S.A. §4352. If notice is posted today (March 12), the earliest meeting date is March 26."
}
```

### 4.5 UI Surface Points

**AgendaBuilder page** — notice compliance banner:
- Always shown when meeting is in `draft` status
- Color-coded by `warningLevel` (green/yellow/red/grey-overdue)
- Shows: "Notice must be posted by [date] — [X days away]. [Statute citation]."
- If overdue: "Notice deadline has passed. This is advisory only — you may still generate and post a notice."

**Meeting creation flow** — when user selects meeting type:
- If the type triggers special notice rules (e.g., special town meeting, zoning hearing), a callout appears:
  - "This meeting type requires [X] days notice per [statute]."
  - "If you'd like to hold this meeting on [selected date], notice must be posted by [deadline date]."
  - "Earliest possible meeting date if you post notice today: [date]."

**"Publish Notice" confirmation dialog** — shows:
- The applicable statute and required notice period
- Whether sufficient notice has been given (advisory warning if not)
- "I understand this is advisory" dismissal if overdue

### 4.6 Extensibility

New states are added by creating a new file in `rules/` exporting an array of `NoticeRule` objects, and registering the state in `rules/index.ts`. The engine selects rules by `state` field.

Town-level charter overrides are not supported in Phase 1 but the data model is structured to accommodate a `charterOverrides?: Partial<NoticeRule>[]` field on the `town` table in a future version.

## 6. Open Questions / Deferred

- **Charter overrides**: Some towns have stricter notice requirements than state minimums. Configuration for charter-specific rules is deferred to a future session.
- **Newspaper publication tracking**: Some meeting types require newspaper publication. Tracking this in the app (upload proof of publication) is deferred.
- **Abuttor notification**: Planning Board subdivision hearings require mailed notice to abutters. This workflow is deferred.
- **NH/VT/MA/CT/RI rules**: Stubs only in Phase 1. Full rule datasets for other New England states are deferred.
