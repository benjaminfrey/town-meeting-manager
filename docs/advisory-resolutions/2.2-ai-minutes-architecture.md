# Advisory 2.2 — AI Minutes Draft — Prompt Architecture & Review UX

**Advisory Item:** The AI Minutes Draft — Prompt Architecture
**Date:** March 2026
**Status:** Resolved with open questions — data pipeline, template system, AI architecture, review UX, hard rules, and agenda packet data model defined. Cross-board template analysis (Select Board vs. Planning Board) completed. 17 questions pending from Newcastle analysis of four meetings across two boards (Section 10).

---

## What This Document Covers

The AI-assisted minutes drafting system is not a single feature — it's a pipeline that spans eight concerns:

1. **Agenda templates** — how meetings are structured before they happen
2. **Live meeting data capture** — what the system records during the meeting
3. **AI transcription & summarization** — how audio and structural data become text
4. **Draft generation** — how the AI produces a minutes document
5. **Review & approval** — how the admin edits, validates, and submits the draft for board approval
6. **Agenda packets** — how staff briefing materials (background, exhibits, recommendations, suggested motions) are structured per agenda item
7. **Real-world patterns** — recusals, absent members, naming conventions, agenda amendments, future items queues, adjournment patterns, presiding officer succession, and items without motions — derived from analysis of three Newcastle Select Board meetings (January 12 & 26, 2026 and November 10, 2025)
8. **Cross-board differences** — how different board types (Select Board vs. Planning Board) structure agendas, record motions, identify items, and produce minutes differently — derived from analysis of Newcastle Planning Board agenda (December 18, 2025) and draft minutes (November 20, 2025)

## 1. Agenda Templates

### What a Template Is

An agenda template is a reusable structure of **sections** that a board uses repeatedly. Each section is a container that can hold 0, 1, or many **agenda items** underneath it. Templates are board-specific — the Select Board template looks different from the Planning Board template.

### Template Structure (derived from the Damariscotta Select Board agenda)

The Damariscotta agenda reveals the real-world hierarchy:

```
MEETING
├── Section: Pledge of Allegiance                    [ceremonial — no minutes]
├── Section: Budget Workshop                         [discussion]
│   └── Item: Review FY27 Proposed Budget
├── Section: Public Hearing                          [formal — statutory]
│   └── Item: New Liquor License: Smashed Burgers
├── Section: Call to Order                           [procedural]
├── Section: Minutes                                 [auto-generated]
│   ├── Item: February 18, 2026 minutes
│   └── Item: February 25, 2026 minutes
├── Section: Financial Reports                       [standard]
│   ├── Item: Payroll Warrants #___                  [placeholder field]
│   └── Item: Accounts Payable Warrants #___         [placeholder field]
├── Section: Citizen Comments & Correspondence       [open — public input]
├── Section: Town Manager's Report                   [informational]
├── Section: Official Action Items                   [action — votes expected]
│   ├── Item: Liquor License Renewal: Bred in the Bone
│   ├── Item: Liquor License Approval: Smashed Burgers
│   └── Item: Board of Appeals Appointment
├── Section: Select Board Discussion Items           [discussion]
└── Section: Adjournment                             [procedural]
```

### Template Data Model

Each template section has the following properties:

| Property | Type | Description |
|----------|------|-------------|
| `title` | Text | Section heading (e.g., "Official Action Items") |
| `sort_order` | Integer | Position in the agenda |
| `section_type` | Enum | Categorization — see below |
| `minutes_behavior` | Enum | How the AI handles this section — see below |
| `is_fixed` | Boolean | If true, this section appears on every agenda built from this template and cannot be removed (only its items change) |
| `description` | Text (optional) | Default description text shown under the section heading |
| `default_items` | Array (optional) | Pre-populated items that appear each time (e.g., "Payroll Warrants #___") |

### Section Types

| Type | Description | Examples |
|------|-------------|---------|
| `ceremonial` | Procedural/symbolic, no substantive content | Pledge of Allegiance, Moment of Silence |
| `procedural` | Meeting mechanics — call to order, adjournment | Call to Order, Adjournment |
| `minutes_approval` | Approval of previous meeting minutes | Minutes |
| `financial` | Financial reports, warrants | Financial Reports |
| `public_input` | Open public comment or correspondence | Citizen Comments, Public Comment Period |
| `report` | Staff or committee reports | Town Manager's Report, Department Reports |
| `action` | Items requiring board action (votes expected) | Official Action Items, New Business |
| `discussion` | Discussion without expected vote | Discussion Items, Workshop |
| `public_hearing` | Statutory public hearing | Public Hearing |
| `executive_session` | Closed session (pre-drafted motion) | Executive Session |
| `other` | Anything that doesn't fit above | Custom sections |

### Minutes Behavior (per section)

Controls how the AI treats this section when generating minutes:

| Behavior | Description | Use For |
|----------|-------------|---------|
| `skip` | AI produces no output for this section. It does not appear in the minutes at all. | Pledge of Allegiance, ceremonial items |
| `timestamp_only` | AI records only the time this occurred (e.g., "Meeting called to order at 4:30 PM") | Call to Order, Adjournment |
| `action_only` | AI records only motions, votes, and decisions. No discussion summary. | Financial Reports (approve warrants), routine action items |
| `summarize` | AI produces a summary of discussion + any actions taken. Follows the per-item transcription preference. | Discussion items, reports, public hearings |
| `full_record` | AI produces the most detailed output available based on the transcription preference. | Contentious items, executive session entry/exit |

The admin sets a default minutes behavior per section in the template. It can be overridden per agenda item and per meeting.

### Placeholder Fields

Agenda items can contain placeholder fields that are blank at publish time and filled in before or during the meeting:

- **Syntax:** `#___` or `[TBD]` or similar — rendered as a fillable field in the agenda builder
- **Examples:** "Payroll Warrants #___", "Appointment of [Name] to [Board]"
- **Behavior:** Placeholders are visually distinct in the agenda builder. The system tracks whether they've been filled in. Unfilled placeholders do not block agenda publishing but generate a warning.

### Auto-Generated Sections

The **Minutes Approval** section is auto-generated when building a new agenda. The system queries for draft minutes from previous meetings of this board that have not yet been approved and lists them as sub-items. The admin can add, remove, or reorder these.

### Multi-Body Meetings

The Damariscotta agenda shows multiple bodies meeting on the same evening (Board of Assessors + Select Board). The system must support both approaches:

- **Separate meetings:** Each body is modeled as a separate meeting in the system with separate minutes, even if they occur on the same evening in the same room. The operator runs them sequentially in the live meeting manager.
- **Combined meeting:** One meeting record with sections for each body, producing one set of combined minutes.

This is a **town-level configuration** — the admin chooses how their town handles multi-body evenings. The system supports both without imposing one model.

### Vote Result Enum

The `result` field on votes supports the following values:
- `passed` — motion approved by the required vote threshold
- `failed` — motion did not achieve the required vote threshold
- `tabled` — item deferred to a specific future meeting (common in Planning Board; see Section 11)
- `continued` — item deferred without a specific date
- `withdrawn` — motion withdrawn by the mover before a vote

### Audio Data (when available)

Audio is **not sent to the Claude API directly.** The pipeline is:

1. **Audio recording** (per agenda item, segmented by operator transitions) → stored in Supabase Storage
2. **Transcription service** (Deepgram, Whisper, or AssemblyAI) processes each audio segment → produces timestamped, speaker-attributed text
3. **Transcription text** is included in the structured meeting record (as a field on each agenda item)
4. **The Claude API receives** the structured meeting record with transcription text embedded — not audio files

This separation is intentional:
- Audio-to-text is a specialized task best handled by a dedicated transcription service
- Claude excels at summarization, formatting, and style-matching — not raw audio processing
- The transcription can be reviewed and corrected before being sent to Claude
- Different transcription services can be swapped without changing the minutes generation pipeline

### Transcription Preferences (per agenda item)

Set by the admin or recording secretary, changeable during the meeting and during post-meeting review:

| Preference | Description | When to Use |
|-----------|-------------|-------------|
| `full_transcription` | Complete word-for-word transcript with speaker attribution | Public hearings, contentious items, items where "who said what" matters |
| `lengthy_summary` | Detailed summary preserving key arguments and attributions | Important discussion items, budget workshops |
| `three_paragraph` | Condensed to 3 paragraphs capturing the essence | Standard discussion items |
| `single_paragraph` | One paragraph summary | Routine items, informational reports |
| `none` | No transcription — AI works from structural data only | Ceremonial, procedural items |

**Key rule:** These preferences can be changed during post-meeting review. If the admin reviews the AI draft and decides the single-paragraph summary of a contentious item is too brief, they can change it to `lengthy_summary` and regenerate just that section. **This requires the full audio recording to be preserved until minutes are approved by the board.**

### Per-Section Generation (not whole-document)

The AI generates minutes **one section at a time**, not as a single monolithic document. This is critical because:

- The admin can regenerate any single section without re-generating the whole document
- Different sections may need different transcription preferences
- Sections with `skip` or `timestamp_only` behavior don't need AI generation at all
- The admin can edit one section while the AI regenerates another
- Errors or hallucinations are isolated to a section, not spread across the entire document

Each section's generation call includes:
- The structured data for that section (items, motions, votes, transcription text)
- The town profile (minutes style, board label, member titles)
- The overall meeting context (date, attendance, who's chairing)
- Instructions specific to the section's minutes behavior and transcription preference

### Prompt Architecture

The AI prompt is assembled from layers:

**Layer 1 — System prompt (constant):**
- Role: "You are a municipal meeting minutes writer for a New England town."
- Constraints: factual accuracy, no embellishment, no hallucination of names/votes/motions not in the data, flag uncertainties rather than guessing
- Legal requirements: motions must include mover, seconder, and vote result

**Layer 2 — Town profile (per-town):**
- Board label and member titles
- Minutes style preference (action/summary/narrative)
- Formality level
- Any town-specific conventions or language preferences (configurable by admin)

**Layer 3 — Section context (per-section):**
- Section type and minutes behavior
- Transcription preference for this section
- Transcription text (if available)
- Structured data (motions, votes, timestamps)
- Operator notes

**Layer 4 — Style examples (per-town, optional):**
- Sample minutes from this town's actual past minutes (uploaded by admin during setup)
- These serve as "write it like this" references for tone, format, and level of detail

### Handling Incomplete Data

When the structured meeting record is missing required data, the AI must **flag it, not fabricate it.** Specific rules:

| Missing Data | AI Behavior |
|-------------|-------------|
| Motion recorded without a mover | Insert: `[MOVER NOT RECORDED — verify before approval]` |
| Motion recorded without a seconder | Insert: `[SECONDER NOT RECORDED — verify before approval]` |
| Vote result recorded without individual votes (roll call expected) | Insert: `[INDIVIDUAL VOTES NOT RECORDED — verify before approval]` |
| Executive session entered but return timestamp missing | Insert: `[RETURN FROM EXECUTIVE SESSION TIME NOT RECORDED — required before approval]` |
| Agenda item with no transcription and no operator notes | Insert: `[NO DISCUSSION RECORDED FOR THIS ITEM]` |
| Guest speaker name not captured | Insert: `[SPEAKER NAME NOT RECORDED]` |
| Recusal recorded without reason | Insert: `[RECUSAL REASON NOT RECORDED — required per 30-A M.R.S.A. §2605]` |
| Item in action section with no motion and no "no action" note | Insert: `[ACTION STATUS NOT RECORDED — was a motion made?]` |
| Member marked absent but no presiding officer set | Insert: `[PRESIDING OFFICER NOT RECORDED — Chair is absent, who presided?]` |
| Suggested motion contains unfilled placeholder | Insert: `[MOTION CONTAINS UNFILLED PLACEHOLDER — verify before approval]` |
| Adjournment with neither motion nor "without objection" | Insert: `[ADJOURNMENT METHOD NOT RECORDED]` |
| Agenda amendment recorded without the requesting member | Insert: `[AMENDMENT REQUESTER NOT RECORDED]` |

These flagged items are **surfaced in the review UI as warnings** (see section 5 below).

## 4. Hard Rules — Legally Required

These are enforced by the system, not just by the AI prompt:

### Vote Integrity

**No vote can be recorded without a motion and a second, with the names of the members who moved and seconded noted.**

- The live meeting manager UI enforces this: the "Record Vote" action is not available until a motion (with mover) and a second (with seconder) have been captured
- If the operator attempts to record a vote without a motion, the system blocks it and shows: "A vote requires a motion and a second. Record the motion first."
- The AI draft will never contain a vote result without the corresponding motion, mover, and seconder — because the structured data won't contain one

### Draft Completeness Validation

Before a draft can be submitted to the board for approval, the system runs a completeness check. The following items generate **blocking warnings** (the admin can override, but must acknowledge each):

| Check | Condition | Warning |
|-------|-----------|---------|
| Executive session return | Executive session entered but `returned_at` is null | "Executive session return time not recorded. This is legally required." |
| Motion completeness | Any motion missing mover or seconder | "Motion on [item] is missing [mover/seconder]. Verify before approval." |
| Attendance | No attendance recorded | "Meeting attendance not recorded. Minutes must include who was present." |
| Quorum | Meeting started without quorum | "Meeting may have started without a quorum. Verify and note in minutes." |
| Call to order time | No start timestamp | "Meeting start time not recorded." |
| Adjournment time | No end timestamp | "Meeting adjournment time not recorded." |

These warnings appear in the review UI as a checklist. The admin can:
- Fix the underlying data (add the missing timestamp, name, etc.) and regenerate the affected section
- Acknowledge the warning and proceed anyway (with the flag text preserved in the draft for the board to see)
- The board may approve minutes with flags, or may vote to table them until corrections are made

## 5. Review UX

### Split-Panel Editor

The review interface is a **split-panel view:**

| Left Panel | Right Panel |
|-----------|------------|
| **Rich text editor** with the AI-generated minutes draft | **Structured data view** showing the source data the AI used for the currently selected section |

**Left panel (editor):**
- Rich text editor with spellcheck
- Formatted as the minutes will appear when published
- Editable — the admin can modify any text directly
- Section headers are clickable to select that section
- Each section shows its generation status: "AI generated", "AI generated (edited)", "Manually written", "Not yet generated"

**Right panel (source data):**
- When a section is selected in the left panel, the right panel shows:
  - The structured data for that section (motions, votes, timestamps, attendance)
  - The transcription text (if available) with speaker attributions
  - The transcription preference used (full/lengthy/3-paragraph/single)
  - Operator notes entered during the meeting
  - Audio playback controls (play the audio segment for this section)
- The admin can change the transcription preference in the right panel and click "Regenerate this section" — only that section is re-generated, the rest of the document is untouched

**Top bar:**
- Completeness warnings (if any) — clickable to jump to the affected section
- Meeting metadata (date, board, attendance summary)
- Actions: "Regenerate All", "Submit for Approval", "Export PDF", "Export DOCX"
- Draft status: "In Progress", "Ready for Review", "Submitted for Approval", "Approved"

### Per-Section Regeneration

The admin can regenerate any individual section without affecting the rest of the document:

1. Select a section in the left panel
2. In the right panel, optionally change the transcription preference (e.g., from `single_paragraph` to `lengthy_summary`)
3. Click "Regenerate Section"
4. The AI re-generates only that section using the structured data + the new transcription preference
5. The new text replaces the old text in the editor
6. If the admin had manually edited the old text, they are warned: "This will replace your manual edits for this section. Continue?"

### Transcription Preference Changes During Review

Because transcription preferences can be changed during review:

- The **full audio recording for each agenda item segment must be preserved** until the minutes are approved by the board
- When the admin changes a preference from `single_paragraph` to `full_transcription`, the system re-runs the transcription service on the stored audio segment (if it hasn't already been fully transcribed)
- The full transcription is then sent to Claude for the regenerated section
- This means the transcription service may be called multiple times for the same segment if preferences change — a cost consideration

### Audio Playback in Review

The right panel includes audio playback controls:
- Play/pause the audio segment for the selected section
- Scrub/seek within the segment
- Playback speed control (0.5x, 1x, 1.5x, 2x)
- This allows the admin to listen to what was said during a specific agenda item while reading/editing the AI's draft of that item

## 6. Audio Lifecycle

### Recording

- Audio is recorded per agenda item segment, driven by operator transitions in the live meeting manager
- When the operator advances to the next agenda item, the current audio segment is closed and a new one begins
- If the operator forgets to advance and the AI detects a topic shift, it suggests a segment boundary (but does not auto-advance)
- Audio is stored locally on the device during the meeting (offline-first) and uploaded to Supabase Storage when connectivity is available

### Storage

- Audio segments are stored in Supabase Storage, organized by meeting ID and agenda item ID
- Each segment has metadata: meeting_id, agenda_item_id, start_timestamp, end_timestamp, duration, file_size
- Estimated storage: ~1 MB per minute of audio (compressed). A 2-hour meeting ≈ 120 MB.

### Retention

- **Audio is retained until the minutes are approved by the board.**
- This is because the admin may change transcription preferences during review, requiring re-transcription from the original audio
- Once the board votes to approve the minutes at their next meeting, the audio MAY be purged — this is a town-level setting:
  - **Option A: Purge after approval** — audio is deleted after minutes are approved. Saves storage.
  - **Option B: Retain for N days after approval** — keeps audio for a configurable period (default 30 days) as a safety buffer.
  - **Option C: Retain indefinitely** — keeps all audio permanently. Maximum storage cost.
- The default should be Option B (30-day retention after approval) as a balance between storage cost and safety.

### Executive Session Audio

- **Audio recording MUST stop automatically when executive session is triggered.** This is legally required.
- The system does not record, store, or transcribe any audio during executive session.
- When the board returns from executive session, audio recording resumes automatically.
- The executive session entry motion, timestamps, and any post-session actions are captured as structured data only.

## 9. Agenda Packet Data Model — Analysis from Newcastle Select Board Packet

### What This Section Covers

Analysis of three Newcastle Select Board meetings — the January 26, 2026 agenda packet (20 pages), the November 10, 2025 approved minutes (2 pages), and the November 10, 2025 agenda packet (36 pages) — reveals critical data structures and patterns not previously captured. The January 26 packet establishes the **Item Commentary** format. The November 10 documents confirm cross-meeting consistency and reveal additional patterns: absent member vote recording, Vice Chair succession, "without objection" adjournment, placeholder fields in suggested motions, redline/clean draft exhibit pairs, fiscal warrant signature blocks, cross-meeting references, and variable staff attendance.

### The Item Commentary Format

Each agenda item in the Newcastle packet has a structured briefing sheet (called "Item Commentary") with the following fields:

| Field | Description | Example from Packet |
|-------|-------------|-------------------|
| **AGENDA ITEM** | Section + item reference with title | "(Unfinished Business) Staff, Board, and Committee Appointments" |
| **STAFF RESOURCE** | The staff member(s) responsible for this item | "Kevin Sutherland, Town Manager" or "Kevin Sutherland, Town Manager \| Michael Martone, Town Planner" |
| **BACKGROUND** | Context for the board — what they need to know before deliberating | Free-form text, typically 1–3 paragraphs explaining the history and current state |
| **EXHIBITS** | Numbered list of attached documents | "1. Updated Draft Dedications and Recognition Policy" or "1. FY27 Initial Budget Memo 2. Budget Timeline FY27 3. FY27 Budget Documents" |
| **RECOMMENDATION** | Staff's recommended course of action | "Staff recommends the Board approve the minutes as presented." or "Provide staff with additional feedback for further modification or support as drafted." |
| **POSSIBLE MOTION** | Pre-drafted motion text, shown in italic | *"To appoint Bonnie Stone to the Newcastle Appeals Board with a term expiring June 30, 2027"* |

### How This Maps to the System

The Item Commentary format is essentially an **agenda item metadata system** — it's the structured briefing data that accompanies each item on the agenda. This maps to a new data layer in the agenda builder:

**Proposed data model per agenda item:**

| Property | Type | Required | Source |
|----------|------|----------|--------|
| `title` | Text | Yes | Agenda builder |
| `section_ref` | Text | Auto | E.g., "6A" — section number + item letter |
| `staff_resource` | Person reference(s) | No | Selected from staff/PERSON records |
| `background` | Rich text | No | Written by staff in the agenda builder |
| `exhibits` | File attachment(s) | No | Uploaded documents (PDFs, memos, proposals) |
| `recommendation` | Rich text | No | Staff's recommended action |
| `suggested_motion` | Text | No | Pre-drafted motion text |
| `minutes_behavior` | Enum | Yes (inherited from section) | Override-able per item |
| `transcription_preference` | Enum | No | Set during or after the meeting |

### The Suggested Motion Field

The POSSIBLE MOTION field is one of the most powerful patterns in the packet. It contains the exact text that a board member can read verbatim to make a motion. In the live meeting manager:

- **Pre-meeting:** Staff writes the suggested motion text in the agenda builder. It appears in the published agenda packet.
- **During meeting:** When the operator reaches this item, the suggested motion text is pre-populated in the motion capture field. A board member can adopt it verbatim, and the operator clicks "Motion Made" — the text is already there.
- **Modification:** If the board modifies the motion during discussion, the operator edits the pre-populated text before recording the vote. The original suggested motion is preserved in the audit trail.
- **No motion:** If the board discusses but takes no action, the suggested motion is simply unused. The structured record shows the item had a suggested motion that was not acted upon.

### The Exhibit/Attachment System

The Newcastle packet includes exhibits as numbered attachments per agenda item. These range from a 1-page draft policy to a 4-page consultant proposal with biographies. The system must support:

- **Per-item file attachments** with labels and exhibit numbers
- **Ordering** — Exhibit 1, Exhibit 2, etc. within each item
- **Public vs. internal attachments** — some exhibits are public (published with the agenda); others may be internal briefing materials
- **Hyperlinks** — the Newcastle packet includes hyperlinked exhibit references (e.g., "FY27 Initial Budget Memo" links to an external document). The system should support both uploaded files and URL references.
- **Downstream to minutes** — when the AI generates minutes, the exhibit list for each item is available in the structured data. The AI can reference exhibits in the minutes where appropriate (e.g., "The Board reviewed the draft policy [Exhibit 1]").

### Amendments to the Agenda

The Newcastle agenda includes "Amendments to the Agenda" as section #2, right after Call to Order. This is a standard pattern where:

1. The Chair asks if there are any amendments to the posted agenda
2. Members can request adding, removing, or reordering items
3. The amendments are typically adopted by consensus or motion
4. The minutes record "None" if there were no amendments, or describe the changes

**System implications:**
- The template should support an "Amendments to the Agenda" section as a standard fixed section
- The minutes behavior for this section is `action_only` — record what was added/removed/changed, or "None"
- If amendments are made, the live meeting manager should allow the operator to modify the running agenda in real-time (add items, move items, remove items)
- The amendment itself is recorded in the structured meeting record so the AI can note it in the minutes

### Future Agenda Items

The Newcastle agenda includes "Future Agenda Items" as section #10 (near the end, before Executive Session). This is a queue of topics that board members or staff want addressed at upcoming meetings:

- The January 12 minutes show this section recording: a brief note with no formal action
- Items mentioned here should be available when building the next meeting's agenda
- This creates a **running backlog** of future items that the system can present to the admin when creating a new agenda

**System implications:**
- The agenda template should support a "Future Agenda Items" section
- Minutes behavior: `summarize` (briefly note what was mentioned)
- The system should maintain a **future items queue** per board — items added here appear as suggestions when building the next meeting's agenda
- Items can be dismissed from the queue, deferred to a specific date, or converted to a full agenda item on a future agenda

### Recusal Patterns

The Newcastle January 12, 2026 minutes contain a clear recusal example: Vice Chair Kostenbader recused himself from discussion and motions regarding the appointment of Angela Kostenbader (his wife) to the ambulance board. The vote was recorded as "Passed 4-0, 1 abstained."

**System implications for the structured meeting record:**

```json
{
  "title": "CLC Ambulance Service Appointment",
  "recusals": [
    {
      "member": "Thomas Kostenbader",
      "reason": "Conflict of interest — spouse is the candidate",
      "recused_from": "discussion_and_vote"
    }
  ],
  "motion": {
    "text": "To appoint Angela Kostenbader to the Central Lincoln County Ambulance Services Board of Directors",
    "moved_by": "Ben Frey",
    "seconded_by": "Rufus Percy",
    "vote": {
      "type": "roll_call",
      "result": "passed",
      "yeas": 4,
      "nays": 0,
      "abstentions": 1,
      "individual_votes": [
        { "member": "Tor Glendinning", "vote": "yea" },
        { "member": "Karen Paz", "vote": "yea" },
        { "member": "Rufus Percy", "vote": "yea" },
        { "member": "Ben Frey", "vote": "yea" },
        { "member": "Thomas Kostenbader", "vote": "abstained", "reason": "recused" }
      ]
    }
  }
}
```

**Key rules for recusal handling:**
- A recused member is marked as abstained (not absent) in the vote count
- The recusal reason should be recorded (maps to Maine conflict-of-interest law per Advisory 1.2)
- The quorum calculation adjusts: if 5 members are present but 1 recuses, quorum is still based on 5 seated members, but the vote threshold may change depending on local rules
- The AI must note the recusal in the minutes: "[Member] recused [himself/herself] from discussion and voting on this item due to [reason]."
- A completeness validation warning should fire if a recusal has no reason recorded

### Member Naming Conventions in Minutes

The Newcastle minutes use a consistent naming pattern:

- **Board members:** Last name only after first reference (Frey, Paz, Percy, Glendinning, Kostenbader)
- **Staff:** Title + last name on first reference ("Town Manager Sutherland"), last name only thereafter
- **Guest speakers/presenters:** Full name + title/affiliation on first reference ("Emma McKearney - Deputy Director of Lincoln County EMA"), last name only thereafter
- **Public commenters:** Full name + identifier ("David Levesque, Lincoln County Commissioner and Newcastle Resident")

**System implications:**
- The town profile should include a **member reference style** setting:
  - `last_name_only` — "Frey motioned..." (Newcastle's style)
  - `title_and_last_name` — "Selectman Frey motioned..."
  - `full_name_first_then_last` — "Ben Frey motioned..." on first reference, "Frey" thereafter
- This setting is passed to the AI in the town profile layer of the prompt
- The AI must maintain consistency within a single minutes document

### Items Without Motions

The Newcastle minutes show several items where discussion occurs but no motion is made:

- **Alewives Fish Committee appointment:** "staff does not have an immediate name for consideration. No motion was made."
- **Town Manager Report:** 8 bullet points of informational updates, no motion
- **Future Agenda Items:** Brief discussion, no motion

**System implications:**
- The structured meeting record must support items with `motion: null` — discussion happened but no action was taken
- The AI should handle this gracefully: summarize the discussion and note that no action was taken, without inserting a fabricated motion
- The completeness validator should NOT warn about missing motions for items in sections typed as `report`, `discussion`, or `public_input` — only for items in `action` sections where a motion was expected but not recorded

### The "Discussion Ensued" Pattern

The Newcastle minutes frequently use the phrase "Discussion ensued" as a placeholder indicating that discussion happened without detailing its content. This is a legitimate summary-style convention.

**AI prompt implication:** When the transcription preference is `single_paragraph` or `none` and the operator has noted that discussion occurred, the AI should use a brief summary phrase rather than fabricating discussion content. The system should recognize that "Discussion ensued" (or equivalent) is a valid and common output for summary-style minutes.

### Post-Executive-Session Actions

The Newcastle minutes show:
1. Motion to enter executive session (with statutory citation)
2. "Exited Executive Session at 8:40pm"
3. Immediately followed by a motion and vote on a separate matter
4. Adjournment

**System implications:**
- The structured record for executive session must support a `post_session_actions` array — actions taken after returning from executive session but before moving to the next agenda section
- The AI must record the exit timestamp and any actions taken after returning
- This validates the existing design in 2.2 section 4, but the `action_taken_after_return` field should be expanded to an array of motions/actions, not a single text field

Based on the Newcastle packet, the standard Select Board template should be updated:

1. Call to Order [procedural, timestamp_only, fixed]
2. Amendments to the Agenda [procedural, action_only, fixed] ← **NEW**
3. Minutes of Previous Meeting(s) [minutes_approval, action_only, fixed, auto-generated]
4. Public Comments on Items Not on the Agenda [public_input, summarize, fixed]

Also note that Newcastle uses "Unfinished Business" / "New Business" rather than "Old Business" / "Official Action Items" — terminology varies by town.

### Cross-Meeting Consistency — Confirmed by November 10, 2025 Analysis

The November 10, 2025 approved minutes and 36-page agenda packet confirm that the patterns identified from January 26, 2026 are **consistent across meetings**, not one-offs:

- **Item Commentary format** — identical structure (AGENDA ITEM, STAFF RESOURCE, BACKGROUND, EXHIBITS, RECOMMENDATION, POSSIBLE MOTION) used across all items in both packets
- **Exhibit numbering** — same per-item numbered exhibits pattern
- **Suggested motion field** — POSSIBLE MOTION present on all action items
- **Section ordering** — same 12-section template structure

The November 10 documents also reveal 10 additional patterns:

### Absent Member Vote Recording

The November 10 minutes record votes as "Passed 4-0, 1 Absent" — this is **distinct from abstention and recusal.** The system's vote data model must distinguish three non-voting categories:

| Category | Description | Vote Record Format | Example |
|----------|-------------|-------------------|---------|
| **Absent** | Member not at the meeting | "Passed 4-0, 1 Absent" | Glendinning absent from Nov 10 meeting |
| **Abstained (recusal)** | Member present but recused due to conflict of interest | "Passed 4-0, 1 Abstained" | Kostenbader recused from Jan 12 ambulance vote |
| **Abstained (other)** | Member present but chooses not to vote (rare) | "Passed 4-0, 1 Abstained" | Board member abstains without conflict |

**Updated vote data model:**

```json
{
  "vote": {
    "type": "voice",
    "result": "passed",
    "yeas": 4,
    "nays": 0,
    "abstentions": 0,
    "absent": 1,
    "individual_votes": [
      { "member": "Thomas Kostenbader", "vote": "yea" },
      { "member": "Karen Paz", "vote": "yea" },
      { "member": "Rufus Percy", "vote": "yea" },
      { "member": "Ben Frey", "vote": "yea" },
      { "member": "Tor Glendinning", "vote": "absent" }
    ]
  }
}
```

**AI prompt implication:** The AI must use the correct phrasing:
- Absent: "Motion passed 4-0, 1 absent."
- Recused: "Motion passed 4-0. Kostenbader recused himself due to [reason]."
- Both in same meeting: A meeting could have 1 absent member and 1 recused member on a specific item — "Motion passed 3-0, 1 recused, 1 absent."

### Presiding Officer Succession

The November 10 minutes show Chair Glendinning absent, with Vice Chair Kostenbader presiding for the entire meeting. This validates a **presiding officer succession** model:

1. **Chair presides** by default
2. **Vice Chair presides** when Chair is absent — this happens automatically per board rules, no motion required
3. **Other member presides** if both Chair and Vice Chair are absent — this may or may not require a motion depending on local rules

**System implications:**
- The meeting record should include a `presiding_officer_succession` object when the presiding officer is not the Chair
- The AI must reference the correct presiding officer throughout the minutes: "Kostenbader called the meeting to order..." not "Glendinning called the meeting to order..."
- The attendance section should clearly note who is absent and who is presiding
- The system should auto-suggest Vice Chair as presiding officer when the Chair is marked absent during meeting setup

### "Without Objection" Adjournment

The November 10 minutes show: "Kostenbader without objection adjourned the meeting at 8:27PM" — no formal motion, no vote, just consensus adjournment. This is a **different adjournment pattern** from a formal motion to adjourn.

**Updated adjournment data model:**

```json
{
  "adjournment": {
    "method": "without_objection",
    "adjourned_by": "Thomas Kostenbader",
    "motion": null,
    "timestamp": "20:27:00"
  }
}
```

Versus a formal adjournment:

```json
{
  "adjournment": {
    "method": "motion",
    "adjourned_by": "Thomas Kostenbader",
    "motion": {
      "text": "to adjourn the meeting",
      "moved_by": "Karen Paz",
      "seconded_by": "Rufus Percy",
      "vote": { "type": "voice", "result": "passed", "yeas": 4, "nays": 0, "absent": 1 }
    },
    "timestamp": "20:27:00"
  }
}
```

**System implications:**
- The live meeting manager should offer both adjournment methods: "Motion to Adjourn" and "Adjourn Without Objection"
- The AI must use the correct phrasing for each method
- The completeness validator should NOT warn about a missing adjournment motion when `method` is `without_objection`

### Placeholder Fields in Suggested Motions

The November 10 packet shows a POSSIBLE MOTION with a **fill-in-the-blank field**: "To make Option 6 by artist Robin Perry with ___ color for the official town logo." This extends the placeholder field pattern beyond item titles — suggested motions themselves can contain blanks.

**System implications:**
- The `suggested_motion` field should support the same placeholder syntax as item titles (`___` or `[TBD]`)
- The live meeting manager should highlight unfilled placeholders in the suggested motion when it pre-populates the motion capture field
- The AI should NOT attempt to fill in placeholder blanks — it should flag them: `[MOTION CONTAINS UNFILLED PLACEHOLDER — verify before approval]`

### Redline / Clean Draft Exhibit Pairs

The November 10 packet shows a common pattern for policy documents: exhibits come in pairs — one **redline draft** (with tracked changes / strikethrough) and one **clean draft**. Example from Item 5A (Provider Agency Funding Request Policy):

- Exhibit 1: Preamble
- Exhibit 2: Red Line Draft
- Exhibit 3: Clean Draft

**System implications:**
- The exhibit system should support an optional `exhibit_type` field to tag exhibits as: `standard`, `redline_draft`, `clean_draft`, `signature_page`, `financial_report`, or `correspondence`
- Tagging is informational — it helps the admin and the AI understand what each exhibit is, but does not change processing behavior
- The AI can reference exhibit types in minutes when relevant: "The Board reviewed the redline draft [Exhibit 2] and the clean draft [Exhibit 3]."

### Fiscal Warrant Details

The November 10 packet includes a multi-page fiscal warrant exhibit — a 5-page "Warrant Preview" showing every vendor, invoice number, account code, and amount. The warrant also includes a **signature block** requiring all Select Board members to sign.

**System implications:**
- Fiscal warrant exhibits are typically the longest exhibits in a packet
- The warrant signature block is a physical/PDF artifact — the system should note that warrants require signature outside the digital system (until/unless e-signatures are supported)
- The AI should record the total warrant amount in the minutes without reproducing the line-item detail: "The Board approved the fiscal warrant in the amount of $125,494.66"
- The discrepancy between the agenda-listed amount ($125,494.66) and the heading in the minutes ($126,494.66) confirms that warrant amounts can be corrected during the meeting — the motion amount is the authoritative number

### Cross-Meeting References

The November 10 packet's BACKGROUND fields contain **hyperlinked references to previous meeting documents**: a Town Manager memo from a prior meeting, a Special Town Meeting Warrant, etc. This creates a web of cross-references between meetings.

**System implications:**
- The `background` field (rich text) should support hyperlinks
- When a background field references a prior meeting by date, the system could eventually auto-link to that meeting's record in the system — but this is a future enhancement, not MVP
- The AI does not need to follow hyperlinks — it uses the background text as context
- Cross-meeting references confirm that the system needs robust meeting history and search to support the staff workflow

### Empty Section Recording

The October 27 minutes (embedded in the November 10 packet as an exhibit) explicitly record "None" for empty sections: "Future Agenda Items: None", "Executive Session: None." This confirms that sections can be explicitly empty, and the convention is to record them with "None" rather than omitting them entirely.

**System implications:**
- The minutes output should include **all fixed sections** from the template, even if nothing happened
- For empty fixed sections, the AI should output: "[Section Title]: None." (matching the convention)
- Non-fixed sections with no items can be omitted entirely
- The operator should have a way to mark a section as "None / Not Applicable" during the live meeting to explicitly confirm nothing happened (rather than it being ambiguous whether the section was forgotten)

### Meeting Cancellation as an Agenda Item

The November 10 packet includes Item 6C: "Cancel Board of Assessors Meeting scheduled for November 24, 2025." This shows that **cancelling a future meeting is itself a board action item** requiring a motion and vote.

**System implications:**
- Meeting cancellation should be modeled as a standard agenda item in an `action` section
- The system should not auto-cancel meetings — it requires a board vote
- When the cancellation motion passes, the system should update the future meeting's status (if it exists in the system) to "Cancelled" with a reference to the authorizing meeting and motion
- The suggested motion pattern works naturally here: "To cancel the Board of Assessors meeting scheduled for November 24, 2025"

### Variable Staff Attendance Per Item

The November 10 minutes list "Seth Hagar — Road Commissioner" and "Peter Drum — Town Attorney" as additional staff present, but these individuals attended for specific items only — not the whole meeting. Meanwhile, the Town Manager and Town Clerk attended the entire meeting.

**Updated attendance model:**

Staff attendance should support **scope** — whether a staff member attended the full meeting or specific items only:

```json
{
  "staff_present": [
    { "name": "Kevin Sutherland", "title": "Town Manager", "scope": "full_meeting" },
    { "name": "Shelly Clifford", "title": "Town Clerk", "scope": "full_meeting" },
    { "name": "Seth Hagar", "title": "Road Commissioner", "scope": "specific_items", "items": ["5C"] },
    { "name": "Peter Drum", "title": "Town Attorney", "scope": "specific_items", "items": ["6A", "6B"] }
  ]
}
```

**AI prompt implication:** The attendance section of the minutes should list all staff, with per-item staff optionally noted: "Also present: Seth Hagar, Road Commissioner (for Item 5C); Peter Drum, Town Attorney (for Items 6A and 6B)."

### Real Agenda Amendment Example

The November 10 minutes contain a **real-world example** of an agenda amendment: "Paz asked to add 5C — Assessors Agent Modification." This validates the "Amendments to the Agenda" section design and shows:

1. A member (not just the Chair or staff) can request an amendment
2. The amendment is recorded with the member's name, the action (add), and the item description
3. The item is inserted into the agenda at a specific position (5C, within section 5)

**System implications for the agenda amendment data model:**

```json
{
  "agenda_amendments": [
    {
      "requested_by": "Karen Paz",
      "action": "add",
      "item_title": "Assessors Agent Modification",
      "inserted_at": "5C",
      "approved_by": "consensus"
    }
  ]
}
```

- `action` enum: `add`, `remove`, `reorder`, `modify`
- `approved_by`: `consensus`, `motion` (with a linked motion object), or `chair_ruling`
- The live meeting manager should allow the operator to record these amendments and modify the running agenda accordingly

### "No Motion Was Made" Pattern

The November 10 minutes explicitly state "No motion was made" for Item 5B (Newcastle Town Logo). This is distinct from items in `report` or `discussion` sections where motions aren't expected — this was an `action` item where a motion was possible but the board chose not to act.

**AI prompt implication:** When an action item has `motion: null` and `operator_notes` contains "No motion was made" (or similar), the AI should include this explicitly: "No motion was made." This is already partially captured in the existing design but should be reinforced: the phrase "No motion was made" is the standard closing for action items where the board deliberately chose inaction.

## 10. Questions Requiring Resolution

The Newcastle packet analysis raises several questions that need decisions before implementation:

### Q1: Should the Item Commentary format be required or optional?

The Item Commentary (staff resource, background, exhibits, recommendation, suggested motion) is a best practice used by towns with professional staff. However, many small volunteer-run towns don't prepare this level of briefing material.

**Decision: B — Template-level default.** The template specifies whether items in each section show the commentary fields by default. The admin can always add or remove them per item. This keeps the UI clean for boards that don't prepare briefing packets (e.g., Planning Board) while making the fields easy to enable for boards where staff prepares full packets (e.g., Select Board). Avoids making the screens feel busy or prescriptive.

### Q2: Should suggested motions auto-populate the live meeting motion capture field?

When staff writes a POSSIBLE MOTION in the agenda item, should it appear pre-filled in the motion capture field during the live meeting?

**Decision: Yes, with visual warning.** The suggested motion text pre-populates the motion capture field, but the field displays a persistent visual indicator (e.g., highlighted border, banner, or label) reminding the operator: "This motion was pre-filled from the agenda packet. Verify the language matches what was actually said before recording the vote." The warning clears once the operator explicitly confirms or edits the text. The original suggested motion is preserved in the audit trail regardless of edits.

### Q3: Should the future items queue be a first-class feature?

Newcastle uses "Future Agenda Items" as a running section. Should the system maintain a persistent queue that carries items forward across meetings?

**Decision: Option A (persistent queue), but the feature is optional per board.** If a board's agenda template includes a "Future Agenda Items" section, the system maintains a persistent per-board queue: items mentioned during that section are stored, appear as suggestions when building the next agenda, and remain in the queue until explicitly placed on an agenda or dismissed. Boards that don't use a Future Agenda Items section are unaffected — the queue feature is not imposed on boards whose templates don't include it.

### Q4: How should real-time agenda amendments be handled?

Newcastle's "Amendments to the Agenda" section means the posted agenda can be modified at the start of the meeting. Should the system support live agenda modification?

**Decision: Option A (full live editing).** The operator can add, remove, and reorder items during the meeting. Changes are logged as amendments in the structured record, including who requested the amendment, what action was taken (add/remove/reorder/modify), and how it was approved (consensus, motion, or chair ruling).

### Q5: Member reference style — who decides?

Newcastle uses last-name-only. Other towns might use title + last name.

**Decision: Option B (board-level setting).** Each board can have its own member reference style convention (`last_name_only`, `title_and_last_name`, or `full_name_first_then_last`). This is passed to the AI in the board's profile layer of the prompt.

### Q6: How should the exhibit/attachment system handle the public portal?

When agenda packets are published to the public portal, should all exhibits be public by default?

**Decision: Option A (all public by default).** Every exhibit attached to a published agenda is visible on the public portal. Admins can mark specific exhibits as internal-only when needed (e.g., draft documents, personnel-related materials). This aligns with open-meeting transparency principles — the default should be public, with explicit opt-out for exceptions.

### Q7: How should the system handle "without objection" adjournment vs. motion to adjourn?

The November 10 minutes show adjournment "without objection" (no motion, no vote). January 12 uses a more formal pattern. Should the system:

**Decision: Option A (support both methods explicitly).** The live meeting manager offers two distinct options: "Motion to Adjourn" (requires mover, seconder, vote) and "Adjourn Without Objection" (records who adjourned and timestamp only). This reflects real-world practice where boards use both conventions.

### Q8: Should the system auto-suggest Vice Chair as presiding officer when Chair is absent?

When the operator marks the Chair as absent during meeting setup, should the system:

**Decision: Option B (prompt the operator).** When the Chair is marked absent, the system shows a prompt: "Chair is absent. Who is presiding?" with Vice Chair pre-selected but requiring confirmation. This avoids silent assumptions while still guiding the operator toward the most likely answer.

### Q9: Should the exhibit type tagging (redline, clean draft, signature page, etc.) be supported?

The November 10 packet reveals consistent exhibit types: redline drafts, clean drafts, preambles, financial reports, signature pages. Should exhibits be taggable?

**Decision: Option A (supported but optional).** The exhibit upload UI includes an optional "type" dropdown (standard, redline_draft, clean_draft, signature_page, financial_report, correspondence). Defaults to "standard." This low-effort addition enables richer exhibit organization and better AI context without burdening operators.

### Q10: How should the system handle meeting cancellation votes?

The November 10 packet shows cancelling a future meeting as a board action item. When the cancellation motion passes, should the system:

**Decision: Option A (auto-update the future meeting).** If the meeting being cancelled exists in the system, automatically set its status to "Cancelled" with a reference to the authorizing vote. Notify users who had agenda items on the cancelled meeting. This reduces manual follow-up and ensures the calendar stays accurate.

### Q11: Should variable staff attendance (full meeting vs. specific items) be tracked?

The November 10 minutes show Road Commissioner and Town Attorney attending for specific items only. Should the attendance model support per-item staff scope?

**Decision: Option B (simple attendance list).** Staff are listed as present or absent, without per-item scope tracking. The operator adds item-specific notes manually if needed. This keeps the attendance model straightforward while still allowing contextual notes in the item-level data.

### Q12: How should empty sections be recorded in the minutes?

The October 27 minutes (embedded in November 10 packet) record empty sections as "None" (e.g., "Executive Session: None", "Future Agenda Items: None"). Should the system:

**Decision: Option A (include all fixed sections, show "None" for empty).** Fixed sections always appear in the minutes even if nothing happened. The AI writes "[Section Title]: None." for empty fixed sections. This provides a complete record showing the board addressed (or had nothing for) every standing section.

### Q13: Should the motion display format be board-level configurable?

The Planning Board uses a structured block format (Motion / Moved by / Second / Discussion / Vote / Result) while the Select Board uses inline narrative. Should the system:

**Decision: Option A (board-level configurable).** Each board template specifies a motion display format: `block_format` or `inline_narrative`. The AI renders motions accordingly. This gives each board control over how motions appear in their minutes, matching their established conventions.

### Q14: Should the system support parcel/application tracking for land-use boards?

Planning Board items are identified by Map/Lot reference and follow an application lifecycle (pre-submittal → submitted → reviewed → decided). Should the system:

**Decision: Option C (deferred).** Don't build parcel or application tracking in MVP. Planning Boards use the same item model as Select Boards with topic-based titles that happen to include map/lot references. This can be revisited post-MVP if land-use boards need structured application lifecycle tracking.

### Q15: Should "Discussion" and "Result" be added to the motion data model?

The Planning Board minutes include "Discussion: none" and "Result: The motion carried and the application is approved" as explicit fields on every motion. Should these be:

**Decision: Option B (not in the motion data model).** "Discussion" is part of the per-item discussion summary, not the motion record itself. "Result" is generated by the AI from the vote data. No new fields needed — the AI can render these as display elements in block-format minutes without storing them as separate data fields.

### Q16: How should public commenter identification work?

Planning Board attributes comments to "a member of the public" (anonymous). Select Board records full names and affiliations. Should this be:

**Decision: Option C (section-type-dependent).** Public hearing comments require named attribution (legally required in some cases); general public comment sections allow anonymous identification. This respects legal requirements for public hearings while accommodating boards that don't always record commenter names in other contexts.

### Q17: Should the minutes certification format be board-level configurable?

Planning Board uses an all-member signature block. Select Board uses "Respectfully submitted" by the recording secretary. Should this be:

**Decision: Option A (board-level configurable).** Each board template specifies a certification format: `prepared_by`, `all_member_certification`, or `both`. This accommodates the different conventions observed across board types (e.g., Planning Board's all-member signature block vs. Select Board's "Respectfully submitted" by recording secretary).

## 11. Cross-Board Template Analysis — Planning Board vs. Select Board

### What This Section Covers

Analysis of two Newcastle Planning Board documents — the December 18, 2025 agenda and the November 20, 2025 draft meeting minutes (4 pages) — reveals fundamental structural differences between how Planning Boards and Select Boards operate within the same town. These differences validate that the template system must be deeply board-specific and that several system features cannot be one-size-fits-all.

### Template Structure Comparison

The Planning Board uses a **6-section structure** compared to the Select Board's 12–14 sections:

**Planning Board (Newcastle):**
```
MEETING
├── 1. Call to Order                        [procedural]
│   └── A. Agenda Amendments (as needed)    [sub-item, not separate section]
├── 2. Review of Minutes                    [minutes_approval]
│   └── A. [Previous meeting date] Minutes
├── 3. Old Business                         [action — applications]
│   └── A. Map/Lot (Address, Applicant)
│       ├── i. Application for Subdivision
│       └── ii. Public Hearing and Comments
├── 4. New Business                         [action — applications]
│   └── A. Map/Lot (Address, Applicant)
│       └── i. Application type
├── 5. Other Items                          [catch-all]
│   ├── A. Public Comment (open to all topics)
│   ├── B. Discussion
│   └── C. Planner's Report
└── 6. Adjourn                              [procedural]
```

**Key structural differences from Select Board:**

| Feature | Select Board | Planning Board |
|---------|-------------|----------------|
| Number of standard sections | 12–14 | 6 |
| Agenda amendments | Separate section (#2) | Sub-item under Call to Order (1.A) |
| Item identification | Topic-based titles | **Map/Lot parcel reference** (e.g., "Map 003, Lot 059") |
| Sub-item hierarchy | Items are flat | Items have **sub-items** (application type, public hearing) |
| Executive session | Standard section | Not used |
| Fiscal warrants | Standard section | Not used |
| Staff report | "Town Manager Report" section | "Planner's Report" as sub-item under Other Items |
| Presentations section | Yes | No |
| Future agenda items section | Yes | No |
| Public comment placement | Separate section (#4) | Sub-item under Other Items (5.A) |
| Meeting schedule | Not on agenda | **Printed on agenda** — full annual calendar of meeting dates |
| Hybrid meeting info | Not on agenda | **Zoom link on agenda** |
| Board size | 5 members | **7 members** |
| Item Commentary / Packet | Full briefing sheets per item | **None** — simple outline agenda only |

### Motion Recording Format

The Planning Board uses a **fundamentally different motion recording format** than the Select Board. This is the single most significant cross-board difference for the system:

**Planning Board format (structured block):**
```
Motion:    To approve the meeting minutes for the October 20, 2025
           Planning Board meeting, as presented.
Moved by:  Wanda Wilcox
Second:    Tyler Tibbets
Discussion: none
Vote:      Yea:     7
           Nay:     0
           Abstain: none
Result:    The motion carried and the October 20, 2025 meeting
           minutes are approved as presented.
```

**Select Board format (inline narrative):**
> Paz moved to approve the minutes of the December 22, 2025 meeting as presented. Percy seconded. Passed 4-0, 1 Absent.

**Key differences:**
- Planning Board includes an explicit **"Discussion:" field** — always recorded, even as "none"
- Planning Board includes a **"Result:" field** — a narrative summary of what the vote means
- Planning Board records **"Abstain: none"** explicitly — Select Board only mentions abstentions when they occur
- Planning Board motions are **in italics** in the minutes document
- Planning Board uses "Second:" — Select Board says "seconded by"
- The Planning Board format reads more like a form; the Select Board format reads more like prose

**System implications:**
- The motion display format must be **board-level configurable** (or at minimum, minutes-style-dependent)
- The structured block format maps more naturally to the system's internal data model — it's essentially a rendering of the vote JSON
- The system should support both rendering styles:
  - `block_format` — Planning Board style with labeled fields
  - `inline_narrative` — Select Board style with prose sentences
- The "Discussion" and "Result" fields are new data elements not in the current motion data model

### Minutes Style Analysis

The Planning Board minutes are clearly **narrative style** — more detailed than the Select Board's summary style:

- Subdivision review discussion: multiple detailed paragraphs covering DEP communications, vernal pools, liquidation harvesting standards, and Findings of Fact
- References to specific Maine statutes by name and chapter
- Applicant and consultant statements recorded in detail
- The Town Planner's (Martone's) explanations are summarized paragraph by paragraph

This contrasts with the Select Board's "Discussion ensued" or brief bullet-point summaries.

**System implications:**
- Minutes style (`action` / `summary` / `narrative`) should be **board-level configurable**, not just town-level
- The Planning Board naturally operates at the `narrative` or `lengthy_summary` level for substantive items
- The Select Board naturally operates at the `summary` or `single_paragraph` level
- Transcription preferences may have different defaults per board

### Parcel-Based Item Identification

Planning Board items are identified by **Map/Lot reference** — "Map 003, Lot 059 (White Road, Shattuck)" — because Planning Board work is inherently land-use-based. This is a fundamentally different item identification pattern:

- Select Board items: topic titles ("Staff, Board, and Committee Appointments — CLC Ambulance Service")
- Planning Board items: **parcel reference + address + applicant name** ("Map 013, Lot 059 (67 Glidden St, Swetland)")

**System implications:**
- Agenda items should support an optional **parcel reference** field: `{ map: "003", lot: "059", address: "White Road", applicant: "Shattuck" }`
- This field is primarily relevant for Planning Boards, ZBAs, and other land-use boards
- When present, the parcel reference should be the primary identifier in the agenda and minutes
- This connects to Advisory 2.3 (Parcel Proximity Data Strategy) — parcel-identified items can link to GIS data and abutter notification requirements

### Application Lifecycle

Planning Board items follow an **application lifecycle** not present in Select Board work:

1. **Pre-submittal meeting** — informal discussion, no vote (Items 4B, 4C in Nov 20 minutes)
2. **Application submission** — formal filing
3. **Staff review** — Planner reviews for completeness
4. **Board review** — discussion of findings of fact
5. **Public hearing** — required for certain application types
6. **Decision** — approve, deny, approve with conditions, or **table**

The November 20 minutes show multiple lifecycle stages in one meeting:
- Item 3A: An ongoing application being reviewed (with public hearing) → **tabled** to next meeting
- Items 4B and 4C: **Pre-submittal meetings** with no formal action — just discussion

**System implications:**
- The system should support an optional **application tracking** layer for land-use boards
- Application types: `subdivision`, `shoreland_zoning`, `site_plan_review`, `conditional_use`, `variance`, etc.
- Application status: `pre_submittal`, `submitted`, `under_review`, `public_hearing_scheduled`, `tabled`, `approved`, `denied`, `approved_with_conditions`, `withdrawn`
- Applications can span **multiple meetings** — the same Map/Lot item appears on successive agendas as the application progresses
- The system should link items across meetings that reference the same application

### Motion Outcomes — "Tabled" as a New Pattern

The November 20 minutes show a motion to **table** an application — a motion outcome not yet captured in our data model. The board voted 7-0 to table the subdivision application to the next meeting.

**Updated motion outcome enum:** `passed`, `failed`, `tabled`, `withdrawn`, `continued`

- `tabled` — the item is deferred to a specific future meeting
- `continued` — the item is deferred without a specific date
- `withdrawn` — the applicant or mover withdraws the motion before a vote

**System implications:**
- When an item is tabled, the system should prompt: "This item was tabled. Add it to the next meeting's agenda?"
- Tabled items should appear in the future items queue for the relevant board

### Public Comment Attribution

The Planning Board minutes attribute public comments to **anonymous members of the public** — "A question was raised by a member of the public" — without recording names. The Select Board minutes, by contrast, record public commenter names and affiliations ("David Levesque, Lincoln County Commissioner and Newcastle Resident").

**System implications:**
- Public commenter identification should be **board-level configurable**:
  - `named` — record commenter name and affiliation (Select Board style)
  - `anonymous` — record as "a member of the public" (Planning Board style)
  - `optional` — operator can choose to record or not per commenter
- This setting affects both the live meeting data capture and the AI output
- Some boards may have legal requirements for named public comment (e.g., public hearings) — the system should allow override per item

### Minutes Certification / Signature Block

The Planning Board minutes include an **all-member signature block**: "We, the undersigned, certify these meeting minutes are true and accurate." with signature lines for all 7 members with date fields.

The Select Board minutes use a simpler "Respectfully submitted, [name], Town Clerk" format without requiring all members to sign.

**System implications:**
- The minutes footer/certification format must be **board-level configurable**
- Options:
  - `prepared_by` — "Respectfully submitted, [Recording Secretary name]" (Select Board style)
  - `all_member_certification` — signature block for all members (Planning Board style)
  - `both` — prepared-by attribution plus member certification
- For digital minutes, the all-member signature block may need a workflow: distribute draft → members review → members sign (electronically or physically) → minutes finalized

### Annual Meeting Schedule on Agenda

The Planning Board agenda includes a **printed meeting schedule** at the bottom — a full-year calendar of meeting dates with past dates struck through and the current date bolded. This is not present on Select Board agendas.

**System implications:**
- The agenda template should support an optional **meeting schedule footer** that auto-generates from the board's scheduled meeting dates
- Past dates are struck through; the current meeting date is highlighted
- This is a presentation/publishing feature, not a data model change

### Sections Not Used by Planning Board

Several sections standard to the Select Board template do **not appear** in Planning Board work:
- Executive Session
- Fiscal Warrants / Financial Reports
- Presentations and Special Guests
- Future Agenda Items
- Town Manager Report (replaced by Planner's Report)

This confirms that the system **must not impose Select Board sections on other boards.** Templates must be built per-board type, with only truly universal sections (Call to Order, Minutes, Adjourn) shared across all boards.

### Updated Standard Planning Board Template

Based on the Newcastle Planning Board agenda and minutes:

1. Call to Order [procedural, timestamp_only, fixed]
   - 1A. Agenda Amendments (as needed) [procedural, action_only]
2. Review of Minutes [minutes_approval, action_only, fixed, auto-generated]
3. Old Business [action, summarize]
4. New Business [action, summarize]
5. Other Items [catch-all]
   - 5A. Public Comment (open to all topics) [public_input, summarize, fixed]
   - 5B. Discussion [discussion, summarize]
   - 5C. Planner's Report [report, summarize]
6. Adjourn [procedural, timestamp_only, fixed]

**Key differences from Select Board template:**
- Agenda Amendments is a sub-item, not a section
- "Other Items" is a catch-all section with fixed sub-items rather than a single-purpose section
- No executive session, fiscal warrants, or presentations sections
- Planner's Report replaces Town Manager Report and is nested under Other Items
