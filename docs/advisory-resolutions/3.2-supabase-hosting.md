# Advisory 3.2 вЂ” Supabase Hosting: Managed vs. Self-Hosted

**Advisory Item:** Supabase Hosting: Managed vs. Self-Hosted
**Date:** March 2026
**Status:** Resolved вЂ” self-hosted from day one

---

## Decision

**Self-hosted Supabase via Docker Compose** on a dedicated server with public IP addresses. The server will handle at least 3 towns for beta testing.

This departs from the original advisory recommendation to start with Supabase Cloud. The rationale:

- **Server is already available** with public IPs, eliminating the primary argument for starting with managed hosting (infrastructure provisioning time)
- **Data locality control** from day one вЂ” municipal clients can be told exactly where their data lives
- **No pricing surprises** вЂ” Supabase Cloud Pro tier charges per-project and has row/storage limits that would complicate multi-town deployments
- **Beta scale is manageable** вЂ” 3 towns is well within what a single server can handle, and the operational burden (backups, upgrades) is reasonable at this scale
- **No migration needed** вЂ” avoids the eventual Cloud-to-self-hosted migration that would have been necessary anyway

### What Self-Hosted Supabase Includes

The Supabase Docker Compose stack provides:

| Component | Purpose |
|-----------|---------|
| PostgreSQL | Primary database (with PostGIS for parcel data) |
| GoTrue | Authentication service (JWT-based) |
| PostgREST | Auto-generated REST API from database schema |
| Realtime | WebSocket server for live meeting updates |
| Storage | File storage for exhibits, agenda packets, minutes PDFs |
| Kong | API gateway |
| Studio | Database admin UI (can be disabled in production) |
| Meta | Metadata management |

### Operational Requirements

Since we're self-hosting from the start, these must be in place before beta:

1. **Automated backups** вЂ” Daily PostgreSQL backups with point-in-time recovery (WAL archiving). Backup retention: 30 days minimum.
2. **SSL/TLS** вЂ” Let's Encrypt certificates via Nginx reverse proxy for all public endpoints.
3. **Monitoring** вЂ” Basic health checks on all Docker containers. Alert if any service goes down.
4. **Upgrade process** вЂ” Documented procedure for updating the Supabase Docker images. Test upgrades on a staging instance before applying to production.
5. **Multi-town isolation** вЂ” All towns share the same Supabase instance and PostgreSQL database. Isolation is enforced via Row Level Security (RLS) policies, not separate databases.

### Scaling Path

- **3 towns (beta):** Single server, single Supabase instance. Plenty of headroom.
- **10вЂ“30 towns:** Same architecture. PostgreSQL handles this scale easily with proper indexing and RLS.
- **30+ towns:** Evaluate whether to add a read replica, move to a larger server, or split into regional deployments. The application code doesn't change вЂ” only infrastructure configuration.
